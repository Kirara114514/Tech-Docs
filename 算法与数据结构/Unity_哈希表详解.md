# 哈希表详解：O(1) 查找、冲突处理、键设计与工程实践

## 摘要

哈希表通过哈希函数把键映射到桶位置，在平均情况下实现 O(1) 查找、插入和删除。本文系统讲解哈希函数、冲突处理、链地址法、开放寻址、负载因子、扩容、Dictionary、HashSet、自定义 Key、字符串 Key、Unity 对象 Key、经典算法题和工程优化。

## 正文

### 背景

哈希表是算法与工程中最常用的数据结构之一。原稿已经覆盖基础概念和若干题解，但对 Key 设计、相等比较、可变 Key、字符串比较器、Unity 对象生命周期等工程问题讲得不够深入。本文将哈希表从“会用字典”扩展为“能设计稳定映射系统”的完整知识。

### 核心内容

#### 哈希表的抽象：键到值的快速映射

哈希表通过哈希函数把 Key 映射到桶数组位置，再通过相等比较确认具体 Entry。理想情况下，它能以平均 O(1) 完成查找、插入和删除。

从算法角度看，哈希表的抽象：键到值的快速映射 的价值在于把问题约束成一种可维护的访问模型。很多复杂问题并不是因为代码不够多，而是因为状态顺序没有被表达清楚；当我们用 哈希表 把顺序、边界和操作入口限制起来后，后续推理会简单很多。工业级代码尤其强调这一点：容器不是“随便装数据”的盒子，而是业务规则的一部分。

在 Unity 或 C# 项目中，哈希表的抽象：键到值的快速映射 常见于配置查询、对象缓存、ID 到对象映射、去重、计数、visited 集合。这些场景通常都不是孤立的一次性操作，而是会被多次调用、跨模块协作，甚至出现在热路径中。因此实现时必须同时考虑正确性、复杂度、GC、生命周期和调试成本。一个能跑通示例的写法，不一定能承受正式项目的数据规模和调用频率。

实现 哈希表的抽象：键到值的快速映射 时，最容易出现的问题包括：误以为 O(1) 无条件成立、依赖遍历顺序、Key 可变、容量规划缺失。这些问题在小样本里可能不明显，但在编辑器工具、运行时 UI、寻路、网络消息、战斗事件或大规模数据处理里会被迅速放大。正式项目里应把这些风险写进封装、测试和 Code Review 清单，而不是靠每个调用者临时记忆。

#### 负载因子与扩容

负载因子表示元素数量与桶容量的比例。负载因子过高会增加冲突，因此哈希表会扩容并重新分布元素。扩容单次成本是 O(N)。

{
    var map = new Dictionary<int, int>();

    for (int i = 0; i < nums.Length; i++)
    {

    }

    return Array.Empty<int>();
}

### 实现方案

#### 1. 默认选型

Key 到 Value 的映射使用 `Dictionary<TKey,TValue>`；去重和存在性判断使用 `HashSet<T>`；多线程写入使用并发集合或外部锁；稠密整数 ID 可以考虑数组替代。

#### 2. 自定义 Key 规范

Key 应不可变，或至少入表期间不修改参与比较的字段。自定义值类型实现 `IEquatable<T>`，并保证 Equals 与 GetHashCode 一致。

#### 3. 容量规划

已知元素数量时预设容量。批量构建大字典应放在初始化、Loading 或离线阶段，避免运行时扩容造成帧尖峰。

#### 4. 字符串 Key 规范

字典构造时指定 `StringComparer.Ordinal` 或 `OrdinalIgnoreCase`。不要通过频繁 ToLower 构造临时字符串实现忽略大小写。

#### 5. 生命周期管理

字典保存 Unity 对象引用时，必须在对象销毁、场景卸载、对象池回收时移除。长期缓存要有清理策略。

#### 6. 检查清单

检查 Key 是否稳定、是否实现相等比较、是否需要预设容量、是否依赖遍历顺序、是否可用数组替代、是否跨线程访问、是否存在引用泄漏。

### 总结

哈希表是用空间换时间的典型数据结构。它通过哈希函数、桶数组和冲突处理，在平均情况下提供 O(1) 查找、插入和删除能力。但这个 O(1) 依赖哈希函数质量、负载因子、扩容策略、Key 稳定性和相等比较正确性。C# 中 Dictionary 和 HashSet 是非常强大的工具，但 Unity 工程中还要考虑字符串 Key、对象生命周期、容量规划和 GC。真正掌握哈希表，不是会写 dict[key]，而是知道 Key 怎么设计、什么时候会退化、什么时候该用数组替代，以及如何避免缓存和引用泄漏。
