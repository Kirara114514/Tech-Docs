### 核心内容

#### C# string 的不可变性

C# 字符串创建后内容不可修改。任何拼接、替换、截取都会产生新字符串。这让字符串更安全，也让高频修改更容易产生 GC。

从算法角度看，C# string 的不可变性 的价值在于把问题约束成一种可维护的访问模型。很多复杂问题并不是因为代码不够多，而是因为状态顺序没有被表达清楚；当我们用 字符串 把顺序、边界和操作入口限制起来后，后续推理会简单很多。工业级代码尤其强调这一点：容器不是“随便装数据”的盒子，而是业务规则的一部分。

在 Unity 或 C# 项目中，C# string 的不可变性 常见于UI 文本、日志、路径、协议字段、本地化、对象名称。这些场景通常都不是孤立的一次性操作，而是会被多次调用、跨模块协作，甚至出现在热路径中。因此实现时必须同时考虑正确性、复杂度、GC、生命周期和调试成本。一个能跑通示例的写法，不一定能承受正式项目的数据规模和调用频率。

实现 C# string 的不可变性 时，最容易出现的问题包括：循环拼接、每帧更新文本、误以为 Replace 原地修改、Substring 分配过多。这些问题在小样本里可能不明显，但在编辑器工具、运行时 UI、寻路、网络消息、战斗事件或大规模数据处理里会被迅速放大。正式项目里应把这些风险写进封装、测试和 Code Review 清单，而不是靠每个调用者临时记忆。

```csharp
public static bool IsPalindrome(string s)
{
    int left = 0;
    int right = s.Length - 1;

    while (left < right)
    {
        while (left < right && !char.IsLetterOrDigit(s[left])) left++;
        while (left < right && !char.IsLetterOrDigit(s[right])) right--;

        if (char.ToLowerInvariant(s[left]) != char.ToLowerInvariant(s[right]))
            return false;

        left++;
        right--;
    }

    return true;
}
```

```csharp
public static int[] BuildLps(string pattern)
{
    int[] lps = new int[pattern.Length];
    int len = 0;

    for (int i = 1; i < pattern.Length;)
    {
        if (pattern[i] == pattern[len])
            lps[i++] = ++len;

        else

    }

}
```

面试表达时，不建议只背 API 名称。更好的回答方式是：先说明抽象模型，再说明底层实现，再说明时间复杂度和空间复杂度，最后结合实际场景解释为什么选择它。这样既能体现算法理解，也能体现工程判断。

#### 本地化 Key 与查表

本地化系统通常用稳定 Key 查找当前语言文本。Key 应独立于原文，不要用中文文案本身作为 Key。查表结构应预构建，避免运行时线性扫描。

从算法角度看，本地化 Key 与查表 的价值在于把问题约束成一种可维护的访问模型。很多复杂问题并不是因为代码不够多，而是因为状态顺序没有被表达清楚；当我们用 字符串 把顺序、边界和操作入口限制起来后，后续推理会简单很多。工业级代码尤其强调这一点：容器不是“随便装数据”的盒子，而是业务规则的一部分。

在 Unity 或 C# 项目中，本地化 Key 与查表 常见于多语言 UI、剧情文本、道具描述、配置表。这些场景通常都不是孤立的一次性操作，而是会被多次调用、跨模块协作，甚至出现在热路径中。因此实现时必须同时考虑正确性、复杂度、GC、生命周期和调试成本。一个能跑通示例的写法，不一定能承受正式项目的数据规模和调用频率。

### 实现方案

#### 1. 高频拼接方案

使用复用的 StringBuilder，预设合理容量，拼接完成后 ToString。注意最终字符串仍会分配，适合减少中间对象，而不是完全无分配。

#### 2. UI 文本刷新方案

使用脏标记或事件驱动。只有数值变化时才刷新文本。TextMeshPro 场景优先使用 SetText 的数值格式化重载，减少中间字符串。

#### 3. 滑动窗口模板

右指针负责扩张窗口，左指针负责在条件不满足或满足后收缩窗口。所有窗口状态都必须在左右指针移动时同步更新。

#### 4. KMP 使用边界

普通业务查找优先使用内置 IndexOf。只有在需要可控线性复杂度、自定义匹配或算法面试时才手写 KMP。不要为了小文本搜索强行上复杂算法。

#### 5. 多模式匹配方案

关键词数量少时可直接查找；关键词多且文本长时考虑 Trie 或 AC 自动机；客户端敏感词系统要权衡内存、更新和误判成本。

#### 6. 检查清单

检查是否每帧拼接字符串、是否重复刷新 UI、是否指定比较规则、是否误用 Length、是否处理 Unicode、是否有大量 Substring、是否真的需要复杂算法。

### 总结

字符串算法连接着算法能力和工程细节。双指针、滑动窗口、KMP、Rabin-Karp、Manacher、编辑距离和 Trie 都是重要工具，但它们不是越复杂越好。工程中应优先使用稳定、可维护且经过验证的内置 API；性能瓶颈明确后，再选择更复杂的算法。Unity 项目尤其要关注 string 不可变性、UI 文本刷新、日志拼接、本地化 Key、字符串比较规则和 GC。真正掌握字符串，不是背出 KMP，而是能根据数据规模、调用频率和业务语义选择合适的处理方式。
