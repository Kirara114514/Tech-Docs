# 队列详解：先进先出的数据结构、循环队列与任务调度模型

## 摘要

队列是一种遵循先进先出规则的线性数据结构。本文从抽象模型、数组队列、循环队列、链表队列、双端队列、BFS、生产者消费者、Unity 主线程派发、事件队列、并发队列和工程背压策略等角度系统讲解队列。

## 正文

### 背景

队列的核心价值是顺序确定性。只要业务强调先到先处理、按层扩展、按提交顺序消费，就很可能需要队列。原稿已经包含队列基础和几道经典题，但存在非循环与循环概念混用、注释复制错误和工程边界不足的问题。本文将队列独立成篇，补充循环缓冲、并发队列、主线程派发和容量治理等正式项目中必须考虑的内容。

### 核心内容

#### 队列的抽象模型：先进先出

队列是一种受限线性表，只允许从队尾入队、从队头出队。它表达的是先到先处理的顺序模型，适合任务排队、消息缓冲和层级遍历。

从算法角度看，队列的抽象模型：先进先出 的价值在于把问题约束成一种可维护的访问模型。很多复杂问题并不是因为代码不够多，而是因为状态顺序没有被表达清楚；当我们用 队列 把顺序、边界和操作入口限制起来后，后续推理会简单很多。工业级代码尤其强调这一点：容器不是“随便装数据”的盒子，而是业务规则的一部分。

在 Unity 或 C# 项目中，队列的抽象模型：先进先出 常见于BFS、任务调度、主线程派发、网络消息、UI 弹窗队列、事件系统。这些场景通常都不是孤立的一次性操作，而是会被多次调用、跨模块协作，甚至出现在热路径中。因此实现时必须同时考虑正确性、复杂度、GC、生命周期和调试成本。一个能跑通示例的写法，不一定能承受正式项目的数据规模和调用频率。

```csharp
public sealed class CircularQueue<T>
{
    private readonly T[] _items;
    private int _head;
    private int _tail;
    private int _count;

    public CircularQueue(int capacity)
    {
        _items = new T[capacity];
    }

    public bool Enqueue(T item)
    {
        if (_count == _items.Length)
            return false;

        _items[_tail] = item;
        _tail = (_tail + 1) % _items.Length;
        _count++;
        return true;
    }

    public bool TryDequeue(out T item)
    {
        if (_count == 0)
        {
            item = default;
            return false;
        }

        item = _items[_head];
        _items[_head] = default;
        _head = (_head + 1) % _items.Length;
        _count--;
        return true;
    }
}
```

```csharp
public static int Bfs(Node start, Node target)
{
    var queue = new Queue<Node>();
    var visited = new HashSet<Node>();
    var distance = new Dictionary<Node, int>();

    queue.Enqueue(start);
    visited.Add(start);
    distance[start] = 0;

    while (queue.Count > 0)
    {
        var current = queue.Dequeue();
        if (current == target)
            return distance[current];

        foreach (var next in current.Neighbors)
        {
            if (visited.Add(next))
            {
                distance[next] = distance[current] + 1;
                queue.Enqueue(next);
            }
        }
    }

    return -1;
}
```

{
    private readonly Stack<T> _in = new();
    private readonly Stack<T> _out = new();

    public void Enqueue(T item) => _in.Push(item);

    public bool TryDequeue(out T item)
    {
        if (_out.Count == 0)
            while (_in.Count > 0)
                _out.Push(_in.Pop());

        if (_out.Count == 0)
        {
            item = default;
            return false;
        }

        item = _out.Pop();
        return true;
    }
}

### 实现方案

#### 1. 普通单线程任务队列

使用 `Queue<T>`，在统一 Tick 中消费。可通过 `maxCount` 限制每帧处理数量，避免任务过多造成单帧卡顿。

#### 2. 固定容量环形队列

用于日志、采样和网络缓冲。满队列时必须明确策略：返回失败、覆盖最旧、丢弃最新或触发告警。不要让固定容量容器偷偷扩容。

#### 3. 主线程派发队列

跨线程场景使用 `ConcurrentQueue<Action>`，子线程只提交数据或回调，主线程消费时才访问 UnityEngine 对象。消费循环应设置每帧上限。

#### 4. BFS 模板

节点入队时就标记 visited，避免同一节点被多个父节点重复入队。若需要路径，额外维护 parent 字典；若需要距离，维护 distance 字典。

#### 5. 事件队列规范

事件队列应区分即时事件和延迟事件。强实时逻辑不应被随意延迟；表现事件和通知事件则适合排队统一派发。Flush 中异常应捕获并记录，避免一个事件阻断整个队列。

#### 6. 检查清单

检查是否空队列安全、是否需要线程安全、是否有容量上限、是否可能堆积、是否需要优先级、是否限制每帧消费、是否错误使用 List.RemoveAt(0) 模拟队列。

### 总结

队列通过先进先出的规则为系统提供公平、稳定和可预测的处理顺序。它是 BFS、消息派发、任务调度、主线程回调、事件缓冲和 UI 弹窗队列的基础。数组队列高效但需要循环利用空间；链表队列动态但有分配成本；Queue<T> 是普通业务首选；ConcurrentQueue<T> 适合跨线程场景。工程上真正重要的是控制队列增长、定义满队列策略、限制每帧消费、区分 FIFO 与优先级需求。队列看似简单，但它经常是系统稳定性的第一道缓冲层。
