# 栈详解：后进先出的受限容器与工程化应用

## 摘要

栈是一种遵循后进先出规则的线性数据结构。本文从抽象模型、数组实现、链表实现、C# Stack<T>、递归、括号匹配、最小栈、单调栈、Undo/Redo、UI 返回栈、表达式求值、DFS、性能优化和 Unity 工程落地等角度系统讲解栈，帮助读者把栈从算法题知识点升级为可用于真实项目的状态管理工具。

## 正文

### 背景

在编程世界中，数据结构不仅是存储方式，更是业务访问规则的表达。栈限制元素只能从一端进入和离开，换来了极强的顺序确定性。函数调用、撤销重做、UI 返回、表达式解析、深度优先搜索都依赖这种“最近状态优先”的规则。原稿已经包含基础概念和部分题解，但存在重复总结、注释混乱和工程边界不足的问题，因此本文将其重新整理为独立、正式、系统的长文档。

#### 栈的抽象模型：后进先出

栈是一种受限线性表，只允许在栈顶进行插入和删除。这个限制让栈天然适合表达最近状态优先处理的业务，例如最近打开的界面最先关闭、最近执行的命令最先撤销、最近进入的函数最先返回。

从算法角度看，栈的抽象模型：后进先出 的价值在于把问题约束成一种可维护的访问模型。很多复杂问题并不是因为代码不够多，而是因为状态顺序没有被表达清楚；当我们用 栈 把顺序、边界和操作入口限制起来后，后续推理会简单很多。工业级代码尤其强调这一点：容器不是“随便装数据”的盒子，而是业务规则的一部分。

在 Unity 或 C# 项目中，栈的抽象模型：后进先出 常见于函数调用、括号匹配、Undo/Redo、UI 面板返回栈、行为树调试路径、DFS 搜索。这些场景通常都不是孤立的一次性操作，而是会被多次调用、跨模块协作，甚至出现在热路径中。因此实现时必须同时考虑正确性、复杂度、GC、生命周期和调试成本。一个能跑通示例的写法，不一定能承受正式项目的数据规模和调用频率。

```csharp
public sealed class ArrayStack<T>
{
    private T[] _items = new T[8];
    private int _count;

    public int Count => _count;

    public void Push(T item)
    {
        if (_count == _items.Length)
            Array.Resize(ref _items, _items.Length * 2);

        _items[_count++] = item;
    }

    public bool TryPop(out T item)
    {
        if (_count == 0)
        {
            item = default;
            return false;
        }

        _count--;
        item = _items[_count];
        _items[_count] = default;
        return true;
    }
}
```

```csharp
public static bool IsValidBrackets(string s)
{
    var stack = new Stack<char>();

    foreach (char c in s)
    {
        if (c == '(' || c == '[' || c == '{')
        {
            stack.Push(c);
            continue;
        }

        if (c == ')' && (!stack.TryPop(out var a) || a != '(')) return false;
        if (c == ']' && (!stack.TryPop(out var b) || b != '[')) return false;
        if (c == '}' && (!stack.TryPop(out var d) || d != '{')) return false;
    }

    return stack.Count == 0;
}
```

```csharp
{
    private readonly Stack<int> _data = new();
    private readonly Stack<int> _mins = new();

    public void Push(int value)
    {
        _data.Push(value);
        _mins.Push(_mins.Count == 0 ? value : Math.Min(value, _mins.Peek()));

        _mins.Pop();
        return _data.Pop();
    }

    public int GetMin() => _mins.Peek();
}
```

```csharp
public static int[] DailyTemperatures(int[] temperatures)
{
    int n = temperatures.Length;
    int[] answer = new int[n];
    var stack = new Stack<int>();

    for (int i = 0; i < n; i++)
    {
        while (stack.Count > 0 && temperatures[i] > temperatures[stack.Peek()])
        {
            int prev = stack.Pop();
            answer[prev] = i - prev;
        }

        stack.Push(i);
    }

    return answer;
}
```

#### Undo/Redo：双栈管理历史命令

撤销与重做通常使用两个栈。执行命令后压入 undoStack，并清空 redoStack；撤销时从 undoStack 弹出并执行反操作，再压入 redoStack；重做则反向移动。

```csharp
{
    void Execute();
    void Undo();
}

public sealed class UndoRedo
{
    private readonly Stack<ICommand> _undo = new();
    private readonly Stack<ICommand> _redo = new();

    public void Execute(ICommand command)
    {
        command.Execute();
        _undo.Push(command);
        _redo.Clear();
    }

    public void Undo()
    {
        if (_undo.TryPop(out var command))
        {
            command.Undo();
            _redo.Push(command);
        }
    }
}
```

### 实现方案

#### 1. 默认选型：优先使用 Stack<T>

普通业务中直接使用 `Stack<T>` 即可。它类型安全、API 清晰、性能稳定。只有在需要无 GC、固定容量、Burst 兼容或特殊调试信息时，才建议自定义栈。不要为了“练手”在项目里到处散落自定义栈实现，容器越基础，越应该统一。

#### 2. 自定义栈接口建议

如果确实需要封装栈，接口建议提供 `Count`、`Push`、`TryPop`、`TryPeek`、`Clear`，热路径优先使用 Try 风格，避免把异常当正常控制流。对于引用类型元素，Pop 和 Clear 时应释放内部引用。

```csharp
public interface IStack<T>
{
    int Count { get; }
    void Push(T item);
    bool TryPop(out T item);
    bool TryPeek(out T item);
    void Clear();
}
```

#### 3. Unity UI 返回栈方案

UI 返回栈应由统一 UI 管理器维护，面板不应直接操作全局栈。入栈对象建议是面板控制器接口，而不是裸 GameObject。面板被关闭、销毁或切换场景时，必须同步栈状态，避免保存无效引用。

#### 4. Undo/Redo 命令方案

命令对象必须记录 Execute 和 Undo 所需的完整状态。连续拖拽、连续输入等高频操作应支持命令合并，否则撤销栈会被大量细碎命令污染。

#### 5. 单调栈模板

单调栈模板的关键是比较条件。求下一个更大元素时维护递减栈；求下一个更小元素时维护递增栈。重复元素是否弹出取决于题目需要严格大于还是大于等于。

#### 6. Code Review 清单

检查是否处理空栈、是否需要反向压栈、是否存在递归栈溢出、是否清理引用、是否每帧创建容器、是否跨线程访问、辅助栈是否与主栈同步、UI 或命令对象生命周期是否可靠。

### 总结

栈通过后进先出的访问规则，把嵌套、回退、撤销、递归和最近状态优先处理等问题表达得非常清楚。它的 API 简单，但工程使用并不简单：要考虑空栈、容量、引用清理、线程安全、辅助状态同步和生命周期。算法层面，栈可以解决括号匹配、表达式求值、DFS、最小栈和单调栈问题；Unity 工程层面，它可以支撑 UI 返回栈、Undo/Redo、编辑器工具和资源依赖扫描。真正掌握栈，不是会调用 Push 和 Pop，而是能识别哪些问题天然具有“最近状态先处理”的结构，并把这种结构封装成稳定、可维护、低 GC 的代码。
