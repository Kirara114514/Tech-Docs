# LRU缓存算法在对象池中的应用

## 摘要
对象池并不意味着资源永不回收。当池内对象类型多、容量受限且使用热点随版本变化时，简单的“只进不出”策略会迅速造成内存浪费。本文讨论 LRU 缓存算法在对象池治理中的应用，重点说明它适用于什么场景、如何与池化策略结合，以及落地时应注意哪些性能与正确性边界。

## 正文

### 背景
对象池的初衷是减少频繁创建和销毁带来的分配开销，但在复杂项目中，对象池本身也会成为资源常驻源。尤其在 UI 特效、子弹、特效实例、临时实体等场景里，如果池容量不受约束，历史上高频使用过的对象可能长期占用内存，即便当前版本已不再需要。此时，池化系统就需要引入“淘汰策略”。

### 核心内容
LRU 的核心思想是优先淘汰“最近最少使用”的对象。在对象池语境下，它适合处理“对象可复用，但同时又不能无限常驻”的场景。相比固定容量但无淘汰依据的池化方案，LRU 更接近真实使用分布，能在保证热点对象命中率的同时回收冷对象占用。

需要注意的是，LRU 不适合所有池化场景。对于创建极其昂贵、但偶发使用仍必须快速响应的对象，盲目淘汰可能导致抖动；对于严格实时系统，维护双向链表和字典索引本身也有成本。是否采用 LRU，应建立在“池中对象存在明显冷热差异”这一前提上。

### 实现方案
典型实现通常由字典与双向链表组成：字典用于快速定位对象节点，链表用于维护最近访问顺序。对象被命中时移动到头部，容量超限时从尾部淘汰。示意如下：

```csharp
public sealed class PoolEntry<T>
{
    public string Key;
    public T Value;
}
```

工程上更重要的是策略设计，而不是容器本身。建议同时定义：

- 最大池容量或最大内存预算。
- 命中时是否续期。
- 淘汰对象是否立即销毁还是延迟释放。
- 是否区分关键对象与普通对象。

此外，必须通过监控验证 LRU 是否真的有效，例如观察池命中率、回收次数、重新创建次数和峰值内存变化。没有数据支撑的淘汰策略，很容易只是“看起来更聪明”。

### 总结
LRU 在对象池中的价值，是让池化从静态容器升级为动态治理机制。但它只有在冷热分布明显、容量受限且可接受一定管理成本的场景下才真正合适。使用得当，LRU 能帮助团队在复用收益和内存预算之间取得更稳定平衡。
