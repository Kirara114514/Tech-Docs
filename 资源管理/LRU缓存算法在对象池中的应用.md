# LRU缓存算法在对象池中的应用

## 摘要
本文围绕“LRU缓存算法在对象池中的应用”在 Unity 项目中的工业化落地展开，重点讨论系统边界、性能账本、构建链路、运行时约束、常见失败模式与治理方法。文档默认读者为已经有一定项目经验的程序员、技术负责人和工具链维护者。

## 正文

### 背景
对象池一旦进入长期运行项目，就会自然演化出缓存问题。资源系统相关问题在小项目中往往还能靠人工经验维持，但在内容体量、构建频率和多平台适配压力上升后，它们会迅速演化为架构问题：包体膨胀、版本兼容困难、依赖不可见、缓存命中率低、运行时释放不稳定、定位问题只能靠猜。因此，这类主题不能只讨论单一 API 或单一工具，而必须把资源生命周期、构建产物、更新策略、目录规范和数据观测一起放进同一张账本里。

### 核心内容
#### 先定义系统边界，再讨论工具选型
资源问题最常见的误区，是在系统边界还没有定义清楚时先讨论工具。比如“是不是要上 Addressables”“要不要拆 Bundle”“要不要做远端热更新”这些问题，如果不先明确资源所有权、共享依赖归属、版本号策略、回滚边界和运行时缓存职责，最终就会变成工具叠工具、脚本叠脚本、例外覆盖例外。真正专业的做法，是先写清楚资源的生产链路和消费链路，再决定工具层如何承载这些规则。

#### 运行时成本和构建成本要一起看
很多团队只盯着运行时加载速度，却忽略了构建成本、补丁粒度和维护成本。举例来说，过度细拆资源可以降低单次下载量，但会放大依赖图复杂度和版本对齐难度；把所有共享依赖集中又能减少重复，但会导致冷启动扇出变大、部分轻量场景被迫加载重依赖。资源方案的质量，不在于某个指标极端优秀，而在于多指标平衡是否稳定。

#### 观测能力决定资源系统是否可维护
没有观测能力的资源系统，本质上是黑盒。成熟项目通常会在构建阶段输出依赖关系、产物大小、重复引用、共享资源、未引用资源、热更分组与下载清单；在运行时输出加载阶段耗时、缓存命中率、重试次数、资源常驻时长和内存峰值。只有这些数据持续可见，团队才能判断优化动作到底是在解决问题，还是只是在重排复杂度。

#### 资源生命周期必须显式建模
很多项目的问题，不在于资源加载不到，而在于加载到了以后谁负责释放、何时释放、释放后谁还能继续持有引用。如果资源生命周期没有显式建模，开发者就会在 UI 打开、场景切换、战斗结算、弱网重试这些节点上各自补丁式地加缓存和防空引用逻辑。最终的结果通常是：表面上谁都觉得自己只做了一点点保护，整体上却形成了难以解释的常驻资源集合。

#### 共享资源与业务资源要分层管理
共享资源和业务资源最大的差异，在于所有权。共享资源应该由基础设施层统一管理，并通过明确的目录、构建规则和版本策略治理；业务资源则应更多贴近模块归属，由模块自己对生命周期负责。混淆这两种资源，会导致项目里既看不清谁能改共享资源，也看不清谁该为业务冗余负责。

#### 工业化治理点
资源治理至少要覆盖五类内容：命名和目录规范、依赖归属规则、构建报告、运行时生命周期策略、异常处理与回滚机制。其中最容易被忽略的是异常链路，例如：资源清单版本不匹配、下载中断、缓存损坏、旧包残留、CDN 回源失败、平台压缩格式不兼容。如果这些链路上线前没有演练，线上问题往往不是“资源没加载到”，而是“资源半加载成功但状态不可恢复”。

#### 常见失败模式
常见失败模式包括：目录语义混乱导致共享依赖无人维护、工具层做了太多魔法导致问题无法解释、把同步加载藏在封装接口里、只在编辑器环境验证而不做真机弱网测试、把“构建能过”误当成“运行时可靠”。这些失败模式的共同点，是团队没有把资源系统视为基础设施，而是把它当作“会用就行”的功能模块。

### 实现方案
#### 推荐落地步骤
建议把落地步骤拆成四阶段。第一阶段，建立命名、目录和依赖规则，禁止新资源以“先放进去再说”的方式进入项目。第二阶段，导出构建报告并建立基线数据，让团队知道当前资源系统的问题分布。第三阶段，围绕热点链路逐步收敛加载与释放接口，避免多个模块各自维护一套资源状态机。第四阶段，建立自动化校验和回滚演练，把治理动作固化到构建流水线中。

#### 推荐实现示例
下面给出一个资源层最小抽象示例。关键点不在于接口长什么样，而在于把加载、引用计数、预热、释放和错误恢复放在同一个边界中管理。

```csharp
public sealed class LruPool<T>
{
    private readonly LinkedList<T> _lru = new();
    private readonly Dictionary<T, LinkedListNode<T>> _map = new();
    private readonly Queue<T> _free = new();

    public void Touch(T item)
    {
        if (_map.TryGetValue(item, out var node))
        {
            _lru.Remove(node);
            _lru.AddFirst(node);
        }
    }

    public T? EvictLeastRecent()
    {
        var node = _lru.Last;
        if (node == null) return default;
        _lru.RemoveLast();
        _map.Remove(node.Value);
        return node.Value;
    }
}
```

#### 验证指标
建议至少持续跟踪以下指标：首包大小、远端更新包大小、中位数加载耗时、95 分位加载耗时、缓存命中率、下载失败率、依赖重复率、Top N 大资源、一次版本迭代中新引入的异常资源数量。指标不一定越多越好，但必须能让团队回答三个问题：问题在哪、趋势如何、这次改动有没有让情况变好。

#### 典型案例拆解
以活动版本为例：活动资源通常具备“时间短、变更快、复用少、上线压力高”的特点。如果活动资源和主线共享依赖没有提前规划，就容易出现一个小活动拉高整包依赖，或者活动下线后残留大量无主资源。因此，活动资源特别适合做独立分组、独立版本标记和独立回收策略，用小周期治理来检验整个资源系统的弹性。

#### 上线前检查清单
上线前建议做一次资源系统专项检查：一是检查重复打包与异常大资源；二是验证弱网和中断恢复；三是确认版本号是否与资源清单、热更代码和配置一致；四是确认缓存目录切换和清理策略不会误删仍在使用的数据；五是确认构建报告已经沉淀到版本记录中。这些检查动作看似繁琐，但它们往往决定了资源系统在上线周是否稳定。

#### 团队协作要求
资源系统只有在跨角色协作中才真正成立。程序需要给出规则和工具，美术需要理解导入与命名边界，策划需要知道资源组织对更新和包体的影响，测试需要参与弱网和异常回滚演练。如果资源规则只掌握在少数程序手里，那么每一次美术资源大量导入、活动版本突击和平台临时适配，都会重新制造混乱。

#### 容量预算与版本账本
LRU缓存算法在对象池中的应用这类主题真正进入项目后，讨论的就不再只是“能不能工作”，而是“用多少预算工作”。资源系统至少要维护首包预算、热更预算、常驻内存预算和峰值瞬时预算四本账。没有这四本账，团队就无法在需求评审阶段知道新增资源究竟是在消耗包体、消耗加载时间，还是在透支未来版本的热更成本。工业化资源治理的第一步不是选工具，而是让预算成为一等公民。

#### 案例拆解：活动资源、主线资源与大厅资源的冲突
活动资源生命周期短、变化快，大厅资源常驻时间长，主线资源复用广但体量大。这三类资源如果没有边界，很容易出现活动偷偷引用大厅图集、主线共享材质被重复携带、活动下线后缓存仍残留旧依赖等问题。正确做法是把资源所有权、共享依赖、分组边界和回收策略写成显式规则，并在构建阶段输出跨域依赖报告和重复打包报告，把错误前移。

#### 故障模型与排障顺序
资源问题在线上最常见的表现并不是明确报错，而是白屏、卡加载、低概率贴图丢失、切场景后内存回不去和补丁异常膨胀。排障时应先看版本与清单是否一致，再看依赖图是否意外扩散，再看缓存与压缩格式兼容性，最后才进入业务加载接口。很多团队一开始就去补重试和判空，最后修出来的是更多偶然成功，而不是更稳定的系统。

#### 治理落地与职责分配
资源治理不是程序员单点工作。TA 负责导入规则和资产规格，美术负责遵守目录与命名边界，策划要理解活动组织方式如何影响包体与更新，测试要参与弱网和中断恢复演练，程序负责把规则固化到构建链路、运行时和审计工具里。只有多角色共同维护，资源文档才不是一份程序侧自说自话的规范。

#### 验收与持续演进
建议把重复打包阈值、首包上限、补丁体积波动、关键加载链路 P95、缓存命中率、资源回收后残留引用数和回滚恢复成功率纳入验收清单。资源治理的终点不是某次构建过了，而是系统能在版本不断迭代时保持可解释、可预测、可回滚。

### 总结
LRU缓存算法在对象池中的应用不是孤立技巧，而是一套资源生产、构建、加载、释放和治理协同工作的工程系统。只有当团队同时具备规则、工具、观测和回滚能力时，资源系统才会随着项目规模增长而更稳定，而不是随着规模增长而更脆弱。
