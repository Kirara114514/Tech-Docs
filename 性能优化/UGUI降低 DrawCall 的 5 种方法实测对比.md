# UGUI降低 DrawCall 的 5 种方法实测对比

## 摘要
DrawCall 是 UGUI 优化中最容易被过度神化的指标之一。很多团队一看到 DrawCall 数字上涨，就本能地开始合图、合材质、砍组件、拆节点，仿佛只要把 DrawCall 压下去，UI 性能就会自然变好。实际项目里，DrawCall 当然重要，但它从来不是唯一账本。特别是在 UGUI 中，Canvas Rebuild、布局刷新、事件射线、文本刷新、Mask 嵌套、列表结构和图集边界往往和 DrawCall 纠缠在一起。只盯 DrawCall，极容易做出“数字下降了，整体体验没变甚至更差”的伪优化。真正成熟的做法，是把降低 DrawCall 放在整体 UI 结构治理之下，再去比较各种手段的真实收益、适用边界和副作用。本文围绕 UGUI 中最常见的 5 种降低 DrawCall 方法，从原理、收益模式、副作用和实战适用性出发做系统对比，并强调如何避免“为了压一个数字，把资源、结构和维护性都拆坏”的典型陷阱。

## 正文

### 背景
UGUI 中 DrawCall 的来源，核心在于：

- 材质不同。
- 图集不同。
- 裁剪 / 蒙版 / 特殊渲染规则打断批处理。
- Canvas 边界打断批处理。
- 某些组件或效果引入额外渲染路径。

从表面看，这很像一个简单资源问题：只要统一材质、统一图集，就能一路合批。但真实项目里，降低 DrawCall 往往会牵一发而动全身：

- 你为了合图，把资源组织搞得很不合理。
- 你为了共材质，牺牲了表现灵活性。
- 你为了减少 Canvas 数量，反而放大了 Rebuild 传播。
- 你为了追求批次极低，导致后续页面维护困难。

因此，DrawCall 优化必须回到一个前提：

> 它是 UI 整体性能治理的一部分，而不是唯一目标。

### 核心内容

### 方法一：统一图集，减少纹理切换
这是最经典也最常见的一种做法。其原理很直接：

- 同一批 UI 元素若引用同一图集，通常更容易合批。

优点：

- 收益稳定。
- 对很多静态或半动态界面都有效。
- 是资源层面的基础优化。

副作用与边界：

- 图集组织若只为了合批，不按界面职责拆分，后期会非常难维护。
- 大图集会带来内存和加载管理压力。
- 把原本不该同时出现的资源塞进同一图集，并不总划算。

实战结论：

- 图集统一是必要基础，但必须按界面职责和共同出现关系组织，而不是盲目“大一统”。

### 方法二：统一材质与避免无必要特效材质
即便图集一致，如果材质不同，批处理依然可能被打断。常见打断来源包括：

- 自定义材质。
- 特效材质。
- 特殊字体材质。
- 混合不同 Mask / Stencil 规则。

优点：

- 一旦收口成功，批次收益明显。
- 对静态面板和常规组件很有效。

副作用与边界：

- 某些视觉需求确实需要独立材质。
- 为了共材质而牺牲关键表现，不一定值得。
- 字体、特效和某些图文混合组件天然更难统一。

实战结论：

- 材质统一非常重要，但应优先清理“不必要的差异”，而不是一刀切消灭所有差异。

### 方法三：减少 Mask、特效和打断批处理的组件
这是很多页面中真正的批次杀手。尤其是：

- 通用 Mask。
- 多层嵌套裁剪。
- 某些特效材质。
- 高复杂度 UI 叠加效果。

优点：

- 对复杂页面收益很大。
- 往往能同时改善 GPU 压力。

副作用与边界：

- 某些需求不可能完全不用裁剪与特效。
- 如果只是换组件，不改结构，收益可能有限。

实战结论：

- 这是典型的“结构性优化点”，单纯小修补通常不够，要和页面结构重构一起看。

### 方法四：合理进行 Canvas 分层与分区
这是一种非常容易被误解的方法。Canvas 分层本来主要是为控制 Rebuild 范围，但它也会影响批次边界。

优点：

- 能控制高频区域刷新。
- 让局部结构更独立。

风险：

- 拆得过细会增加 DrawCall。
- 排序与管理复杂度明显上升。

实战结论：

- Canvas 分层不是单纯降 DrawCall 技术，而是结构性权衡。它可能降低整体帧时间，也可能让 DrawCall 变高。所以它应服务于整体结构，而不是只为了压 DrawCall。

### 方法五：控制节点复杂度和可见元素数量
很多人谈 DrawCall 时只想资源层，却忽略节点数量和可见元素数量本身也会间接影响批次与渲染压力。典型高价值做法包括：

- 列表虚拟化。
- 减少同屏可见元素。
- 简化列表项结构。
- 对低价值装饰层做合并或去除。

优点：

- 往往不只是 DrawCall 下降，整体 CPU / GPU 账都会一起改善。

副作用与边界：

- 有时会牺牲界面丰富度，需要产品与美术协作。

实战结论：

- 这是最容易带来“整体收益”的方法，因为它不只优化批次，而是直接减少渲染负担来源。

### 为什么“实测对比”比“道听途说”更重要
DrawCall 优化特别容易陷入经验主义，因为很多方法在一个页面有效，在另一个页面可能几乎无效，甚至副作用更大。原因在于：

- 页面结构不同。
- 资源组织不同。
- 是否高频动态不同。
- GPU / CPU 瓶颈不同。

因此，真正靠谱的对比必须基于：

- 同一页面结构。
- 相同交互路径。
- 相同设备。
- 相同 Profile 窗口。

否则，所谓“方法 A 比方法 B 更好”通常没有工程意义。

### 正确的比较维度，不止 DrawCall 本身
如果要做 5 种方法的实测对比，至少应比较：

- DrawCall 数变化。
- GPU 帧时间变化。
- CPU 主线程变化。
- Rebuild 次数变化。
- 真机连续交互时的稳定性。
- 维护复杂度变化。

有些方法能把 DrawCall 压得很低，但会让结构更难维护；有些方法 DrawCall 下降不算夸张，却显著改善整体帧时间和波动。后者往往更有价值。

### 常见反模式

#### 反模式一：为了压 DrawCall，把所有资源都塞进超大图集
批次也许好了，资源管理和内存却变糟了。

#### 反模式二：为了少一个批次，牺牲页面结构合理性
短期数字漂亮，长期开发维护代价很高。

#### 反模式三：看到 DrawCall 降了就宣布优化成功
如果 GPU 帧时间没改善、CPU 更差、结构更乱，那不算成功。

#### 反模式四：把 Canvas 分层当降 DrawCall 通用技巧
它首先是结构优化，不是专职批次优化。

### 典型结论：5 种方法的真实优先级
从多数项目的工程收益看，通常更合理的优先级是：

1. 先整理图集和材质边界。
2. 再清理不必要的 Mask 和特效打断。
3. 再通过虚拟化和结构简化减少可见复杂度。
4. 最后谨慎做 Canvas 分层权衡。

也就是说，最有效的 DrawCall 优化通常先发生在资源组织与结构治理层，而不是炫技级局部技巧。

### 调试与验收
如果要判断某种降 DrawCall 手段是否值得保留，至少应验证：

- 它是否稳定降低了批次数。
- 真实真机帧时间是否改善。
- 页面后续是否更难维护。
- 它有没有把问题从 GPU 账转移到 CPU 账。

### 实现方案

### 推荐的落地顺序
#### 一、先做页面级审计
明确当前批次打断来自图集、材质、蒙版还是结构。

#### 二、优先做资源层统一
这是收益最稳、风险最可控的一步。

#### 三、对高风险页面做专项实测
尤其是主界面、长列表和高动态交互页面。

#### 四、把结论写成页面设计和资源规范
避免每个新页面都重新犯同类错误。

### 总结
UGUI 降低 DrawCall 的 5 种方法，真正的价值不在于谁能把数字压得最低，而在于谁能在真实项目中以可维护、可复用、可扩展的方式让整体账本更划算。统一图集、统一材质、减少打断批处理的组件、控制可见复杂度，以及谨慎使用 Canvas 分层，都有价值，但它们的适用边界完全不同。只有在相同页面、相同设备、相同交互路径下进行实测，并把 DrawCall 与 CPU、GPU、Rebuild 和维护成本一起看，团队才能真正做出有效的 DrawCall 优化，而不是陷入“数字优化成功、工程质量退化”的假胜利。
