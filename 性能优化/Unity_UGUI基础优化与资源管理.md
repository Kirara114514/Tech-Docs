# Unity UGUI 基础优化与资源管理

## 摘要

本文是 UGUI 性能优化系列的第一篇，主题是“基础优化与资源管理”。本文会从 UGUI 的渲染链路、Draw Call、合批、Sprite Atlas、字体、纹理导入、图片尺寸、RawImage、资源加载与释放等基础问题出发，重写为一篇更专业、规范、正式、工业化、易懂且深入的技术文档。

原稿中存在一些重复段落，并且部分说法已经不适合现代 Unity 项目，例如继续以 Sprite Packer 作为主要方案、把使用图集等同于必然合批、把 UI 遮挡理解成类似 3D 遮挡剔除、用 PNG 文件大小判断运行时纹理内存等。本文会统一修正这些问题，并以 Sprite Atlas、TextMeshPro、平台纹理压缩、Frame Debugger 验证和资源生命周期管理为主线，建立一套可以在团队中落地的 UGUI 基础优化方法。

## 正文

### 背景

UGUI 是 Unity 长期使用的 GameObject 组件式 UI 系统。它的优势是使用门槛低、编辑器工作流成熟、与运行时游戏 UI 场景高度契合；它的缺点也很明显：性能高度依赖层级组织、资源导入、材质复用、Canvas 结构和运行时更新方式。一个 UI 界面在设计稿里看起来只是几张图片和几行文字，但进入 Unity 后，它会变成纹理、材质、字体图集、Mesh、CanvasRenderer、事件射线、批处理和资源引用链。

很多项目的 UGUI 性能问题并不是后期突然出现的，而是在资源进入项目时就埋下了。图片尺寸过大、透明边没有裁剪、Read/Write 误开、MipMap 误开、Sprite 没进正确图集、公共图集混入低频资源、字体字符集过宽、TMP 材质实例泛滥、RawImage 滥用、动态图片没有缓存释放策略，这些问题会共同造成内存膨胀、Draw Call 增加、加载变慢、Canvas 重建扩大和 GPU 填充率上升。

因此，本篇不把优化写成零散技巧，而是从资源治理角度建立基础规则。只有资源侧是干净的，后续 Canvas 分层、Overdraw 优化、动画优化和 Profiler 定位才有意义。如果资源本身混乱，后面再靠代码补救，就像在漏水的船上擦地板，忙是很忙，但治不了本。

### 核心内容

#### 1. UGUI 从元素到屏幕的基本链路

一个 UGUI 元素最终显示到屏幕上，大致会经历四个阶段。第一，开发者创建 Image、RawImage、TextMeshProUGUI、Button、Toggle 等组件，这些组件大多继承自 Graphic 或依赖 Graphic。第二，UGUI 根据 RectTransform、CanvasScaler、布局组件、父子层级和 Canvas 参数计算元素在屏幕上的区域。第三，Graphic 生成顶点、UV、颜色等网格数据，并通过 CanvasRenderer 进入 Canvas 的批处理流程。第四，Canvas 按材质、纹理、裁剪状态、渲染顺序等条件组织批次，向渲染管线提交绘制。

这条链路里，CPU 主要承担布局计算、网格生成、文本排版、批处理构建和事件检测；GPU 主要承担纹理采样、透明混合、像素填充和 Shader 运算。资源管理会同时影响两端：图集会影响纹理切换和合批，字体资产会影响文本生成和材质数量，图片尺寸会影响纹理内存和填充率，材质实例会影响批次拆分，动态加载策略会影响内存峰值。优化时如果只盯代码，往往会错过真正的源头。

#### 2. Draw Call 与合批的正确理解

Draw Call 是 CPU 向 GPU 提交一次绘制命令。UGUI 会尽量把满足条件的 UI 元素合并为批次，以减少 Draw Call。可合批的前提包括相同或兼容的 Canvas 批处理范围、相同材质、相同纹理、相同 Shader 关键字、相同裁剪状态，以及连续的渲染顺序。

这里有一个经典误区：用了图集并不等于必然合批。Sprite Atlas 解决的是多个 Sprite 共享同一张纹理的问题，但如果这些 Sprite 中间插入了不同材质、不同 TMP 字体材质、不同 Mask、不同 Render Queue 或 RawImage，批次仍然会被打断。真正判断合批是否生效，必须使用 Frame Debugger 查看 UI Batch 的拆分原因，而不是只看资源是否被放进同一个图集。

另一个误区是 Canvas 越少越好。过大的 Canvas 会扩大重建范围，一处动态文本变化可能让大面积 UI 参与重建；Canvas 过多又会增加批次和排序管理成本。基础优化阶段先要做到资源可合批，下一篇再通过 Canvas 分层控制重建范围。

#### 3. Sprite Atlas 是现代 UGUI 项目的基础设施

现代 Unity 项目应优先使用 Sprite Atlas，而不是继续围绕旧式 Sprite Packer 或手工大图切割组织资源。Sprite Atlas 是显式资产，可以配置打包对象、平台压缩、Include in Build、变体图集和运行时加载策略。它不仅是美术资源容器，更是资源生命周期和批处理策略的一部分。

图集通常分为公共图集、模块图集和动态运营图集。公共图集只放全项目高频复用的小图标、通用按钮、通用边框、默认头像、货币图标等。模块图集放背包、商城、战斗、抽卡、任务等只在特定功能中出现的资源。动态运营图集放活动、下载、远程配置或版本化资源。公共图集必须严格审核，因为一旦常驻，它的每一个低频资源都会变成长期内存成本。

图集也不是越大越好。超大图集会增加加载峰值、内存占用和压缩风险，也会导致一个小界面引用整张大图。图集太碎又会造成纹理切换频繁，削弱合批收益。实用原则是：同一界面高概率同时出现、生命周期接近的 Sprite 进同一图集；不会同时出现或释放策略不同的 Sprite 不要硬塞进同一图集。

#### 4. 纹理内存不能按 PNG 文件大小判断

很多新人会看 PNG 文件只有几十 KB，就认为它运行时也很省内存，这是错误的。运行时纹理内存主要取决于导入后的平台格式、分辨率、MipMap、Read/Write 和压缩方式。比如 1024×1024 的 RGBA32 纹理，未压缩运行时约 4MB；如果开启 MipMap，额外增加约三分之一；如果开启 Read/Write，CPU 侧还可能保留一份可读副本，内存进一步增加。磁盘大小只能说明文件压缩效果，不能代表显存成本。

UI 纹理通常应关闭 Generate Mip Maps。MipMap 主要服务 3D 物体随距离变小的采样质量，而屏幕空间 UI 大多没有这种需求。Read/Write Enabled 也应默认关闭，除非确实要运行时读写像素、生成截图或做图像算法。任何开启 Read/Write 或 MipMap 的 UI 资源都应该进入审核清单。

#### 5. 平台压缩格式要按目标设备设计

纹理压缩不是“越压越好”，而是画质、内存、包体、加载和兼容性的平衡。移动端常见选择围绕 ASTC 和 ETC2 展开，ASTC 可以通过不同块大小在质量与内存之间调节；PC 平台通常使用 BC/DXT 系列；旧设备或特殊平台需要根据硬件支持制定回退。带 Alpha 的 UI 图标、渐变、细边框、文字类纹理要特别检查压缩伪影。

项目中不应让每张图由个人手动选择格式，而应使用 Texture Import Preset 或 Editor 脚本批量控制。公共图集、模块图集、运营大图、头像缩略图、像素风图标、高清背景可以有不同 Preset。资源审核时必须在目标设备上看效果，因为 Editor 预览无法代表真实采样和压缩质量。

#### 6. 图片尺寸应接近实际显示需求

一张图片在界面中最大只显示 64×64，就不应该导入 512×512 再缩小使用。缩小显示不会减少纹理内存，也不会自动减少图集占用。过大图片会浪费显存、增加加载时间、加大图集尺寸，还可能提升 Overdraw 区域。相反，如果图片太小再放大，会导致模糊和锯齿。

图片尺寸应由目标分辨率、CanvasScaler、Reference Resolution 和最大显示状态共同决定。比如道具图标、货币图标、按钮背景、页签、全屏背景、活动 Banner 都应有不同尺寸规范。透明边也要裁剪。大量透明像素虽然不可见，但仍会占据纹理面积和渲染矩形，造成内存和填充率浪费。

#### 7. TextMeshPro 不是免成本，但应作为默认文本方案

TextMeshPro 提供 SDF 字体、高质量缩放、描边、阴影、富文本和更现代的文本控制能力，正式项目中通常应优先使用 TMP 而不是旧 Text。但 TMP 仍然需要治理。字体图集尺寸、字符集范围、Fallback 链、材质 Preset、动态字符生成和频繁文本刷新都会影响性能。

中文项目最重要的是字符集管理。不要直接生成全量中文字库，而应从本地化表、配置表、剧情文本和 UI 预制体中提取实际使用字符，生成 Custom Character 文件，再生成 Font Asset。玩家昵称、聊天输入、运营文本等不可预测字符可以使用受控 Fallback，但要限制范围并记录缺字日志。TMP 材质也要复用，避免每个文本复制一份材质实例导致合批失败。

#### 8. RawImage 要有明确边界

RawImage 直接引用 Texture，适合头像、远程下载图、RenderTexture、视频帧、WebView 截图和特殊动态纹理。它不适合替代普通 Image。大量 RawImage 引用不同 Texture 会导致纹理切换和批次增加，尤其在列表中显示头像、商品图、卡牌图时很容易失控。

普通静态图标应使用 Sprite + Image，并进入 Sprite Atlas。动态图片应由统一加载器管理，包含下载、缓存、引用计数、缩略图尺寸、失败占位图和释放策略。列表中动态图要按可见区域懒加载，并在 item 复用时做版本检查，避免旧图片写回新 item。

#### 9. 资源生命周期必须与 UI 模块绑定

UI 资源不应该散落在各个脚本里随意加载。常驻 HUD、公共图标、全局字体属于基础资源域；背包、商城、活动、战斗结算属于模块资源域；头像、运营图、下载图属于动态资源域。不同资源域应有不同加载和释放策略。

对于 Addressables 或 AssetBundle 项目，打开界面时加载依赖资源，关闭界面时 Release 句柄或减少引用计数；是否立即卸载内存要看界面复用频率。频繁打开关闭的界面可以短期缓存，低频大型界面关闭后可以释放。Resources.UnloadUnusedAssets 是重操作，适合加载页、场景切换或明确内存整理时机，不适合普通 UI 关闭时频繁调用。

#### 10. 工程化细则：公共图集准入标准

公共图集准入标准不是单一技术点，而是一条流程约束。项目应为它定义负责人、检查时机、异常处理和回归方式。例如资源提交时先由自动脚本扫描，再由对应模块负责人确认是否符合图集和内存策略，最后在真机包中通过 Profiler 或 Memory Profiler 观察运行时表现。对于 UI 资源而言，越早发现问题，修复成本越低；等到界面已经接入业务、动画和异步加载后再改图集或字体，往往会牵动大量 Prefab 和代码。

落地时建议把规则写成可执行检查，而不是只写在文档里。文档告诉团队为什么要做，工具保证团队真的做到。对于例外情况，例如高质量活动图不能压缩、特殊图标必须独立纹理、某个字体需要更大字符集，应要求提交说明和真机截图。这样既保留灵活性，也避免“特殊需求”逐渐变成无规范状态。

### 实现方案

#### 1. 推荐目录结构

```text
Assets/Game/UI
  Common/
    Sprites/
    Atlas/UI_Common.spriteatlasv2
    Fonts/
    Materials/
  Battle/
    Sprites/
    Atlas/UI_Battle.spriteatlasv2
    Prefabs/
  Bag/
    Sprites/
    Atlas/UI_Bag.spriteatlasv2
    Prefabs/
  Activity/
    Sprites/
    Atlas/UI_Activity_xxx.spriteatlasv2
```

目录结构不是为了好看，而是为了资源生命周期清晰。公共资源、模块资源和活动资源分开后，图集归属、加载策略和卸载策略才容易自动化。

#### 2. UI 纹理导入检查伪代码

```pseudo
for texture in uiTextures:
    if texture.isSprite:
        assert texture.mipmap == false
        assert texture.readWrite == false
        assert texture.maxSize <= rule.maxSize(texture.category)
        assert texture.platformFormat in allowedFormats
        if texture.transparentPaddingTooLarge:
            report("透明边过大")
        if texture.notInAtlas and texture.category != "DynamicRaw":
            report("未归属图集")
```

这类检查应进入 Editor 菜单、CI 构建前检查和资源提交检查。不要依赖人工逐个 Inspector 点开看。

#### 3. 字符集生成流程

```pseudo
texts = collect(localization, config, story, prefabDefaultText)
chars = unique(texts)
chars += commonRuntimeSymbols
write(chars, "TMP_Chars_CN.txt")
generateTmpFontAsset(charsFile)
validateMissingGlyphs()
```

多语言项目建议按语系拆分 Font Asset。运行时不可预测文本使用 Fallback，但 Fallback 链必须受控，避免动态扩展失控。

#### 4. Frame Debugger 验证合批

资源整理后必须在真实界面中验证。打开 Frame Debugger，查看 UI Batch 是否因为 Texture Changed、Material Changed、Clipping Changed、Shader Keywords Changed、Stencil State Changed 等原因拆分。每个核心界面都应记录 Draw Call 基线，并随版本对比。

#### 5. 动态图片加载模板

```pseudo
async loadIcon(id, target, token):
    version = target.version
    sprite = await iconCache.getOrLoad(id, token)
    if token.cancelled or target.version != version:
        iconCache.release(sprite)
        return
    target.image.sprite = sprite
```

对象池列表必须检查 version。否则 item 被复用后，旧异步请求可能把 A 的头像写到 B 的位置。

#### 6. 资源审核清单

```text
图片尺寸是否超过实际显示需求？
是否开启 Read/Write？
UI 纹理是否误开 MipMap？
Sprite 是否归属正确 Atlas？
公共图集是否混入低频模块资源？
TMP Font Asset 是否字符集过宽？
材质是否被 Prefab 复制成实例？
RawImage 是否可以改成 Image + Sprite？
动态图片是否有缓存和释放策略？
Frame Debugger 中批次是否符合预期？
```

#### 1. 公共图集准入

公共图集准入应被视为资源进入项目的质量门槛，而不是事后优化项。对于 UGUI 来说，资源一旦进入 Prefab、图集和资源包，后续修改会牵动引用、加载路径、图集重新打包和真机验收，因此越早约束越省成本。团队可以把这一项拆成三个动作：导入时自动检测，Prefab 制作时人工确认，构建后通过报告复核。

落地时建议为该项设置明确阈值。例如图片最大尺寸、是否允许 Alpha、压缩格式、图集归属、是否允许进入公共资源域、是否必须按需加载等，都要有规则。特殊资源可以例外，但必须说明原因、目标平台和内存影响。这样项目既有规范，也保留必要灵活性。

#### 资源治理补充

在团队执行时，建议把该项拆成开发期检查和发布前检查两层。开发期检查偏快速反馈，例如 Editor 扫描、Prefab Review、资源导入提示；发布前检查偏真实环境，例如真机 Profiler、Memory Profiler 快照、Frame Debugger 截图、低端机回归。这样既不会把日常开发拖得太重，又能在发版前守住性能底线。

### 总结

UGUI 基础优化的根基是资源治理。图集规划、纹理压缩、字体字符集、RawImage 边界、资源加载释放、材质复用和合批验证，决定了 UI 系统的初始性能上限。本文修正了原稿中对 Sprite Packer、图集合批、遮挡剔除和纹理内存的过度简化说法，并将内容整理为可执行的工程规范。

一个成熟项目不应等到界面卡顿后再找原因，而应在资源导入、Prefab 制作、图集归属、字体生成和真机验收阶段就把问题挡住。后续 Canvas 管理、Overdraw 优化和高级动画优化，都会建立在这套基础之上。
