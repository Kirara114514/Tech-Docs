# Canvas 分层的正确姿势与性能收益

## 摘要
Canvas 分层是 UGUI 优化中最常被讨论、也最容易被误解的一项技术。很多团队一听到“Canvas Rebuild 很贵”，就会下意识地把界面一层层拆成很多 Canvas，仿佛拆得越碎性能越好。结果往往相反：层级变复杂、批次变多、管理更乱、排序问题更频繁，最终既没获得理想收益，还增加了维护成本。问题的关键在于，Canvas 分层不是“把界面拆散”这么简单，而是在控制 UI 脏区传播范围、重建范围、排序边界和资源组织方式。它的收益建立在一个前提上：高频变化元素与低频稳定元素之间必须有足够清晰的刷新边界。如果这个边界判断错了，Canvas 分层就会从性能优化变成性能噪音制造器。本文围绕 Canvas 分层真正解决的问题、CPU 与 GPU 两本账、分层的正确粒度、典型适用场景、错误拆分方式、调试与验收方法，系统说明什么才是 Canvas 分层的“正确姿势”，以及它为什么既不是银弹，也不是可有可无的小技巧，而是 UI 架构层必须认真设计的基础策略。

## 正文

### 背景
UGUI 的很多性能问题，最终都绕不开 Canvas。原因很简单：Canvas 是批处理、顶点生成、Graphic 重建和布局更新的重要边界之一。当某个 UI 元素发生变化时，变化不会只影响它自己，而可能沿着某些层次关系把更多元素卷进重建。

这也是为什么项目里常出现这种现象：

- 一个小红点变化，整个主界面都有明显开销。
- 一个数字文本刷新，连带一大片静态区域也触发 Rebuild。
- 一个高频动画区域拖慢了整个页面。

此时，Canvas 分层自然会成为优化候选，因为它能帮助把“高频变化区域”从“大面积稳定区域”中隔离出来。但问题在于，很多团队对它的理解过于粗糙，只记住了：

- “多个 Canvas 可以减少重建传播。”

却忽略了另一半：

- “多个 Canvas 也会增加批次边界、排序边界和管理复杂度。”

因此，Canvas 分层不是单向收益，而是明确的工程权衡。

### 核心内容

### 第一原则：Canvas 分层的目标是控制脏区传播，而不是追求层级数量
Canvas 分层的核心收益来自“把高频脏区隔离在更小范围内”。这句话非常重要。真正应该问的不是：

- 这个界面该拆几个 Canvas？

而是：

- 哪些元素高频变化？
- 这些变化是否值得把重建局限在局部？
- 被隔离出去的那部分是否足够独立？

如果一个界面里有明显两类区域：

- 大部分区域长期稳定。
- 少量区域高频刷新。

那把这两者分开通常是有价值的。典型例子包括：

- 主界面中的静态背景与高频红点区域。
- 常驻面板中的计时器、货币数字、Buff 倒计时。
- 战斗 HUD 中的高频数值飘动和稳定底板。

但若所有区域都低频变化，或强依赖统一层级和统一排序，那么强行拆 Canvas 很可能得不偿失。

### 第二原则：Canvas 分层要同时看 CPU 账和 GPU 账
很多团队在做 Canvas 分层时只盯着 Rebuild 次数，忽略 GPU 侧成本，这会导致片面优化。Canvas 分层至少要同时考虑两本账：

#### CPU 账
- Graphic Rebuild 范围是否缩小。
- Layout 传播是否减少。
- 高频刷新是否被局部化。
- 排序与脏区更新是否更可控。

#### GPU 账
- DrawCall 是否增加。
- 材质 / 图集边界是否被打断。
- Overdraw 是否因层次关系变化变得更糟。
- 排序关系是否导致更多渲染切换。

换句话说，Canvas 分层常见的真实收益是：

- CPU 降了，GPU 可能升了。

若不同时算两本账，就可能做出“Profile 某一项更好，但整体帧率没改善”的错误决策。

### 第三原则：最值得分层的是“高频变化的小块”，不是“视觉上看起来是一块”的区域
很多 UI 设计师和程序初看分层时，会按视觉区域拆：

- 左边一块一个 Canvas。
- 右边一块一个 Canvas。
- 顶栏一个 Canvas，底栏一个 Canvas。

这种拆法有时会对，但真正更稳的判断标准应是“刷新频率和刷新边界”，而不是视觉布局边界。

例如：

- 一个视觉上在同一区域的金币数值和静态背景，如果金币高频变化、背景稳定，那它们更适合拆。
- 一个视觉上上下分离的区域，如果它们总是同时刷新、共享同一资源边界，拆开未必有收益。

也就是说，Canvas 分层本质上是按“变化模式”拆，而不是按“美术版面”拆。

### 第四原则：不要为了减少 Rebuild，制造无法收拾的排序与批次碎片
Canvas 一旦拆开，就天然会带来更多独立排序和渲染边界。这在以下情况尤其容易出问题：

- 动态生成的浮层很多。
- 弹窗、提示条、引导高亮、特效混在一起。
- 各层 UI 资源组织本就不够规整。
- 大量子 Canvas 共用关系复杂。

这时常见问题包括：

- DrawCall 突然明显上升。
- 某些元素层级关系变难维护。
- 动态特效和普通 UI 排序互相穿插。
- 同一个逻辑块被拆到多个 Canvas 后，调试和改需求都很痛苦。

所以成熟团队在做 Canvas 分层时，一定会同时问：

- 它带来的排序复杂度能不能承受？
- 它会不会打散原本可合批的资源组织？
- 它的管理边界是否足够清晰？

### 第五原则：静态 Canvas、半动态 Canvas、高动态 Canvas 应该被区别对待
这是实际项目里非常好用的一种分层思路。

#### 静态 Canvas
几乎不变或极少变化，例如：

- 固定底板。
- 背景装饰。
- 长时间不动的框架结构。

目标：

- 尽量少重建。
- 保持资源组织稳定。

#### 半动态 Canvas
偶尔变化、局部刷新，例如：

- 顶栏资源数。
- 页面状态切换。
- 常规提示。

目标：

- 控制变化传播。
- 保持适度合批。

#### 高动态 Canvas
频繁变化、可能每帧刷新，例如：

- 倒计时。
- 战斗飘字。
- 高频数值区。
- 动画驱动密集区域。

目标：

- 尽量局部隔离。
- 接受一定批次代价，换取更小 Rebuild 范围。

这种分层方式往往比“每个区域随便拆个 Canvas”更有工程价值。

### 第六原则：子 Canvas 不是越多越好，越细越容易反噬
这是很多团队最容易踩的坑。子 Canvas 当然可以隔离脏区，但如果拆得过细：

- Canvas 数量膨胀。
- 批次边界过碎。
- 排序和引用关系变复杂。
- 页面结构阅读性下降。

这种做法短期也许能把某个 Rebuild 热点压下去，但长期很容易形成：

- 难以理解的 UI 树。
- 维护人员不敢动。
- 新需求一来不知道该挂在哪个 Canvas 下。

所以真正的“正确姿势”是：只拆那些真的具备高频变化边界且值得独立承担成本的区域。

### 典型适用场景

#### 一、主界面常驻高频区
如顶栏货币、红点、计时器，适合和大面积稳定底板隔离。

#### 二、战斗 HUD
高频血条、伤害、倒计时和特效区，往往适合局部独立，而不是拖整个 HUD 一起 Rebuild。

#### 三、复杂面板中的滚动列表旁边的静态框架
列表区和固定框架分离，通常收益明显。

#### 四、运营活动页中的局部动态组件
尤其是数字刷新、转盘、倒计时和领奖状态高频变化区域。

### 不适合盲目分层的场景

#### 一、低频变化页面
如果页面本身几乎不刷新，拆 Canvas 价值很低。

#### 二、需要强统一层级和统一合批的区域
拆了之后可能 GPU 账比 CPU 账更差。

#### 三、生命周期高度一致、总是一起刷新的区域
如果本来就一起脏、一起刷新，分层未必有意义。

### 常见反模式

#### 反模式一：所有模块都默认挂一个子 Canvas
看起来模块化，实际上层级和批次都被打碎。

#### 反模式二：以美术视觉区块为唯一拆分标准
忽略真实刷新模式。

#### 反模式三：只看 Rebuild，不看 DrawCall 和图集边界
最后 CPU 好一点，GPU 却明显更差。

#### 反模式四：动态子 Canvas 四处漂，排序关系全靠试
维护和联调都会越来越困难。

### 调试与验收
评估 Canvas 分层是否真的有效，至少要比较：

- 分层前后的 Rebuild 次数和耗时。
- 高频交互时主线程波动。
- DrawCall 与材质切换变化。
- 真机，尤其是中低端设备上的整体帧时间。
- 页面结构复杂度是否显著增加。

必须强调：Canvas 分层是结构优化，不是单指标优化。若 Rebuild 降了但整体工作流更乱、GPU 更差、层级难以维护，那就不是合格方案。

### 实现方案

### 推荐的落地顺序
#### 一、先找高频变化热点
不要先拆层，先用 Profiler 找出真正高频刷新的区域。

#### 二、按变化模式而不是版面拆分
确认哪些区域应该独立承担脏区传播。

#### 三、控制拆分粒度
每多一个子 Canvas，都要回答“它为什么值得独立存在”。

#### 四、同步检查图集和材质策略
避免把可合批结构拆碎。

#### 五、真机回归
尤其看连续交互时的帧时间波动，而不只是编辑器下的平均值。

### 总结
Canvas 分层的正确姿势，不是把界面越拆越碎，而是围绕高频变化边界控制脏区传播，用合理的局部隔离换取更可控的 Rebuild 范围。它的收益永远建立在 CPU 账与 GPU 账同时平衡的前提上：高频变化区域确实被局部化了，排序与批次又没有被拆得难以收拾。只有按变化模式而不是按视觉版面拆分、只隔离真正值得隔离的区域，并结合资源组织与真机验证一起评估，Canvas 分层才会成为可靠的结构优化，而不是“听起来懂优化、实际上制造新问题”的表面动作。
