# 避免 UI Rebuild 的技巧与案例

## 摘要
在 Unity 的 UGUI 体系里，很多团队一看到界面卡顿，就把锅直接甩给 DrawCall，或者把优化动作全部集中在图集、合批和裁剪上。但在大量项目里，真正把 CPU 压爆、把主线程拖慢、把交互延迟放大的，往往不是渲染提交本身，而是 UI Rebuild。所谓 Rebuild，本质上是界面树在“我变了”之后，为了重新计算布局、顶点、裁剪和绘制数据而进行的一系列重建动作。只要脏区扩散过大、重建频率过高、重建范围失控，再少的 DrawCall 也救不了帧率。

这篇文档的核心不是罗列零散技巧，而是建立一套工业化的 UI Rebuild 控制方法：先理解 Rebuild 到底在重建什么，再识别哪些操作最容易触发大范围脏传播，最后通过分层、隔离、批处理、延迟刷新、虚拟化和职责重构，把“每改一点就整棵树重算”的粗暴模式改造成“只让真正发生变化的局部付费”的精细模式。只有这样，UI 系统才能在长生命周期、多状态、多人协作的项目里持续保持稳定。

## 正文

### 背景
很多项目在开发前期并不会明显暴露 UI Rebuild 问题。界面数量少、节点数量低、交互节奏慢时，即便写法粗糙，性能往往也还过得去。问题通常在中后期集中爆发：系统页叠加得越来越多，弹窗越来越复杂，列表越来越长，文本更新越来越频繁，状态动画越来越多，结果就是一个看似不起眼的数值变化、一个按钮显隐、一次节点尺寸变化，都会引发一连串布局重算和图形重建。开发者在 Profiler 里看到大量时间耗在 `Layout`、`Canvas.SendWillRenderCanvases`、`Graphic` 更新、顶点重建上，才意识到 UI 的真正成本不是“画出来”，而是“准备把它画出来”。

更麻烦的是，UI Rebuild 具备很强的隐蔽性。许多业务代码本身并不复杂，只是调用了 `text.text = xxx`、`SetActive`、`SetSizeWithCurrentAnchors`、修改父子层级、刷新 Layout Group，表面上只是“改个显示”，实际上却在悄悄触发整片区域的重新布局和几何重建。团队如果没有明确的边界约束，很容易演化成一种危险模式：程序把 UI 当成数据对象任意驱动，策划不断提出实时刷新需求，美术要求复杂层级和动态适配，最终所有变化都直接打到同一棵大 Canvas 树上，任何一处高频更新都会把整块 UI 系统拖下水。

因此，避免 UI Rebuild 的真正目标，不是追求“完全没有重建”。那是不现实的，因为界面只要变化，就一定会有某种形式的重建。真正的目标是把 Rebuild 控制在合理频率、合理范围、合理时机之内，让每一次重建都服务于必要的用户反馈，而不是被无序的数据流和错误的层级设计反复放大。

### 核心内容
要优化 UI Rebuild，首先必须区分几类不同的成本。第一类是布局重建。它关注的是节点尺寸、锚点、父子约束、自动布局、内容自适应等问题，代表组件包括 Layout Group、Content Size Fitter、Layout Element 等。第二类是图形重建。它关注的是顶点、颜色、UV、材质、文本字形等图形数据是否需要重新生成。第三类是 Canvas 级的数据整理与提交，它把前面产生的变化汇总成可渲染的数据。这三类成本彼此关联，但触发源并不完全一样。团队如果把它们混成一个概念，就会出现“明明没加 DrawCall，为什么还卡”“只是换个文案，为什么整页抖动”的认知错位。

在工程实践里，最常见的 Rebuild 诱因有以下几类。第一，高频文本更新。比如战斗飘字、血量数字、倒计时、货币面板、在线人数，这些都可能在短时间内频繁修改文本内容。文本一旦变化，不仅图形需要重建，若布局依赖文本尺寸，布局链条也会被一并拉起来。第二，自动布局滥用。Layout Group 和 Content Size Fitter 在原型阶段极其好用，但一旦进入复杂页面、高频列表、动态面板，它们往往会变成性能黑洞，因为它们会让一个局部尺寸变化沿父层级持续传播。第三，频繁显隐与重排。通过 `SetActive`、修改 sibling index、动态挂载和移除节点来表达状态切换，虽然业务上直观，但会让树结构和布局关系不断变化。第四，把高频更新元素和大量静态元素放在同一 Canvas、同一布局域里。只要局部变化发生，静态区也会被迫参与重建。

真正成熟的做法，是把“界面变化”从“界面树重建”里拆开看。不是所有业务变化都应该立刻转成 UI 改动，更不是所有 UI 改动都应该立刻落到完整节点树上。很多所谓的 Rebuild 问题，根因并不是 Unity 算得慢，而是上层系统把变化组织得太粗糙，导致本可合并、延迟、局部化的更新，被写成了即时、离散、全域性的树操作。

工业化项目里，一般会建立以下几个基本原则。第一，静态区域、低频区域、高频区域必须隔离。第二，布局系统优先服务“易开发”，但上线结构必须服务“低脏传播”。第三，状态刷新应尽量经过聚合层，不让每个业务事件直接打 UI。第四，任何会牵动层级、尺寸、文本和材质的操作，都要先问一句：这是必须实时更新，还是可以批量更新？第五，列表、排行榜、聊天、背包等高节点数系统，不允许依赖整树自动布局去“自动算出来”，而要主动控制可见区、复用池和局部刷新边界。

### 实现方案
第一步是做界面树分区，把“变化频率”而不是“视觉归属”作为首要切分原则。很多团队喜欢按页面结构切 Canvas，比如主界面一个、顶部栏一个、底部栏一个，听起来合理，但如果顶部栏有货币跳动、体力计时、活动红点，而底部栏几乎静止，这两者的性能风险完全不同。正确的做法是把会高频变化的区域单独隔离出来，让它们的重建只影响自己；把长期稳定的背景、装饰、边框、静态图标留在低频 Canvas 中。这样做的收益不是“绝对减少所有重建”，而是把本来会扩散到整个页面的重建局限在小范围内。

第二步是给数据刷新加“收口层”。不要让每个系统事件都直接改 UI 控件。更好的模式是由 Presenter、ViewModel 或统一的刷新调度器先合并业务变化，再在合适时机一次性提交给视图。例如顶栏的金币、钻石、体力、邮件红点、活动状态经常在一个短时间窗口内同时变化，如果每个系统消息都立刻改一个 Text 或一个节点状态，就会触发多次连续 Rebuild。相反，可以把这一帧内的所有状态先汇总，再统一刷新顶栏。这种做法本质上是在降低 UI 的写入频率，把“十次小改”变成“一次合并更新”。

第三步是减少对自动布局链的依赖。自动布局不是不能用，而是不能在高频区域和长列表里滥用。对于相对固定的结构，优先采用手工锚点、预设尺寸和有限的布局约束，把“运行时持续计算”变成“设计时一次确定”。特别是列表项内部，如果每次文本变化都会驱动父级自适应、再驱动滚动容器重排，那么性能会随着节点数线性甚至更糟地放大。工业项目中常见的做法，是把复杂列表拆成固定项高、固定间距、少量动态字段的结构，再用虚拟化控制可见项数量，而不是寄希望于 Vertical Layout Group 和 Content Size Fitter 自动处理一切。

第四步是避免用“树结构变化”表达普通状态变化。很多界面为了切换状态，会频繁 `SetActive` 子节点、增删对象、改兄弟顺序，或者在不同面板之间来回挂载节点。这样做短期方便，但树一动，布局和渲染缓存就都可能失效。能用透明度、可交互状态、局部内容切换解决的问题，尽量不要通过大范围显隐和重排来表达。这里不是绝对禁止 `SetActive`，而是强调应把它当作重操作。对于高频状态，比如按钮冷却、可领取标记、数值闪烁，更适合通过轻量的图形状态切换解决，而不是频繁改层级。

第五步是为文本系统建立规则。文本是 Rebuild 的高发源头，尤其在中文项目里更明显，因为文案长度波动大、字体和换行策略复杂。首先，高频数值文本应与复杂布局解耦，不要让一个金币数字变化拖动整个父节点重新布局。其次，能固定宽度和排版的数字区尽量固定，避免内容长度变化牵动周围结构。再次，滚动列表中的文本更新必须谨慎，尤其在复用项里，要避免每次绑定数据都触发多层布局刷新。最后，对于跳字、伤害数、计时等极高频内容，应评估是否需要专门的轻量渲染方案，而不是混入常规页面层级中。

第六步是用案例驱动治理，而不是只靠规范口号。以一个常见的主界面顶栏为例，原始实现可能是：金币变化立刻改 Text，体力恢复每秒改一次时间文本，邮件红点来自异步消息立刻显隐，活动入口根据多个条件切换子节点，结果顶部栏整个区域每几百毫秒就有脏传播。如果顶部栏又和整页背景、页签、资源条放在同一 Canvas 上，就很容易在低端机上产生明显抖动。优化方式不是一句“拆 Canvas”就结束，而是组合治理：将顶部栏独立为高频层；把多来源数据聚合成单次刷新；体力时间显示采用按秒驱动但只更新必要文本；红点系统采用状态缓存，只在值真正变化时刷新；活动入口避免频繁重建树结构。这样优化后，CPU 时间下降往往不只是百分之几，而是能从“肉眼卡顿”变成“几乎感觉不到”。

再看背包或装备页。很多团队的问题是每次筛选、排序、数量变化都直接全量重绑列表，再让 Layout Group 重新排一遍，导致滚动容器和所有项一起重建。正确做法应该分层处理：数据排序在数据层完成；UI 只接收最终可见区数据；列表项池化复用；数量变化只刷新受影响项；筛选结果变化时才触发可见窗口重算，而不是整页整树全量刷新。这里的关键不只是“减少实例化”，而是把 Rebuild 控制成“局部、有限、可预测”的更新。

在战斗 HUD 场景里，错误更常见。血条、能量条、Buff 图标、伤害跳字、目标状态、提示文本常常被堆在同一层里同步刷新。只要其中有一块高频波动，就会把整个 HUD 拖入高频重建。成熟做法会把高度动态的部分单独隔离，比如飘字系统走独立层或专用渲染通道，核心状态条集中但局部更新，Buff 区域限制图标数量和刷新频率，文本变化避免牵动布局。换句话说，避免 UI Rebuild 的关键不是“禁止动态 UI”，而是把动态 UI 的影响半径控制住。

第七步是建立排障方法。很多团队知道“UI Rebuild 很贵”，却不知道问题到底出在哪。正确的排查思路应从“谁最频繁改”“谁传播最远”“谁最不可控”三个维度入手。首先在 Profiler 中确认问题是布局重建、图形重建还是 Canvas 提交；然后定位高频变化的控件和页面；再看这些控件是否处于错误的布局链和错误的 Canvas 边界中。之后要回到代码层检查：是否存在每帧赋值、重复刷新、即使值没变也照样写 UI、列表项反复全量绑定、动画驱动尺寸变化、滥用 Layout Group 等。只有把“现象”还原为“触发链条”，团队才能真正建立可复制的优化经验。

最终，避免 UI Rebuild 不是某个技巧的胜利，而是一种系统化的工程观。UI 不应该是任由业务随意写入的被动树，而应当是一套对变化频率、作用范围、更新节奏都高度敏感的前端系统。你越早建立这种边界，项目越不容易在后期被 UI 拖垮；你越晚意识到这件事，越可能在系统爆炸后陷入“哪里都能改、哪里都不敢动”的维护泥潭。

### 总结
UI Rebuild 优化最容易出现两个误区。第一个误区是把问题简单归因成“Canvas 太大”或“DrawCall 太多”，结果只做表层拆分，却没有治理高频写入和布局传播。第二个误区是把所有动态能力都视为禁忌，最后把 UI 写成一套极难维护的硬编码系统。真正有效的方法是在性能和可维护性之间找到工程平衡：高频区域单独隔离，自动布局只用于合适场景，状态变化经过聚合层统一提交，大列表走虚拟化，文本和布局解耦，重操作谨慎使用。

对于成熟团队来说，避免 UI Rebuild 不是一次性的专项优化，而应该沉淀成长期规范：界面评审时看变化频率，程序实现时看脏传播链路，性能测试时看重建范围与触发次数，线上问题复盘时看是谁打破了局部更新边界。只有把这些经验制度化，UI 性能才不会在项目后期反复回退。
