# Unity UGUI Canvas 与元素管理

## 摘要

本文是 UGUI 性能优化系列的第二篇，主题是 Canvas 与 UI 元素管理。本文会重写 Canvas 重建机制、Canvas 分层策略、RectTransform 使用规范、LayoutGroup 风险、显示隐藏策略、UI 对象池、虚拟列表和事件生命周期管理。

原稿中存在重复段落，同时也有一些需要更谨慎表达的内容，例如“任意子元素变化必然重建整个 Canvas”“CanvasGroup alpha 完全无成本”“DOTween 会自动解决 RectTransform 成本”“SetActive 一定比 CanvasGroup 慢”等。本文会将这些内容改写为更准确的工程判断：不同操作触发的重建类型不同，不同隐藏方式适用于不同生命周期，Canvas 分层必须通过 Profiler 验证收益。

## 正文

### 背景

Canvas 是 UGUI 的核心组织单元。它既是渲染批处理的重要边界，也是重建影响范围的重要边界。很多 UI 性能问题表面上看是某个 Text 更新、某个 Image 换图、某个列表滚动卡顿，实质上是 Canvas 结构设计不合理：静态背景、动态数字、滚动列表、弹窗动画、特效遮罩全部混在一个巨型 Canvas 内，一点小变化就可能牵动大量批处理和布局计算。

但 Canvas 也不是拆得越细越好。每个 Canvas 都有独立批处理、排序和管理成本。把每个按钮、每个图标、每个列表项都拆成独立 Canvas，通常会让 Draw Call、层级复杂度和维护成本上升。Canvas 优化真正考验的是平衡：哪些 UI 变化频率相近，可以放在一起；哪些 UI 需要隔离重建；哪些 UI 需要独立排序；哪些 UI 只是为了心理安全而被过度拆分。

### 核心内容

#### 1. Canvas 重建要区分 Layout 与 Graphic

UGUI 重建不是单一动作。Layout Rebuild 关注 RectTransform、LayoutElement、HorizontalLayoutGroup、VerticalLayoutGroup、GridLayoutGroup、ContentSizeFitter 等布局关系；Graphic Rebuild 关注 Image、RawImage、TextMeshProUGUI 等可渲染组件的顶点、UV、颜色、材质和文本几何。

修改文本可能只触发文本网格更新，也可能因为 preferred size 改变进一步触发布局；修改 RectTransform 的 sizeDelta 更容易影响布局链；修改 Image.sprite 会导致图形数据变化；在 LayoutGroup 下增删子项会引发布局重新计算。优化时必须知道自己在处理哪类成本，否则会把问题归因错。

#### 2. 子 Canvas 的作用是隔离变化，不是魔法加速

子 Canvas 可以把一部分 UI 从父 Canvas 的批处理和重建影响中隔离出来。高频变化区域放进独立 Canvas，能避免它牵连大面积静态 UI。但子 Canvas 也会带来额外批次和排序成本。是否拆分，需要通过 Profiler 对比 Canvas.BuildBatch、Layout 耗时和 Draw Call 变化。

判断是否拆 Canvas，可以问四个问题：这个区域是否高频变化？它变化时是否导致大 Canvas BuildBatch 明显升高？它是否本来就因为材质、Mask 或排序无法与周围合批？它是否需要独立显示隐藏或排序？如果答案多为是，可以拆；如果只是静态小图标，拆 Canvas 通常没必要。

#### 3. 按变化频率分层比按视觉层级更关键

UI 分层不能只按“背景、按钮、弹窗”这种视觉层级来做。真正影响 Canvas 重建的是变化频率。静态背景、低频按钮、高频倒计时、滚动列表、弹窗动画应该分开管理。常见结构包括 StaticBackgroundCanvas、CommonHudCanvas、DynamicHudCanvas、ListCanvas、PopupCanvas、TopTipCanvas 和 GuideCanvas。

动态数字、倒计时、血条、聊天输入、列表滚动这些高频元素，应放在较小的 Canvas 中。静态背景和低频装饰应放在几乎不变的 Canvas 中。弹窗层则应由窗口栈管理排序、遮罩、交互和底层界面暂停。

#### 4. RectTransform 修改是布局成本入口

RectTransform 的 anchoredPosition、sizeDelta、anchorMin、anchorMax、pivot、localScale 等变化，都可能让 UI 被标记为脏。尤其在 LayoutGroup、ContentSizeFitter、AspectRatioFitter 参与时，一个子元素变化可能向上传播到父级，再向下影响兄弟节点。

运行时应避免无条件写 RectTransform。即使写入的是相同值，也可能造成不必要的脏标记。需要动画的元素尽量不放在复杂 LayoutGroup 内；需要尺寸变化的动画最好隔离到小 Canvas；拖拽或滚动应控制刷新范围；初始化阶段一次性设置完整布局，运行时只改真正变化的属性。

#### 5. LayoutGroup 是便利工具，也是性能风险

LayoutGroup、ContentSizeFitter 和 LayoutElement 对自适应 UI 很有帮助，尤其是多语言、聊天气泡、动态列表。但它们也很容易形成复杂布局链。动态列表里每次新增、删除、改文本，如果都让 LayoutGroup 重新计算大量子项，CPU 峰值会非常明显。

工程建议是：静态 UI 可以编辑期使用 LayoutGroup，运行时固定布局；动态列表优先固定 item 尺寸和虚拟化；可变高度文本区域单独隔离；ContentSizeFitter 不要与多层 LayoutGroup 随意嵌套；批量刷新时先收集数据，再集中应用，避免每次修改都触发一次布局链。

#### 6. 显示隐藏方式要按生命周期选择

SetActive(false) 会禁用对象及子对象，停止 Update、交互和渲染，但重新激活复杂层级可能触发较大初始化和重建。CanvasGroup alpha 适合短时淡入淡出，但 alpha 为 0 的对象仍可能参与布局、事件或渲染管理，需要同步设置 interactable 和 blocksRaycasts。禁用 Graphic.enabled 可以隐藏某个图形，但对象和布局仍在。禁用 Canvas 可以停止整层渲染，但恢复时也要测试成本。

短时间动画隐藏用 CanvasGroup；长期关闭的大界面用 SetActive 或资源卸载；频繁复用的列表项用对象池；纯装饰图可以禁用 Graphic；整个层暂停显示可以禁用 Canvas。不要把一种方式套所有场景。

#### 7. UI 对象池必须处理状态重置

对象池不是简单把 GameObject 放进 Queue。一个合格的 UI 对象池要处理创建、预热、获取、绑定数据、版本号、归还、重置、取消异步加载、事件解绑、动画停止、资源引用释放和父节点归位。否则会出现旧数据残留、按钮监听重复、Tween 继续播放、旧头像加载完成后写到新 item 上等问题。

池化 UI 建议实现 OnSpawn 和 OnDespawn。OnSpawn 负责绑定新数据和版本号，OnDespawn 负责停止动画、取消异步、清理图片、移除事件、恢复颜色和缩放。列表 item 必须做版本检查，因为异步图片和网络数据经常晚于 item 复用完成。

#### 8. 动态列表应虚拟化

背包、聊天、排行榜、邮件、商城这类列表不应该为所有数据创建 GameObject，再靠 Mask 裁剪不可见区域。不可见 item 即使被裁掉，仍然可能参与布局、占用内存、维持组件和事件。虚拟列表只创建可见区域附近的少量 item，并根据滚动位置复用它们显示不同数据。

固定高度列表最容易虚拟化：根据滚动偏移计算首尾索引，复用固定数量 slot。可变高度列表需要缓存高度和前缀和。复杂瀑布流需要更强的数据结构。不要每次滚动都全量遍历所有 item 判断可见性。

#### 9. 事件生命周期必须成对管理

UI 元素经常订阅按钮事件、全局事件、数据模型事件、网络回调和异步任务。如果界面关闭后没有解绑，可能导致对象无法释放，或隐藏 UI 继续响应数据变化。OnEnable 订阅、OnDisable 解绑是一种基本模式，但对象池 item、异步任务和一次性弹窗还需要 OnDespawn、OnDestroy 和 CancellationToken。

匿名 lambda 订阅尤其危险，因为取消订阅时很难引用同一个委托实例。按钮点击也要在归还对象池时 RemoveListener。建议使用 Disposable 集合统一管理订阅，界面关闭时一次性 Dispose。

#### 10. 工程化细则：Canvas 命名规范

Canvas 命名规范需要明确写入团队 UI 规范。它不只是一个建议，而是为了让多人协作下的 UI 行为保持一致。比如弹窗栈管理要统一决定遮罩、排序、底层暂停和返回键处理；高频文本隔离要规定哪些文本必须放入动态 Canvas；对象池容量上限要防止池无限增长；界面关闭清理要覆盖事件、Tween、异步任务和资源句柄。

每条规则都应配合检查工具或 Review 清单。只靠口头约定，项目变大后必然失效。对于确实需要违反规范的特殊界面，应保留说明、测试截图和性能数据。这样后续维护者才能知道这是有意设计，而不是历史遗留问题。

### 实现方案

#### 1. Canvas 分层模板

```text
UIRoot
  Canvas_Static_Background
  Canvas_HUD_Static
  Canvas_HUD_Dynamic
  Canvas_Window_Normal
  Canvas_Window_Popup
  Canvas_ModalMask
  Canvas_TopTip
  Canvas_Guide
```

新增 Canvas 必须说明目的：隔离哪类重建、解决哪类排序、是否增加 Draw Call、是否经过 Profiler 验证。

#### 2. UI 更新规范

```pseudo
onGoldChanged(newGold):
    if currentGold == newGold:
        return
    currentGold = newGold
    goldText.SetText("{0}", currentGold)
```

禁止在 Update 中无条件写 UI。倒计时只显示整数秒时，只在整数秒变化时更新文本。

#### 3. 对象池接口

```csharp
public interface IPoolableUI<TData>
{
    void OnSpawn(TData data, int version);
    void OnDespawn();
}
```

OnDespawn 必须停止 Tween、取消异步、解绑事件、清理图片和恢复默认状态。

#### 4. 固定高度虚拟列表伪代码

```pseudo
visibleCount = ceil(viewportHeight / itemHeight) + buffer
firstIndex = floor(scrollOffset / itemHeight)
for slot in visibleSlots:
    dataIndex = firstIndex + slot
    item = slotItems[slot]
    item.position = calcPosition(dataIndex)
    item.bind(data[dataIndex], version)
contentHeight = dataCount * itemHeight
```

#### 5. 显示隐藏决策表

```text
短时淡入淡出：CanvasGroup alpha + blocksRaycasts
```

#### Canvas 管理补充

下面这些补充项用于处理项目后期最常见的 UI 性能退化问题。它们看似偏流程，但本质上是在约束运行时成本。UGUI 的许多问题并不会在功能刚完成时立刻暴露，而是在资源越来越多、界面越来越复杂、活动频繁热更、低端机覆盖扩大后集中爆发。

##### 1. 动态区域边界

动态区域边界需要有明确的输入、输出和验收口径。输入指哪些资源、Prefab、脚本或运行时路径会触发该项检查；输出指发现问题后应该形成怎样的报告、截图、日志或修复任务；验收口径指修改后要用什么数据证明它真的变好了。没有验收口径的优化只能算经验建议，不能算工业化规范。

在团队执行时，建议把该项拆成开发期检查和发布前检查两层。开发期检查偏快速反馈，例如 Editor 扫描、Prefab Review、资源导入提示；发布前检查偏真实环境，例如真机 Profiler、Memory Profiler 快照、Frame Debugger 截图、低端机回归。这样既不会把日常开发拖得太重，又能在发版前守住性能底线。

Canvas 分层回滚需要有明确的输入、输出和验收口径。输入指哪些资源、Prefab、脚本或运行时路径会触发该项检查；输出指发现问题后应该形成怎样的报告、截图、日志或修复任务；验收口径指修改后要用什么数据证明它真的变好了。没有验收口径的优化只能算经验建议，不能算工业化规范。

在团队执行时，建议把该项拆成开发期检查和发布前检查两层。开发期检查偏快速反馈，例如 Editor 扫描、Prefab Review、资源导入提示；发布前检查偏真实环境，例如真机 Profiler、Memory Profiler 快照、Frame Debugger 截图、低端机回归。这样既不会把日常开发拖得太重，又能在发版前守住性能底线。


### 总结

Canvas 与 UI 元素管理是 UGUI 优化的中枢。资源侧解决“画什么”，Canvas 管理解决“如何组织、何时重建、如何复用”。本文将原稿中比较绝对的说法修正为工程判断：Canvas 分层要按变化频率和功能边界设计，子 Canvas 是隔离工具不是魔法加速器，CanvasGroup 适合短时透明控制但不是完全零成本，对象池必须处理状态和异步生命周期，动态列表应虚拟化。

最终落地时，不要凭感觉拆 Canvas，也不要为了简单把所有 UI 塞进一个巨型 Canvas。用 Profiler 看数据，用 Frame Debugger 看批次，用对象池和虚拟列表控制生命周期，用事件解绑和取消令牌控制引用链，这样 UGUI 的 CPU 峰值和复杂界面卡顿才会真正可控。
