# Unity 几何计算与常用算法：从公式理解到玩法落地

## 摘要

本文系统重写 3D 游戏开发中的几何计算与常用算法，包括向量投影、平面投影、反射、点到线段距离、点到平面距离、点在三角形内判断、AABB 与球体相交、扇形范围检测、插值、贝塞尔曲线、样条曲线、平面与包围盒表示、路径跟随、平面约束、浮点精度与性能优化等内容。文章将数学公式转化为 Unity 工程中可直接使用的思路，强调边界条件、数值稳定性、性能成本和调试方式，避免把几何算法写成只能看懂但不敢用的公式墙。

几何计算是物理系统、AI、技能判定、程序化路径、相机控制、编辑器工具和图形调试的基础。Unity 提供了大量封装 API，但项目中经常会遇到内置组件无法直接覆盖的需求，例如自定义扇形技能、线段范围攻击、角色斜坡移动、曲线路径跟随、空间约束、点击三角形区域、地图区域检测等。本文将原稿中重复和不严谨的部分清理后，重构为一篇更专业、规范、深入、易懂的工业化几何算法手册。

## 正文

### 背景

当开发者掌握了向量、旋转、坐标系和基础物理查询之后，很快会遇到更细的问题：目标是否在技能扇形内？点到路径的最近距离是多少？物体碰到墙之后反射方向怎么计算？角色在斜坡上应该沿哪个方向移动？相机应该如何沿曲线平滑推进？这些问题都不是简单拖一个组件就能完全解决的，它们需要几何计算。

原文已经覆盖了投影、反射、距离、三角形判断、AABB、球体、插值、贝塞尔曲线和平面等内容，但重复段落较多，公式排版不够规范，部分示例缺少退化情况处理。比如点到线段距离需要处理线段长度为零；点在三角形内判断需要考虑点是否在同一平面、三角形是否退化和浮点误差；贝塞尔路径跟随如果直接使用参数 t 匀速推进，并不等价于沿曲线弧长匀速移动。本篇会从工程角度重写这些算法。

### 核心内容


#### 一、几何算法的工程价值

几何算法的价值不在于炫公式，而在于把空间问题转化成稳定、可复用、可测试的代码。技能范围、AI 感知、碰撞修正、路径跟随、镜头约束、程序化生成，都需要数学工具支撑。

推荐把几何算法沉淀成工具库，而不是每次临时写。例如 `DistancePointToSegment`、`IsPointInTriangle`、`ProjectPointOnPlane`、`IsInCone`、`GetCubicBezierPoint`。这样业务代码表达语义，边界条件集中处理，浮点误差策略统一，后续还能写单元测试和性能基准。

#### 二、向量投影：提取一个向量在某方向上的分量

向量投影用于回答：“向量 A 在方向 B 上有多少分量？”如果 B 是单位向量，投影向量为 `dot(A, B) * B`；如果 B 不是单位向量，则为 `dot(A, B) / dot(B, B) * B`。Unity 提供 `Vector3.Project(vector, onNormal)`。

典型用途包括速度分解、斜坡移动、沿墙滑动、判断前后关系。比如角色撞墙后保留沿墙速度：

```csharp
Vector3 intoWall = Vector3.Project(velocity, wallNormal);
Vector3 slideVelocity = velocity - intoWall;
```

这里 wallNormal 必须是合理的表面法线。投影算法本身很简单，难点在于输入向量和法线语义是否正确。

#### 三、平面投影：把向量压到一个平面上

`Vector3.ProjectOnPlane(vector, planeNormal)` 可以把向量投影到由法线定义的平面。它等价于 `vector - Vector3.Project(vector, planeNormal)`。斜坡移动时，玩家输入方向应投影到地面平面，避免角色穿透或浮空：

```csharp
Vector3 moveOnGround = Vector3.ProjectOnPlane(inputDirection, groundNormal).normalized;
```

常见用途包括角色在斜坡上移动、相机在固定平面内移动、2.5D 游戏限制角色行动、目标方向忽略高度差、特效沿表面扩散等。

#### 四、向量反射：入射方向与表面法线的镜面关系

反射公式为 `R = V - 2 * dot(V, N) * N`，其中 V 是入射向量，N 是单位法线。Unity 提供 `Vector3.Reflect`：

```csharp
Vector3 reflected = Vector3.Reflect(velocity, hitNormal);
```

它适合弹球、激光反射、子弹跳弹、碰撞速度修正。能量损失可以乘系数：

```csharp
velocity = Vector3.Reflect(velocity, normal) * bounceFactor;
```

如果需要摩擦，可以分解法向分量和切向分量，分别处理反弹和衰减。

#### 五、点到点距离：平方距离优先用于比较

点到点距离可用 `Vector3.Distance(a,b)`，但它包含开方。如果只比较距离大小，应使用平方距离：

```csharp
float sqrDistance = (a - b).sqrMagnitude;

if (sqrDistance <= radius * radius)
{
    // 在范围内
}
```

平方距离只能用于比较，不能直接替代真实距离参与伤害衰减、速度计算或线性插值。否则曲线会变形，这是很多优化误用的来源。

#### 六、点到线段距离：路径和线形技能的核心算法

给定点 P 和线段 AB，最近点算法是：`AB = B - A`，`AP = P - A`，`t = dot(AP, AB) / dot(AB, AB)`，将 t 限制到 `[0,1]`，最近点为 `A + AB * t`。工程实现必须处理线段退化：

```csharp
public static Vector3 ClosestPointOnSegment(Vector3 p, Vector3 a, Vector3 b)
{
    Vector3 ab = b - a;
    float abSqr = ab.sqrMagnitude;

    if (abSqr < 1e-8f)
        return a;

    float t = Vector3.Dot(p - a, ab) / abSqr;
    t = Mathf.Clamp01(t);

    return a + ab * t;
}
```

用途包括直线型技能命中、AI 到巡逻路径的偏离距离、鼠标选择路径段、绳索和轨道编辑器。

#### 七、点到平面距离与点投影到平面

平面可由一点和法线定义。点 P 到平面的有符号距离为 `dot(P - planePoint, normal)`，要求 normal 是单位向量。正负号表示点在平面哪一侧，绝对值才是几何距离。

```csharp
public static float SignedDistancePointToPlane(Vector3 p, Vector3 planePoint, Vector3 planeNormal)
{
    Vector3 n = planeNormal.normalized;
    return Vector3.Dot(p - planePoint, n);
}
```

点投影到平面：

```csharp
public static Vector3 ProjectPointOnPlane(Vector3 p, Vector3 planePoint, Vector3 planeNormal)
{
    Vector3 n = planeNormal.normalized;
    float d = Vector3.Dot(p - planePoint, n);
    return p - n * d;
}
```

用途包括地面约束、水面上下判断、战斗区域限制和自定义裁剪。

#### 八、射线与平面相交

射线与平面求交常用于无 Collider 地面点击、编辑器平面拖拽、棋盘放置等。Unity 的 `Plane` 已经提供 `Raycast`：

```csharp
Plane plane = new Plane(Vector3.up, Vector3.zero);
Ray ray = camera.ScreenPointToRay(Input.mousePosition);

if (plane.Raycast(ray, out float enter))
{
    Vector3 point = ray.GetPoint(enter);
}
```

如果场景地面有复杂起伏，应使用 Physics.Raycast 或地形高度采样；如果只是固定高度平面，数学平面求交更轻量。

#### 九、点在三角形内部判断：重心坐标更适合扩展

点在三角形内判断常用同向法或重心坐标法。重心坐标法还能用于插值 UV、法线、颜色和权重。实现时要处理三角形退化和浮点误差：

```csharp
public static bool IsPointInTriangle(Vector3 p, Vector3 a, Vector3 b, Vector3 c, float epsilon = 1e-5f)
{
    Vector3 v0 = b - a;
    Vector3 v1 = c - a;
    Vector3 v2 = p - a;

    float d00 = Vector3.Dot(v0, v0);
    float d01 = Vector3.Dot(v0, v1);
    float d11 = Vector3.Dot(v1, v1);
    float d20 = Vector3.Dot(v2, v0);
    float d21 = Vector3.Dot(v2, v1);

    float denom = d00 * d11 - d01 * d01;

    if (Mathf.Abs(denom) < epsilon)
        return false;

    float v = (d11 * d20 - d01 * d21) / denom;
    float w = (d00 * d21 - d01 * d20) / denom;
    float u = 1f - v - w;

    return u >= -epsilon && v >= -epsilon && w >= -epsilon;
}
```

该方法默认点已在三角形所在平面附近。若不在平面，应先投影或改用射线三角形求交。

#### 十、AABB 相交：粗略检测和空间剔除基础

AABB 是轴对齐包围盒，Unity 的 `Bounds` 就是典型表示：

```csharp
if (boundsA.Intersects(boundsB))
{
    // 粗略相交
}
```

AABB 优点是计算便宜、易于合并和扩展，适合粗略剔除、空间分区和预检测。缺点是物体旋转后包围不够紧，对细长或倾斜物体误判较多。工程中常用“先 AABB 粗判，再精确检测”的策略减少开销。

#### 十一、球体相交：范围检测的简洁模型

两个球体相交条件是中心距离小于等于半径和。优化为平方距离：

```csharp
public static bool IntersectsSphere(Vector3 a, float ra, Vector3 b, float rb)
{
    float r = ra + rb;
    return (a - b).sqrMagnitude <= r * r;
}
```

球体适合 AI 感知范围、AoE 粗判、音效影响范围、简单投射物和轻量碰撞近似。它不够精确，但足够便宜，适合做第一层过滤。

#### 十二、扇形范围判断：技能和 AI 视野常用算法

扇形判断通常由距离 + 点积组成：

```csharp
public static bool IsInCone(Vector3 origin, Vector3 forward, Vector3 target, float radius, float halfAngleDeg)
{
    Vector3 toTarget = target - origin;

    if (toTarget.sqrMagnitude > radius * radius)
        return false;

    if (toTarget.sqrMagnitude < 1e-8f)
        return true;

    float cos = Mathf.Cos(halfAngleDeg * Mathf.Deg2Rad);
    return Vector3.Dot(forward.normalized, toTarget.normalized) >= cos;
}
```

优化时可预计算 `cosHalfAngle`，并保证 forward 已归一化。大量目标时先做距离过滤，再做角度判断，最后需要遮挡时再做 Raycast。

#### 十三、Lerp：线性插值与指数趋近的区别

线性插值公式是 `start + (end - start) * t`。`Vector3.Lerp(start, end, t)` 中 t 通常在 `[0,1]`。常见误区是：

```csharp
transform.position = Vector3.Lerp(transform.position, target, Time.deltaTime * speed);
```

这不是固定速度移动，而是指数趋近。适合平滑跟随，不适合规定速度到达。固定速度应使用 `Vector3.MoveTowards`。旋转同理，指数趋近用 `Quaternion.Slerp`，固定角速度用 `Quaternion.RotateTowards`。

#### 十四、贝塞尔曲线：由控制点定义平滑路径

三次贝塞尔曲线公式为 `(1-t)^3P0 + 3(1-t)^2tP1 + 3(1-t)t^2P2 + t^3P3`。实现如下：

```csharp
public static Vector3 CubicBezier(Vector3 p0, Vector3 p1, Vector3 p2, Vector3 p3, float t)
{
    t = Mathf.Clamp01(t);
    float u = 1f - t;

    return u * u * u * p0
         + 3f * u * u * t * p1
         + 3f * u * t * t * p2
         + t * t * t * p3;
}
```

曲线切线可由导数得到，用于路径朝向。注意按 t 匀速推进不等价于按弧长匀速移动；如果要求真实匀速，需要预采样曲线长度表。

#### 十五、Catmull-Rom 样条：经过控制点的平滑路径

贝塞尔曲线不一定经过中间控制点，而 Catmull-Rom 通常经过路径点，适合相机轨迹、AI 巡逻路径和编辑器路径工具。

```csharp
public static Vector3 CatmullRom(Vector3 p0, Vector3 p1, Vector3 p2, Vector3 p3, float t)
{
    float t2 = t * t;
    float t3 = t2 * t;

    return 0.5f * (
        2f * p1 +
        (-p0 + p2) * t +
        (2f * p0 - 5f * p1 + 4f * p2 - p3) * t2 +
        (-p0 + 3f * p1 - 3f * p2 + p3) * t3
    );
}
```

使用时每段通常从 p1 走到 p2，p0 和 p3 提供切线影响。

#### 十六、浮点误差：几何算法必须处理 epsilon

浮点数不是精确实数，几何计算不要直接用 `==` 判断。应使用 epsilon：

```csharp
const float Epsilon = 1e-5f;

if (Mathf.Abs(value) < Epsilon)
{
    // 近似为 0
}
```

常见误差场景包括点是否在平面上、两向量是否平行、点是否在三角形边界、投影参数是否接近端点、路径是否到达终点、法线归一化前长度接近 0。epsilon 大小要结合项目尺度决定。

#### 十七、几何计算性能策略

优化顺序建议是：减少调用次数，先粗后细，使用平方距离避免开方，缓存静态数据，批量处理数据，必要时考虑 Job/Burst，最后再做公式级微优化。不要为了省一个开方把代码写到没人敢维护。几何工具函数应先正确、可测试，再根据 Profiler 数据优化。


### 实现方案


#### 一、建立 GeometryUtil 工具库

建议集中管理几何算法：

```csharp
public static class GeometryUtil
{
    public const float Epsilon = 1e-5f;

    public static bool IsZero(float value)
    {
        return Mathf.Abs(value) < Epsilon;
    }

    public static bool IsZero(Vector3 value)
    {
        return value.sqrMagnitude < Epsilon * Epsilon;
    }
}
```

统一 epsilon 策略可以减少不同系统边界不一致的问题。

#### 二、封装常用距离函数

点到线段平方距离是很常用的基础函数：

```csharp
public static float SqrDistancePointToSegment(Vector3 p, Vector3 a, Vector3 b)
{
    Vector3 ab = b - a;
    float abSqr = ab.sqrMagnitude;

    if (abSqr < Epsilon * Epsilon)
        return (p - a).sqrMagnitude;

    float t = Vector3.Dot(p - a, ab) / abSqr;
    t = Mathf.Clamp01(t);

    Vector3 closest = a + ab * t;
    return (p - closest).sqrMagnitude;
}
```

业务层调用它时语义明确，也更容易统一测试。

#### 三、封装范围判断

常用范围判断建议语义化：

```csharp
public static bool IsInRadius(Vector3 origin, Vector3 target, float radius)
{
    return (target - origin).sqrMagnitude <= radius * radius;
}

public static bool IsInCone(Vector3 origin, Vector3 forwardUnit, Vector3 target, float radius, float cosHalfAngle)
{
    Vector3 toTarget = target - origin;
    float sqrDistance = toTarget.sqrMagnitude;

    if (sqrDistance > radius * radius)
        return false;

    if (sqrDistance < Epsilon * Epsilon)
        return true;

    return Vector3.Dot(forwardUnit, toTarget.normalized) >= cosHalfAngle;
}
```

`forwardUnit` 和 `cosHalfAngle` 尽量在外部预处理，减少高频重复计算。

#### 四、实现曲线采样与可视化

曲线算法必须配合可视化。贝塞尔绘制可以这样做：

```csharp
public static void DrawBezier(Vector3 p0, Vector3 p1, Vector3 p2, Vector3 p3, int segments)
{
    Vector3 prev = CubicBezier(p0, p1, p2, p3, 0f);

    for (int i = 1; i <= segments; i++)
    {
        float t = i / (float)segments;
        Vector3 current = CubicBezier(p0, p1, p2, p3, t);
        Debug.DrawLine(prev, current, Color.magenta);
        prev = current;
    }
}
```

路径、投影、法线、三角形边界都建议画出来。几何问题先画图，别硬猜，真的能少掉很多离谱调试时间。

#### 五、单元测试建议

几何工具函数非常适合单元测试。建议覆盖：点到线段的中间、端点外侧和退化线段；点到平面的正侧、负侧和平面上；点在三角形的内部、边上、顶点、外部和退化三角形；球体相离、相切和相交；扇形判断的正前方、边界角、背后和距离外；贝塞尔曲线 t=0 返回 p0、t=1 返回 p3。

#### 六、性能验收建议

几何算法优化应按真实规模测试：100、1000、10000 个对象分别验证。记录 CPU 耗时、GC Alloc、调用次数、是否使用开方、是否可分帧、是否可批量化、是否需要 Job/Burst。如果只是几十次调用，不要过度优化；如果是每帧数万次，则要从数据布局和批处理层面考虑。


### 总结

几何计算是 3D 游戏开发中连接数学与玩法的关键层。投影、反射、距离、相交、插值、曲线、包围体这些概念看似基础，但组合起来可以支撑角色移动、技能判定、AI 感知、相机路径、编辑器工具、程序化生成和自定义碰撞等大量系统。

本篇的核心建议是：不要把几何算法写成散落在业务代码里的公式片段，而要沉淀为统一工具库、统一误差策略、统一可视化调试和统一测试用例。数学代码越底层，越要讲工程规范。正确性、边界条件、性能和可维护性缺一不可。公式会背只是第一步，能在项目里稳定落地，才是真本事。

#### 工程化补充 1：几何计算与算法 的边界验证

【通用原则】 在正式项目中，几何计算与算法 不能只依赖一次手动测试。建议至少覆盖默认参数、极端参数、空数据、无效引用、父子层级变化、运行时切场景、不同帧率、不同分辨率以及目标平台真机表现。很多数学和物理问题在编辑器里看起来正常，是因为数据规模小、帧率稳定、对象层级简单；一旦进入真实战斗、复杂 UI、多人同屏或者低端设备，隐藏问题就会暴露出来。

【验证原则 1】应同时观察正确性和性能。正确性关注结果是否符合玩法预期，性能关注主线程耗时、GC Alloc、物理查询次数、内存增长和帧峰值。若该模块属于基础设施，还应提供可视化调试开关，让使用者能够直接看到空间方向、检测范围、命中点、法线、包围盒、曲线采样点或数据分区。没有可视化的底层系统，排查成本会随着项目规模指数级上升。

【验证原则 14】应同时观察正确性和性能。正确性关注结果是否符合玩法预期，性能关注主线程耗时、GC Alloc、物理查询次数、内存增长和帧峰值。若该模块属于基础设施，还应提供可视化调试开关，让使用者能够直接看到空间方向、检测范围、命中点、法线、包围盒、曲线采样点或数据分区。没有可视化的底层系统，排查成本会随着项目规模指数级上升。

最后，所有优化都应写明适用范围。比如平方距离适合比较距离大小，却不能直接替代真实距离参与线性衰减；降低检测频率适合 AI 感知，却不适合即时格挡；使用缓存可以减少计算，但必须处理数据失效。把这些约束写进注释和文档，比单纯写一段看似聪明的代码更重要。
