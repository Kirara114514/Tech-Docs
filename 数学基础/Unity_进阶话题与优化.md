# Unity 3D 数学进阶话题与性能优化：从算法到工程治理

## 摘要

本文作为 Unity 3D 数学与物理基础系列的进阶收束篇，聚焦空间数据结构、物理数值稳定性、数学计算优化、批量数据布局、GPU/Job/Burst 方向、程序化生成、AI 数学基础、调试可视化、团队规范和性能验收方法。文章不再重复基础 API，而是把前文的向量、旋转、坐标变换、射线碰撞、几何算法提升到项目级工程治理角度，回答“如何在复杂项目中长期稳定地使用这些数学能力”。

随着项目规模扩大，真正困难的问题往往不是某个公式不会写，而是对象数量暴涨后如何查询，物理步长不稳定后如何避免穿透和抖动，数学优化如何不破坏规则正确性，几何工具如何沉淀为团队基础设施，以及如何通过 Profiler、Gizmos、自动化测试和性能基线证明优化有效。本文会清理原文重复内容，并把高阶话题重写为一篇更正式、专业、工业化、深入且可落地的工程指南。

## 正文

### 背景

完成向量、旋转、坐标系、射线碰撞和几何算法学习后，开发者已经拥有处理大多数 3D 玩法问题的工具箱。但在真实项目中，问题不会停在“怎么求一个距离”或“怎么发一条射线”。当对象数量从几十变成几千，当物理对象从简单盒子变成复杂角色控制器，当 AI 感知从单个目标变成大规模群体，当地图从小关卡变成开放世界，基础算法就需要上升为系统设计。

原文涉及空间数据结构、FixedUpdate、碰撞检测模式、物理材质、数学性能优化、计算着色器、程序化生成和成长建议等内容，但仍然存在重复堆叠、概念浅尝辄止、工程边界不够清楚的问题。本篇重写会把这些内容重新整理为四条主线：数据如何组织，模拟如何稳定，计算如何优化，团队如何沉淀。目标不是让读者一次性掌握所有高阶技术，而是建立正确的判断框架，知道什么时候用什么技术，以及如何验证它真的有收益。

### 核心内容


#### 一、进阶的核心不是更复杂的公式，而是更清晰的系统边界

初学 3D 数学时，重点是理解点积、叉积、四元数、射线、矩阵。进入项目级开发后，重点会变成哪些计算属于规则层，哪些属于表现层，哪些是热点，哪些应该依赖物理引擎，哪些应该自己做数学过滤，哪些需要空间结构加速，哪些逻辑应该放在 FixedUpdate，哪些优化必须用目标设备验证。

这就是工程视角。它不像公式那样有唯一答案，但它决定项目能不能长期稳定运行。

#### 二、空间数据结构：当遍历所有对象不再可接受

如果场景里只有 20 个敌人，遍历全部目标做距离判断完全没问题。如果场景有 5000 个可交互物、2000 个单位、10000 个掉落物，每帧全量遍历就会变成灾难。空间数据结构的目标是减少候选对象数量，把“全场遍历”变成“只查附近”。

常见结构包括 Uniform Grid、Quadtree、Octree、BVH、Spatial Hash、KD-Tree。它们不是越高级越好，而是要根据对象分布、动态程度、查询类型和实现成本选择。

#### 三、统一网格与空间哈希：动态对象的实用选择

统一网格把世界划分为固定大小的格子，每个对象根据位置放入某个格子。查询附近对象时，只需要检查周围几个格子。空间哈希是统一网格的变体，不为所有格子分配数组，而是通过哈希表只存有对象的格子，适合大世界稀疏对象。

适用场景包括 2D/2.5D 地图、大量均匀分布单位、掉落物、弹幕、采集物和服务端 AOI 粗略管理。优点是实现简单、插入删除便宜、查询可预测；缺点是格子大小难选，对象分布极不均匀时效率下降，超大对象跨多个格子需要特殊处理。

#### 四、四叉树与八叉树：递归划分空间

四叉树用于 2D，八叉树用于 3D。它们将空间递归划分为更小区域，当某个区域对象数量超过阈值时继续细分。它们适合对象分布不均匀、大范围空间查询、静态或低频更新对象、场景剔除和编辑器工具。

缺点是实现复杂，动态对象频繁移动会导致更新成本，树退化时需要重建或平衡策略，阈值和最大深度需要调参。工程中，静态物体适合构建树；高频动态单位不一定适合八叉树，空间哈希可能更稳。树结构一定要可视化，否则调试困难。

#### 五、BVH：按物体包围体建立层次

BVH 不固定切空间，而是根据对象包围体构建层次。父节点包围子节点，查询时如果射线或范围不与父节点相交，就能跳过整个子树。BVH 适合光线追踪、大量三角形求交、静态复杂网格、自定义碰撞加速和程序化网格查询。

一般 Unity 玩法开发不一定要手写 BVH，但理解 BVH 有助于理解 MeshCollider、光线追踪、渲染剔除和复杂物理查询的加速思想。

#### 六、空间结构选择原则

选择空间结构可以按以下思路：对象少时直接遍历；对象多但分布均匀时用统一网格或空间哈希；对象多且静态时考虑四叉树、八叉树或 BVH；大量射线对静态几何时 BVH 更合适；动态单位频繁移动时空间哈希优先；开放世界分块通常用网格加分块加载。

不要为了“高级”而上八叉树。空间结构本身有维护成本，如果候选数量不大，直接遍历可能更快、更简单。

#### 七、物理步长与 FixedUpdate：稳定模拟的时间基础

Unity 的物理模拟按固定时间步推进。`FixedUpdate` 与渲染帧率不是一回事。帧率高时，可能多个渲染帧之间没有物理步；帧率低时，一帧内可能执行多次 FixedUpdate 追赶物理时间。

物理相关逻辑建议放在 FixedUpdate，包括给 Rigidbody 加力、设置速度、MovePosition、MoveRotation 和与物理状态强相关的检测。输入采集和表现更新仍然适合 Update：Update 中读取输入，FixedUpdate 中应用到 Rigidbody。

#### 八、fixedDeltaTime 不是越小越好

默认固定步长通常是 0.02 秒，也就是 50Hz。减小 fixedDeltaTime 可以提高物理精度，但会增加 CPU 成本。比如 0.01 秒意味着物理频率变成 100Hz，物理计算量大约翻倍。

普通项目不轻易改全局 fixedDeltaTime。高速物体优先考虑连续碰撞、Raycast 或 SphereCast，而不是全局提高物理频率。局部精度问题应优先用局部方案解决。改动后必须在目标设备做压力测试，并观察低帧率下多次 FixedUpdate 是否造成峰值卡顿。

#### 九、碰撞检测模式：离散、连续与高速物体

高速物体穿透薄墙是离散碰撞的典型问题。Rigidbody 的 Collision Detection Mode 可以缓解，但成本不同。Discrete 便宜，适合大多数物体；Continuous 适合高速动态物体对静态或运动学物体的穿透防护；Continuous Dynamic 更贵，适合高速动态物体之间；某些模式可能带来预测接触问题，必须测试。

高速子弹的最佳方案未必是动态刚体连续碰撞。很多射击游戏会使用 Raycast 或 SphereCast 表示一帧内的弹道段，这更可控，也更便于做穿透、反弹和命中排序。

#### 十、数值稳定性：抖动、穿透和爆炸的根源

物理不稳定常见表现包括物体抖动、角色卡墙、高速穿透、堆叠物体爆炸、关节疯狂甩动、斜坡移动异常、网络同步状态发散。常见原因是直接修改 Transform 移动 Rigidbody、FixedUpdate 和 Update 混用状态、碰撞体太薄或太复杂、质量比例极端、时间步过大、Solver Iterations 不足、非均匀缩放影响碰撞体、物体初始重叠或施加力过大。

处理时先简化碰撞体，确认移动方式正确，降低速度或使用扫掠检测，检查 Layer Collision Matrix，检查质量、阻尼、摩擦、弹性，必要时调 Solver 参数，并使用 Physics Debugger 和 Gizmos 可视化。

#### 十一、数学性能优化：先减少规模，再优化公式

优化顺序非常重要。不要一上来手写近似三角函数，先看调用规模。推荐顺序是：用 Profiler 确认热点；减少调用次数；降低查询候选数量；缓存不变结果；使用平方距离；避免 GC Alloc；批量化数据布局；使用 Job/Burst 或 Compute Shader；最后才考虑公式近似。

比如 AI 找最近敌人，先不要对所有敌人 `Vector3.Distance`。应先通过空间结构得到附近候选，再用平方距离比较，最后必要时才计算真实距离。

#### 十二、GC 优化：数学代码周边也会分配内存

数学本身不一定产生 GC，但周边写法会产生：每帧 new List、RaycastAll 返回数组、LINQ 查询、字符串拼接日志、装箱、闭包、迭代接口集合、临时数组排序。高频数学与物理查询中应复用数组和 List，使用 NonAlloc 查询，避免 LINQ，日志加条件编译，缓存比较器和委托，预分配容量，用 Clear 而不是反复 new。

#### 十三、数据布局：从对象思维到批量思维

当数量上来后，瓶颈常常不是单个公式，而是数据访问方式。MonoBehaviour 分散在内存中，组件查询和对象引用跳转会降低缓存效率。面向批量计算的思路是：从对象上采集必要数据，写入连续数组，批量计算，再写回结果。

例如 AI 感知评分可以把目标位置、阵营、威胁值快照为结构体数组。这样后续更容易迁移到 Job/Burst，也更方便做性能测试。

#### 十四、Unity.Mathematics、Job System 与 Burst

`UnityEngine.Vector3` 适合常规 MonoBehaviour 业务。对于大规模数学计算，`Unity.Mathematics` 提供了 `float3`、`quaternion`、`float4x4` 和 `math` 函数，更接近 SIMD 和 Shader 风格，也更适合 Burst 编译。

适用场景包括大量点距离计算、批量可见性测试、程序化网格生成、大量弹幕轨迹、粒子或单位行为评分、空间结构更新、自定义物理近似。但不要为了高级把所有代码都改成 Job。Job/Burst 有数据转换成本、调度成本和调试成本，只有批量规模足够大且数据能整理成 NativeArray 等连续结构时，收益才明显。

#### 十五、Compute Shader：GPU 大规模并行计算

Compute Shader 适合 GPU 上的大规模并行任务，例如粒子模拟、布料近似、群体行为、图像处理、体素或地形生成、大量无强分支的并行计算。但它不适合所有数学计算，因为存在数据传输成本、平台兼容性、调试难度、与 CPU 玩法逻辑同步复杂等问题。

选择 Compute Shader 前要问：数据规模是否足够大，是否可以并行，是否需要频繁读回 CPU，目标平台是否支持，调试工具是否具备，是否已有更简单的 Job/Burst 方案。

#### 十六、程序化生成：数学作为内容生产工具

程序化生成依赖大量数学，包括噪声函数、细胞自动机、L-System、Voronoi、Delaunay、随机采样等。工程要点是随机数种子必须可控，生成过程应可复现，参数要可视化调试，离线生成与运行时生成要分开评估，大规模生成需要分块和异步，生成结果要能序列化或缓存，规则层随机与表现层随机要区分。

#### 十七、AI 与数学：从距离判断到决策评分

AI 系统大量使用数学：距离用于接近和威胁评估，点积用于视野角度，射线用于视线遮挡，曲线用于路径平滑，图论用于寻路，概率用于行为选择，评分函数用于目标选择和技能释放，空间结构用于快速找附近目标。

AI 评分必须可调试。建议在编辑器或调试面板中显示每项分数，否则策划只会感觉“AI 很怪”，程序也很难定位。

#### 十八、可视化调试：高阶系统必须可见

越高阶的数学系统，越需要可视化。应可视化空间网格、八叉树节点、AI 视野、查询半径、射线命中、路径曲线、法线和切线、速度和加速度、包围盒、生成区域、物理接触点。

调试可视化不应只是临时 Debug.DrawLine，而应成为工具：支持开关，支持只显示选中对象，支持颜色区分，支持数值标签，支持运行时开关，必要时支持录制关键帧。

#### 十九、数学优化不能破坏规则正确性

优化有适用范围。平方距离适合比较，但不适合代替真实距离进入线性衰减公式。近似角度适合表现，但不适合严格命中判定。降低检测频率适合 AI 感知，但不适合即时格挡判定。

错误示例：

```csharp
float damage = baseDamage - sqrDistance * falloff;
```

如果设计需要线性距离衰减，应先得到真实距离：

```csharp
float distance = Mathf.Sqrt(sqrDistance);
float damage = baseDamage - distance * falloff;
```

如果要避免开方，就需要重新设计衰减曲线，而不是偷偷替换。

#### 二十、性能优化必须有基线

没有基线的数据不叫优化，叫玄学。每次优化前后应记录测试场景、对象数量、平台设备、平均帧耗时、峰值帧耗时、GC Alloc、CPU/GPU 瓶颈、内存变化、误差范围和玩法影响。

优化文档至少说明问题、方案、结果和限制。比如 AI 感知每帧遍历 3000 个目标，改为空间哈希加分帧检测后，主线程耗时从 4.8ms 降到 0.7ms，但感知最多有 0.2 秒延迟。这才是工程化优化闭环。

#### 二十一、团队数学基础设施

成熟团队可以建立 MathUtil、GeometryUtil、PhysicsQueryService、SpaceConvertUtil、DebugDrawSystem、BenchmarkScene、数学单元测试、编码规范和 Cookbook。这样常见算法不再依赖个人发挥，而是团队共享能力。

别让每个程序都手写一遍扇形检测、线段距离和 UI 跟随。重复造轮子还造得半圆不圆，真的会把项目折磨成恐怖游戏，程序员限定版。


### 实现方案


#### 一、空间哈希落地方案

空间哈希适合动态对象附近查询。核心是把世界位置映射到格子坐标：

```csharp
Vector2Int GetCell(Vector3 position)
{
    return new Vector2Int(
        Mathf.FloorToInt(position.x / cellSize),
        Mathf.FloorToInt(position.z / cellSize));
}
```

实现时要维护格子到对象列表、对象到当前格子的反向索引。对象移动后更新格子，查询半径换算成格子范围，返回候选集后仍用平方距离精确过滤。

#### 二、物理移动规范

Rigidbody 控制建议采用输入缓存 + FixedUpdate 应用：

```csharp
public sealed class RigidbodyMotor : MonoBehaviour
{
    [SerializeField] private Rigidbody rb;
    [SerializeField] private float speed = 5f;

    private Vector3 desiredMove;

    public void SetInput(Vector3 move)
    {
        desiredMove = Vector3.ClampMagnitude(move, 1f);
    }

    private void FixedUpdate()
    {
        Vector3 delta = desiredMove * speed * Time.fixedDeltaTime;
        rb.MovePosition(rb.position + delta);
    }
}
```

规则是输入在 Update 采集，物理位移在 FixedUpdate 应用，不直接修改动态 Rigidbody 的 Transform，高速位移使用扫掠检测，表现平滑通过插值或渲染层处理。

#### 三、数学工具库目录建议

建议目录：

```text
Runtime/
  Math/
    MathConst.cs
    FloatCompare.cs
    GeometryUtil.cs
    CurveUtil.cs
    SpaceConvertUtil.cs
    PhysicsQueryUtil.cs
  Debug/
    DebugDraw.cs
    GeometryGizmos.cs
  Tests/
    GeometryUtilTests.cs
    CurveUtilTests.cs
```

每个工具函数都应说明输入空间、输出空间、是否允许零向量、是否返回平方距离、epsilon 策略、典型用途和不适用场景。

#### 四、性能基准场景

建立 BenchmarkScene，例如 RangeQuery、Raycast、DistanceCheck、CurveSampling、AIScoring、SpatialHash。每个场景提供对象数量配置：100、1000、5000、10000、50000。输出耗时、GC Alloc、命中数量、查询次数、峰值和平均值。性能测试必须能在目标设备运行，不能只看编辑器。

#### 五、优化决策流程

推荐流程：

```text
发现问题
 -> Profiler 确认热点
 -> 判断是算法规模、数据布局、GC、物理查询还是渲染问题
 -> 设计最小优化方案
 -> 验证正确性
 -> 压测性能
 -> 记录结果和限制
 -> 沉淀工具或规范
```

没有 Profiler 数据时不要先优化。感觉某段代码“应该很慢”并不可靠，现代硬件和 Unity 内部优化经常会打脸。嗯，小雅也被打过，但我不承认很多次。

#### 六、代码评审清单

进阶数学和物理代码提交时检查：是否有明确空间语义；是否区分规则层和表现层；是否使用平方距离但没有误用到真实距离公式；是否处理零向量和退化几何；是否有 epsilon 策略；是否产生 GC；是否可视化调试；是否有单元测试或测试场景；是否在目标规模验证；是否需要空间结构；是否真的需要 Job/Burst/Compute；是否写明适用范围和限制。

#### 七、团队落地路线

初期团队先做 GeometryUtil、SpaceConvertUtil、PhysicsQueryService、DebugDraw 和常见算法文档。中期补空间哈希、AI 感知批处理、曲线工具、性能基准场景和自动化测试。后期再考虑 Job/Burst 批量化、Compute Shader、自定义空间树、程序化生成工具链和运行时性能监控。路线不要反过来。没有基础工具和测试，就急着上 Compute Shader，通常会得到一坨很难调的高科技麻烦。


### 总结

进阶数学与优化的核心，不是掌握更多听起来高级的名词，而是能在复杂项目中做正确选择。对象少时直接遍历就够；对象多时考虑空间结构；物理不稳定时先检查时间步、移动方式和碰撞体；数学热点出现时先减少规模，再考虑平方距离、缓存、批量化、Job/Burst 或 Compute Shader；优化前后必须有数据验证。

本篇将系列知识从单点技术提升到工程治理。真正成熟的 Unity 开发者，不只是会写公式，还要知道公式属于哪个系统、运行在哪个生命周期、会不会分配内存、在目标设备上是否稳定、能不能被调试、能不能被团队复用。数学基础设施一旦沉淀下来，项目中的空间问题、物理问题和性能问题都会更可控。
