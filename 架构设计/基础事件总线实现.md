# 基础事件总线实现

## 摘要
基础事件总线的意义，从来不只是“把消息发出去”。真正值得作为架构基础设施的事件总线，必须同时解决模块解耦、事件语义清晰、生命周期安全、发布过程稳定和后续可扩展性这几个核心问题。许多团队第一次写事件总线时都能在几分钟内做出一个能跑的版本，但真正进入项目后，很快会遇到重复订阅、忘记取消、遍历过程中修改监听表、事件对象语义混乱、消息泛滥和调试困难等问题。本文以 Unity 项目为语境，从架构设计视角系统说明基础事件总线为什么需要存在、基础实现必须解决哪些问题、事件如何建模、订阅和发布应遵守哪些规则，以及一个“可作为起点的基础版本”应具备哪些最小能力。

## 正文

### 背景
随着项目规模增长，模块之间的通信会越来越频繁。最初，一个按钮点击可能只需要调用一个页面方法；后来，一个购买动作可能同时需要通知 UI、音频、埋点、奖励系统、红点系统、引导系统和资源刷新逻辑；再往后，一个角色死亡事件可能需要驱动状态机切换、特效播放、掉落生成、统计记录、网络同步和战斗日志。若这些关系都通过直接引用和同步调用实现，代码很快会演变成复杂的网状结构：发布者必须知道所有接收者，新增功能必须回头修改旧逻辑，模块边界越来越脆弱。

事件总线正是在这种背景下被广泛引入。它试图把“发生了什么”和“谁对此感兴趣”拆开。发布者只负责表达一个事实或通知，不需要知道有哪些消费者；订阅者根据自己的职责决定是否响应。看起来这是一个简单的抽象，但它对项目的影响很深。它不仅减少直接依赖，还会改变团队组织行为的方式：很多扩展型需求不再需要回头修改原始业务逻辑，而只需增加新的订阅者；很多过去依赖巨型控制器或单例的联动逻辑，可以被拆回各自负责的模块。

不过，事件总线也非常容易被误用。一些团队把它当成“万能解耦工具”，结果任何逻辑都通过消息广播完成，系统变得更加不可见；另一些团队实现了一个简单字典加回调列表的版本，但没有处理生命周期、安全性和调试能力，最终在中型项目里就迅速暴露出内存泄漏、顺序不明和异常难查的问题。因此，真正值得讨论的并不是“事件总线能不能写出来”，而是“一个基础事件总线至少应该具备哪些工程品质，才能成为可持续使用的底座”。

### 核心内容
#### 一、事件总线要解决的首要问题是“通知解耦”，而不是“流程替代”
理解基础事件总线的第一前提，是明确它适合解决哪类问题。事件总线最适合表达“某件事发生了，感兴趣的系统可以自行响应”。例如购买成功、角色受击、场景加载完成、配置更新、任务状态变化等，都属于典型事件语义。它不适合取代所有流程控制，也不适合承担强顺序、强返回值依赖或复杂事务一致性的逻辑。换句话说，事件总线解决的是通知传播问题，而不是完整业务流程编排问题。

这个边界非常重要。很多团队的问题并不是事件总线太弱，而是把它用到了不合适的地方。比如请求-响应型逻辑被硬塞成广播消息，最后谁先处理、谁回写状态、谁负责失败回退都不清楚。基础事件总线若没有明确适用边界，很快就会从“解耦工具”变成“流程迷雾”。

#### 二、事件的核心是语义，而不是参数列表
一个成熟的事件系统不会把事件理解成“若干参数凑在一起”。事件应当被视为一类具备明确语义的消息对象。`PlayerDiedEvent` 与 `InventoryChangedEvent` 的重要性，不在于字段多寡，而在于它们准确表达了某个事实发生了什么、发生在何时、订阅者可以根据这些信息作出怎样的反应。若事件只是一堆临时参数，例如 `Publish("shop_buy", id, count, price)`，那么事件语义会严重依赖调用方与订阅方的口头默契，长期极难维护。

事件对象的设计应尽量做到三点。第一，名称表达事实，而不是命令。例如“购买已完成”比“刷新商城”更适合作为事件。第二，字段只携带订阅者理解该事实所需的最小必要信息，不要把发布者内部状态整包外露。第三，事件结构要考虑未来扩展，不因新增一个字段就被迫修改所有监听者的行为模型。

#### 三、基础事件总线必须首先解决生命周期安全
基础版本里最常被忽略的，是生命周期问题。Unity 项目中大量对象会被启用、禁用、销毁、复用、跨场景保留或进入对象池。如果某个对象订阅事件后没有正确取消订阅，那么后续常见问题至少有三类：无效对象仍被调用、对象因委托引用无法回收、业务逻辑在已离开场景的对象上继续触发。很多团队一开始觉得事件总线写起来很轻松，真正用一段时间后却认为它“不稳定”，原因往往正是在生命周期管理没有被纳入基础设计。

因此，一个真正能作为起点的基础事件总线，必须让订阅和取消订阅成为显式且可管理的动作。团队也必须明确：哪些系统是常驻订阅者，哪些是临时订阅者，订阅应放在 `OnEnable` 还是初始化阶段，取消应放在 `OnDisable`、`OnDestroy` 还是显式释放流程里。若这些问题只停留在开发者经验层面，而没有进入规范与模板，生命周期问题一定会重复出现。

#### 四、基础实现还必须处理“遍历时修改”的稳定性
一个非常经典的基础事件总线 bug，是在发布事件时，某个监听器又取消了自己的订阅、增加了新的订阅，或者触发了另一个会影响监听表的逻辑。这会导致原始监听列表在遍历过程中被修改，引发异常或造成同一轮分发行为不确定。很多初学版事件总线都忽略了这个问题，因为在最简单的演示里看不出来，但项目一复杂就会频繁踩到。

因此，基础实现至少要保证当前分发使用的是稳定快照，而不是直接遍历原始容器。这样即便监听表在回调过程中发生变化，也只影响后续分发，而不会破坏当前轮事件传播的完整性。这个设计看似基础，却是基础事件总线能否进入真实项目的重要分水岭。

#### 五、事件总线的价值不仅是解耦，还在于扩展弹性
事件总线的一个常被低估的好处，是让系统更容易承接“附加型需求”。例如某个原本只有购买逻辑的页面，后续需要增加埋点、音效、红点刷新、限时活动提醒和新手引导埋点。如果没有事件总线，这些逻辑往往只能不断回头修改原始购买流程代码；有了事件总线，许多附加能力可以通过新增监听器来接入，而不必破坏原始业务主路径。

这种扩展弹性对长期项目非常有价值，因为项目真正复杂起来之后，新增需求很少是完全独立的功能，更多是对既有链路附加新的响应动作。事件总线若设计得当，可以显著降低“老代码不断被新需求侵入”的概率。

#### 六、但基础事件总线绝不等于“什么都发事件”
许多团队在吃到解耦甜头之后，容易走向另一个极端：任何模块通信都走事件总线，连同步请求、即时返回值依赖和短路径函数调用也想通过消息统一。最终系统会变成“看似没有直接依赖，实际上谁都不知道谁在监听谁”。这种过度消息化会让可读性、调试效率和行为确定性都下降。

因此，基础事件总线文档里必须明确写出“不适用场景”。例如：只在局部模块内部发生、调用链极短且无需扩展的逻辑，直接调用通常更清晰；需要明确返回值和事务控制的流程，不适合用广播式事件；强顺序依赖流程也不应完全依赖松散消息完成。越是基础设施，越需要同时说明“能做什么”和“不要拿它做什么”。

#### 七、基础版本就应为后续扩展留出空间
虽然这里讨论的是基础事件总线，但基础版本若完全不考虑未来扩展，后期就很容易推倒重来。基础实现至少应在结构上为以下能力留出可能性：一次性订阅、优先级、调试追踪、订阅统计、弱引用监听、高频与低频通道分层、异常隔离和性能分析。基础版本不一定立刻实现全部，但数据结构和接口风格最好不要一开始就把这些方向彻底堵死。

例如若基础版本统一把所有事件都做成字符串键和无类型参数列表，后续再想做类型安全、编译期检查和高频优化就会非常痛苦。相反，如果从一开始就采用类型化事件设计，后续无论是做高性能分发还是更好的调试可视化，成本都会低很多。

### 实现方案
#### 一、推荐一个可作为起点的最小实现
基础事件总线的目标不是一次做到最强，而是先保证事件语义、生命周期管理和分发稳定性成立。以下示例展示一个适合项目起步的最小版本：

```csharp
using System;
using System.Collections.Generic;

public sealed class EventBus
{
    public static EventBus Instance { get; } = new EventBus();

    private readonly Dictionary<Type, List<Delegate>> _listeners = new Dictionary<Type, List<Delegate>>();

    private EventBus() { }

    public void Subscribe<T>(Action<T> handler)
    {
        var eventType = typeof(T);
        if (!_listeners.TryGetValue(eventType, out var list))
        {
            list = new List<Delegate>();
            _listeners[eventType] = list;
        }

        if (!list.Contains(handler))
        {
            list.Add(handler);
        }
    }

    public void Unsubscribe<T>(Action<T> handler)
    {
        var eventType = typeof(T);
        if (_listeners.TryGetValue(eventType, out var list))
        {
            list.Remove(handler);
            if (list.Count == 0)
            {
                _listeners.Remove(eventType);
            }
        }
    }

    public void Publish<T>(T eventData)
    {
        var eventType = typeof(T);
        if (!_listeners.TryGetValue(eventType, out var list))
        {
            return;
        }

        var snapshot = new List<Delegate>(list);
        foreach (var listener in snapshot)
        {
            ((Action<T>)listener).Invoke(eventData);
        }
    }
}
```

这段代码并不复杂，但它具备几个关键点：事件按类型组织、订阅是显式行为、重复订阅可控、发布时使用快照保证稳定性。对许多项目而言，这已经足以作为基础版本起步。

#### 二、配套规范比代码本身更重要
基础事件总线真正落地时，代码只是其中一部分，配套规范更重要。团队至少应明确：事件命名规范、事件对象字段设计原则、哪些场景允许走事件总线、临时对象的订阅/取消模板、评审中如何检查事件边界、事件是否允许跨模块传播。没有这些配套，哪怕代码实现本身不错，也很容易在不同开发者的使用习惯中逐步失控。

一个实用做法是配一份事件总线使用清单。例如：事件必须命名成事实；不得在事件对象中塞无关上下文；页面类订阅默认在 `OnEnable`，取消在 `OnDisable`；广播型事件不得承担返回值逻辑；高频事件须经过专项评估。规范一旦形成，基础设施才算真正进入团队实践。

#### 三、在基础版本中加入最小调试信息
虽然完整调试面板可以后续再做，但基础版本非常建议至少保留最小调试钩子。例如开发环境下能够输出：某类事件当前监听者数量、某次发布是否无监听者、某个事件是否被异常高频触发。哪怕只是内部调试日志，也会比完全黑盒好得多。事件系统一旦完全不可见，后续排查“为什么这个功能没反应”或“为什么同一个动作被触发多次”会非常痛苦。

#### 四、事件使用边界的典型判断方法
为了避免团队过度依赖事件总线，可以提供一套判断问题是否适合走事件的快速标准：

- 如果这是“某个事实发生后可能有多个系统感兴趣”的场景，适合事件。
- 如果这是“必须同步返回结果才能继续”的场景，不适合事件。
- 如果逻辑只在局部对象之间发生且没有扩展需求，直接调用更清晰。
- 如果未来大概率会新增附加响应逻辑，事件总线更有价值。

这种判断方法越早写进团队共识，事件总线就越不容易被误用。

#### 五、基础版本后的自然演进路径
当基础版本稳定后，可以逐步考虑以下演进方向：加入一次性订阅能力，减少某些临时监听模板代码；增加优先级或阶段调度能力，应对少量顺序依赖场景；加入统计面板，帮助观测事件热点；为高频事件设计更轻量的数据通道；对异常处理做隔离策略，防止单个监听器破坏整轮分发。关键是按真实问题推进，而不是一开始就做成过度复杂的全能消息中心。

#### 六、典型反模式
基础事件总线落地时，有几类反模式特别常见。第一，用字符串和 object 数组拼装事件，导致类型安全彻底丢失。第二，匿名订阅随手写，取消订阅时找不到原句柄。第三，把所有跨模块调用都改成事件，造成系统黑箱化。第四，事件对象承载过量上下文，变成谁都顺手拿一点的信息包。第五，没有调试钩子，出了问题只能靠猜。基础版本越早警惕这些反模式，后续维护成本越低。

#### 七、事件对象设计还应考虑“谁来拥有事件生命周期”
虽然基础版本通常不会引入复杂对象池，但事件对象本身仍然存在生命周期设计问题。对于低频事件，直接创建轻量事件对象往往足够；对于可能逐步升高频率的事件，则应至少在文档中记录其当前频率预期和未来升级路径。这样团队后续在性能治理时，能较快判断某类事件是否需要进入更轻量或更高性能的通道，而不会在发现问题时才意识到该事件早已被大量使用。

#### 八、基础事件总线也需要最小的组织约束
很多团队写完一个 `EventBus` 类就默认事件架构已经完成，实际上这还远远不够。基础设施真正落地的标志，是团队已经形成最小组织约束：事件命名规则、订阅模板、哪些模块允许对外广播全局事件、哪些事件属于局部模块事件、谁负责在评审中检查事件边界。这些约束即使只是一页短文档，也比完全靠口头共识稳定得多。

#### 九、案例拆解：为什么基础事件总线最容易在“方便”中失控
一个常见过程是这样的：团队先为了解耦引入事件总线，初期只用于少量全局通知，效果很好；随后更多开发者发现它很方便，于是开始用它替代直接调用、替代局部状态同步、替代模块接口，甚至把某些业务流程节点也都做成广播事件。几个月后，系统里虽然直接依赖减少了，但谁在监听谁、事件顺序为何如此、某次功能为什么会连带触发多个系统，反而越来越难看清。

这说明基础事件总线最大风险不是“不够强”，而是“太方便”。也正因为如此，基础版本更需要清楚写明边界和用法，避免便利性本身反过来制造新复杂度。

#### 十、验收标准：基础版本什么时候算可以进入项目共用
一个基础事件总线若想进入团队共用，至少应满足以下条件：类型化事件设计成立；重复订阅与取消订阅规则明确；发布时监听表修改不会破坏当前分发；基础调试信息可用；团队对使用边界达成最小共识；至少有一到两个真实模块试点并完成复盘。只有满足这些条件，基础事件总线才算从“示例代码”迈入“可共用基础设施”。

#### 十一、基础版本也要考虑“事件治理”而不只是“事件分发”
很多团队写事件总线时只关注技术分发层，忽略了治理层。但对基础设施而言，治理往往决定它能活多久。所谓事件治理，至少包括：哪些事件是全局事件、哪些是模块局部事件；是否允许跨模块直接订阅；事件命名是否有统一前缀或语义规范；哪个层次的代码有权发布哪些级别的事件；评审中如何判断一个事件是否真的有存在必要。没有这些治理约束，事件总线很快会因为使用范围无限扩张而失控。

也就是说，一个基础版本真正能进入项目共用，不仅因为它能发消息，更因为它为后续组织治理留下了足够稳定的抓手。

#### 十二、案例拆解：为什么很多团队的事件总线最终变成“广播式单例依赖”
一种非常常见的情况是：团队最初为了减少直接引用引入事件总线，结果后续所有模块都把它当成默认通信方式，久而久之谁都不再设计清晰接口，而是习惯于“先发个事件看看有没有人接”。表面上直接依赖少了，实际上系统依赖只是从显式调用迁移到了隐式广播。最终任何功能异常都要先猜是谁没发事件、谁多发了事件、谁在监听链上做了额外事情。

这说明基础事件总线如果没有边界和治理规则，很容易退化成另一种形式的全局单例依赖。它减少了对象引用，但没有减少系统复杂度。

#### 十三、案例拆解：为什么“事件总线很好用”往往也是它最危险的时候
当团队第一次吃到事件总线的甜头时，最容易发生的事就是过度扩张。因为开发者会发现：加个监听器就能接入新功能，太方便了。于是越来越多事情开始走广播：页面初始化、局部刷新、按钮回调、模块请求、错误通知、缓存更新、配置读取，甚至局部对象之间的简单沟通也想走总线。短期看扩展速度快了，长期看调试与理解成本却迅速飙升。

真正成熟的团队，会把“方便”视为需要被约束的能力，而不是任其扩散的理由。基础事件总线之所以要有规范，就是为了防止这种成功后的失控。

#### 十四、从基础版本到稳态使用的关键动作
如果团队已经做出一个基础版本，接下来最关键的不是继续加功能，而是做几件小而重要的事情：选两个真实模块试点；在评审模板里加入事件边界检查；沉淀一页使用规则；记录几类高频反模式；建立最小调试输出。只要这些动作完成，基础事件总线就会从“某位程序员写的工具类”转变为“团队可以共同依赖的基础设施”。

#### 十五、基础事件总线的长期价值，取决于它能否帮助团队形成统一语言
如果一个团队在几个月后已经能自然地说出“这是全局事件，不该由页面直接广播”“这个事件语义像命令，不该这么命名”“这个订阅生命周期有问题，应跟对象启停绑定”，那说明事件总线真正产生了组织价值。反之，若几年后团队仍然只是“有个 EventBus 工具类”，却对事件边界、命名和订阅规则没有共同语言，那么基础设施本身其实并没有真正扎根。基础事件总线最终能否长期稳定，不只取决于代码设计，更取决于它是否帮助团队建立了共享的架构判断标准。

### 总结
基础事件总线实现的真正目标，不是写出一个“看起来很架构”的消息分发器，而是为项目建立一套语义清晰、生命周期安全、分发稳定且可持续扩展的通知基础设施。它要解决的是通知解耦问题，而不是取代所有流程控制；它强调事件是事实语义，而不是临时参数列表；它要求订阅和取消订阅成为显式且可管理的动作；它还必须从一开始就为后续调试、扩展和规则治理留出空间。只有在这些前提都满足时，事件总线才会真正成为长期项目中的稳定底座，而不是新的复杂度来源。
