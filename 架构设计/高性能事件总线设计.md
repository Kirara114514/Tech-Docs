### 第三篇：精益求精：如何打造高性能、易维护的事件总线

上一篇我们成功实现了一个基础的事件总线。它具备了核心功能，能够帮助我们解耦代码。但是，一个“可用”的工具和“生产级”的工具之间，往往隔着性能优化和健壮性设计。在这一篇，我们将深入探讨如何解决可能遇到的性能瓶颈和维护问题，让我们的事件总线从“可用”变为“生产级”。

#### 3.1 性能优化：GC Allocations 的克星

在上一篇中，我们特意将事件数据类型约束为 `struct`（值类型），以避免 GC Allocations（垃圾回收分配）。但如果你的事件需要传递更复杂的数据，比如 `string` 或 `List`，或者你就是习惯用 `class` 来封装事件，那么你将不可避免地遇到 GC 问题。

**问题剖析：为什么频繁发布 `class` 事件会导致 GC？**

在 C# 中，`class` 是**引用类型（Reference Type）**。每次我们通过 `new` 关键字创建一个 `class` 实例时，都会在**托管堆（Managed Heap）**上分配一块内存。当这个对象不再被引用时，垃圾回收器（Garbage Collector, GC）就会介入，回收这块内存。

如果你的游戏每秒发布几十甚至上百个事件（比如玩家移动、UI 动画更新等），每次都 `new` 一个 `class` 实例，那么就会在短时间内产生大量的内存分配。这会频繁地触发 GC，而 GC 是一个耗时操作。在 GC 运行时，你的游戏主线程可能会被暂停，从而导致卡顿（“Hiccup”或“Spike”）。

**解决方案：使用对象池（Object Pool）**

为了解决这个问题，我们可以采用**对象池（Object Pool）**模式。它的核心思想很简单：**提前创建好一批事件对象，而不是在需要时才创建。当需要发布事件时，从池中“借用”一个；事件发布完成后，再将其“归还”给池子，以供下次使用。**

我们来设计一个简易的对象池。

C#

```
// 一个简单的泛型对象池
public class ObjectPool<T> where T : new()
{
    private readonly Stack<T> _pool = new Stack<T>();

    public T Get()
    {
        // 如果池子有对象，就直接返回
        if (_pool.Count > 0)
        {
            return _pool.Pop();
        }
        // 否则就创建一个新的
        return new T();
    }

    public void Return(T obj)
    {
        // 将对象归还到池子中
        _pool.Push(obj);
    }
}

```

现在，我们改造一下事件总线，让它能够使用对象池。

C#

```
// 改造后的 EventBus，支持 class 事件
public sealed class EventBus
{
    public static EventBus Instance { get; } = new EventBus();

    // 存储 class 事件订阅者
    private readonly Dictionary<Type, List<Delegate>> _classEventDictionary = new Dictionary<Type, List<Delegate>>();

    // 存储每个 class 事件类型的对象池
    private readonly Dictionary<Type, object> _objectPools = new Dictionary<Type, object>();

    // 订阅 class 事件
    public void Subscribe<T>(Action<T> action) where T : class
    {
        Type eventType = typeof(T);
        if (!_classEventDictionary.ContainsKey(eventType))
        {
            _classEventDictionary[eventType] = new List<Delegate>();
            // 为新的 class 事件类型创建对象池
            _objectPools[eventType] = new ObjectPool<T>();
        }
        _classEventDictionary[eventType].Add(action);
    }

    // 取消订阅 class 事件
    public void Unsubscribe<T>(Action<T> action) where T : class
    {
        Type eventType = typeof(T);
        if (_classEventDictionary.ContainsKey(eventType))
        {
            _classEventDictionary[eventType].Remove(action);
        }
    }

    // 发布 class 事件
    public void Publish<T>(T eventData) where T : class
    {
        Type eventType = typeof(T);
        if (_classEventDictionary.ContainsKey(eventType))
        {
            // 遍历并调用订阅者
            foreach (var actionDelegate in new List<Delegate>(_classEventDictionary[eventType]))
            {
                (actionDelegate as Action<T>)?.Invoke(eventData);
            }
            
            // 重要：将事件对象归还给对象池
            // 这里我们假设 publish 方法里拿到的是从池子里借出的对象
            if (_objectPools.ContainsKey(eventType))
            {
                (_objectPools[eventType] as ObjectPool<T>)?.Return(eventData);
            }
        }
    }

    // 获取一个可用的事件对象
    public T GetPooledEvent<T>() where T : class, new()
    {
        Type eventType = typeof(T);
        if (!_objectPools.ContainsKey(eventType))
        {
            _objectPools[eventType] = new ObjectPool<T>();
        }
        return (_objectPools[eventType] as ObjectPool<T>)?.Get();
    }
}

```

**使用对象池发布事件**

现在，发布事件的流程稍微改变了：

1.  从事件总线中**获取**一个事件对象。
    
2.  **填充**事件对象的数据。
    
3.  **发布**事件。
    
4.  事件总线在发布完成后，**自动将事件对象归还**给池子。
    

C#

```
// 玩家攻击事件的 class 版本
public class PlayerAttackClassEvent
{
    public string attackerName;
    public int damage;
}

// 攻击按钮的脚本
public class AttackButtonWithPool : MonoBehaviour
{
    public void OnClick()
    {
        // 1. 从事件总线中获取一个事件对象
        PlayerAttackClassEvent eventData = EventBus.Instance.GetPooledEvent<PlayerAttackClassEvent>();
        
        // 2. 填充数据
        eventData.attackerName = "玩家B";
        eventData.damage = 10;

        // 3. 发布事件
        EventBus.Instance.Publish(eventData);
    }
}

```

通过这种方式，我们可以在不产生任何 GC Allocations 的前提下，高频率地发布 `class` 类型的事件。

#### 3.2 生命周期管理：告别内存泄漏

在第二篇我们强调了在 `OnDisable()` 或 `OnDestroy()` 中取消订阅的重要性。但这只是基础，在实际项目中，我们还需要一些更智能的机制来避免内存泄漏。

**问题剖析：为什么忘记取消订阅会导致内存泄漏？**

想象一下，`Monster` 脚本订阅了事件，但玩家在游戏中途离开了这个场景，`Monster` `GameObject` 被 `Destroy` 了。如果它忘记在 `OnDestroy()` 中调用 `Unsubscribe`，那么 `EventBus` 的字典中仍然保留着一个指向这个 `Monster` 实例方法的委托。

-   **结果**：当事件再次发生时，事件总线会尝试调用一个已经不存在的对象上的方法。虽然 C# 的 `?.Invoke()` 会防止程序崩溃，但这个`Monster` 实例本身因为还被`EventBus`中的委托所引用，无法被 GC 回收。这就会导致内存泄漏。
    

**解决方案：一次性订阅和自动管理**

1.  实现 SubscribeOnce() 方法：
    
    有时候我们只需要监听事件发生一次，比如 UI 上的一个弹出窗口，只在“任务完成”事件第一次发生时弹出。我们可以实现一个 SubscribeOnce() 方法，让订阅者在事件首次触发后自动取消订阅。
    
    C#
    
    ```
    public void SubscribeOnce<T>(Action<T> action) where T : class
    {
        // 封装一个匿名方法
        Action<T> wrappedAction = null;
        wrappedAction = (eventData) =>
        {
            action.Invoke(eventData); // 执行用户的方法
            Unsubscribe(wrappedAction); // 自动取消订阅
        };
        Subscribe(wrappedAction);
    }
    
    ```
    
    > **注意**：这里的 `wrappedAction` 是一个临时的委托，它包含了原始的 `action`。在 `Unsubscribe` 时，我们必须取消的是 `wrappedAction`，而不是原始的 `action`。
    
2.  利用弱引用（Weak Reference）：
    
    这是一种更高级的、可以自动防止内存泄漏的方案。我们可以在订阅时，不直接存储订阅者对象的强引用，而是存储一个弱引用（WeakReference）。当订阅者对象不再被其他地方引用时，GC 就可以正常回收它，即使事件总线中还保留着这个弱引用。在发布事件时，我们再检查这个弱引用是否还指向一个有效的对象。
    
    C#
    
    ```
    // 概念演示，实际实现会更复杂
    public void SubscribeWithWeakReference<T>(Action<T> action)
    {
        // 存储一个弱引用
        // ...
    }
    
    ```
    
    这种方式实现起来比较复杂，容易出错，但它能从根本上解决“忘记取消订阅”导致的内存泄漏问题。对于初学者来说，**严格遵循“订阅-取消订阅”的约定，并使用 `SubscribeOnce()` 这样的辅助方法，是更实际且推荐的做法。**
    

#### 3.3 调试和扩展：让事件总线更智能

一个好的工具不仅要功能强大，还要易于调试和扩展。

1.  可观测性：可视化事件流
    
    在开发过程中，我们经常需要知道：
    
    -   哪个对象发布了什么事件？
        
    -   这个事件有多少个订阅者？
        
    -   某个订阅者是否成功接收到事件？
        
        我们可以在 Publish() 方法中，加入一些 Debug.Log()，来打印这些关键信息，比如：
        
    
    C#
    
    ```
    public void Publish<T>(T eventData) where T : class
    {
        Type eventType = typeof(T);
        if (_classEventDictionary.ContainsKey(eventType))
        {
            var listeners = _classEventDictionary[eventType];
            Debug.Log($"<color=cyan>[EventBus]</color> Publishing event '{eventType.Name}' to {listeners.Count} listeners.");
    
            // ...遍历和调用
        }
    }
    
    ```
    
    你甚至可以更进一步，在 Unity Editor 中编写一个**自定义窗口**，实时显示事件总线中的所有事件类型和订阅者数量，让事件流变得可视化。
    
2.  优先级：控制事件响应顺序
    
    在某些复杂场景中，你可能需要让特定的订阅者优先于其他订阅者响应事件。比如，在一个“玩家掉血”事件中，你希望先更新血条 UI，再触发“玩家受伤”动画。
    
    我们可以通过在 Subscribe 方法中加入一个 int priority 参数来实现：
    
    C#
    
    ```
    public void Subscribe<T>(Action<T> action, int priority = 0)
    {
        // 订阅者列表不再是简单的 List<Delegate>，
        // 而是 Dictionary<int, List<Delegate>>，key 是优先级
        // ...
    }
    
    ```
    
    在 `Publish()` 时，我们先按优先级从高到低排序，然后再依次执行订阅者的委托。
