# C#到C++语言迁移要点

## 摘要
Unity 程序员从 C# 迁移到 Unreal Engine 的 C++，最容易掉进的误区，就是把这件事理解成“语法替换练习”。好像只要把 `class` 写法、泛型、属性、委托、协程、GC、访问修饰符和 API 习惯换一套，语言迁移就完成了。实际上，真正困难的部分根本不在语法，而在于：UE 的 C++ 并不是一门孤立的“纯 C++”，它是被 UObject 反射系统、宏元数据、序列化规则、蓝图暴露、模块系统、构建链路、资产体系和运行时生命周期共同塑形过的一种“带框架语义的工程语言”。

这意味着，Unity 程序员迁移时面对的不是“从托管语言到原生语言”这么简单，而是同时面对三层变化：

- 从 C# 语言习惯切换到现代 C++ 语言习惯。
- 从 Unity 的脚本运行模型切换到 UE 的对象反射和模块模型。
- 从“写业务逻辑”切换到“定义框架边界、暴露扩展点、服务编辑器工作流”。

本文会把这些变化逐层拆开，重点讨论哪些 C# 习惯在 UE C++ 中必须放弃，哪些经验仍然有价值，哪些看似只是语法差异、实则会引发架构偏差，哪些地方是迁移中最容易造成长期维护灾难的深坑。目标不是让读者把 C# 写法机械翻成 C++，而是让读者知道：在 UE 中写 C++，到底是在做什么。

## 正文

### 背景
很多 Unity 程序员第一次接触 UE C++ 时，会产生一种双重错觉。

第一种错觉是“其实没那么难”。因为从表面看，一个带几个属性和函数的类确实很快就能写出来，编辑器里也能跑起来。你会觉得：“无非就是写法繁琐一点，语法严格一点，本质还是写类、写逻辑、调接口。”

第二种错觉是“其实特别难”。因为当你刚开始看到 `UCLASS`、`UPROPERTY`、`UFUNCTION`、`GENERATED_BODY()`、`TArray`、`TSubclassOf`、软引用、Build.cs、模块、反射和蓝图可见性这些东西同时出现时，又会觉得这根本不是单纯的 C++，而是一大堆奇怪语法和宏拼起来的引擎方言。

这两种错觉都只看到了局部。

说“没那么难”的人，低估了迁移的工程边界；说“特别难”的人，则没抓到真正的主线。真正的主线是：UE C++ 的难点不是“语言本身难到无法学习”，而是你必须一边学语言，一边接受它如何被引擎框架塑造成一套服务编辑器、资产、反射、多人和构建的工程工具。

Unity 时代的 C# 经验并不是没用。你仍然会从中受益，例如：

- 对面向对象和组件化的理解。
- 对抽象边界和模块拆分的意识。
- 对事件、配置、资源、生命周期的经验。
- 对调试、测试、性能和团队协作的工程习惯。

但这些经验不能直接原封不动带过去。因为在 UE 里，“一个类如何被创建、如何暴露给编辑器、如何被蓝图继承、如何参与反射、如何参与打包和序列化”本身就是设计的一部分。你如果还按 Unity 中“先把逻辑写通、后面再看怎么接编辑器和资源”的习惯推进，后面通常都会被迫返工。

### 核心内容

#### 一、语言迁移的真正对象，不只是 C++，而是“UE 风格的 C++”
很多 C# 程序员会先去学一套通用 C++ 教程，然后带着“我已经会 C++ 了”的心态再进入 UE。这样当然比完全零基础好，但很容易产生认知偏差，因为 UE C++ 的日常工作内容并不等同于写一堆标准库算法题或纯原生项目代码。

UE C++ 的几个核心特征必须先建立：

##### 1. 它深度依赖反射系统
普通 C++ 类不会天然参与编辑器属性面板、蓝图暴露、序列化、垃圾回收和网络复制。UE 中，只有纳入 UObject 系列与反射系统的类型，才会真正进入这条工作流。

这意味着，写 UE C++ 不是“先写纯类，必要时再暴露给引擎”，而常常是“从一开始就要决定这个类是否属于反射与资产生态的一部分”。

##### 2. 它是模块化和构建系统绑定的
Unity 程序员很容易把“写代码文件”与“引擎会自动发现”联系起来。但在 UE 中，代码所在模块、依赖声明、Build.cs 配置、导出宏、插件边界，都会影响类是否能被正确编译、链接和使用。

所以，UE C++ 的一项基本能力不是“会写类”，而是“知道这个类应属于哪个模块，以及它对外暴露的边界是什么”。

##### 3. 它服务于编辑器和内容工作流
在 Unity 中，脚本当然也会服务于 Inspector、Prefab 和资源系统，但很多时候你仍然可以先把逻辑写成纯代码，再逐步挂接到编辑器。UE 中这种顺序常常更危险，因为很多系统一开始就要考虑：

- 哪些属性需要编辑器可见。
- 哪些函数需要蓝图调用。
- 哪些类要允许 Blueprint 继承。
- 哪些资产要让设计师参与配置。

如果你忽略这些，往往不是“少一点方便性”，而是整个系统边界就错了。

#### 二、C# 到 C++ 迁移中最容易踩的第一层坑：把语法差异误当成核心难点
语法当然有门槛，但它不是迁移中最致命的部分。真正危险的是某些 C# 习惯会在 C++ 中悄悄诱导你写出高风险代码。

##### 1. 垃圾回收习惯
Unity 程序员习惯了托管对象和 GC 世界，因此很容易低估对象所有权和生命周期的明确性需求。到了 UE C++，你会同时面对几种不同世界：

- 标准 C++ 对象。
- 栈对象和局部变量。
- 智能指针管理的非 UObject 对象。
- UObject 体系下受引擎管理的对象。
- 软引用、弱引用和资产路径。

如果你没有明确区分这些对象类别，就很容易在“谁负责持有、谁负责释放、谁会被 GC、谁只是引用一个资产路径”上频繁犯错。

UE 不是“没有 GC”，而是“GC 发生在特定对象体系中，而且需要你遵守该体系的约束”。这比 Unity 那种相对统一的托管对象世界更复杂，也更需要边界意识。

##### 2. 容器和集合习惯
C# 程序员很自然会想找 `List<T>`、`Dictionary<TKey, TValue>`、`HashSet<T>`、`IEnumerable<T>`、LINQ 对应物，然后用差不多的思维继续写逻辑。

在 UE 中，你当然会接触 `TArray`、`TMap`、`TSet` 等容器，但如果你仍然带着“反正写法差不多”的心态，就会忽略：

- 它们与反射系统、序列化和蓝图暴露的关系。
- 它们在性能、拷贝、移动、引用稳定性上的行为差异。
- 它们在热路径中的分配和访问成本。
- 某些方便写法是否会带来不必要复制。

语言迁移在这里不是“会不会声明容器”，而是“你是否知道容器行为会如何影响引擎工作流和性能”。

##### 3. 委托、事件与回调习惯
C# 的委托和事件模型在 Unity 项目里非常常见，程序员往往已经形成比较成熟的事件流设计习惯。迁移到 UE 后，这份经验仍然有价值，但不能直接照搬，因为：

- UE 的 Delegate 体系有多种形态，是否动态、是否反射可见、是否蓝图可绑定、是否序列化可见，都会影响选择。
- 有些在 C# 中理所当然的高层事件流，在 UE 中可能更适合通过 Gameplay Framework、组件事件、蓝图实现事件或资产驱动方式表达。
- 如果你不考虑编辑器工作流和反射边界，就会错误地选用事件机制，最终让设计师、蓝图或序列化系统无法接入。

##### 4. 协程与异步习惯
Unity 程序员往往很依赖协程或者 Task 风格异步逻辑。到 UE 后，如果仍然想着“我要找一个等价协程机制”，往往会把问题想歪。

迁移时更应该问的是：

- 这段逻辑究竟是时序流程问题、状态机问题，还是异步资源问题？
- 它应该由 Tick、Timer、Latent Action、Async Task、行为树、动画事件还是状态驱动来表达？
- 这段流程是否需要进入 Blueprint 工作流？

也就是说，异步迁移的关键不是“找协程替代”，而是重新理解时序表达的系统边界。

#### 三、第二层坑：把 C++ 当成“业务实现语言”，而不是“框架边界语言”
这可能是迁移过程中最容易长期伤害项目质量的一点。

在 Unity 里，C# 既是业务实现语言，也是工程组织语言。很多业务需求都可以直接通过脚本快速落地。UE 中，C++ 当然也能写业务，但它更重要的职责通常是：

- 定义稳定的系统边界。
- 提供可被 Blueprint 和编辑器安全消费的接口。
- 为资产、配置和表现层提供可靠骨架。
- 承担性能敏感和长期维护的核心逻辑。

如果你把 UE C++ 理解成“我原来写在 C# 里的业务代码，现在换个语法继续写”，你很快就会遇到两个问题。

##### 1. 一切都写得能跑，但没有边界
例如技能系统、交互系统、角色状态系统、任务系统都可以写出来，但：

- 哪些函数应暴露给蓝图不清楚。
- 哪些配置应进入资产不清楚。
- 哪些能力应由组件承载，哪些应由 Subsystem 承载不清楚。
- 哪些类是给设计师配的，哪些是给程序扩展的不清楚。

结果就是项目越做越多，框架却越来越薄。

##### 2. 你会下意识排斥蓝图和资产参与
因为在 Unity 阶段，很多程序员已经习惯“逻辑都在脚本里最安全”。如果不主动重建认知，就会下意识把所有关键逻辑都锁死在 C++ 中，最终让内容团队、设计师和技术策划失去可参与入口。短期看这似乎可控，长期看却会严重降低团队产能。

所以，语言迁移真正需要学会的，是“哪些东西应该用 C++ 固化，哪些东西应该通过 C++ 建边界后交给蓝图或资产扩展”。

#### 四、第三层坑：忽略 UObject 世界与普通 C++ 世界的差异
很多迁移问题看起来像语言 bug，实则是世界观混淆。

在 UE 中，普通 C++ 与 UObject 世界不是同一个层面。二者的差异会体现在：

- 是否参与反射。
- 是否能在编辑器中可见。
- 是否可被蓝图引用。
- 是否可被 GC 管理。
- 是否能被序列化。
- 是否能被网络复制系统理解。

如果你没有明确区分“这个类型只是纯 C++ 辅助结构”还是“这个类型必须成为引擎生态的一部分”，就很容易出现这些问题：

- 设计明明要让蓝图配，但你定义成了无法暴露的纯类型。
- 逻辑明明是运行时 UObject，却被你当作普通对象管理。
- 性能敏感辅助结构明明不该进入反射，却被你错误纳入 UObject 体系。
- 某些引用关系你以为引擎会帮你管理，其实根本不会。

迁移时，最重要的判断之一就是：这个东西应该属于哪一层世界。

#### 五、C# 经验中真正值得带过去的部分
虽然很多习惯要改，但并不意味着 Unity 时代的经验都得丢掉。相反，有些经验在 UE C++ 中依然非常有价值。

##### 1. 抽象与边界意识
成熟 Unity 程序员通常已经习惯处理模块边界、依赖方向、配置驱动、事件流、状态同步。这些经验在 UE 中依然重要，只是落位方式不同。

##### 2. 可维护性优先的工程习惯
会写清晰命名、会拆小函数、会控制副作用、会做日志与排障、会考虑后续接手成本，这些能力不会因为换了语言而失效。

##### 3. 内容驱动意识
如果你已经习惯把参数和配置从代码中抽出来，那么迁移到 UE 后，只需要把这种意识进一步升级为“资产驱动工作流意识”，而不是从零开始。

##### 4. 性能敏感与数据结构意识
有过 Unity 性能治理经验的程序员，通常已经知道热路径、GC、容器选择和数据局部性的重要性。到 UE 后，这些意识仍然有效，只是要换成更适合 C++ 和引擎运行模型的表达方式。

### 实现方案

#### 一、建议的学习顺序
如果你准备系统迁移 C# 到 UE C++，不建议一上来就啃整本现代 C++ 大全，也不建议直接在真实项目里边写边猜。更合理的顺序是：

##### 第一步：掌握与 UE 日常最相关的 C++ 子集
先把真正会高频用到的部分学会：

- 类、继承、虚函数
- 引用、指针、const 语义
- RAII 基本思想
- 拷贝、移动与值语义基础
- 模板的使用层面理解
- 标准类型与 UE 类型的常见差异

目标不是成为纯 C++ 专家，而是先让你具备读懂 UE 代码和写出安全基础类的能力。

##### 第二步：把 UObject 反射体系单独当一个专题学
这一部分必须独立学习，而不是顺手理解：

- `UCLASS`、`USTRUCT`、`UENUM`
- `UPROPERTY`、`UFUNCTION`
- `GENERATED_BODY`
- 蓝图暴露、编辑器可见性、序列化与网络标签

很多人以为这些只是“宏写法”，实际上它们决定了你的类型是否真正进入 UE 生态。

##### 第三步：用小样例理解边界，而不是上来写业务
建议先做几个最小样例：

- 一个可编辑属性的组件。
- 一个 DataAsset 配置 + C++ 消费的例子。
- 一个 BlueprintImplementableEvent 扩展点。
- 一个带简单容器和引用关系的运行时模块。

这些样例比直接写业务更能帮你看清“UE C++ 到底在服务什么”。

##### 第四步：再进入真实项目专题
此时再去处理输入、资源、动画、UI、网络等专题，语言和边界才能真正合上。

#### 二、推荐的实践准则
为了避免 C# 惯性持续污染 UE C++ 设计，建议至少坚持下面这些实践准则。

##### 1. 先判断类型归属，再写代码
任何新类型出现前先问：

- 它是不是 UObject 生态的一部分？
- 它是否需要反射、序列化、编辑器可见、蓝图可见？
- 它是否只是纯 C++ 辅助结构？

这个判断比语法正确更重要。

##### 2. 先判断边界，再决定写在 C++ 还是 Blueprint
先问：

- 这段逻辑是否长期稳定？
- 是否性能敏感？
- 是否需要设计师可编辑？
- 是否需要在编辑器或蓝图中扩展？

只有这些问题答完，才值得决定语言或落位。

##### 3. 先定义扩展点，再写业务细节
好的 UE C++ 不只是“把业务做出来”，而是先想清楚：

- 哪些属性要给编辑器。
- 哪些函数要给蓝图。
- 哪些数据要给资产。
- 哪些地方只允许程序接触。

这决定了系统是否能长期协作。

##### 4. 不要在迁移初期过度模仿 C# API 风格
很多人会下意识写出“特别像 C#”的 UE C++ 接口，例如过度依赖自定义高层包装、隐藏引擎语义、用自己熟悉的命名和架构模式替代引擎原生表达。这样短期舒服，长期成本极高，因为后来者看不懂引擎原生意图，也难以对照官方资料排查问题。

#### 三、示例：一个更合理的语言迁移落点
下面这个例子演示的不是复杂功能，而是“如何用 C++ 定义边界、用反射暴露能力、为 Blueprint 留可控扩展口”。

```cpp
UCLASS(ClassGroup=(Inventory), meta=(BlueprintSpawnableComponent))
class UInventoryComponent : public UActorComponent
{
    GENERATED_BODY()

public:
    UPROPERTY(EditAnywhere, BlueprintReadOnly, Category="Inventory")
    int32 MaxSlotCount = 32;

    UFUNCTION(BlueprintCallable, Category="Inventory")
    bool TryAddItem(int32 ItemId, int32 Count);

    UFUNCTION(BlueprintImplementableEvent, Category="Inventory")
    void OnItemAdded(int32 ItemId, int32 Count);
};
```

这个例子里真正要看的，不是 `int32`、宏或者函数声明本身，而是几个迁移原则：

- 组件是稳定能力入口，而不是临时脚本容器。
- 属性有限暴露给编辑器和蓝图，不是全部裸奔。
- C++ 定义核心边界，Blueprint 只承接表现和扩展点。
- 命名和函数职责强调“业务意图”，不是强调“模仿 C# 写法”。

这就是 UE C++ 与 Unity C# 在工程语义上的差别。

#### 四、验收标准
语言迁移是否到位，不应以“会不会写出一个能编译的类”为标准，而应从以下几个层面判断：

- 是否能区分普通 C++ 类型与 UObject 生态类型。
- 是否能解释 `UPROPERTY` 和 `UFUNCTION` 背后的工作流意义，而不是只会背修饰词。
- 是否能判断什么该用 C++ 固化，什么该留给 Blueprint 或 DataAsset。
- 是否能在不强行模仿 C# API 风格的前提下，写出符合 UE 习惯的边界代码。
- 是否能在阅读 UE 官方示例与源码时理解其设计意图，而不是只盯着语法细节。

如果这些做不到，就说明语言迁移还停留在“换了语法皮”的阶段。

### 总结
C# 到 C++ 的迁移，真正难的从来不是大括号、指针、模板或者宏，而是承认一件事：你现在写的不是一门孤立语言，而是一种深度嵌入引擎生态、服务框架边界与编辑器工作流的工程表达方式。

如果只把迁移理解成“把 C# 改写成 C++”，你会不断被语法、内存和宏系统绊住；如果把迁移理解成“学会用 UE C++ 定义稳定边界、暴露扩展点、承接资产和蓝图工作流”，那很多原本混乱的点就会连成主线。

真正成熟的迁移结果，不是“一个 Unity 程序员勉强会写 UE C++”，而是“一个 UE 项目的核心骨架可以被他稳定地定义、解释、扩展和维护”。这才是语言迁移真正该达到的标准。
