# 渲染管线与图形技术迁移要点

## 摘要
渲染管线从 Unity 迁移到 Unreal Engine 的难点，不在于找到材质、后处理、光照、阴影或特效功能的对应入口，而在于重新理解 UE 的渲染架构、默认画面假设、资源组织方式、平台画质策略和性能分析方法。Unity Built-in、URP、HDRP、Shader Graph、SRP Batcher、Volume、Lightmap、Reflection Probe 与自定义 Render Feature 的经验，不能简单映射到 UE 的 Deferred/Forward Renderer、Material System、Material Instance、Post Process Volume、Lumen、Nanite、Virtual Shadow Maps、Reflection、Lightmass/GPU Lightmass、Scalability、Render Dependency Graph、Niagara 和平台渲染设置中。

本文围绕 UE 迁移中的图形技术体系展开，重点说明如何建立视觉目标、如何拆分材质与后处理职责、如何选择光照方案、如何治理 Shader 与材质复杂度、如何处理透明和特效、如何制定平台画质档位、如何建立渲染调试和性能验收流程。迁移目标不是把 Unity 里每个效果“做出类似版本”，而是让视觉目标、内容生产、渲染成本和平台发布形成稳定闭环，避免项目长期停留在“效果能做出来，但系统不稳、性能不可控”的状态。

## 正文

### 背景
Unity 团队通常熟悉 Built-in、URP 或 HDRP 的渲染假设。Built-in 强调传统材质、Forward/Deferred 选择和大量项目自定义；URP 强调跨平台、移动端友好和 Scriptable Renderer Feature；HDRP 强调高端画质、物理光照和复杂后处理。团队可能已经积累了自己的 Shader Graph 库、后处理栈、光照烘焙流程、材质规范、图集策略和性能经验。

UE 的默认渲染哲学不同。UE 在桌面和主机项目中常以高质量 PBR、Deferred、Lumen、Nanite、Virtual Shadow Maps、Post Process Volume 和强材质编辑器为基础；移动端和低端平台又需要单独考虑 Forward、移动渲染路径、纹理压缩、光照简化和 Scalability。材质、光照、后处理、反射、阴影、特效和资源流送之间的耦合更强，很多图形功能不是一个开关，而是一组内容生产、平台设置和性能预算的组合。

迁移时如果只按旧问题找新按钮，容易形成表面对照：Unity 的 Shader 换成 UE Material，Post Processing Volume 换成 Post Process Volume，Reflection Probe 换成 Reflection Capture，Lightmap 换成 Lightmass，Particle System 换成 Niagara。这样的对照可以帮助入门，但不足以支撑量产。真正需要迁移的是画面目标定义、资产制作规范、渲染路径选择、性能指标、调试流程和跨平台降级策略。

### 核心内容

#### 一、先定义视觉目标和平台边界
渲染迁移的第一步不是调材质节点，而是定义视觉目标。项目希望达到写实、半写实、卡通、低多边形、二次元、电影化、移动端轻量还是竞技清晰风格，不同目标决定材质复杂度、光照方案、后处理强度、特效策略、阴影质量和平台预算。没有视觉目标，团队会在每个功能上追求“UE 能做到的最高效果”，最终性能和风格都失控。

视觉目标必须和平台边界一起定义。PC 高端、主机、移动端、VR、云游戏和低端设备对分辨率、帧率、内存、显存和渲染特性要求不同。Lumen、Nanite、Virtual Shadow Maps、体积雾、高质量透明、复杂后处理和大量动态光源在不同平台上的可用性和成本差异很大。迁移文档应明确每个平台的目标帧率、默认画质档位、最低支持设备和禁用特性。

建议建立视觉基准场景，包括室内、室外、白天、夜晚、角色近景、战斗峰值、UI 叠加、透明特效、反射材质和大场景视距。所有渲染决策都在这些场景中验证，而不是只在单个模型预览窗口中判断。

#### 二、材质系统迁移要从 Shader 逻辑转向材质资产治理
Unity 团队可能习惯将 Shader 视为渲染效果的核心。UE 中材质编辑器同样强大，但量产项目更强调材质函数、主材质、材质实例和参数治理。迁移时不应为每个旧 Shader 都重写一个独立 UE 材质，而应先提炼材质族：角色皮肤、角色布料、金属、石头、植被、水体、UI 特效、透明溶解、武器、地形、建筑、特效材质等。

每个材质族应有主材质和实例策略。主材质承担稳定结构和功能开关，材质实例承担颜色、贴图、粗糙度、金属度、法线、遮罩和少量变体。不要把所有可能功能塞进一个超级主材质，也不要为每个资产复制一套几乎相同的材质。超级材质会导致 Shader 变体和指令成本上升，过度分散又会让维护困难。

材质迁移还要关注 PBR 参数一致性。Unity 旧项目中的贴图通道、颜色空间、金属度/光滑度、法线格式、遮罩贴图和自定义 BRDF 可能与 UE 默认材质模型不同。技术美术应建立转换规范：Smoothness 如何转 Roughness，Metallic/Roughness/AO 如何打包，法线是否翻转绿色通道，透明和双面如何处理，Subsurface、Clear Coat、Hair、Cloth 等模型何时使用。

#### 三、后处理应服务画面风格，不应补救所有资源问题
Unity 项目常通过后处理调整色调、Bloom、景深、运动模糊、色差、曝光、环境光遮蔽和抗锯齿。UE 的 Post Process Volume 也非常强大，但迁移时要明确后处理的职责。后处理应统一画面风格、镜头语言和整体观感，而不应成为修补材质、光照和资源问题的万能层。

曝光和色彩管理是迁移重点。UE 的自动曝光、Tonemapper、ACES 风格、Color Grading、LUT 和后处理体积会显著影响画面。Unity 旧项目如果使用自定义色彩曲线或后处理栈，迁移时应建立对照样片，在相同光照和材质条件下调整。不要在每个关卡里临时改不同后处理参数，否则项目风格会碎片化。

后处理还要有性能预算。Bloom、DOF、Motion Blur、SSR、AO、体积雾、局部后处理材质和高质量抗锯齿都会影响 GPU。平台画质档位应定义哪些后处理可用、强度范围、是否随动态分辨率调整。后处理资源也应有 owner，核心全局 Volume 不应被关卡随意覆盖。

#### 四、光照方案要按项目类型选择
UE 提供静态光照、Stationary Light、Movable Light、Lightmass、GPU Lightmass、Lumen、反射捕捉、天空光、虚拟阴影等多种能力。迁移时不能简单把 Unity 的 Baked GI、Realtime GI、Light Probe 或 Reflection Probe 对应到某个 UE 功能。应根据项目类型选择光照路线。

线性关卡、室内场景和移动端项目可能更适合烘焙光照或混合光照，以获得稳定性能；开放世界、高动态时间、破坏和高端平台项目可能选择 Lumen 或更多动态光照；竞技项目可能为了清晰度和性能限制复杂光照；二次元或卡通项目可能使用更可控的定制光照和后处理。不同路线会影响关卡制作、材质、动态物体、阴影、反射和包体。

光照迁移还要建立验收标准。美术需要知道光照强度、色温、曝光、阴影软硬、反射、间接光和 Lightmap 密度如何控制；程序和 TA 需要知道每类光源数量、阴影策略和性能成本。不要让每个场景独立摸索，否则画面一致性和性能都会失控。

#### 五、Lumen、Nanite 和 Virtual Shadow Maps 不能无条件启用
UE 现代图形特性很有吸引力，但迁移团队必须按项目和平台评估。Lumen 提供动态全局光照和反射，但有成本和内容要求；Nanite 提供高密度几何渲染能力，但对材质、透明、变形和平台有边界；Virtual Shadow Maps 可提升阴影质量，但也有内存和 GPU 成本。启用这些特性不是“升级画质”的简单开关，而是生产方式变化。

Nanite 适合大量静态高面数几何，但不等于所有模型都应该 Nanite。植被、透明物、骨骼网格、世界位置偏移、特殊材质和移动端平台需要单独评估。Lumen 需要控制场景尺度、材质反射、光源数量和性能预算。Virtual Shadow Maps 需要关注阴影投射对象数量、缓存失效、动态物体和画质档位。

迁移时建议建立特性开关矩阵：每个平台是否启用 Lumen、Nanite、VSM、Virtual Texture、TSR、硬件光追、体积雾；不同画质档位如何降级；资源制作有哪些限制。没有矩阵时，美术可能按最高特性制作内容，最终低端平台无法承载。

#### 六、透明、特效和 UI 渲染要专项治理
透明渲染在大多数引擎中都是性能和排序难点，UE 也不例外。粒子、溶解、半透明材质、玻璃、水、雾、屏幕空间特效和 UI 叠加可能造成 overdraw、排序问题、深度冲突和 GPU 成本。Unity 中某些透明 Shader 或粒子效果迁移到 UE 后，成本结构可能完全不同。

Niagara 特效应与材质和性能预算联动。CPU 粒子、GPU 粒子、碰撞、光照、透明层数、材质复杂度、同屏数量和 LOD 都要有规范。特效美术不能只在空场景预览效果，必须在战斗峰值和目标平台验证。高频技能、环境特效和 UI 特效应有不同预算。

UI 渲染也会影响整体画面。UMG、材质 UI、Retainer、3D Widget、后处理 UI 和透明叠加要与渲染预算协同。不要把大量复杂材质和实时模糊放进常驻 UI。UI 视觉目标应有自己的材质和后处理边界。

#### 七、Shader 复杂度和变体管理是长期成本
Unity 迁移中，很多旧 Shader 功能会被转换成 UE 材质节点、材质函数和静态开关。若没有治理，材质复杂度和 Shader 编译时间会快速上升。Shader 成本不仅影响运行时 GPU，也影响编辑器迭代、Cook、构建时间和包体。

材质规范应限制高成本节点、透明复杂度、动态分支、纹理采样数、世界位置偏移、像素深度偏移、复杂数学和过多静态开关。材质函数应复用，但要避免隐藏成本。静态开关会生成变体，不能为了所有可能需求无限增加。

团队应定期使用 Shader Complexity、Material Analyzer、GPU Visualizer 和构建报告检查。对公共主材质建立 owner，新增功能必须说明成本和用途。对于移动端或低端平台，应有简化材质路线，而不是依赖同一套高端材质自动降级。

#### 八、资源 Streaming 与渲染质量紧密相关
渲染质量不仅取决于材质和光照，也取决于资源何时加载、以什么精度加载。纹理 Streaming、Virtual Texture、Nanite Streaming、关卡 Streaming、HLOD、World Partition、LOD 和 Mip 策略都会影响画面清晰度、内存和卡顿。迁移团队如果只关注单帧效果，可能忽略移动和切场景时的质量波动。

纹理组、Mip Bias、LOD Group、Streaming Pool、Virtual Texture 使用范围和平台压缩格式应统一设计。角色近景、UI、地形、远景、特效和活动资源应有不同策略。过高纹理规格会爆显存，过低又会造成模糊。Streaming 问题要在真实移动路径中验证，而不是静态截图。

HLOD 和 World Partition 对开放世界或大场景迁移尤其重要。它们不仅是关卡技术，也会影响模型制作、材质合并、光照、碰撞和包体。迁移早期应建立大场景样例，验证视距、加载、卡顿和内存。

#### 九、图形调试需要工具链而不是主观截图
渲染问题必须可测量、可复现。UE 提供 View Modes、Buffer Visualization、Shader Complexity、Quad Overdraw、Light Complexity、Nanite Visualization、Lumen Visualization、Virtual Shadow Map Debug、GPU Visualizer、RenderDoc 和平台 GPU 工具。迁移团队应建立工具使用流程，明确每类问题看什么视图。

例如材质过重看 Shader Complexity 和 GPU Pass，透明过重看 Quad Overdraw，光源过多看 Light Complexity，Nanite 问题看 Nanite Visualization，Lumen 漏光看 Lumen Scene 和 Surface Cache，阴影问题看 VSM 调试，反射问题看 Buffer 和 Reflection 视图。测试反馈“画面不对”时，应附带场景、相机、画质档位、后处理 Volume、光照配置和截图。

图形调试还应区分编辑器和运行包。编辑器视口可能启用调试、未 Cook 资源和不同 Scalability；正式包才代表目标平台。所有最终结论应在目标平台构建中确认。

#### 十、跨平台 Scalability 是渲染迁移必需品
UE Scalability 系统可以控制分辨率、视距、阴影、后处理、纹理、特效、全局光照、反射和抗锯齿等质量档位。迁移项目如果没有清晰档位，低端平台适配会变成后期手工删效果。应从项目早期定义 Low、Medium、High、Epic 或自定义档位的特性差异。

Scalability 不应只改控制台变量，还要配合内容规范。低档位需要简化材质、特效 LOD、低分辨率纹理、短视距、简化阴影和替代后处理；高档位可以启用更高质量光照和反射。某些资源可能需要平台专用版本。美术和 TA 应知道不同档位会如何影响最终画面。

动态分辨率、TSR、FSR、DLSS 或平台超分策略也应纳入画质规划。它们可以帮助性能，但不能替代基础优化。分辨率策略要考虑 UI 清晰度、透明特效、后处理和输入延迟。

#### 十一、美术工作流要和 UE 渲染假设一致
渲染迁移不是程序独自完成。美术资产制作必须适配 UE 的材质、光照、LOD、Nanite、纹理压缩、碰撞、HLOD 和平台规则。旧 Unity 资源可能需要调整贴图通道、材质参数、模型面数、Lightmap UV、Pivot、LOD、法线和源文件规范。

技术美术应建立材质样例库、光照样例关卡、后处理基线、贴图打包规范、LOD 规范、主材质使用说明和性能预算。美术提交资源时应通过校验工具检查命名、贴图尺寸、材质复杂度、LOD、Nanite 标记、虚拟纹理和平台限制。

工作流顺畅比单个效果更重要。如果每个效果都需要程序临时支持，每个材质都由不同人复制修改，项目后期会失去风格一致性和性能控制。UE 的优势在于强工具链，但前提是团队建立统一规范。

#### 十二、渲染迁移要与构建、资源和性能系统联动
渲染内容最终会进入 Cook、包体、内存、显存和补丁。材质变体、Shader 编译、纹理规格、Nanite 数据、虚拟纹理、光照数据、HLOD、反射捕捉和后处理资源都会影响构建和发布。图形技术不能只看编辑器效果。

构建系统应报告 Shader 编译时间、Shader 数量、纹理体积、光照数据体积、Nanite 数据、HLOD 体积和平台裁剪。资源管线应能追踪大纹理、复杂材质和异常引用。性能系统应能在基准场景中监控 GPU Pass 和显存。发布系统应知道哪些高清资源可下载，哪些低端平台需要裁剪。

迁移成功的渲染体系应形成闭环：视觉目标决定内容规范，内容规范进入资源管线，资源管线进入构建发布，性能系统验证目标平台，调试工具定位问题，规范再反哺生产。

#### 十三、截图回归与美术验收应成为渲染交付的一部分
渲染迁移很容易出现“单次评审看起来正确，但后续某次材质、光照、后处理或配置修改悄悄改变整体画面”的问题。UE 项目中全局后处理、主材质、光照模板、Scalability、引擎版本、Shader 编译和平台配置都可能影响大量场景。仅依赖人工记忆和临时截图，很难发现渐进式画面漂移。因此，渲染体系应建立截图回归与美术验收流程。

截图回归不要求一开始做到完全自动化，但应固定一组相机点、角色姿态、时间、天气、画质档位和平台构建。每次修改公共材质、后处理、光照模板、渲染配置或引擎版本后，都应在这些点位生成对照截图。美术负责人和 TA 需要判断差异是否符合预期，并记录变更原因。对于竞技、UI 清晰度、角色近景和商业展示场景，还应建立更严格的对比标准。

截图验收要和性能验收绑定。某个新光照方案如果画面更好但 GPU 成本明显超预算，不能只按视觉通过；某个低档位如果性能达标但角色辨识度下降，也不能只按性能通过。正式的渲染交付应同时说明画面差异、性能差异、平台差异和风险。这样渲染迁移才不会变成一次性美术调参，而是可回归、可解释、可持续维护的工程流程。

### 实现方案

#### 一、建立视觉基准与对照样片
选择一组 Unity 旧项目代表画面和 UE 目标画面，建立对照样片。样片应覆盖角色、场景、材质、光照、后处理、特效、UI 和性能峰值。每张样片记录平台、分辨率、时间、光照条件、后处理配置和预期指标。

基准样片用于统一审美和验收，不用于机械复刻。迁移目标可以调整风格，但必须明确差异是主动选择还是迁移误差。

#### 二、制定渲染路径和平台矩阵
为每个平台确定使用 Deferred 还是 Forward，是否启用 Lumen、Nanite、Virtual Shadow Maps、Virtual Texture、TSR、动态分辨率、硬件光追、体积雾和高质量反射。矩阵应说明默认档位、降级路径和禁用特性。

平台矩阵是内容制作依据。美术和 TA 在制作资源前应知道目标平台能承载什么，而不是后期再砍效果。

#### 三、建立材质族和主材质库
按项目风格建立主材质库：角色、布料、金属、植被、地形、建筑、水、透明、特效、UI、武器、特殊表现。每个主材质有使用说明、参数范围、实例示例、性能成本和平台限制。

旧 Unity Shader 迁移时先归类，再决定是否进入主材质、材质函数或特殊材质。不要为每个旧 Shader 单独复制一套 UE 材质。

#### 四、统一光照和后处理规范
建立光照样例关卡，定义曝光、天空光、主光、间接光、阴影、反射、雾、色彩分级和后处理基线。核心 Post Process Volume 由 TA 或图形负责人维护，关卡局部调整需要说明原因。

不同场景类型可以有不同模板，例如室内、室外、夜景、地下城、竞技地图、剧情镜头。模板化能减少风格漂移。

#### 五、建立渲染性能预算
为目标平台定义 GPU 帧时间、主要 Pass 成本、材质指令、透明 Overdraw、动态光源数量、阴影数量、粒子数量、显存和纹理规格预算。预算写入资产规范和评审流程。

新增高成本效果必须在基准场景中验证。没有数据的“看起来还好”不能作为上线依据。

#### 六、建设图形调试流程
整理常用视图和工具：Shader Complexity、Quad Overdraw、Light Complexity、Buffer Visualization、GPU Visualizer、Nanite、Lumen、VSM 调试、RenderDoc、平台 GPU 工具。为每类问题建立排查模板。

测试问题单应包含画质档位、平台、相机位置、截图、相关资源和可复现步骤。图形问题越可复现，修复越快。

#### 七、将渲染规范接入资源管线
资源导入时检查贴图尺寸、压缩、通道、sRGB、法线、LOD、Nanite、材质复杂度、虚拟纹理和平台限制。CI 或编辑器工具输出异常报告。公共材质、后处理、光照模板和 Shader 函数有 owner。

资源规范和渲染规范应使用同一套术语。不要让美术文档和程序文档各说各话。

#### 八、建立截图回归与画面对比流程
为核心基准场景设置固定相机、固定角色、固定光照和固定画质档位。公共渲染配置变化后，输出截图对比、GPU 成本对比和显存占用对比。差异必须标注为预期变化、风险变化或异常变化，并由对应 owner 确认。

截图回归流程应覆盖高端和低端档位。高端档位检查画质上限，低端档位检查清晰度、辨识度和降级质量。这样可以避免低配画面在长期迭代中被忽视。

#### 九、建立渲染迁移验收标准
渲染管线与图形技术迁移完成应满足以下标准：

- 视觉目标、目标平台、帧率和画质档位已明确。
- 材质族、主材质、材质实例和贴图通道规范已建立。
- 光照路线、后处理基线、曝光和色彩管理有统一模板。
- Lumen、Nanite、VSM、Virtual Texture、TSR 等特性有平台矩阵。
- 透明、Niagara、UI 渲染和后处理有专项预算。
- Shader 复杂度、变体、材质函数和公共材质有 owner。
- 资源 Streaming、LOD、HLOD、纹理组和显存预算可验证。
- 图形调试工具和问题反馈格式已进入团队流程。
- 基准场景在目标平台上通过画质和性能验收。
- 核心相机点有截图回归流程，公共渲染变更可对比、可解释、可追踪。
- 渲染内容与资源管线、构建发布和性能回归形成闭环。

### 总结
渲染管线与图形技术迁移，本质上是视觉目标、性能预算和内容生产方式的共同迁移。Unity 的 Built-in、URP、HDRP 经验能帮助团队理解问题类型，但不能替代 UE 的材质体系、光照假设、后处理流程、现代渲染特性和平台画质治理。只要团队先定义目标和边界，再迁移具体效果，就能避免长期停留在表面对照阶段。

成熟的 UE 渲染迁移结果，应让美术知道如何制作资源，TA 知道如何维护材质和光照，程序知道如何分析性能，测试知道如何复现图形问题，发布系统知道如何裁剪平台内容。这样渲染系统才不是单个效果集合，而是可量产、可调试、可降级、可持续迭代的图形生产体系。
