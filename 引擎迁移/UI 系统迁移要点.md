# UI 系统迁移要点

## 摘要
Unity UI 迁移到 UE，并不是把 UGUI 或 UIToolkit 的界面结构照搬到 UMG，而是重新理解界面层级、数据驱动、输入焦点和资源组织方式。本文总结 UI 系统迁移时最需要关注的心智切换与工程边界。

## 正文

### 背景
Unity 团队在 UI 开发中通常已经形成自己的分层习惯，例如 View、Presenter、配置数据和界面管理器等。进入 UE 后，UMG、Widget Blueprint、Slate 与输入焦点体系带来了新的工作流和约束。若只是把原有界面逻辑原样搬运，迁移后的 UI 往往会出现边界不清、蓝图过重和资源组织混乱等问题。

### 核心内容
UI 迁移首先要明确数据层、界面层和交互层的职责边界。UE 的 Widget 适合承载表现和交互编排，但不应成为所有业务逻辑的容器。其次，要理解 UE 的输入焦点、界面生命周期和可视化编辑工作流与 Unity 存在明显差异，特别是在界面创建、销毁、层级管理和可编辑参数暴露方面。

常见误区包括：把每个界面都做成庞大蓝图；忽视数据与 Widget 的分离；用熟悉的 Unity 管理器强行覆盖 UE 原生输入和焦点机制；低估资源命名与界面资产组织的重要性。

### 实现方案
建议先迁移一个中等复杂度界面作为样例，重点验证：

- 数据如何进入 Widget。
- 输入和焦点如何切换。
- 界面显示、隐藏和销毁的责任归谁。
- 蓝图与 C++ 如何协同。

在样例稳定前，不应急于迁移大规模复杂 UI。界面迁移最怕前期边界没立住，后面所有页面都按错误方式复制。

### 总结
UI 系统迁移的关键，是从“复刻界面”转向“重建界面工作流与职责分层”。只有接受 UE 的 Widget 生态和输入模型，再结合团队自身架构做有边界的适配，迁移后的 UI 才能既好用又可维护。
