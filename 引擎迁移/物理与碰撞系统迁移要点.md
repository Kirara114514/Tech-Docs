# 物理与碰撞系统迁移要点

## 摘要
物理与碰撞系统是 Unity 向 Unreal Engine 迁移时最容易产生“经验错用”的领域之一。之所以危险，并不是因为开发者不知道什么叫刚体、碰撞体、射线检测、约束、触发器，而是因为这些概念在两个引擎里虽然都存在，却被放进了不同的求解器、不同的生命周期、不同的角色移动框架、不同的网络语义和不同的调试手段之中。很多迁移团队会自然地带着 Unity 时代的手感与经验去解释 UE 中的碰撞和物理行为，结果得到的往往是“看起来像同一种问题、修起来却完全不对路”的连续挫败。

尤其是在 Unreal Engine 已经切换到 Chaos 物理体系之后，物理系统更不应被看成“另一套 PhysX 配置面板”。它不只是更换了若干参数名称，而是意味着你要重新理解：物理求解器如何与 CharacterMovement、动画根运动、碰撞通道、Query 与 Simulate 分工、网络同步、约束系统、破碎系统以及性能预算共同构成一套新的运行时边界。

本文重点讨论物理与碰撞系统迁移中最关键的认知切换：为什么同样叫“碰撞”但行为差异巨大，为什么 Query 和 Simulate 的边界在 UE 中尤其重要，为什么 CharacterMovement 不能简单当作“包了层刚体的角色控制器”，为什么碰撞通道与对象类型治理会决定项目后期维护成本，以及怎样按工业化方式规划物理、移动、碰撞、动画和网络之间的职责分工。

## 正文

### 背景
Unity 程序员对物理系统通常并不陌生。多数人都做过刚体驱动、Trigger、射线检测、CharacterController 或 NavMesh 与碰撞配合，也见过各种经典问题：穿透、抖动、台阶卡住、胶囊体与动画不同步、刚体受力不稳定、触发器顺序问题等等。正因为积累了大量经验，迁移到 UE 时反而更容易被这些经验误导。

最常见的误区不是“完全不会用”，而是过于快速地把现象类比为旧问题。例如：

- 角色移动不稳定，就下意识当作刚体参数问题。
- 碰撞不到，就以为只是 Layer / Mask 对应关系没配好。
- 射线结果怪异，就假设只是 API 用法差异。
- 角色和动画打架，就觉得是根运动设置不对。

这些判断有时会碰巧正确，但在 UE 项目中，很多物理问题其实位于更深的结构层：

- Query 与 Simulate 根本不是同一种碰撞语义。
- CharacterMovement 有自己的控制模型，不等于单纯物理刚体。
- 碰撞通道不仅是“筛选条件”，还是整个项目对象交互契约的一部分。
- Chaos 求解行为、约束和调试方法不能直接按 Unity 用户熟悉的经验解释。
- 多人项目中的角色移动和物理状态同步边界，比单机项目更严格。

也就是说，物理迁移最怕的并不是不会，而是“看起来很会”。因为你越快把新问题解释成旧问题，就越容易在错误方向上持续修补。

### 核心内容

#### 一、物理迁移首先要区分：你在处理的是“物理模拟”还是“空间查询”
很多项目在早期就把“碰撞”统称为同一类问题，这是后期所有混乱的源头之一。在 UE 中，这种区分尤其重要：很多系统压根不是在做物理模拟，而是在做空间查询。

常见的空间查询包括：

- Line Trace
- Sphere Trace
- Box Trace
- Sweep
- Overlap Query

这些行为和真正的刚体模拟、碰撞求解、约束响应不是一回事。迁移时如果仍然把所有东西都叫“物理”，就会不断混淆两个问题：

- “我需要世界几何与对象信息吗？”
- “我需要真实物理响应吗？”

举例来说，近战攻击判定、交互检测、落点预测、可视检测、命中校验、脚底探测，这些大多更接近 Query 问题，而不是刚体模拟问题。如果你一上来就用“让碰撞系统自己解决”或“让物理驱动行为”来思考，会平白给系统增加不必要的不确定性和成本。

成熟的迁移第一步，应该是先把 Query 和 Simulate 分开。只有这样，后面碰撞体、通道、移动、动画和网络的边界才谈得清楚。

#### 二、CharacterMovement 不是“UE 版 CharacterController + Rigidbody 混合物”
这是从 Unity 迁移到 UE 时最危险的误判之一。

很多 Unity 程序员会自然寻找：

- `CharacterController` 对应什么
- 刚体移动对应什么
- 胶囊体碰撞对应什么

然后试图把 CharacterMovement 理解成一种“封装了刚体和角色碰撞的组件”。这种理解只能用于最初入口，不能用于工程判断。

CharacterMovement 的真正意义，在于它定义了一套角色移动与碰撞交互模型。它并不是单纯把物理刚体交给你配置，而是为“角色”这种特殊对象提供一套更偏游戏逻辑、可控性更强的移动和碰撞行为。也正因为如此：

- 很多角色移动问题并不是纯物理参数问题。
- 很多你以为该交给模拟求解的事情，实际上应交给 CharacterMovement 或更上层状态逻辑控制。
- 将动画、根运动、输入、导航、网络预测与角色移动接上时，CharacterMovement 的边界会比刚体模拟更重要。

如果迁移时没有先把这层边界看清，团队很容易做出“既想要完全物理模拟，又想要角色行为绝对可控”的混合设计，最终两边都做不好。

#### 三、碰撞通道与对象类型，不是配置细节，而是项目级交互协议
Unity 团队很习惯 Layer / Mask 思维，因此迁移到 UE 时看到 Object Channel、Trace Channel、Response 设置，往往会觉得“也就是另一套 Layer 配法”。这只对了一半。

在 UE 中，碰撞通道如果管理得当，会成为整个项目的交互协议层；如果管理混乱，会变成后期最难维护的全局泥潭。

为什么这么说？因为碰撞通道并不只是决定“打不打得中”，它还决定：

- 哪些对象会被哪些查询看到。
- 哪些对象参与阻挡，哪些只参与 Overlap。
- 哪些交互是物理级，哪些是逻辑级。
- 哪些系统共享同一通道语义，哪些必须分离。

如果项目早期没有统一规则，后期最常见的情况就是：

- 新系统来了就新建一个通道。
- 不同开发者对同一通道的语义理解不同。
- 某个修复临时把响应关系改了，结果误伤别的系统。
- Query 和 Simulate 的响应关系越改越乱。

因此，迁移时一定要尽早定义碰撞协议，而不是把通道当成本地配置问题。

#### 四、根运动、物理、导航和角色碰撞是四套容易互相抢控制权的系统
很多物理与碰撞问题表面看起来是“卡住了”“抖了”“穿了”，实则是控制权争夺。

最常见的冲突包括：

- CharacterMovement 想控制角色位移，但动画根运动也在推位置。
- AI 导航想控制移动方向，但攻击动画或受击动画也在抢控制权。
- 物理模拟想给对象响应，但上层逻辑又试图手动修正位置。
- 客户端预测想平滑，权威修正又在不断打断。

如果没有显式边界，后果往往不是某一个系统完全错，而是所有系统都“有点像在工作”，最终结果却特别难解释。

迁移时必须提前回答：

- 这个对象是逻辑驱动、移动组件驱动、物理驱动，还是动画驱动？
- 不同阶段控制权如何切换？
- 控制权切换时哪些状态必须同步更新？
- 失败时如何回退？

只要不明确这些，物理与碰撞问题就会一直伪装成各种“奇怪手感 bug”。

#### 五、Chaos 体系下，求解与调试思路必须重新学习，不能机械套用 Unity 经验
即便不深入引擎实现，迁移团队也必须承认一件事：Chaos 的行为和你在 Unity / PhysX 里积累的“手感经验”并不能直接照抄。你当然仍然可以参考过去的稳定性原则，例如：

- 控制连续高速物体穿透风险。
- 谨慎处理过轻或过重质量差。
- 关注约束稳定性。
- 避免把所有问题都交给真实模拟。

但一旦进入具体排障，你必须重新建立一套 UE 语境下的问题阅读方式。例如：

- 当前问题是角色移动模型问题，还是 Chaos 模拟问题？
- 当前碰撞不稳定，是 Query 层误判，还是 Simulate 层求解结果？
- 约束、破碎、刚体叠放或复杂交互是否进入了 Chaos 本身的特性边界？

如果继续凭旧引擎经验拍脑袋，很容易在错误参数上反复试错。

#### 六、多人项目中的物理迁移，核心不是“同步物体位置”，而是“同步哪一层语义”
和输入、动画、UI 一样，物理在多人项目里也不能直接照搬单机思路。

许多团队在单机里对物理对象的理解是：

- 位置变了就更新位置。
- 受力了就看结果。
- 碰撞发生了就触发逻辑。

到了多人环境，这种理解会迅速失效，因为你必须重新回答：

- 这次碰撞结果由谁权威决定？
- 客户端是否允许本地先模拟？
- 同步的是刚体状态、离散事件，还是高层结果？
- 某个物理对象在多人里到底是不是值得做全模拟同步？

如果不先定义这些，迁移时特别容易出现两类极端：

- 想同步所有东西，带宽和复杂度爆炸。
- 只同步最终位置，结果表现层与交互层严重割裂。

所以，多人物理迁移真正要做的，是先区分“哪些物理行为必须成为多人语义的一部分，哪些只应是本地表现细节”。

### 实现方案

#### 一、推荐的迁移顺序
物理与碰撞系统迁移不建议从最复杂的战斗或场景交互开始，而建议按下面顺序推进。

##### 第一步：建立 Query 与 Simulate 的最小样例
先分别验证：

- 射线 / Sweep / Overlap 查询
- 一个简单刚体对象的碰撞与受力

重点不是功能多，而是让团队明确“查询”和“模拟”根本不是一回事。

##### 第二步：建立角色移动样例
做一个最小角色，验证：

- CharacterMovement 的移动与胶囊碰撞
- 台阶、坡度、地面判定
- 与简单阻挡物和触发区交互

这一步的目标是把“角色控制模型”和“普通物理对象”分开。

##### 第三步：引入控制权冲突样例
专门验证：

- 动画根运动与 CharacterMovement
- 导航与移动控制权切换
- 受击位移与物理 / 动画 / 逻辑的关系

如果这一步不做，后面战斗系统一定会在这里反复返工。

##### 第四步：再进入复杂物理场景
包括：

- 可破坏对象
- 约束系统
- 大量可交互刚体
- 多人同步物理行为

这些内容必须建立在前面几层边界清晰的基础上。

#### 二、推荐的项目级碰撞协议
建议团队尽早制定并文档化以下规则：

- 角色、敌人、投射物、场景阻挡、触发区、交互对象、UI 射线体分别属于哪些对象类型。
- 哪些查询应该使用 Trace Channel，哪些交互应走 Object Response。
- 哪些对象只做 Query，哪些允许 Simulate。
- 哪些碰撞结果只是表现层触发，哪些会改变核心状态。

这些规则一旦文档化，碰撞系统就从“每个人自己配一个响应关系”升级为“团队共用的交互契约”。

#### 三、最小代码骨架示例
下面这个示例强调的是：很多交互更适合先作为 Query 处理，而不是直接交给物理模拟。

```cpp
FHitResult HitResult;
FCollisionQueryParams Params(SCENE_QUERY_STAT(InteractTrace), false);

bool bHit = GetWorld()->LineTraceSingleByChannel(
    HitResult,
    Start,
    End,
    ECC_Visibility,
    Params
);
```

这个例子的重点不是射线本身，而是迁移思路：

- 先明确需求是不是查询问题。
- 先通过正式通道语义约束交互对象。
- 不要一上来就把一切都交给物理响应。

这类克制的建模方式，会显著减少后续系统混乱。

#### 四、验收标准
物理与碰撞迁移是否真正到位，不应只看“撞得上”“能推动”，而应至少满足：

- 团队能明确区分 Query 问题与 Simulate 问题。
- CharacterMovement 与刚体模拟边界清晰。
- 碰撞通道和对象类型具备稳定协议，不靠临时修补。
- 根运动、导航、角色移动和物理之间控制权可解释。
- 多人项目里物理行为的权威语义已被定义，而不是只同步表面位置。
- 常见问题已有统一排障路径，而不是持续凭旧经验盲调参数。

只有这些成立，物理与碰撞系统才算真正完成迁移，而不是只是“用起来差不多了”。

#### 五、典型故障案例与排障顺序
迁移期最常见的物理问题，不是“完全不工作”，而是“看起来大致对，但总有边角错”。下面列几个特别典型的故障模式。

第一类是角色在坡面、台阶和边缘处表现异常。很多团队第一反应是调胶囊、调摩擦、调重力，实际上问题往往在 CharacterMovement 的地面判定、坡度限制、步高设置和动画/根运动干扰。正确排障顺序通常应是：先确认这是 CharacterMovement 行为问题还是刚体问题，再看胶囊体和地面交互规则，最后才去微调手感参数。不要一开始就把它当成“物理手感不对”。

第二类是攻击、交互或技能判定时好时坏。很多人会以为“碰撞系统又抽风了”，其实经常是 Query 与 Simulate 混用导致的：一部分逻辑依赖 Trace，一部分逻辑又期待真实碰撞事件，两者时序和对象过滤规则不一致，自然就会出现偶发错误。更稳妥的做法是先明确这条链路到底是查询问题还是模拟问题，统一判定入口，再看通道和对象类型是否符合协议。

第三类是动画播放时角色位移和碰撞不同步。这类问题经常发生在根运动、CharacterMovement、导航和技能位移共同作用时。排障时一定要先问“当前是谁拥有位移权”，不要先去改碰撞参数。只要控制权没搞清楚，再多参数也只是掩盖问题。

第四类是多人项目里本地看起来撞上了，但权威结果不认。这个问题通常不是“网络同步慢了一点”这么简单，而是本地表现、权威判定和碰撞查询使用了不同语义层。正确做法是把命中和碰撞决策的权威层级文档化，并在日志中明确记录：谁发起、谁判定、谁展示。

#### 六、推荐的项目级物理治理规则
物理和碰撞只要进入中大型项目，就不能靠“谁遇到问题谁顺手调一下”来维持。建议至少建立下面几条治理规则。

第一，所有对象在进入项目时必须明确属于哪一类交互角色：纯查询对象、纯模拟对象、角色碰撞对象、导航辅助对象、触发区对象、投射物对象、可破坏对象。不要让对象同时在多个语义层里摇摆。

第二，碰撞通道必须有命名规范和使用手册。不是写给新人看的装饰文档，而是团队真正执行的协议。谁能挡谁、谁能查谁、谁只做 Overlap、谁绝不做 Simulate，必须提前定义，而不是靠个人经验记忆。

第三，角色移动、根运动、技能位移、AI 导航和物理模拟的控制权切换必须有专项样例。只要一个团队没有专门验证过这些切换，正式项目里几乎一定会在战斗和复杂移动上踩坑。

第四，物理问题应当有专门的可视化和日志约定。包括 Trace 可视化、碰撞盒显示、关键命中日志、控制权切换日志和多人权威判定日志。物理问题最怕靠感觉改参数，最稳的方式永远是让空间关系和判定链路可见。

### 总结
物理与碰撞系统迁移最难的地方，不是概念陌生，而是概念太熟。正因为刚体、碰撞体、触发器、射线、约束这些词你早就认识，所以最容易误把新引擎中的现象解释成旧引擎中的经验问题。迁移真正要做的，不是忘掉旧知识，而是把旧知识重新放进 UE 的 Query/Simulate 分层、CharacterMovement 模型、Chaos 求解、碰撞协议、动画与导航控制权以及多人语义中重新理解。

一旦这套理解建立起来，很多原本看似玄学的问题都会变得可解释：为什么有些东西不该做真实物理，为什么有些交互只该做查询，为什么碰撞通道是一种项目协议，为什么角色移动和物理模拟不能混成一锅。那时，物理迁移就不再只是“把旧角色放到新引擎里也能撞起来”，而是重建了一整套面向长期项目的交互基础设施。
