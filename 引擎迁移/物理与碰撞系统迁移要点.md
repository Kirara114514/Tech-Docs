# 物理与碰撞系统迁移要点

## 摘要
物理与碰撞系统从 Unity 迁移到 Unreal Engine，并不是把 Collider、Rigidbody、Trigger、Layer 和 CharacterController 替换成 UE 中看似对应的组件，而是重新设计碰撞通道、对象类型、查询规则、响应矩阵、物理材质、角色移动模型、刚体模拟边界、触发器语义、网络权威和验证流程。Unity 的 Layer Collision Matrix、Physics Query、Rigidbody、CharacterController、NavMeshAgent 与脚本位移组合，在 UE 中通常需要拆解为 Collision Channel、Object Type、Collision Profile、Trace Channel、Query/Sweep/Overlap、Chaos Physics、CharacterMovementComponent、Physics Asset、Physical Material、Movement Mode 和网络复制策略。

本文围绕“碰撞语义如何重建、角色移动与物理模拟如何分层、查询与触发如何规范、高速碰撞如何验证、物理材质如何迁移、网络物理如何约束、性能与调试如何治理”展开，帮助团队避免把 UE 物理系统用成 Unity 旧碰撞配置的表面映射。迁移目标不是让角色和物体“差不多能撞”，而是建立一套可预测、可调试、可扩展、可验收、可跨平台稳定运行的物理与碰撞工程体系。

## 正文

### 背景
Unity 团队通常已经熟悉一套物理组合方式：用 Layer 控制碰撞矩阵，用 Collider 表示形状，用 Trigger 表示区域事件，用 Rigidbody 表示动力学对象，用 CharacterController 或自定义脚本控制角色移动，用 Physics.Raycast、OverlapSphere、SphereCast 等查询接口完成技能判定、拾取检测、地面检测和交互检测。很多玩法逻辑会直接依赖 Unity 的回调语义，例如 `OnCollisionEnter`、`OnTriggerEnter`、`FixedUpdate` 和 Rigidbody 约束。

UE 的物理与碰撞系统组织方式不同。碰撞通道分为对象通道和查询通道，组件通过 Collision Profile、Object Type 和 Response 决定与其他对象或查询如何交互；查询可通过 Line Trace、Shape Trace、Sweep、Overlap 等方式执行；角色移动通常由 CharacterMovementComponent 处理胶囊、地面、斜坡、台阶和网络预测；刚体模拟由 Chaos Physics 负责；骨骼物理通过 Physics Asset、Constraint、Physical Animation、Ragdoll 等机制组织；物理材质影响摩擦、弹性、脚步声和表面反馈；网络场景还必须区分服务端权威、客户端表现和物理状态复制。

迁移时如果继续沿用旧引擎思维，最容易出现“看起来差不多但手感不对”的问题。角色上下台阶、坡面滑动、高速投射物穿透、触发区域重复触发、技能判定命中异常、摄像机碰撞抖动、多人状态不一致、刚体堆叠不稳定，都可能来自碰撞语义没有重新设计。物理系统的风险在于它通常不是孤立 Bug，而是会影响移动、战斗、AI、动画、音频、特效、关卡和网络。

### 核心内容

#### 一、先重建碰撞语义，而不是照搬 Unity Layer
Unity Layer 常同时承担渲染分组、物理碰撞、射线过滤、逻辑分类和编辑器选择等多个职责。UE 中更适合把碰撞语义拆成对象类型、查询通道、碰撞 Profile 和响应规则。对象类型回答“这个组件是什么”，例如 Pawn、WorldStatic、WorldDynamic、Projectile、Interactable、Pickup、HitBox、TriggerVolume。查询通道回答“这次检测想找什么”，例如 WeaponTrace、CameraTrace、InteractionTrace、GroundTrace、VisibilityTrace。

迁移时不要直接把 Unity Layer 一对一转换成 UE Channel。应先列出项目中所有碰撞语义：玩家角色、敌人角色、NPC、地形、建筑、可破坏物、投射物、近战判定、拾取物、交互物、触发区域、摄像机遮挡、导航障碍、物理道具、载具、UI 3D 点击、技能范围、脚步材质检测。再决定哪些是对象通道，哪些是查询通道，哪些只需要 Profile。

响应矩阵要控制规模。UE 自定义通道数量不是无限治理空间，通道过多会导致配置难读、测试困难和误用频繁。应优先建立少量稳定通道，再用 Tag、组件类型、接口或 Gameplay 逻辑做二次过滤。碰撞通道适合处理低层物理和查询过滤，不应承载所有业务语义。

#### 二、Collision Profile 应成为团队共享契约
UE 的 Collision Profile 可以将 Object Type、Collision Enabled、Response、Trace 设置封装为可复用配置。迁移团队如果每个蓝图和组件都手动改 Response，很快会出现同类对象碰撞行为不一致。应为常见对象建立项目级 Profile，例如 PlayerPawn、EnemyPawn、Projectile、MeleeHitBox、PickupItem、Interactable、TriggerOnly、CameraBlocker、RagdollBody、Destructible、NoCollision。

每个 Profile 都要说明用途、允许修改范围和验收方式。比如 PlayerPawn 的胶囊负责角色移动碰撞，不应同时作为近战受击盒；MeleeHitBox 只用于查询或 Overlap，不应阻挡角色；TriggerOnly 只产生 Overlap，不应阻挡物理；CameraBlocker 只响应摄像机查询，不应影响角色移动。Profile 是跨程序、关卡、美术和策划的语言，必须稳定。

Profile 变更应走评审。某个公共 Profile 改动可能影响全项目角色、投射物、交互物和关卡触发。迁移过程中可以允许试验性 Profile，但量产后核心 Profile 不应随意修改。最好在文档中记录 Profile 名称、Object Type、默认响应、典型使用组件和禁止事项。

#### 三、角色移动与刚体模拟必须分层
Unity 项目中角色移动可能由 CharacterController、Rigidbody、NavMeshAgent 或脚本 Transform 混合实现。UE 中普通角色移动通常应优先使用 CharacterMovementComponent，因为它处理胶囊移动、地面检测、坡度、台阶、跳跃、下落、网络预测和模拟代理平滑。不要轻易把玩家角色做成完全物理刚体，除非项目玩法明确需要。

角色胶囊代表玩法碰撞，不等于视觉网格，也不等于受击盒。角色 Mesh 可以用于动画和骨骼表现，Physics Asset 可以用于布娃娃或局部命中，额外 HitBox 可以用于战斗判定，但角色移动的主要阻挡应由胶囊承担。若把战斗命中、角色阻挡、摄像机遮挡和交互检测都塞进同一个碰撞体，会导致响应规则不可控。

刚体模拟适合物理道具、破碎物、可推动物、投掷物、布娃娃和环境动态物体。它们与角色移动互动时要明确权威：角色能否推动刚体，刚体能否阻挡角色，角色站在刚体上是否稳定，刚体是否参与网络复制。脚本位移、物理力和 CharacterMovement 不应随意混用，否则会出现穿透、抖动和校正异常。

#### 四、查询、Sweep 与 Overlap 要按用途建立接口
Unity 中常用 Raycast、SphereCast、OverlapSphere 等接口。UE 中有 Line Trace、Sphere Trace、Capsule Trace、Box Trace、Sweep、Overlap、Multi Trace 等对应能力。迁移时应为常见用途建立统一查询接口，而不是让每个系统直接写不同参数。常见用途包括地面检测、交互检测、武器命中、投射物路径、摄像机遮挡、AI 感知辅助、技能范围、拾取检测和导航前置检查。

查询接口应明确使用 Trace Channel 还是 Object Query，是否忽略自己，是否返回复杂碰撞，是否命中 Trigger，是否返回多个结果，是否排序，是否需要调试绘制。比如近战武器命中常需要 Sweep 或多段 Trace，并忽略施法者；交互检测可能使用 Camera 或 InteractionTrace 并过滤接口；地面检测要考虑坡度和物理材质；摄像机检测要阻挡 WorldStatic 和特定遮挡物，但不应被拾取物干扰。

查询结果不能只看是否命中。还要处理命中组件、命中骨骼、物理材质、表面类型、距离、法线、穿透初始状态、Actor 是否有效、是否已被重复命中。对于战斗系统，命中查询只是候选结果，最终伤害仍应由权威规则判断。

#### 五、Trigger 与 Overlap 事件需要生命周期治理
Unity Trigger 迁移到 UE Overlap 时，要注意事件语义和生命周期差异。UE 中 Overlap 依赖组件碰撞启用、Generate Overlap Events、响应规则、初始重叠、移动更新和组件激活状态。迁移后常见问题包括初始重叠不触发、重复触发、离开事件缺失、对象销毁时未清理、多人客户端和服务端触发不一致。

触发区域应有明确 owner 和状态机。任务区域、伤害区域、拾取区域、交互区域、剧情触发、检测体积和音频区域虽然都使用 Overlap，但业务语义不同。触发区域应决定是否只服务端有效，是否客户端也做表现，是否允许重复触发，是否需要冷却，是否记录已进入对象，Actor 销毁时如何清理。

不要把复杂业务直接写进 Overlap 回调。Overlap 应产生事件或候选事实，业务组件再判断权限、状态和条件。例如玩家进入门区域后，交互系统显示提示；真正开门仍需玩家输入和服务端验证。这样可以避免触发器既做检测又做业务导致难以复现。

#### 六、物理材质与表面类型应纳入表现和玩法规范
Unity 项目中可能通过 Physic Material 控制摩擦和弹性，也可能通过材质或地形信息决定脚步声、粒子和命中特效。UE 中 Physical Material 和 Surface Type 可以同时影响物理响应和表现反馈。迁移时应统一设计表面类型，例如 Concrete、Grass、Metal、Wood、Water、Mud、Sand、Snow、Flesh、Armor、Shield。

表面类型应服务多个系统：脚步声、落地声、子弹命中特效、近战命中特效、脚印、摩擦、弹跳、轮胎抓地、AI 感知噪声。不要让音频、特效和物理各自维护一套表面枚举。查询命中后读取 Physical Material，可以将同一表面事实传给表现和玩法系统。

物理材质配置要区分物理参数和表现参数。摩擦、恢复系数、密度等物理参数会影响模拟稳定性；脚步声和特效选择更偏表现。若所有内容都写进材质蓝图或临时表，后续跨系统难以维护。建议建立表面类型数据资产，统一映射音频、特效、Decal、脚印、噪声和调试颜色。

#### 七、高速碰撞和连续检测必须专项验证
高速投射物、快速近战挥砍、冲刺、车辆、击退和物理碎片是物理迁移高风险点。离散帧检测可能导致穿透，碰撞体过小或速度过高时尤其明显。UE 中可使用 CCD、Sweep、ProjectileMovementComponent、Sub-Stepping、多段 Trace 或服务器回溯等方式处理，但具体策略取决于玩法和性能预算。

投射物不应一律做真实刚体模拟。子弹类可使用 Trace 或服务器命中检测；慢速飞行物可用 ProjectileMovementComponent；带物理反弹的物体才需要更完整模拟。近战武器通常不建议只依赖武器碰撞体 Overlap，因为动画帧间可能漏检；可以沿武器轨迹做多点 Sweep 或使用 Notify Window 驱动判定。

高速碰撞验收必须在目标帧率、目标平台和网络条件下进行。编辑器高帧率下不穿透，不代表低端设备或 Dedicated Server 下稳定。应建立专门测试关卡，包含高速投射物、薄墙、移动目标、旋转武器、斜坡、台阶和复杂碰撞体。

#### 八、复杂碰撞体要平衡精度、性能和编辑成本
UE 支持简单碰撞、复杂碰撞、凸包、自动碰撞、Physics Asset 和自定义碰撞体。迁移时不能直接使用渲染网格作为所有碰撞依据。复杂碰撞精度高，但查询和物理成本也高；简单碰撞性能好，但可能不符合视觉形状。团队应按资源类型制定碰撞策略。

静态环境可使用简化碰撞或按需求使用复杂作为简单，但要关注性能和导航；角色使用胶囊和 Physics Asset 分层；可交互物使用简单碰撞加交互 Trace；投射物命中可对关键对象使用更精细 HitBox；大型地形要关注坡度、台阶和导航一致性。美术导入时应提供合理碰撞，不应让程序后期逐个修。

碰撞体也影响关卡设计。窄门、楼梯、斜坡、栏杆、低矮障碍、移动平台和可破坏物都需要碰撞规范。关卡美术如果只按视觉搭建，不按角色胶囊、摄像机和导航要求验证，后期会出现大量卡住和穿帮问题。

#### 九、物理与动画、战斗、AI、音频要建立协作边界
物理系统不是孤立模块。动画系统需要 Root Motion、布娃娃、物理动画和碰撞胶囊协调；战斗系统需要命中查询、受击盒、击退和阻挡；AI 需要导航、动态障碍和感知查询；音频和特效需要表面类型；UI 可能需要 3D 点击和交互检测。迁移时必须为这些系统建立共同协议。

例如战斗命中应使用专门 HitBox 或查询通道，不应直接依赖角色移动胶囊；动画布娃娃启用时要处理角色控制、碰撞 Profile、网络复制和恢复站立；AI 动态障碍要明确是否影响 NavMesh；脚步声应从地面检测或动画 Notify 中获得表面信息；摄像机碰撞不应被所有 Trigger 干扰。

跨系统协议应写成规范，而不是靠口头约定。每个系统都应知道自己可以修改哪些碰撞设置，哪些 Profile 不能动，哪些查询通道可用，如何调试命中结果。

#### 十、网络物理要控制权威与表现范围
多人项目中的物理迁移尤其复杂。完全同步所有刚体物理通常成本高且不稳定。团队必须决定哪些物理对象服务端权威，哪些客户端本地表现，哪些只同步结果，哪些不参与网络。玩家角色移动通常使用 CharacterMovement 的网络模型；战斗命中由服务端权威；小型碎片、尘土和局部特效可客户端本地表现；关键可推动物或载具需要专门同步策略。

刚体复制要谨慎。服务端模拟并复制物体位置可保证权威，但高频同步成本高，客户端表现可能抖动。客户端本地模拟手感好，但多人一致性差。对玩法关键物体，应减少数量、降低复杂度、设置休眠和相关性；对纯表现物体，应避免进入网络同步。

物理网络验收应覆盖延迟、丢包、服务器低帧率、客户端预测失败、玩家同时推动、物体休眠唤醒、掉落物拾取和关卡重置。不要只在 Listen Server 中验证，因为 Listen Server 很容易掩盖客户端与服务端差异。

#### 十一、性能预算要覆盖碰撞查询与物理模拟
物理性能问题常来自大量 Overlap、频繁 Trace、复杂碰撞、未休眠刚体、骨骼物理、布娃娃、破碎模拟和调试绘制。迁移时应为不同系统设定预算：每帧最大武器 Trace 数、AI 查询频率、场景 Overlap 数、同时模拟刚体数量、布娃娃数量、物理约束数量和复杂碰撞使用范围。

查询优化要从语义开始。不要每个系统都每帧全场 Overlap；交互检测可以降低频率或只在输入时执行；AI 感知可以使用服务和缓存；近战检测只在攻击窗口执行；摄像机检测只对必要通道执行。通道和 Profile 设计越清晰，查询过滤越高效。

物理模拟也要设置休眠、最大速度、约束、LOD 和销毁策略。大量可破坏物或掉落物不应永久高频模拟。性能分析时要结合 stat 命令、Insights、Chaos 调试和场景复现，而不是凭感觉关闭功能。

#### 十二、调试可视化是物理迁移交付物
物理问题必须可视化。迁移时应建立调试工具或流程，能显示碰撞体、通道、Profile、Overlap、Trace、命中点、法线、物理材质、角色胶囊、Movement Mode、Floor Result、刚体睡眠状态、约束、网络角色和服务端结果。只有看到实际碰撞和查询，团队才能避免凭感觉调参数。

UE 提供多种可视化方式，包括碰撞显示、Trace 调试绘制、物理调试、控制台命令、日志和 Gameplay Debugger。项目应将常用调试开关封装成团队可用工具。测试人员反馈“角色卡住”时，最好能附带碰撞截图、位置、Profile、关卡资产和复现步骤。

物理调试日志应包含对象类型、Profile、通道、响应结果、查询参数、命中对象和业务请求 ID。尤其是战斗命中和交互检测，日志越清晰，越容易判断是碰撞配置问题、查询过滤问题还是业务校验问题。

### 实现方案

#### 一、建立碰撞语义清单
迁移前盘点旧项目所有 Layer、Trigger、Collider、Rigidbody、Physics Query、CharacterController、NavMeshAgent、脚本位移和物理材质用途。将它们按语义分类：角色阻挡、战斗命中、交互检测、拾取、摄像机、地面、投射物、触发区域、AI、物理道具、导航、表面反馈。

基于清单设计 UE 的 Object Type、Trace Channel、Collision Profile 和响应矩阵。每个通道和 Profile 都要写明用途、使用者和禁止事项。不要让通道名称只反映旧 Layer 名称。

#### 二、构建标准碰撞验证关卡
建立一个专用验证关卡，包含平地、斜坡、台阶、窄门、低矮障碍、移动平台、薄墙、复杂碰撞体、Trigger 区域、可推动物、高速投射物、近战靶、不同物理材质地面和多人测试点。所有角色移动、战斗命中、交互和物理道具都在该关卡中回归。

验证关卡应随项目迭代更新。每新增一种特殊移动、武器判定、物理道具或表面类型，都应加入对应测试点。这样物理问题不会只在真实关卡中偶发。

#### 三、重构角色移动边界
优先用 CharacterMovementComponent 建立玩家和普通 AI 的基础移动。明确胶囊尺寸、坡度、台阶高度、跳跃、下落、摩擦、制动、网络预测和 Movement Mode。战斗受击盒、交互检测和摄像机碰撞使用独立组件或查询，不与移动胶囊混用。

对旧项目中的脚本位移、Rigidbody 角色和特殊控制器逐项评估。能转为 CharacterMovement 的尽量转，确实需要物理控制的对象要单独设计同步和碰撞策略。

#### 四、迁移查询接口
建立项目统一的物理查询封装，例如交互 Trace、武器 Sweep、地面检测、摄像机 Trace、技能范围 Overlap、AI 可见性辅助查询。封装中固定通道、忽略规则、返回格式、调试绘制和日志字段。

业务系统调用封装接口，而不是各自写零散查询。这样未来修改通道、调试或优化时有统一入口。

#### 五、迁移 Trigger 与区域逻辑
对旧 Trigger 逐项分类：任务触发、伤害区域、拾取区域、交互区域、音频区域、剧情区域、AI 区域。为每类区域定义是否服务端触发、是否客户端表现、是否重复触发、是否需要冷却、是否记录进入者、销毁时如何清理。

Overlap 回调只负责检测和事件分发，具体业务由对应系统处理。多人项目中，关键 Trigger 以服务端为准，客户端只做提示和预测表现。

#### 六、建立物理材质与表面反馈表
统一表面类型，并为每种类型配置摩擦、弹性、脚步声、落地声、命中特效、Decal、脚印和 AI 噪声参数。地面检测、武器命中和投射物命中都从命中结果读取表面信息，再分发给音频和特效系统。

表面类型由技术美术和音频、特效、程序共同维护。新增表面必须有默认表现和缺失兜底，避免运行时查不到资源。

#### 七、专项验证高速和网络物理
为高速投射物、近战武器、冲刺、击退、载具、布娃娃和可推动物建立专项测试。验证低帧率、不同 Tick、目标平台、Dedicated Server、延迟和丢包条件下是否稳定。对漏检场景使用 Sweep、多段 Trace、CCD 或服务器权威判定。

网络物理只同步玩法必要对象。纯表现碎片、局部粒子和非关键物理反馈尽量客户端本地生成。玩法关键刚体设置数量限制、休眠策略和重置逻辑。

#### 八、建立物理性能与调试流程
在性能测试中记录 Trace 数量、Overlap 数量、物理模拟耗时、刚体数量、休眠数量、布娃娃数量和复杂碰撞查询成本。设置预算并在高风险系统中加日志或统计。

调试流程要求问题单包含复现场景、对象 Profile、通道、查询参数、调试截图和网络模式。对战斗命中和交互问题，必须能复现命中查询结果，而不是只描述“没打到”或“触发不了”。

#### 九、形成物理迁移验收标准
物理与碰撞系统迁移完成应满足以下标准：

- 碰撞通道、对象类型、查询通道和 Profile 有正式矩阵。
- 角色移动、刚体模拟、战斗命中、Trigger、摄像机和交互检测边界清晰。
- 旧 Unity Layer、Trigger、Rigidbody、CharacterController 用途已完成语义迁移。
- 查询接口统一封装，支持调试绘制和日志。
- 物理材质和表面类型被音频、特效、战斗和移动系统共享。
- 高速投射物、近战 Sweep、冲刺、击退和薄墙场景经过专项验证。
- Dedicated Server 下的关键物理行为与客户端表现一致或可解释。
- 复杂碰撞、刚体数量、Trace 和 Overlap 有性能预算。
- 标准验证关卡覆盖移动、碰撞、触发、材质、物理和网络场景。
- 核心 Profile 有 owner，变更需要评审和回归。


#### 工业化补充：迁移的核心不是“找到对应功能”，而是重建一套新的工程工作方式
引擎迁移类文档最常见的误区，是把迁移理解成大规模 API 对照或功能搬运。事实上，Unity 到 UE 这样的迁移真正困难的地方，从来不是“某个功能能不能做出来”，而是原来建立在旧引擎工作方式上的组织习惯、调试习惯、资源流转、代码边界、性能判断和协作接口，都需要重新落位。也正因为如此，成熟的迁移文档绝不会只列功能映射，而会把重点放在“为什么原来的做法在新引擎里不再成立”“应该先建立什么基础心智”“哪些能力必须按新引擎哲学重构，而不是照抄旧习惯”。只有当团队接受迁移本质上是工程工作方式迁移，而不是语法迁移，这类文档才真正有指导价值。

从项目管理角度看，迁移也是一项高风险系统工程。语言、脚本协作、UI、输入、网络、资源、物理、渲染、构建发布、性能分析和工具链，看似分别属于不同专题，实际上都共享同一个迁移前提：旧项目里哪些抽象可以保留，哪些必须舍弃，哪些需要通过中间层过渡，哪些必须在新引擎原生能力之上重新组织。文档若不能回答这些问题，就很容易让团队陷入“局部上手很快、整体越来越乱”的假熟悉状态。

#### 双轨验证、分阶段退出与学习顺序：迁移不能只靠勇气，更要靠可回退的计划
大型迁移项目最怕的不是进度慢，而是没有可验证、可回退的阶段边界。很多团队一开始雄心很大，计划把旧系统整体搬过去，结果中途才发现角色框架、资源引用、蓝图协作、打包发布和网络复制并没有想象中那么容易对齐。真正成熟的做法，是在文档里提前定义双轨验证策略：哪些模块先做概念验证，哪些模块先做最小可用版本，哪些关键路径需要同时保留旧实现以便对照，哪些阶段满足什么条件后才能正式切换。只有当退出条件和切换条件被写清楚，迁移才不会在版本压力下变成一场只能向前硬冲的赌博。

学习顺序同样关键。Unity 开发者转向 UE 时，最危险的不是学不会，而是太快找到表面对应关系，从而过早获得虚假的熟悉感。文档应反复强调，正确顺序应优先建立对象模型、资源引用、编辑器工作流、调试与分析工具、Gameplay Framework 和代码/蓝图边界，然后再逐步扩展到 UI、网络、物理、输入、特效和构建链路。只要顺序错了，团队就会在早期形成大量看似能跑、实则违背新引擎哲学的做法，后续返工代价极高。

#### 协同面、资产面与工具链：迁移永远不是程序单线任务
引擎迁移失败的另一个高频原因，是把它误判成程序组自己的技术升级。实际上，只要项目进入实战迁移阶段，美术、策划、技术美术、测试、构建发布和工具链都会被牵动。UI 不只是控件换一套；资源管理不只是目录重新组织；蓝图与 C++ 协作不只是脚本语言换个写法；构建发布不只是新打包工具；性能优化也不只是换个 Profiler。每一项迁移都会改变上下游的交付接口和验证方式。因此，文档必须把协同面显式写出来：哪些改动只影响程序内部，哪些会改变内容团队的工作流，哪些需要培训支持，哪些需要先补工具再迁移，哪些要同步建立回归样例。

资产与工具链尤其需要被正式化管理。新引擎中的资源引用、Cook、软硬引用、DataAsset、Primary Asset、蓝图继承链和自动化构建链路，都决定了项目能否在规模扩大后仍然维持稳定。迁移文档不能只写“在 UE 里对应哪个概念”，还必须要求团队回答：资源如何组织、引用如何约束、编辑器配置如何沉淀、自动构建与校验如何接入、历史资产如何批量迁移、团队如何确认迁移后内容生产效率并未明显倒退。只有当这些问题被纳入主线，迁移才真正拥有工业化落地条件。

#### 评审、风险清单与验收矩阵：把迁移从“专家经验”变成“团队流程”
迁移项目之所以容易依赖少数专家，是因为很多判断天然跨越多个系统。某个模块该先迁还是后迁，某个旧框架该保留还是拆掉，某项蓝图逻辑是否应该转回 C++，某个资源系统是否已经适应新管线，如果缺少统一评审问题，团队就会持续依赖“最懂的人拍板”。更稳定的做法，是把迁移评审问题写进文档：这项能力在新引擎中的原生承载点是什么；旧项目里的假设在新引擎里是否还成立；迁移后上下游工作方式如何变化；最小验证样例是什么；失败时如何回退；成功后如何验收；哪些风险需要先补工具或先建约束再迁移。持续围绕这些问题工作，团队才会逐步从“依赖专家直觉”转向“依赖共享流程”。

验收矩阵也必须足够具体。一个专题完成迁移，不应只以“功能表现看起来差不多”为标准，而应同时检查：行为一致性是否通过；资源/配置是否能被新流程稳定承载；性能与内存是否满足目标机预算；调试入口是否明确；编辑器和自动化链路是否可用；新人是否能按照文档接手；旧实现是否已经具备退出条件；相关岗位是否完成最小培训。只有当这些维度一起通过，迁移才算真正完成，而不是停留在 Demo 成功阶段。


#### 迁移回归矩阵与培训补充
迁移类文档如果要真正支撑团队落地，还应补一层回归矩阵与培训计划。也就是说，每个专题迁移完成后，不仅要验证功能是否能跑，还要验证旧功能行为、新引擎工作流、调试入口、资源流程和协作接口是否已经被团队真正接住。同时，新旧思维切换往往不是一次讲解就能完成，团队需要为对象模型、蓝图/C++ 边界、资源引用、调试分析工具和构建流程安排持续培训与示例沉淀。没有回归矩阵和培训闭环，迁移就很容易停留在“少数人已经会了”，而不是“团队整体已经稳定迁过去了”。


#### 迁移完成判定补充
迁移专题在项目里真正“完成”的标志，不应只是功能可运行，而应包括团队是否掌握了新的默认做法、旧路径是否具备退出条件、文档和示例是否足以支撑接手、以及相应回归是否已纳入常规流程。把这层完成判定写进去，能避免迁移长期停留在半新半旧、谁都不敢彻底切换的状态。


#### 文档治理与版本维护补充
一份技术文档是否真正成熟，最终并不只取决于它有没有把知识点讲明白，还取决于它能否在项目生命周期里持续发挥作用。也就是说，这份文档在版本推进、人员流动、需求变化、线上问题和技术债积累的过程中，是否仍然能被团队拿来指导判断、约束实现、解释问题和组织回归。很多团队的文档一开始写得并不差，后面之所以失效，往往不是知识本身过时，而是缺少文档治理意识：没有固定更新入口，没有明确责任人，没有版本相关的补充规则，没有把新问题和新例外回写进文档，最终导致文档逐渐变成“历史上某一次讨论的快照”，而不是项目的正式工程资产。

因此，无论主题更偏架构、性能、算法、数学、物理、资源、协作、迁移还是职业成长，这类文档都应当补一层共同的维护要求。第一，关键结论要能被回到真实项目中验证，而不是只停留在抽象建议。第二，文档应有最小责任归属，知道谁来更新、何时更新、什么变更会触发更新。第三，涉及高风险主题时，应当能指向对应的回归样例、排障入口或验收清单，而不是只告诉读者“这里很重要”。第四，历史妥协、临时方案和版本性例外不应被默默埋掉，而应被显式记录并定期审视。第五，文档要能服务接手和培训，让后来者不必依赖作者口头补充才能理解核心边界。

当这些治理要求被写进正式文档之后，知识才会真正转化为组织能力。团队也会逐步从“出了问题再找最懂的人”转向“先回文档、回基线、回约定，再定位责任和修正方案”。这不仅能提高技术判断的一致性，也能显著降低协作中的解释成本和重复犯错概率。从这个意义上说，技术文档的成熟，并不只是内容变长，而是它已经具备了被长期维护、被多人复用、被版本持续校准的能力。

### 总结
物理与碰撞系统迁移的核心，是重新接受 UE 对碰撞语义、查询机制、角色移动和物理模拟的组织方式，而不是寻找一组一对一替换关系。Unity 中 Layer、Trigger、Rigidbody 和 CharacterController 的经验仍有价值，但必须转化为 UE 的通道、Profile、Trace、Overlap、CharacterMovement、Chaos 和网络权威模型。

成熟的迁移结果应让团队能清楚解释每一次阻挡、触发、命中和查询：它使用什么通道，命中了什么对象，响应规则来自哪里，业务结果由谁决定，网络上谁是权威，性能成本是否可接受。只要碰撞矩阵、验证关卡、调试工具和跨系统协议先建立，UE 物理系统就能稳定支撑复杂玩法，而不是成为后期反复试错的黑箱。
