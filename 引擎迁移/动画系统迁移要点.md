# 动画系统迁移要点

## 摘要
动画系统从 Unity 迁移到 Unreal Engine 时，真正需要迁移的不是若干组件名称，也不是把 Animator Controller 的状态图照搬成 Animation Blueprint，而是角色表现架构、动画资产生产方式、运行时驱动边界、调试验收标准和团队协作流程的系统性重建。Unity 项目中常见的 Animator、脚本参数、状态机、Timeline、Animation Event 与第三方插件组合，在 UE 中通常需要拆解为 Animation Blueprint、State Machine、Blend Space、Anim Montage、Anim Notify、Control Rig、IK Rig、Retargeter、Gameplay Ability、Character Movement、Gameplay Tag、数据资产和编辑器工具链等多个协作单元。

本文围绕“动画职责如何划分、数据如何驱动、资产如何迁移、逻辑如何落点、调试如何闭环、团队如何工业化生产”展开，帮助团队避免把 UE 动画系统用成另一套低效脚本容器。迁移后的目标不是“看起来能播动画”，而是建立一套可复用、可追踪、可调试、可扩展、可验收的动画工程体系，使角色移动、战斗、技能、受击、交互、过场和多人同步都能在统一规范下稳定演进。

## 正文

### 背景
Unity 团队通常习惯以 Animator Controller 为核心组织动画表现。角色脚本负责计算速度、方向、是否在地面、是否攻击、是否受击、是否释放技能等参数；Animator 根据参数进入不同状态；Animation Event 在关键帧回调脚本；必要时再通过 Avatar Mask、Layer、Playable、Timeline、Cinemachine 或第三方动画插件补足复杂需求。这种模式在中小规模项目中上手快，状态与脚本之间的关系也比较直观，但当角色数量、武器类型、技能组合、网络同步、战斗判定和资产复用要求上升时，脚本层、状态机层和动画事件层很容易互相缠绕。

UE 的动画体系更强调职责分层和资产化协作。Animation Blueprint 负责每帧姿态求值与动画状态组织；Anim Instance 从角色或组件读取稳定输入；State Machine 处理连续状态切换；Blend Space 处理移动方向、速度、倾斜角、瞄准偏移等连续插值；Anim Montage 承载攻击、技能、换弹、交互、处决等一次性或可分段动作；Anim Notify 负责在动画时间线上发出表现事件或 Gameplay 事件；Control Rig、IK Rig 与 Retargeter 处理骨骼控制、重定向和运行时修正；Character Movement、Ability System 或业务 C++ 组件负责产生权威状态。迁移时如果仍按 Unity 习惯把所有业务判断塞进动画蓝图，短期内也许能快速复刻画面，但后续会在调试、复用、联网、性能和协作上付出很高成本。

因此，动画系统迁移的第一原则是先迁移“动画工程方法”，再迁移具体资产。团队需要明确：哪些状态属于 Gameplay 权威，哪些参数只是表现输入；哪些动作应由 Montage 触发，哪些应由状态机自然过渡；哪些事件可以留在 Anim Notify，哪些必须回到服务端或业务层校验；哪些骨骼调整属于资产问题，哪些属于运行时 IK 或 Control Rig 问题。只有这些边界清楚，动画蓝图才不会变成不可维护的巨型逻辑图。

### 核心内容

#### 一、迁移动画系统时应先重建职责边界，而不是先复刻 Animator
Unity Animator Controller 往往同时承担状态组织、参数判断、动画播放、事件回调和部分表现逻辑。UE 中应把这些职责拆开：Gameplay 层决定角色真实状态，Animation Blueprint 负责把这些状态翻译成姿态，Montage 负责一次性动作流程，Notify 负责时间点通知，资产管线负责动作来源和命名规范。职责拆得越清楚，后续加入新武器、新角色、新怪物、新技能时越不容易产生连锁修改。

推荐将动画相关职责划分为五层：

- 角色状态层：由 Character、Pawn、Movement Component、Ability、Combat Component 或状态组件维护，表达角色是否移动、跳跃、攻击、硬直、死亡、交互、被控制等权威状态。
- 动画输入层：由 Anim Instance 或中间数据结构读取并缓存速度、加速度、朝向差、地面状态、瞄准角、武器类型、姿态标签、动作标签等表现参数。
- 姿态求值层：由 Animation Blueprint、State Machine、Blend Space、Layered Blend、Aim Offset、Pose Cache、Linked Anim Graph 等机制生成最终骨骼姿态。
- 动作片段层：由 Montage、Section、Slot、Notify Window、Root Motion 策略组织攻击、技能、换弹、处决、采集、开门等短周期动作。
- 生产治理层：由命名规范、骨骼规范、导入规范、重定向规范、验收用例、性能预算和变更流程保证团队可持续生产。

这五层应保持单向依赖：动画蓝图可以读取权威状态，但不应反向决定关键业务事实；Notify 可以请求触发事件，但不能替代服务端判定；Montage 可以表达攻击窗口，但不能单独决定伤害是否成立。否则迁移后的系统会把 Unity 时代的脚本耦合换成 UE 蓝图耦合。

#### 二、Animation Blueprint 应承载姿态逻辑，不应承载完整业务逻辑
Animation Blueprint 的强项是姿态计算、状态切换、层级混合和编辑器可视化调试，而不是复杂业务编排。迁移时常见错误是把 Unity 脚本里的 `if`、冷却、技能条件、武器规则、Buff 判断全部搬进 Anim Graph 或 Event Graph。这样做会导致业务逻辑分散在角色 C++、蓝图、动画蓝图和 Notify 中，任何一次状态异常都需要跨多个图排查。

更合理的做法是让 Animation Blueprint 读取经过业务层整理后的最小输入。例如动画蓝图不需要知道“角色是否拥有某个 Buff 且技能冷却完成且目标距离小于阈值”，它只需要知道当前是否处于某个动作标签、武器姿态、移动模式或受控状态。复杂条件应在 Gameplay 层得出结论，再以稳定枚举、Gameplay Tag、布尔值或数据结构传给 Anim Instance。这样动画图可以保持可读，动画师也能理解状态切换的含义。

Animation Blueprint 的 Event Graph 也应保持轻量。它适合做每帧参数缓存、速度计算、角色引用校验、LOD 相关开关和少量调试输出，不适合执行重型查找、动态加载、复杂容器遍历或跨系统指令。高频 Tick 中的逻辑越复杂，动画线程优化越难发挥作用，性能问题也越难定位。

#### 三、State Machine 与 Montage 的边界必须在迁移初期定下来
Unity 团队容易把所有动画都放进状态机，因为 Animator Controller 本身就是最熟悉的组织方式。在 UE 中，连续、可循环、可被移动状态自然驱动的动作适合放入状态机，例如待机、行走、奔跑、跳跃、下落、落地、游泳、攀爬、骑乘、持武器姿态等。短周期、强触发、需要分段、需要中断、需要事件窗口的动作更适合由 Montage 承载，例如普攻、连击、技能释放、换弹、装填、交互、处决、受击、死亡、表情动作和剧情手势。

边界混乱会引发两个典型问题。第一，把攻击技能放进状态机，导致连招切段、取消窗口、攻击判定、受击打断和网络同步很难维护。第二，把基础移动也做成 Montage，导致角色移动表现与 Movement Component 脱节，Blend Space、速度插值和姿态修正能力被浪费。迁移时应对原 Unity Animator 的每个状态做分类：它是持续状态、一次性动作、覆盖层动作、上半身动作、过场动作，还是纯事件触发动作。分类结果决定迁移落点，而不是由原状态图形态直接决定。

Montage 迁移还要明确 Section 和 Slot 规范。连击类动作应通过 Section 表达动作阶段，通过业务层或 Ability 控制跳转；上半身射击、换弹或挥手应进入指定 Slot，并通过 Layered Blend 按骨骼层覆盖基础移动；全身硬直、死亡和处决应占用更高优先级 Slot 或切换到专用状态，避免与移动层互相抢权。没有 Slot 规范的 Montage 会在项目中快速变成“谁先播谁赢”的混乱系统。

#### 四、Root Motion 策略不能逐资源临时决定
Unity 项目可能使用脚本移动、动画位移、NavMeshAgent 位移或插件 Root Motion。迁移到 UE 后，团队必须明确哪些动作使用代码驱动位移，哪些动作使用 Root Motion，哪些动作使用 Motion Warping 或 Distance Matching 修正。Root Motion 不是美术偏好问题，而是移动权威、网络同步、碰撞一致性和技能手感问题。

基础移动通常建议由 Character Movement 或自定义 Movement Component 驱动，动画跟随速度与方向进行混合。攻击、闪避、冲刺、处决、攀爬等动作可根据玩法要求选择 Root Motion 或 Motion Warping，但必须有统一规则。例如服务端权威多人游戏中，Root Motion 的复制、预测、校正与打断策略必须提前验证；单机动作游戏中，可以更积极使用 Root Motion 保证动作质感，但仍需处理碰撞、斜坡、台阶和目标对齐。

迁移资产时应建立 Root Motion 审核表：资源是否包含根骨位移，根骨方向是否统一，导入后是否启用提取，Montage 是否允许 Root Motion，网络场景是否可复制，动作被打断时角色位置如何处理，碰撞胶囊是否与视觉位置一致。没有审核表时，团队常常在后期发现“动画看起来对，角色胶囊却不在正确位置”，这类问题修复成本很高。

#### 五、动画参数应来源清晰、命名稳定、更新时机一致
动画参数迁移不是简单把 Unity Animator 参数换成 UE 变量。每个参数都要回答三个问题：谁维护、何时更新、谁消费。速度、加速度、朝向差和是否在地面通常来自 Movement Component；武器类型、姿态、战斗状态来自装备或战斗组件；动作标签来自 Ability 或状态机；瞄准角可能来自控制器旋转、摄像机或目标锁定系统；受击方向来自战斗判定结果。

参数命名应避免业务含义漂移。例如 `IsAttacking` 看似简单，但在复杂项目中可能同时表示正在播放攻击动画、处于攻击输入窗口、伤害判定有效、不能移动、不能转向等多个含义。更稳妥的做法是拆成 `CurrentActionTag`、`bIsInAttackMontage`、`bCanComboInput`、`bDamageWindowActive`、`bMovementLocked` 等明确字段。动画蓝图只消费自己需要的字段，业务字段不应被表现层随意重定义。

更新时机也必须统一。Anim Instance 每帧从角色读取参数时，应先处理空引用、权威状态快照和网络模拟代理差异，再进入动画求值。不要在 Notify 中临时改动关键状态，也不要在多个蓝图 Tick 中重复计算同一参数。对于多人游戏，应区分本地预测参数、服务端确认参数和模拟代理插值参数，避免不同客户端看到完全不同的动作阶段。

#### 六、Anim Notify 应是时间轴事件，不应成为业务真相来源
Unity Animation Event 常被用来触发伤害、音效、特效、脚步声、武器显示、位移锁定等逻辑。迁移到 UE 后，Anim Notify 与 Notify State 仍然非常重要，但要区分表现事件与权威事件。脚步声、布料音、镜头震动、特效、武器 Trail、材质参数变化等可以由 Notify 直接触发表现；攻击判定、消耗资源、施加 Buff、生成投射物、开关无敌帧等则应由 Gameplay 层最终确认，Notify 只能作为时间点输入。

推荐把 Notify 分为三类：

- 表现型 Notify：只触发本地视觉、声音、镜头和 UI 反馈，不改变核心玩法结果。
- 请求型 Notify：向 Ability、Combat Component 或事件总线发出“到达某个动画窗口”的请求，由权威层判断是否执行。
- 窗口型 Notify State：表达连击输入窗口、伤害有效窗口、转向允许窗口、Root Motion 特殊窗口、霸体窗口等持续区间。

每类 Notify 都应有命名、参数和调试规范。攻击类 Notify 最好携带动作标识、阶段标识、Socket 名称、窗口 ID 或 Gameplay Tag，而不是只写一个泛化的 `DoDamage`。这样在日志、回放和调试器中可以追踪“哪个动画、哪个 Section、哪个窗口触发了什么请求”。

#### 七、Blend Space、Aim Offset 与分层混合要按表现问题设计
Unity 中常通过 Animator Blend Tree 处理移动混合。UE 的 Blend Space 与 Aim Offset 能力更细，但也更容易被滥用。迁移时应根据表现问题选择维度：基础地面移动常用速度和方向二维 Blend Space；侧身、锁定、持枪移动可能需要方向角、速度和姿态枚举组合；瞄准系统应使用 Aim Offset 或分层骨骼混合处理上半身；受伤、疲劳、负重等表现可以通过额外层或姿态 Additive 实现。

Blend Space 的输入必须稳定，否则角色会出现脚步抖动、方向跳变和姿态闪烁。方向角应有插值策略，速度应区分真实速度、期望速度和动画驱动速度，急停急转应决定是依赖状态机转身动画，还是依赖 Orientation Warping、Stride Warping、Distance Matching 等高级机制。迁移初期不要为了“看起来全能”建立过多维度的 Blend Space，维度越多，美术资产覆盖成本越高，调参空间越难维护。

分层混合要提前定义骨骼层级。上半身攻击、持枪瞄准、换弹和交互如果都各自定义骨骼过滤，很快会出现肩部断裂、手臂穿插和优先级冲突。建议建立项目级骨骼层模板，例如全身层、上半身层、右臂层、左臂层、头颈层、面部层、附加物层，并明确不同 Slot 对应哪些层级。动画师与程序都使用同一套层级语言，沟通成本会明显降低。

#### 八、重定向与骨骼规范是动画迁移的基础设施
从 Unity 迁移到 UE 时，很多团队会把注意力放在动画蓝图节点上，却低估骨骼命名、比例、根骨、姿态和重定向规范的重要性。原项目可能存在多套角色骨架、不同外包批次、不同 T-Pose 或 A-Pose、不同根骨方向、不同武器 Socket 命名。UE 对 Skeleton、Retargeter、IK Rig、Socket、Physics Asset 和 Control Rig 的依赖更强，基础规范不统一会直接影响资产复用效率。

迁移前应清点所有角色骨架并分类：主角骨架、通用人形骨架、大体型怪物骨架、四足骨架、飞行单位骨架、载具或机械骨架。每类骨架应确认根骨层级、单位比例、朝向、默认姿势、Socket 命名、虚拟骨骼、IK 骨骼、武器挂点和面部骨骼策略。对于人形角色，应优先建立标准 IK Rig 和 Retargeter，并用一批代表性动作验证走跑跳、攻击、转身、受击、死亡和交互。

重定向验收不能只看单帧姿势。至少要检查脚底滑动、膝盖方向、肩肘扭曲、手持武器位置、根骨位移、骨盆高度、头部朝向和极端动作。对比例差异大的角色，可能需要专用修正动画、Control Rig 约束或独立动作，而不能指望通用重定向完全解决。工业化迁移必须承认这一点，并在资源计划中预留修正成本。

#### 九、动画资产命名与目录结构必须支持规模化协作
动画资产一旦迁移到 UE，资产浏览器、引用关系、重定向、Cook、包体裁剪和批量检查都会依赖稳定命名。没有命名规范时，项目后期会出现大量 `Attack_01_New_Final`、`Run_Copy`、`Montage_Test`、`ABP_Character2` 之类资产，既影响搜索，也影响自动化审计和跨团队协作。

建议按角色类别、动作域、资源类型和变体组织命名。例如动画序列可采用 `A_角色或骨架_动作_方向_变体`，Montage 使用 `AM_角色或骨架_动作_用途`，Animation Blueprint 使用 `ABP_角色或骨架_层级`，Blend Space 使用 `BS_角色_移动模式`，Aim Offset 使用 `AO_角色_姿态`，Notify 使用语义化类名。目录层面应区分 `Characters`、`Common`、`Montages`、`BlendSpaces`、`Retarget`、`ControlRig`、`Tests`、`Deprecated` 等区域。

迁移时还应制定废弃策略。旧资产不能简单留在生产目录中，否则美术和策划很难判断哪些可用。建议为废弃资产设置专用目录、标记说明和删除窗口；为核心动作建立 owner；为公共动画蓝图和骨骼资产建立变更审批。动画资产不是“内容文件”这么简单，它们是运行时逻辑的一部分，必须按工程资产治理。

#### 十、动画蓝图复杂度需要预算和拆分策略
UE Animation Blueprint 很强，但图越大越难维护。迁移团队如果把所有移动、战斗、武器、受击、表情、骑乘、过场、特殊状态都塞进一个图，很快会出现状态线交叉、变量含义不清、调试耗时和性能不可控的问题。复杂角色应采用分层设计：基础 Locomotion、武器层、上半身动作层、特殊状态层、面部层、IK 修正层、后处理层分别有明确入口。

可以使用 Linked Anim Graph、Anim Layer Interface、Sub Anim Instance 或模块化蓝图方式拆分公共逻辑。多人形角色共享基础移动图，武器或职业通过层接口提供差异；怪物角色按类型共享基础骨架和状态结构；Boss 角色可拥有独立的技能 Montage 和特殊后处理，但仍遵循统一参数规范。拆分的目标不是为了“图更小”，而是让公共能力可复用、差异能力可隔离、调试路径可追踪。

复杂度预算应写入规范。例如单个动画蓝图中状态机数量、状态数量、高频变量数量、每帧 Cast 次数、动态查找次数、Pose Cache 使用规则、调试开关和 LOD 策略都应有建议。没有预算的图会随着需求自然膨胀，直到没人敢改。

#### 十一、网络同步下的动画迁移要关注权威状态与视觉预测
Unity 单机项目迁到 UE 多人项目时，动画系统往往暴露出大量隐藏假设。单机里可以直接在本地脚本触发攻击动画、伤害和位移；多人里必须区分服务端权威、拥有者客户端预测和其他客户端模拟显示。动画表现要顺滑，但不能让本地视觉结果伪造玩法事实。

攻击、技能、受击、死亡、换弹等动作应由权威状态驱动，并通过可靠的状态复制或 Ability 事件同步。拥有者客户端可以提前播放预测 Montage 提升手感，但必须处理服务端拒绝、阶段修正和打断回滚。模拟代理不应依赖本地输入，而应根据复制状态播放对应 Montage 或状态。Root Motion 在网络中尤其敏感，需要明确是否由服务端提取、是否允许客户端预测、校正时如何平滑。

动画 Notify 在多人中也要分级。本地脚步声和镜头震动可以客户端触发；投射物生成、伤害窗口和资源消耗必须由权威层处理；命中特效可以根据服务端命中结果或预测命中结果分别触发，并具备修正策略。迁移验收必须包含网络延迟、丢包、预测失败、角色加入中途、观战、回放等场景，否则上线后会出现“本地看已攻击，服务器未承认”的严重体验问题。

#### 十二、调试与可视化能力应作为迁移交付物
动画系统问题常常跨越资产、状态、代码、蓝图和输入。只靠肉眼看角色动作很难定位原因。迁移时应把调试能力作为正式交付物，而不是后期救火工具。UE 提供了 Animation Blueprint 调试、Montage 状态查看、Pose Watch、骨骼显示、Gameplay Debugger、Visual Logger、Network Profiler 等能力，团队应把它们纳入日常流程。

每个核心角色样例应提供调试视图，至少显示当前移动模式、动画状态、正在播放的 Montage、当前 Section、Slot 权重、Root Motion 状态、关键 Gameplay Tag、输入方向、速度、瞄准角、是否本地预测、是否服务端确认。对于攻击或技能，还应显示当前 Notify Window、可取消窗口、伤害窗口、连击输入缓存和打断原因。

日志也要语义化。不要只输出“Play Montage Failed”，而应输出角色 ID、动作 ID、Montage 名称、Section 名称、失败原因、当前状态、网络角色类型和请求来源。调试信息越接近领域语言，策划、动画、程序之间越容易共同排查问题。

#### 十三、性能优化要从动画架构阶段介入
动画系统的性能问题通常不是某一个节点造成，而是资源、骨骼、图复杂度、LOD、Tick 策略和可见性策略共同作用。迁移时应建立性能预算：每类角色允许的骨骼数量、同时可见数量、动画蓝图复杂度、Montage 层数、IK/Control Rig 使用范围、物理骨骼开关、更新频率、远距离简化规则等。

UE 中应合理使用 Update Rate Optimization、LOD、动画预算分配、可见性裁剪、Pose Cache、Linked Graph 拆分和 Skeletal Mesh Merge 等手段。近景主角可以保留高质量 IK、足底贴合和上半身层；远景 NPC 可以降低更新频率、关闭复杂后处理或使用简化骨骼；群体怪物应避免每个个体都运行完整复杂动画蓝图。

性能验收要绑定实际场景，而不是单个角色预览窗口。应在目标平台上测试多人同屏、战斗峰值、怪物群、过场镜头、低端设备和长时间运行。动画系统若不设预算，内容团队会持续增加层、Notify、曲线、IK 和物理效果，直到性能问题变成项目后期的系统性风险。

### 实现方案

#### 一、建立迁移清单并按动作语义分类
迁移前先梳理 Unity 项目中的 Animator Controller、动画片段、Avatar Mask、Animation Event、Timeline 片段、脚本参数、Root Motion 用法和第三方插件依赖。不要直接按文件夹搬运，而要按动作语义分类：基础移动、姿态切换、攻击技能、受击死亡、交互采集、武器操作、表情口型、剧情演出、特殊载具、怪物专属动作等。

每个动作条目应记录原文件、使用角色、触发条件、是否循环、是否 Root Motion、是否有事件、是否参与连击、是否需要网络同步、是否可被打断、是否有上半身覆盖、是否需要重定向。这个清单是后续决定 State Machine、Montage、Blend Space 和 Notify 的依据。没有清单就开始迁移，最终往往会遗漏隐藏事件或重复制作动作。

#### 二、先做一条完整角色样例链路
建议选择一个复杂度适中的核心角色作为样例，覆盖待机、移动、转向、跳跃、攻击、受击、死亡、交互和一个技能。样例的目标不是做出最终美术质量，而是验证职责边界、资产导入、骨骼规范、动画蓝图结构、Montage 流程、Notify 回传、Root Motion 策略、调试面板和性能预算。

样例稳定后，团队再把规范扩展到其他角色。这样可以避免每个角色各自探索一套做法。样例角色应成为“黄金路径”：新角色接入时必须能回答如何命名资源、如何接入 Anim Instance、如何播放 Montage、如何添加 Notify、如何配置 Slot、如何做重定向、如何跑验收用例。

#### 三、定义动画输入数据结构
为 Anim Instance 设计稳定输入结构，避免动画蓝图直接到处读取业务对象。输入结构可以包含移动速度、局部方向、加速度、是否在地面、是否跳跃、当前姿态、武器类型、当前动作标签、瞄准角、目标锁定状态、当前 Montage 状态、受击方向、网络角色类型等字段。

这些字段应由角色或组件统一写入，并在动画更新阶段读取。字段命名要表达事实，不要表达临时实现。对于复杂项目，可以把动画输入分成移动输入、战斗输入、装备输入、瞄准输入、网络输入和调试输入。这样动画蓝图读到的是稳定接口，而不是业务对象内部细节。

#### 四、重建基础 Locomotion 图
基础移动是所有角色表现的地基。迁移时应优先完成待机、起步、行走、奔跑、急停、转身、跳跃、下落、落地和坡面适配。根据项目类型决定是否引入 Orientation Warping、Stride Warping、Distance Matching、Turn In Place 和足底 IK。不要在基础移动尚未稳定时就大量接入技能和特殊动作，否则后续问题难以判断是移动层还是动作层造成。

Locomotion 图应有清晰状态：地面移动、空中、落地、特殊移动或受控移动。速度与方向输入应具备平滑策略，避免网络模拟代理抖动。移动层还要考虑武器姿态和锁定状态的差异，例如自由移动、锁定移动、持枪移动和潜行移动可能需要不同 Blend Space 或状态子图。

#### 五、建立 Montage 与 Slot 规范
为所有一次性动作建立 Montage 规范。普攻、连击、技能、换弹、交互、受击、死亡、表情和剧情动作应分别定义 Slot、优先级、可打断规则、Section 命名和 Notify 规则。业务层触发 Montage 时必须知道当前是否允许播放、是否需要停止已有 Montage、是否跳转 Section、是否锁定移动、是否允许转向。

Slot 规范应覆盖全身动作、上半身动作、右手动作、左手动作、头部表情和特殊覆盖层。不同 Slot 的冲突策略要明确，例如死亡和处决优先级高于攻击，受击可打断普通攻击但不一定打断霸体技能，换弹可以被受击打断但可能保留弹药状态。把这些规则写在规范中，比散落在蓝图节点里更可靠。

#### 六、迁移 Animation Event 到 Notify 体系
对原 Unity Animation Event 逐项迁移。脚步声、落地声、武器挥动音、特效挂点、镜头震动、布料或材质变化可迁移为表现型 Notify。攻击窗口、连击窗口、取消窗口、位移锁定、霸体窗口、投射物生成请求应迁移为请求型 Notify 或 Notify State，并由 Gameplay 层处理最终结果。

Notify 类应有统一命名、参数结构和日志输出。不要让每个动画师创建一套临时 Notify。项目级 Notify 库应由动画技术负责人维护，新增 Notify 必须说明用途、触发层级、是否网络相关、是否可在 Dedicated Server 执行、是否需要预测和回滚。

#### 七、制定 Root Motion 与 Motion Warping 验收流程
将所有含位移动作列为专项清单，确认其 Root Motion 策略。普通移动尽量保持 Movement Component 驱动；闪避、冲刺、处决、攀爬、翻越、跳劈等动作按玩法需求选择 Root Motion、代码位移或 Motion Warping。每种策略都要验证碰撞胶囊、斜坡、台阶、墙体、网络延迟和打断结果。

对于需要目标对齐的动作，例如处决、抓取、近战技能和交互，建议建立 Motion Warping 或对齐组件流程。动作开始前锁定目标点，播放期间根据窗口修正根运动，动作结束后把角色胶囊和视觉姿态校准到一致状态。不要用硬编码瞬移掩盖对齐问题，否则多人和回放会很难稳定。

#### 八、建立重定向资产标准
为主要骨架建立标准 Skeleton、IK Rig、Retargeter、Preview Mesh、Socket 和虚拟骨骼。迁移动画前先跑一组基准动作，包括走、跑、跳、转身、攻击、受击、死亡、持武器和交互。每个新角色接入时必须通过基准动作验收，确认脚底、肩肘、手持物、骨盆高度和根骨方向没有明显错误。

对外包或历史资产要建立导入检查项：单位是否正确，帧率是否统一，根骨是否干净，循环首尾是否连续，曲线是否保留，事件是否需要迁移，是否包含多余骨骼，是否符合命名规范。导入不是美术单点工作，而是动画工程链路的一部分。

#### 九、设计动画调试面板和验收关卡
为样例角色建立动画调试面板或 Gameplay Debugger 扩展，显示当前状态、参数、Montage、Section、Slot、Notify Window、Root Motion、网络角色类型和关键标签。再建立专用验收关卡，包含平地、斜坡、台阶、窄路、碰撞墙、移动平台、目标锁定、多人延迟和大量同屏场景。

每次迁移动画资产或修改动画蓝图后，都应在验收关卡跑固定检查。动画问题越早被发现，修复越便宜。特别是脚步滑动、上半身断层、攻击窗口偏移、受击打断异常、Root Motion 穿墙和网络模拟不同步，必须在迁移阶段就进入常规检查。

#### 十、把性能预算写入动画规范
为不同角色等级制定预算。主角可以使用高质量 IK、复杂层混合和更多 Notify；普通 NPC 需要限制骨骼数、层数和更新频率；远景群体单位需要简化图、降低 Tick 或使用替代显示策略。预算应包含目标平台、同屏数量、平均耗时、峰值耗时、内存占用和包体影响。

性能检查不应只由程序后期承担。动画师在新增复杂动作、曲线、物理骨骼或 Control Rig 时，也需要知道成本。技术动画应负责维护预算基线和分析工具，程序负责提供运行时指标，策划负责在玩法需求与性能限制之间做取舍。

#### 十一、建立动画变更评审机制
核心动画蓝图、骨骼、Slot 规范、Notify 类和 Root Motion 策略不应随意修改。建议为这些资产设置 owner，并建立变更评审流程。任何影响公共角色、公共骨架或公共动画层的修改，都应说明变更原因、影响范围、回滚方案和验收结果。

对高风险修改，例如更换骨架、调整根骨、重命名 Socket、修改公共 Anim Layer、删除 Notify、改变 Montage Section，应在分支中验证并通知相关团队。动画系统一旦工业化，很多资产之间存在隐性引用；没有评审机制会让迁移项目频繁出现“某个角色突然坏了但没人知道为什么”。

#### 十二、形成动画迁移验收标准
每篇迁移动画文档和每个角色样例都应落到可验收项。建议至少包含以下标准：

- 所有目标动画资产已按语义分类，并明确迁移落点。
- Animation Blueprint 只承载姿态与表现逻辑，核心业务状态由 Gameplay 层维护。
- State Machine、Montage、Slot、Notify、Root Motion 和重定向规则已有书面规范。
- 样例角色覆盖移动、攻击、受击、死亡、交互和至少一个复杂技能。
- 动画参数来源清晰，命名稳定，更新时机一致。
- 攻击窗口、连击窗口、取消窗口和受击打断可通过调试工具追踪。
- Root Motion、Motion Warping 和胶囊位置在目标场景中表现一致。
- 多人场景下拥有者、服务端和模拟代理的动画表现可解释、可修正。
- 核心动画资产有 owner，变更有记录，废弃资产可追踪。
- 目标平台上的动画性能满足预算，并有回归检查流程。

### 总结
动画系统迁移的核心，不是学会更多 UE 动画节点，而是接受 UE 对动画职责、资产协作和运行时边界的重新划分。Unity 时代习惯把大量表现与业务逻辑挤进 Animator 和脚本之间，迁移到 UE 后必须拆成角色状态、动画输入、姿态求值、动作片段和生产治理多个层级。只有边界清楚、参数稳定、Montage 规范、Notify 分级、Root Motion 策略明确、重定向基础扎实，动画系统才不会在项目扩大后失控。

优秀的迁移结果应当让动画师能安全生产动作，让策划能理解动作窗口，让程序能追踪状态来源，让测试能复现问题，让性能负责人能看到预算。也就是说，动画系统最终服务的不是单个角色“能动起来”，而是一整套角色表现工业化能力。只要团队从样例角色开始建立规范，再用验收关卡和调试工具持续约束，UE 动画体系就能从迁移风险转化为长期生产优势。
