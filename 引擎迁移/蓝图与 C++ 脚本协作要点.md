# 蓝图与 C++ 脚本协作要点

## 摘要
蓝图与 C++ 的协作边界，是 UE 迁移中最早出现分歧、也最容易长期失控的主题之一。本文从职责划分、扩展点设计、团队协作与维护成本四个角度，总结蓝图与 C++ 协作时应遵守的关键原则。

## 正文

### 背景
Unity 团队在进入 UE 后，常常会在两个极端间摇摆：要么过度依赖蓝图，把大量逻辑做成难维护的图；要么对蓝图缺乏信任，试图把所有东西都收进 C++。这两种做法都忽略了 UE 的设计初衷，即让蓝图与 C++ 形成稳定的协作分层，而不是彼此替代。

### 核心内容
协作边界的核心原则是：稳定基础能力放在 C++，内容侧可调参与组合能力放在蓝图。C++ 适合承担框架、性能敏感路径和统一扩展点；蓝图更适合承担内容编排、参数配置和可视化逻辑衔接。两者之间应通过清晰接口连接，而不是互相突破边界。

常见误区包括：把蓝图当成快速原型垃圾桶并长期保留；没有设计可暴露参数与扩展点；让 C++ 直接依赖过多具体蓝图实现；不同模块对边界理解完全不一致。边界一旦不统一，团队后续每个系统都会重新争论同一件事。

### 实现方案
建议团队尽快形成一套蓝图与 C++ 协作准则，至少明确：

- 哪类逻辑默认进入 C++。
- 哪类内容允许在蓝图扩展。
- 参数暴露、事件回调和数据资产如何设计。
- 评审时如何判断边界是否合理。

最好配合若干标准样例，让团队在角色、UI、交互和 AI 等不同系统中都看到相似的协作模式，而不是每个模块各自发明一套规则。

### 总结
蓝图与 C++ 协作的关键，不在于谁更强，而在于边界是否稳定。只要团队尽早统一规则、沉淀样例并坚持评审，蓝图与 C++ 就能形成高效互补；反之，它们只会成为迁移过程中最持续的摩擦源之一。
