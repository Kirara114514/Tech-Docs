# 网络与多人游戏迁移要点

## 摘要
多人项目从 Unity 迁移到 Unreal Engine 时，最容易犯的错误并不是“不会写 RPC”，而是把多人系统理解成“把单机逻辑同步出去”这么简单。许多团队在单机或伪联机阶段建立起来的直觉，会让他们下意识把网络问题看成状态广播、对象生成、远程调用和延迟处理的问题。到了 UE，这些当然仍然重要，但它们都只是更高层语义的表象。真正决定多人系统是否健康的，是 Authority、Ownership、Replication、Prediction、State Source 和 Gameplay Framework 如何共同构成一套稳定边界。

也正因为如此，多人迁移最危险的地方不是 API，而是语言。你如果没有建立 UE 的多人语言，就很容易在系统设计时不断问错问题：这个变量怎么同步？这个按钮怎么发 RPC？这个 Actor 为什么客户端上看不到？这些问题固然会出现，但它们往往只是更深层错误设计的结果。真正该先问的问题通常是：这段状态到底属于谁、谁有权决定、谁只负责表现、谁负责预测、谁负责回收、谁应该知道、谁不应该知道。

本文将重点讨论 Unity 团队迁移到 UE 多人框架时最需要完成的几次认知切换：为什么 Authority/Replication 不是工具细节而是世界观，为什么 PlayerController / PlayerState / GameMode / GameState 的分层会决定多人项目边界质量，为什么“客户端先做一点再同步”与“服务端权威 + 本地预测”不是一回事，以及怎样按可扩展、可调试、可回归的方式建设多人语义，而不是只把网络当成补丁层。

## 正文

### 背景
在 Unity 项目里，多人能力的来源常常差异很大。不同团队可能用不同网络库、不同同步模型、不同房间逻辑、不同状态管理方式。正因为工具链差异大，很多程序员会形成一种经验性思维：多人无非就是“把该同步的东西同步出去，把不一致修掉”。这套经验在某些框架下未必完全错误，但切到 UE 后，如果还用它来理解多人系统，很容易在项目中长期累积错误边界。

原因在于，UE 的多人框架并不只是“给你几个网络 API”，而是把多人语义深度嵌进了 Gameplay Framework 和对象生命周期里。一个系统在单机里能不能工作，和它在多人里是不是可持续维护，并不是同一回事。很多看似在单机里十分顺手的设计，一旦放到多人环境，就会出现以下问题：

- 客户端和服务端都能改同一份状态，最后谁才是权威没人说得清。
- 角色控制权、玩家身份、对局状态和可见表现被混在一个类里。
- UI 显示的是本地猜测、缓存还是权威状态，边界完全模糊。
- 行为和表现同步耦合在一起，出了问题很难定位是逻辑错还是复制错。
- 一切都能“先做出来”，但带宽、延迟、重连、回放、预测和调试问题越积越多。

这些问题在迁移初期通常不会全部爆发，因为功能量还小，玩家数量有限，系统边界也还没互相缠死。可一旦职业、技能、UI、AI、道具、结算、匹配和观战系统相互叠加，多人框架就会迅速决定整个项目的维护上限。

### 核心内容

#### 一、多人迁移第一步，不是学 RPC，而是建立 Authority 语言
这是所有迁移中最重要也最容易被跳过的一步。

很多 Unity 团队切到 UE 多人时，会先学：

- 变量如何复制。
- RPC 怎么写。
- Actor 如何生成和同步。
- 客户端怎么响应服务端结果。

这些当然重要，但如果没有先建立 Authority 语言，它们只会变成工具性补丁。所谓 Authority 语言，本质上是在问：

- 这个状态最终由谁说了算？
- 哪一端有权改变它？
- 其他端拿到的是权威结果、推导结果还是表现结果？
- 哪些对象本来就不该在所有端上拥有相同语义？

一旦这个问题不清楚，你后面所有的网络动作都会变得混乱。比如：

- “技能释放失败”到底应该由哪一端先决定？
- “位置修正”是服务器纠正客户端，还是客户端视觉平滑？
- “背包更新”是本地先展示还是等权威确认？
- “角色死亡”是动画先播还是权威状态先改？

这些都不是简单的“同步方式问题”，而是 Authority 设计问题。

#### 二、Replication 不是“自动同步开关”，而是状态传播协议
很多迁移者第一次接触 UE 复制系统时，会自然产生一种误解：某个 Actor 或变量只要勾上复制，后面就由引擎自动帮我保持一致。技术上，这个理解有一小部分没错；工程上，它会非常危险。

Replication 真正要解决的不是“自动同步”，而是“哪些状态以何种语义传播给谁”。这里面至少有几个层次必须明确：

- 这个状态是不是权威状态。
- 这个状态是否值得传播，还是应该让客户端本地推导。
- 这个状态传播给所有人，还是只传播给特定拥有者或相关观察者。
- 状态变化本身就足够了，还是还需要额外时序事件支持。
- 状态更新频率和时机如何影响带宽、延迟和表现一致性。

如果团队只是把 Replication 当成“同步按钮”，最终一定会出现两类问题：

- 不该同步的东西同步了，系统越来越重。
- 该清晰表达的语义没表达，系统表面一致、实际上逻辑脆弱。

因此，Replication 的设计不是“会不会写宏”，而是“是否知道哪些信息值得进入网络协议层”。

#### 三、多人项目的对象分层，在 UE 中比单机项目更不能乱
Unity 团队切到 UE 后，最容易忽视但最关键的多人好处之一，是 Gameplay Framework 给出了更明确的对象分工。尤其在多人项目里，以下几类对象的职责边界非常重要：

- `PlayerController`
- `Pawn / Character`
- `PlayerState`
- `GameMode`
- `GameState`

如果团队在迁移时不认真对待这组分工，很快就会把多人项目写成一个谁都在存状态、谁都在消费状态的混合体。

例如：

- 玩家个人身份、队伍、分数、背包权限等更适合放在 PlayerState 还是 Character？
- 对局规则、胜负条件和服务端控制逻辑为什么不该塞进任意 Actor？
- 全局比赛阶段为什么应该在 GameState 中表达，而不是让每个客户端自己猜？
- 玩家控制权为什么不能和实体状态完全捆绑？

这些边界如果在迁移初期不建立，后续多人问题不会表现成“写错了一个函数”，而会表现成“整个系统到处都像临时修补”。

#### 四、客户端预测不是“先做了再说”，而是一套严格受限的设计
很多 Unity 团队在延迟高一点的时候，会自然采用一种简单策略：客户端先做表现，服务器再慢慢同步回来。这种经验并不是完全不能用，但迁移到 UE 后，如果还把它理解成“预测”，就会很危险。

真正的客户端预测至少要回答：

- 哪些行为允许客户端先行。
- 客户端先行时，本地到底在预测什么。
- 权威结果回来后，如何校正。
- 校正后，表现层和状态层如何收口。
- 哪些系统根本不应该预测。

如果没有这些边界，你做的就不是预测，而是“让客户端先假装成功”。这种做法在小场景中也许看不出问题，但在战斗、技能、位移、命中、背包和 UI 状态上会持续制造错觉。

真正成熟的多人设计会非常克制地使用预测。预测不是为了让客户端随意先跑，而是为了在不牺牲权威前提下改善体验。

#### 五、UI 在多人项目里最容易暴露权威边界是否清楚
很多团队以为多人问题主要体现在角色、技能和同步变量上，实际上 UI 往往是最早把问题暴露出来的地方。因为玩家看见的第一层不一致，常常就体现在 UI：

- 血量条和真实状态不一致。
- 技能冷却本地已经转好了，但服务器不认。
- 背包图标先显示了物品，确认后又被打回。
- 结算倒计时和对局阶段不同步。
- 队友状态面板显示和真实网络状态延迟错位。

这些现象表面上看像 UI 问题，实际上往往说明团队没有定义清楚：

- 哪些状态是权威广播。
- 哪些状态是本地预测。
- 哪些状态只适合本地展示，不应该被当成真实结果。

所以，多人迁移时，UI 不能最后再补。它应当在网络语义设计阶段就被纳入考虑，否则表现层会不断放大底层边界问题。

#### 六、多人调试最容易失败的原因，是团队仍然在用单机思路查问题
很多迁移者都经历过这样的时刻：单机逻辑看起来没问题，客户端本地也能运行，日志也有输出，但结果一到联机环境就完全不对。这个时候最容易发生的坏事是，团队仍然试图用单机思路排查：

- 看某一端当前值是不是对。
- 在某个函数里打断点看有没有执行。
- 把状态改一改看看现象会不会变。

这些方法不是没用，但对于多人问题远远不够。多人调试必须同时回答：

- 哪一端先发生了状态变化？
- 哪个端才是权威源？
- 这次复制是没发生、发生晚了，还是本来就不该复制？
- 客户端看到的是权威状态、本地预测还是残留缓存？
- 当前这个 Actor 在不同端上拥有的语义是否一致？

只有当团队真正学会用这种方式提问，多人系统才会开始变得可调试。

#### 七、带宽与频率问题，不能在项目后期才开始补课
很多迁移团队在前期会因为功能验证优先，把“先同步出来再说”当作默认策略。等项目渐渐成型后，才发现：

- 状态太多。
- 更新太频。
- 广播范围太大。
- 客户端本地推导能力太弱。
- 整体带宽和延迟表现开始恶化。

这个问题的本质，不是“后期优化做得不够”，而是前期没有把网络成本当成系统边界的一部分。多人项目中，任何“值得同步”的状态都应该从一开始就被视为有成本的正式设计决策，而不是事后再清理的临时行为。

### 实现方案

#### 一、推荐的迁移顺序
多人迁移不建议一上来就把完整战斗系统、完整背包系统或完整匹配系统搬到 UE。更合理的顺序是：

##### 第一步：做最小权威样例
只验证以下问题：

- 服务端生成对象，客户端能否正确看到。
- 某个核心状态由服务端改变后，客户端如何观察。
- PlayerController / Character / PlayerState 的基本职责能否跑通。

这一阶段的重点不是玩法，而是 Authority 语言。

##### 第二步：做最小交互样例
例如：

- 玩家与场景中的一个可交互物体同步交互。
- 一个简单动作由客户端发起、服务端裁决、客户端表现更新。

目标是理清“输入意图 -> 服务端裁决 -> 状态传播 -> 表现反馈”这条最小链。

##### 第三步：做一个带失败分支的样例
多人系统最怕只测 happy path。建议尽早验证：

- 输入被拒绝怎么办。
- 对象已失效怎么办。
- 客户端本地先行但服务端不认怎么办。
- 某状态迟到或丢包时表现如何收口。

##### 第四步：再进入完整系统
只有这些最小链路稳了，才适合逐步接入：

- 技能系统
- 背包和道具
- 结算和对局阶段
- 观战
- 重连

否则越早堆完整系统，返工越大。

#### 二、推荐的边界划分
一个更健康的 UE 多人项目，建议至少明确下面这些边界：

- 输入层
  - 只表达玩家意图，不直接替代权威执行。

- 权威执行层
  - 明确哪些逻辑只在服务端决定。

- 状态传播层
  - 明确哪些状态需要复制、传播给谁、以什么频率传播。

- 表现层
  - 负责消费状态和做视觉反馈，不偷偷承担权威逻辑。

- UI 层
  - 明确本地显示与权威结果的关系。

只要这些层没有清晰分开，多人系统迟早会在复杂度上反噬项目。

#### 三、最小代码骨架示例
下面这个例子强调的不是完整实现，而是多人语义入口：客户端通过控制器表达意图，真正执行边界在服务端。

```cpp
UCLASS()
class AGamePlayerController : public APlayerController
{
    GENERATED_BODY()

public:
    UFUNCTION(Server, Reliable)
    void ServerTryInteract(AActor* TargetActor);
};
```

这个示例里最重要的不是 `Server` 宏，而是三个原则：

- 输入层表达的是“尝试交互”意图，而不是本地直接改状态。
- 真正执行边界位于权威端。
- 本地表现与权威结果之间必须通过正式状态链路收口。

如果团队能把这种最小语义坚持下去，后面的多人系统才有可能稳定。

#### 四、验收标准
多人迁移是否真的到位，不应只看“两个客户端能连起来”，而应至少满足：

- 团队能清楚解释 Authority、Ownership、Replication、Prediction 各自含义。
- PlayerController、PlayerState、GameMode、GameState 的职责没有混乱。
- 输入、状态、表现、UI 在多人语义下边界清楚。
- 带失败分支的最小样例能跑通，不只测 happy path。
- 多人问题的排查方式已经从单机函数视角升级到跨端状态链视角。
- 系统开始从一开始就考虑网络成本，而不是后期再想办法减同步。

做到这些，才说明团队真正进入了 UE 多人框架语义，而不是只会调用几组网络 API。

### 总结
网络与多人游戏迁移真正难的地方，不是你会不会写复制变量、会不会发 RPC，而是你是否愿意承认：多人项目首先是一套世界观，而不是一组技术接口。谁有权决定，谁只负责观察，哪些状态值得传播，哪些状态只应表现，哪些行为允许预测，哪些绝不能本地拍板，这些问题如果不被前置定义，再多的网络代码也只是在错误结构上补砖。

Unity 团队迁移到 UE 后，真正应该学习的不是“怎样在新引擎里重复旧多人习惯”，而是怎样接受 Gameplay Framework 与多人语义共同定义系统边界。当这一步完成时，你才会发现，UE 的多人框架不是“限制多”，而是终于给了团队一套能稳定沟通、稳定扩展、稳定调试的语言。

多人迁移的终点，不是项目“联网成功”，而是项目中以后每一项玩法、每一个 UI、每一个技能和每一次结算，都能在统一的权威语义下被设计、实现和验证。这才是多人迁移真正应该交付的能力。
