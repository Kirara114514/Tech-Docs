# 网络与多人游戏迁移要点

## 摘要
多人系统迁移是 Unity 转 Unreal Engine 中最容易低估复杂度的专题之一。Unity 团队可能使用过自研同步框架、Mirror、Photon、Netcode for GameObjects、FishNet、Socket 协议或完全服务端自定义方案，因此容易把 UE 网络系统理解为“另一套同步 API”。实际上，UE 的多人模型与 Gameplay Framework 深度绑定，`GameMode`、`GameState`、`PlayerState`、`Controller`、`Pawn`、`Actor`、`Component`、Replication、RPC、Ownership、Relevancy、Role、Dormancy、Prediction 和 Level Travel 共同构成了一套强约束架构。迁移时如果不先统一权威端、所有权、复制边界和生命周期模型，上层战斗、技能、背包、UI、AI、动画和关卡逻辑都会长期处在“局部能同步、整体很混乱”的状态。

本文围绕 UE 多人迁移的关键认知展开：谁有权修改状态，状态应该放在哪里，属性复制与 RPC 如何分工，客户端预测与服务端校正如何设计，玩家生命周期如何组织，网络性能如何预算，调试工具如何建设，跨系统如何协作。目标不是罗列网络接口，而是帮助团队建立可维护、可扩展、可调试、可性能治理的多人架构，使项目从单机思维或 Unity 旧同步框架平稳过渡到 UE 的权威复制模型。

## 正文

### 背景
许多 Unity 多人项目的网络架构由团队自选方案决定。有的项目采用服务端权威，客户端只提交输入；有的项目采用房主主机或客户端主导；有的项目把状态同步封装在网络对象组件里；有的项目通过 RPC、事件、属性同步和自定义消息混合完成同步；还有一些项目在单机原型后期才加入多人，导致本地逻辑与网络逻辑交错。迁移到 UE 后，团队如果沿用“每个对象自己同步自己”的习惯，很容易忽略 UE 对对象生命周期、所有权、复制频率和框架类职责的默认设计。

UE 的网络模型默认强调服务端权威。服务端拥有游戏真实状态，客户端提交请求和输入，服务端验证后改变权威状态，再通过复制和 RPC 把结果传播给相关客户端。`GameMode` 只存在于服务端，适合保存规则和流程；`GameState` 复制给所有客户端，适合保存全局可见状态；`PlayerState` 代表玩家跨 Pawn 的持久状态；`PlayerController` 承载玩家连接和输入控制；`Pawn` 或 `Character` 承载可被控制的实体；Actor 和 ActorComponent 根据复制配置进入网络同步体系。这些类不是命名风格，而是状态归属和生命周期的基础。

Unity 迁移中最常见的问题，是把单机对象的全部字段继续挂在角色上，然后通过 RPC 或 Replicated 变量局部修补。短期看功能能跑，长期会出现权限不清、重复同步、丢包异常、加入中途状态缺失、断线重连困难、观战状态错误、UI 读取错误和性能不可控。多人迁移必须先从架构层回答：哪些状态是权威状态，哪些只是本地表现；哪些对象由服务端生成，哪些由客户端本地创建；哪些字段需要新玩家加入时也能看到，哪些只是一瞬间事件；哪些请求允许客户端发起，哪些必须由服务器主动计算。

### 核心内容

#### 一、权威端模型是多人迁移的第一原则
多人迁移首先要重建“谁有权修改状态”的认知。UE 默认情况下，服务端才是游戏真实状态的权威来源。客户端可以输入移动、请求交互、请求释放技能或提交 UI 操作，但不应直接决定血量、资源、装备、任务进度、命中结果、掉落、比分和胜负。服务端验证请求后修改状态，再通过复制让客户端看到结果。

这种模式与很多 Unity 单机或弱联网项目差异很大。Unity 脚本里常见的 `TakeDamage`、`AddItem`、`PlaySkill`、`OpenDoor` 可能在本地直接执行；迁移到 UE 后，需要重新拆成客户端请求、服务端校验、权威状态修改、结果复制和本地表现反馈。任何没有明确权威端的状态，都会在多人环境中变成作弊风险、不同步风险或回放困难。

权威端模型并不意味着客户端没有主动性。为了手感，客户端可以做预测表现，例如本地先播放开火动画、移动反馈或 UI 冷却；但预测必须能被服务端结果修正。迁移时要为每类玩法定义权威级别：移动可预测但服务端校正，射击可本地表现但命中服务端确认，背包和经济必须服务端权威，纯 UI 动效可客户端本地处理。

#### 二、状态归属必须遵循 Gameplay Framework 职责
UE 多人系统中，状态放错位置比同步接口用错更危险。`GameMode` 只存在于服务端，不能让客户端 UI 直接依赖它；`GameState` 适合全局可见状态，例如比赛阶段、剩余时间、队伍比分、当前地图规则；`PlayerState` 适合玩家身份、昵称、队伍、分数、段位、跨死亡保留的资源；`PlayerController` 适合玩家连接、输入、客户端专属 UI 通信和拥有者 RPC；`Pawn` 适合当前被控制角色的生命、移动、技能执行、动画状态和临时战斗状态。

Unity 迁移团队常把所有字段放在角色对象上，因为原来一个 `PlayerController` 脚本可能同时管输入、角色、UI、背包和分数。UE 中这样会导致玩家死亡换 Pawn 后丢失持久状态，观战或重连时拿不到正确信息，非拥有者客户端无法读取应当复制的玩家信息。迁移时应把旧字段逐项分类：它属于比赛、玩家、连接、角色、装备、队伍、关卡，还是纯本地 UI。

状态归属还决定复制范围。所有人都需要看到的状态应放在可复制且相关性正确的对象上；只有拥有者需要看到的状态可以只复制给 Owner；只服务端使用的状态不应复制；临时表现事件不应长期存储。状态放对位置，后续 RPC 和属性复制自然会简化。

#### 三、属性复制和 RPC 是不同工具，不能互相替代
UE 提供属性复制和 RPC，两者用途不同。属性复制适合表达“状态最终是什么”，例如血量、比分、当前武器、门是否打开、任务阶段、当前动作标签。RPC 适合表达“发生了一个过程或请求”，例如客户端请求开火、服务端通知播放一次爆炸、拥有者收到交互失败提示。迁移时常见错误是用 RPC 同步所有状态，或用复制变量表达所有瞬时事件。

用 RPC 同步状态的问题是加入中途的客户端看不到历史 RPC，丢包或顺序问题也会让状态难以恢复。应该复制的长期状态，如果只通过 Multicast RPC 通知一次，新玩家加入时就无法知道门已经开了、Boss 已经半血、比赛已进入下一阶段。反过来，瞬时表现如果只靠复制布尔值，可能因为值未变化或复制频率错过而不触发。

迁移规则可以简化为：需要新客户端加入后仍能恢复的，用属性复制；只对一次表现有意义的，用 RPC 或 OnRep 触发；客户端想改变权威状态，发 Server RPC 请求；服务端想通知所有相关客户端播放一次表现，用 NetMulticast 或状态复制驱动表现；服务端只想通知拥有者，用 Client RPC。每次使用 RPC 都应回答可靠性、调用方向、调用者权限、目标客户端和失败处理。

#### 四、Ownership 决定客户端能否向服务端发起请求
UE 的 RPC 权限与 Actor Ownership 密切相关。客户端通常只能在自己拥有的 Actor 上发起 Server RPC，例如自己的 PlayerController、Pawn 或由其拥有的组件。Unity 团队如果不理解 Ownership，常会遇到“客户端调用 Server RPC 没有执行”或“请求必须绕很多对象”的问题。

迁移时应建立明确的请求入口。玩家输入和 UI 请求优先通过 PlayerController、Pawn 或拥有者组件发往服务端；服务端再转给权威系统处理。不要让客户端直接在不拥有的世界物体、敌人、掉落物或门上调用 Server RPC。对于交互对象，可以由玩家拥有对象发起“请求与目标交互”，服务端验证距离、权限、状态后修改目标对象。

Ownership 也影响复制可见性和 OwnerOnly 状态。背包详细内容、个人任务、私密冷却、局内提示等可以只复制给拥有者；全局比分、角色位置、公开生命值则应复制给相关玩家。迁移时要把“谁能发起请求”和“谁能看到状态”分开设计，二者都不能靠默认行为猜。

#### 五、角色移动迁移要理解预测、校正和输入压缩
UE `CharacterMovementComponent` 已经提供成熟的客户端预测、服务端校正和移动复制机制。Unity 项目迁移时，不应轻易用自研 Transform 同步替代它，除非项目有非常特殊的移动需求。普通角色移动应尽量接入 UE 移动组件，利用其预测和校正能力，再扩展冲刺、攀爬、飞行、游泳、根运动和特殊状态。

移动迁移的关键是区分输入、期望移动和权威位置。拥有者客户端根据输入本地预测移动，服务端验证并校正，模拟代理客户端通过复制结果平滑显示。若团队把位置每帧 RPC 给服务端，或直接复制 Transform 覆盖移动组件，会破坏 UE 内置预测，导致抖动、回弹、带宽浪费和作弊风险。

特殊移动要单独设计。闪避、冲刺、击退、Root Motion、平台移动、载具、同步交互、技能位移可能需要扩展 Movement Mode、Root Motion Source、Ability、Montage 或自定义移动组件。每种特殊移动都要验证高延迟、丢包、碰撞、打断和动画同步，不要只在本地 PIE 中验证。

#### 六、客户端预测与服务端校正要按玩法分级
并非所有系统都需要预测，也并非所有系统都能预测。高频、强手感系统适合预测，例如移动、开火表现、近战起手、技能按钮反馈和 UI 冷却显示。权威性强、低频或经济相关系统不应预测最终结果，例如抽卡、交易、背包结算、任务奖励、匹配结果和排行榜。

预测设计必须包含失败处理。客户端预测释放技能后，服务端可能因冷却、距离、资源、状态或目标无效拒绝请求；此时客户端要撤销表现、修正冷却、提示原因或平滑过渡。没有失败处理的预测，只是在本地制造假象。对于战斗系统，可以采用“预测表现，权威结算”的模式：本地先播放动画和音效，伤害、命中、Buff 和资源最终由服务端确认。

迁移时建议为每类玩法建立同步等级表：纯本地表现、OwnerOnly 复制、服务端权威无预测、客户端预测服务端校正、完全服务端驱动。同步等级明确后，程序、策划、动画和 UI 才知道哪些反馈可以立即显示，哪些必须等待确认。

#### 七、生命周期迁移要覆盖登录、生成、死亡、重连和换图
多人项目的问题常常出现在生命周期边界，而不是普通运行过程。UE 中玩家连接、登录、PostLogin、生成 PlayerState、生成 PlayerController、Possess Pawn、死亡、Respawn、Seamless Travel、断线重连和关卡切换都有特定顺序。Unity 迁移时如果只验证“两个玩家进房间能移动”，后续会在换图、观战、重生、重连和中途加入时暴露大量状态缺失。

迁移时应绘制玩家生命周期图。哪些数据在登录前由平台或账号系统提供，哪些由 GameMode 分配，哪些写入 PlayerState，哪些在 Pawn 生成时初始化，哪些在死亡时保留，哪些在换图时迁移，哪些在退出时清理，都要明确。特别是背包、队伍、分数、任务、角色选择和临时 Buff，必须决定它们跨 Pawn、跨关卡和跨连接的保留策略。

加入中途是检验复制模型的重要场景。新加入客户端应能通过当前复制状态恢复世界，而不是依赖历史 RPC。门的状态、Boss 阶段、地图目标、队伍比分、已拾取道具、活动阶段和当前任务都应有持久复制来源。若中途加入无法正确恢复，说明状态归属和复制方式存在问题。

#### 八、网络相关性、复制频率和休眠决定性能上限
UE 复制不是“标记 Replicated 就结束”。每个可复制 Actor 都会消耗带宽和 CPU，复制频率、相关性、优先级、Dormancy、Net Cull Distance、OwnerOnly 条件和 Replication Graph 都会影响性能。Unity 迁移团队如果按单机思维给大量对象开启复制，很快会遇到带宽爆炸、服务器帧耗时上升和客户端卡顿。

应按对象类型制定复制策略。玩家角色高频复制移动和关键战斗状态；远处 NPC 降低频率或只在相关时复制；静态交互物只在状态变化时复制并进入休眠；掉落物按距离和可见性复制；纯表现对象本地生成；大量投射物可根据玩法选择服务端生成并复制、客户端预测显示或命中结果同步。不是所有对象都需要完整网络实体。

大型多人或开放世界项目应尽早评估 Replication Graph 或自定义相关性策略。依赖默认相关性可能在小规模测试中正常，但在真实地图、真实人数和大量 Actor 下失控。性能预算应包含每秒复制 Actor 数、属性数量、RPC 数量、带宽、服务器网络耗时和客户端接收耗时。

#### 九、数据结构复制要关注粒度和差异
Unity 项目常通过自定义协议精细控制数据包。迁移到 UE 后，如果简单复制大型数组、复杂结构或整个背包对象，可能导致带宽浪费和状态抖动。UE 提供条件复制、OnRep、Fast Array Serializer 等机制，可用于高效同步动态列表和增量变化。

背包、装备、Buff、队伍成员、任务条目、排行榜局部数据、技能冷却列表等动态集合，应设计复制粒度。客户端是否需要完整列表，是否只 Owner 可见，是否需要增量，是否需要顺序，是否允许预测插入，是否需要可靠确认，都要提前决定。不要把整个 DataTable 或完整配置复制给客户端，配置应通过本地资源或登录数据管理。

OnRep 函数应只处理表现和本地缓存更新，不应再次修改权威状态。复杂 OnRep 逻辑容易造成递归、顺序依赖和客户端状态漂移。复制结构应尽量表达最终事实，表现层根据事实变化做动画、音效和 UI 更新。

#### 十、RPC 可靠性要谨慎使用
Reliable RPC 不是“更安全的 RPC”，而是有队列和拥塞风险的可靠传输。高频 Reliable RPC 可能导致网络队列堵塞，反而让后续关键消息延迟。迁移时不能把所有 RPC 都标记 Reliable。可靠性应由业务语义决定：登录、购买、关键交互请求、确认类消息需要可靠；高频输入、瞄准方向、临时表现、可被下一帧覆盖的信息通常不应 Reliable。

Server RPC 要做校验。客户端请求释放技能、拾取物品、开门、交易或进入区域，服务端都应检查距离、权限、冷却、状态、资源和目标有效性。不要因为 UE 提供 RPC 就信任客户端传来的结果。客户端可以告诉服务端“我想做什么”，不能告诉服务端“结果已经是什么”。

Multicast RPC 也要谨慎。它适合服务端通知相关客户端播放一次表现，但不适合恢复长期状态，也不适合大量高频事件。若一个事件只对附近玩家相关，应通过 Actor 相关性或局部消息控制范围；若只对拥有者相关，应使用 Client RPC 或 OwnerOnly 复制。

#### 十一、Gameplay Ability 与网络迁移要统一规则
如果项目采用 Gameplay Ability System 或自定义技能系统，网络规则必须在迁移早期统一。技能激活、预测、冷却、消耗、命中、Buff、标签、属性和动画 Montage 都可能涉及复制。GAS 提供了较完整的预测与复制模型，但团队仍需明确哪些 Ability 允许本地预测，哪些只允许服务端执行，哪些效果复制给所有人，哪些只给拥有者。

Unity 旧技能系统常把技能配置、释放条件、动画事件、伤害判定和表现混在一起。迁移到 UE 后，应拆成输入请求、Ability 激活、服务器校验、Montage 播放、Notify 窗口、命中检测、Gameplay Effect 应用和结果复制。这样技能网络行为才可控。

如果不使用 GAS，也应建立类似边界。技能请求入口、预测表现、权威结算、失败回滚、冷却同步和 UI 更新必须有统一模式。否则每个技能各写一套 RPC，会很快变成难以维护的网络脚本集合。

#### 十二、UI 不能直接依赖服务端专属对象
迁移中常见问题是客户端 UI 读取 `GameMode`、服务端组件或未复制字段，导致编辑器单机正常，多人客户端为空。UE 多人 UI 应读取客户端可见的复制状态、PlayerState、GameState、拥有者组件或本地 ViewModel。服务端专属规则只能通过复制摘要、Client RPC 或响应结果传给客户端。

UI 更新也应避免频繁 RPC。血量、弹药、冷却、比分、任务目标等适合通过复制状态或本地预测缓存驱动；一次性提示、交互失败原因、奖励弹窗可以通过 Client RPC 或消息系统。UI 只需要展示事实和反馈，不应成为网络状态的权威来源。

迁移时应建立 UI 数据源清单。每个界面字段来自哪里，是否 OwnerOnly，是否需要预测，是否支持加入中途，是否支持断线重连，都要明确。这样可以避免 UI 团队在蓝图中临时 Cast 到各种对象读取状态。

#### 十三、调试与日志必须网络语义化
多人 Bug 通常高度依赖角色类型、网络延迟、调用方向和时间顺序。普通日志如果不包含网络语义，很难定位问题。迁移时应建立网络调试规范：每条关键日志包含 World、NetMode、LocalRole、RemoteRole、Owner、Actor 名称、玩家 ID、请求 ID、时间戳和状态摘要。

UE 的 Network Profiler、Stat Net、Packet Lag/Loss 模拟、Replication Graph 调试、Actor 复制查看、日志类别和 Gameplay Debugger 都应纳入日常流程。测试不能只在本地 Listen Server 验证，还要覆盖 Dedicated Server、模拟延迟、丢包、多客户端、加入中途、重连和低帧率。

对 RPC 和复制问题，应能回答：调用发生在哪端，Actor 是否复制，客户端是否拥有该 Actor，Actor 是否 relevant，属性是否满足复制条件，是否进入 Dormancy，RPC 是否可靠队列阻塞，OnRep 是否执行，加入中途是否能恢复。调试工具的目标是把这些问题显性化。

#### 十四、安全与反作弊要从请求边界开始
多人迁移不能只关注同步效果，还要关注安全。客户端传来的位置、命中、资源、购买、拾取、任务完成、冷却结束、伤害数值都不应被直接信任。服务端应验证关键请求，并对异常频率、异常距离、异常状态、非法目标和非法参数记录日志。

反作弊不是后期外挂系统一个人的工作，网络架构本身要减少可作弊面。客户端只提交意图和输入，服务端计算结果；敏感数据不复制给不该知道的客户端；随机结果由服务端生成；关键经济数据不保存在客户端；调试命令和开发 RPC 不进入正式包。迁移早期若不建立这些边界，后期补安全会牵动大量玩法代码。

#### 十五、多人架构需要跨系统协议
网络系统会影响几乎所有模块。输入系统需要决定哪些输入本地预测、哪些发服务端；动画系统需要区分本地预测 Montage 与服务端确认 Montage；AI 通常服务端决策，客户端表现；物理系统需要决定哪些物体服务端权威、哪些只本地表现；资源系统要保证所有客户端加载同一版本资源；UI 系统要有复制数据源；音频和特效要决定本地触发还是网络事件触发。

迁移时应建立跨系统协议，而不是每个模块各自同步。协议可以包括：网络请求统一入口、状态复制命名规则、OwnerOnly 数据规范、RPC 使用规范、预测失败处理、OnRep 表现规则、Debug 信息格式、多人验收场景。跨系统协议越早建立，后续功能越少返工。

### 实现方案

#### 一、先完成最小多人样例
正式迁移复杂业务前，先完成一个最小多人样例。样例至少包含 Dedicated Server、两个客户端、玩家登录、PlayerState 初始化、Pawn 生成与 Possess、角色移动、一次交互请求、一个复制状态、一个 Client RPC、一个 Multicast 或 OnRep 表现、一次死亡重生和一次加入中途恢复。

样例的目的不是做玩法，而是验证权威模型和生命周期。团队应通过样例明确哪些状态放 GameState，哪些放 PlayerState，哪些放 Pawn，客户端如何发请求，服务端如何验证，复制如何触发 UI 和表现。样例稳定后，再把战斗、背包、任务和技能迁移到同一模式下。

#### 二、建立状态归属表
盘点 Unity 项目中的所有多人相关状态：角色血量、护盾、弹药、技能冷却、Buff、装备、背包、分数、队伍、任务、交互物、门、机关、掉落、Boss 阶段、地图目标、房间设置、聊天、观战状态。逐项标注权威端、存放对象、复制范围、是否持久、是否 OwnerOnly、是否支持中途加入、是否需要预测。

状态归属表是多人迁移的核心设计文档。没有这张表，团队会不断在代码里争论某个字段到底放哪。表建立后，代码审查也能检查状态是否按归属落地。

#### 三、定义 RPC 使用规范
为 Server RPC、Client RPC、NetMulticast 制定使用规则。Server RPC 只能表达客户端请求，参数必须可校验，服务端必须做权限和状态检查。Client RPC 用于只通知拥有者的结果或提示。Multicast 用于服务端通知相关客户端的一次性表现，不用于持久状态恢复。

每个 RPC 应说明是否 Reliable、调用频率、调用端、目标端、失败处理和日志字段。高频输入和可覆盖状态避免 Reliable；关键交互可 Reliable，但要防止重复请求。RPC 命名应表达方向和意图，例如 `ServerRequestInteract`、`ClientNotifyInteractFailed`、`MulticastPlayImpactEffect`。

#### 四、重构角色移动和特殊位移
优先接入 UE CharacterMovement 的标准复制，不要直接同步 Transform。对原 Unity 中的冲刺、闪避、击退、攀爬、飞行、载具和 Root Motion 位移逐项评估。能用 Movement Mode 表达的就进入移动组件，属于技能动作的由技能系统和 Root Motion Source 或 Montage 协作，纯表现位移不影响权威位置。

移动验收应在 Dedicated Server 下进行，并模拟延迟和丢包。检查拥有者手感、模拟代理平滑、服务端校正、碰撞、动画和打断。特殊位移必须验证预测失败和服务端拒绝后的表现修正。

#### 五、设计预测与校正策略
为高频玩法定义预测策略。移动使用内置预测；开火和近战可预测动画、音效和 UI，但命中服务端确认；技能释放可根据风险决定是否本地预测；背包、交易和经济不预测最终结果。每个预测行为都要有请求 ID 或可关联上下文，便于服务端结果回来后匹配。

预测失败要有用户体验方案。可以撤销效果、播放失败提示、回滚冷却、修正位置或进入短暂过渡。不要让客户端静默修正，让玩家感到“按钮没反应”或“突然被拉回”。

#### 六、迁移战斗和技能同步
战斗迁移应先建立一个标准攻击链路：客户端输入，拥有者对象发 Server RPC 或激活 Ability，服务端验证，服务端设置动作状态，客户端播放或校正 Montage，Notify 窗口请求判定，服务端结算伤害，复制血量和受击表现。然后再扩展到投射物、范围技能、Buff、召唤物和连击。

命中判定要明确权威。射击游戏可能需要服务端回溯或延迟补偿；动作游戏可能使用服务端范围检测；弹幕或投射物可能服务端生成权威弹体，客户端预测显示。迁移时不能把 Unity 本地命中逻辑直接搬过来，否则作弊和不同步风险很高。

#### 七、处理加入中途、重连和换图
建立加入中途测试用例。新客户端进入正在进行的局，应看到正确的比赛阶段、玩家分数、角色状态、地图目标、已打开的门、已拾取物、Boss 阶段和剩余时间。任何只能靠历史 RPC 恢复的状态都要改为复制属性或可查询状态。

断线重连要决定保留哪些 PlayerState 数据、Pawn 是否销毁、位置是否恢复、队伍和分数是否保留。换图要验证 Seamless Travel 或非 Seamless Travel 下的数据迁移。生命周期测试应成为多人回归的一部分。

#### 八、建立网络性能预算
为 Actor 数量、复制频率、RPC 数量、带宽和服务器网络耗时建立预算。不同对象类型使用不同策略：玩家高频，交互物低频并休眠，远处 NPC 降频，纯表现本地生成，大量临时对象避免复制。项目规模较大时尽早引入 Replication Graph 或自定义相关性。

每次新增可复制 Actor 或高频 RPC，都应评估预算。CI 或性能测试应记录固定场景下的网络指标，防止内容增长导致同步成本失控。

#### 九、建设网络调试工具
在开发环境中提供网络调试面板，显示本地 NetMode、角色 Role、Owner、关键复制状态、RPC 请求、延迟、丢包、预测状态、校正次数和带宽。对关键业务请求生成请求 ID，并在客户端、服务端和结果日志中串联。

测试人员应能一键开启延迟、丢包和多客户端场景。多人问题如果只能靠程序手动复现，迁移效率会很低。调试工具要服务团队，而不是只服务网络程序。

#### 十、形成多人迁移验收标准
网络与多人迁移完成应满足以下标准：

- 权威端模型清晰，客户端请求和服务端结果边界明确。
- GameMode、GameState、PlayerState、Controller、Pawn 的状态归属符合 UE 生命周期。
- 属性复制与 RPC 分工明确，持久状态不依赖历史 RPC。
- Ownership 规则清楚，客户端只通过拥有对象发起合法请求。
- 角色移动使用可预测、可校正、可调试的网络方案。
- 战斗、技能、交互、背包、UI 均有统一同步模式。
- 加入中途、重连、死亡重生、换图和观战场景可恢复正确状态。
- 网络相关性、复制频率、Dormancy、带宽和 RPC 数量有预算。
- Dedicated Server、延迟、丢包、多客户端下完成回归测试。
- 关键日志包含 NetMode、Role、Owner、请求 ID 和失败原因。
- 安全边界明确，客户端不能直接决定敏感结果。

### 总结
网络与多人游戏迁移的本质，是重新接受 UE 对状态所有权、对象生命周期和复制边界的约束。Unity 项目中可自由组织的同步逻辑，迁移到 UE 后必须落到服务端权威、Gameplay Framework 职责、属性复制、RPC、Ownership、预测校正和网络相关性这些基础概念上。只要这些认知先统一，具体玩法同步就能逐步落到合适层级；反之，系统会长期停留在“局部能同步，但整体不可解释”的状态。

成熟的多人迁移应从最小样例和状态归属表开始，再扩展到移动、战斗、技能、UI、AI 和资源加载。每个同步决策都应能说明权威端、复制范围、加入中途恢复方式、失败处理和性能成本。这样 UE 的网络框架才会成为项目的结构优势，而不是一组难以驾驭的接口。
