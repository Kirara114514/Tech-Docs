# 输入系统迁移要点

## 摘要
输入系统迁移的核心不是把按键映射重新配置一遍，而是重建“输入意图、控制权和上下文切换”的整体模型。本文从 Unity 到 UE 的迁移视角，总结输入系统最关键的心智变化与工程要点。

## 正文

### 背景
Unity 项目中的输入实现常常随着项目成长逐步演化，可能混合旧输入系统、新输入系统和大量自定义封装。进入 UE 后，Enhanced Input 等机制提供了更明确的输入映射与上下文管理方式。若团队只关注按键对应关系，而不重建输入架构，后续角色切换、UI 抢占和多人权限问题会集中爆发。

### 核心内容
输入迁移首先要区分输入设备事件、输入意图和业务动作。输入系统的职责应停留在采集与映射层，而不直接承担玩法结果。其次，要重新理解 UE 中 `Controller`、`Pawn`、UI 焦点和输入上下文的关系，它们共同决定了输入该在何处被解释和消费。

常见误区包括：把输入逻辑继续塞在角色脚本里；未明确 UI 与角色的输入抢占规则；低估上下文切换在技能、载具和菜单中的重要性；只迁移单机输入而未同步考虑网络权限。

### 实现方案
建议先以一个最小样例完成以下链路：

- 输入映射与上下文切换。
- 角色控制与 UI 焦点互斥。
- 输入意图如何传入 Gameplay 层。
- 多设备输入如何统一表达。

只有当这些边界在样例中稳定后，再去迁移更复杂的交互系统，才不会在后续版本反复返工输入架构。

### 总结
输入系统迁移的重点，从来不是按键表本身，而是重新定义输入在系统中的职责与流向。只有意图、上下文和控制权被正确建模，迁移后的输入系统才能真正支撑复杂玩法与多端交互。
