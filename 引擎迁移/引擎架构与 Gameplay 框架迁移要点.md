# 引擎架构与 Gameplay 框架迁移要点

## 摘要
从 Unity 迁移到 Unreal Engine 时，最容易被低估、但对项目成败影响最大的部分，往往不是单个模块功能，而是引擎架构与 Gameplay 框架的整体迁移。Unity 团队通常习惯于围绕 `Scene + GameObject + Component + MonoBehaviour` 组织业务逻辑，再通过若干 Manager、Service、事件系统与 ScriptableObject 补出工程层能力；而 UE 则天然提供了更明确也更强约束的运行时骨架，包括 `GameInstance`、`World`、`GameMode`、`GameState`、`PlayerController`、`PlayerState`、`Pawn/Character`、`Subsystem`、`ActorComponent`、`UObject` 以及围绕复制、反射、资产和编辑器协作形成的一整套机制。迁移的真正难点不是“把 Unity 类对应到 UE 类”，而是重建团队对职责分层、生命周期、联机边界、数据驱动和编辑器协作的系统认知。本文围绕对象模型、框架职责、系统分层、联机架构、代码与蓝图边界、团队落地策略和常见反模式，说明迁移时真正需要建立的工业化心智。

## 正文

### 背景
很多 Unity 项目早期架构其实并不复杂。团队常见的组织方式是：

- 一个启动场景承载全局单例。
- 游戏对象通过组件组合功能。
- 若干 `Manager` 负责输入、UI、音频、配置、战斗、存档等全局服务。
- 业务逻辑通过事件总线、直接引用或脚本回调串联。

这种模式的优点是灵活、上手快、低门槛。随着项目规模增大，它的成本也会逐渐显现：对象职责模糊、生命周期边界不清、联机时状态归属混乱、全局单例越来越多、调试和重构成本陡增。

UE 并不是简单提供另一套 API，而是给出一套更明确的运行时框架。它的优势在于：

- 每种职责有更清晰的宿主。
- 联机同步的边界天然进入框架设计。
- 编辑器、资产和运行时对象的关系更稳定。
- 可通过反射、Subsystem、组件、DataAsset、Blueprint 等机制把技术和内容协作串起来。

但这套优势只有在团队真正接受新框架分层时才会发挥出来。如果仍然试图在 UE 中复刻 Unity 时代“一个万能 GameManager + 一堆脚本驱动”的组织方式，最后得到的只会是一套名字变了、问题没变、排错更难的架构。

### 核心内容

### 迁移的第一原则是先重建对象模型
Unity 与 UE 的最大差异之一，不是语法，而是对象模型。Unity 项目里，团队经常把“组件挂在哪个物体上”视为主要组织方式；而在 UE 中，更应先问清楚：

- 这段状态属于整个应用、单个 World、单局游戏、单个玩家还是单个实体。
- 这段逻辑是否需要网络同步。
- 这段能力是否是编辑器和资产可见的。
- 这段功能应当以 Actor、Component、Subsystem、UObject 还是 DataAsset 形式存在。

如果对象模型不先厘清，后面无论怎么搬系统，都会出现“暂时能跑，但长期不可维护”的情况。

### Unity 常见架构习惯在 UE 中为什么容易出问题

#### 一、万能 Manager 习惯会和框架层冲突
Unity 项目中，一个 `BattleManager`、`UIManager`、`AudioManager`、`InputManager` 往往可以无处不在地被引用。进入 UE 后，如果继续这么做，就会与原生框架层产生重复职责：

- 原本属于 `GameInstance` 的跨局状态，又被单独 manager 保存一份。
- 原本应在 `PlayerController` 或 `PlayerState` 的玩家侧状态，被外部管理器代持。
- 原本适合 `WorldSubsystem` 的世界级服务，被做成全局单例。

最终系统不是没有功能，而是职责开始重叠，谁都能改、谁都说不清边界。

#### 二、生命周期错位会引发大量隐性 bug
Unity 团队常习惯用 `Awake/Start/OnEnable/Update/OnDisable/OnDestroy` 推导对象生命周期。UE 虽然也有初始化与 Tick，但对象创建、GC、反射、复制、关卡加载、Play In Editor、多 World 并存等行为更复杂。如果团队没有建立新的生命周期认知，就很容易出现：

- 在错误阶段访问尚未就绪的对象。
- 以为对象会“全局常驻”，实际上它跟随 World 被销毁。
- 把跨关卡状态放在局内对象上。
- 忘记某些逻辑既有服务端实例又有客户端实例。

这类问题在小样例中不明显，进入真实项目后会持续反噬。

#### 三、联机边界必须前置，而不是后补
很多 Unity 单机项目架构是先按单机方式组织，再在后期补网络。UE 的框架天然把联机语义放在更前面，例如 `GameMode` 只存在于服务端、`GameState` 用于同步公共状态、`PlayerController` 与 `PlayerState` 分别承担控制和玩家状态职责。迁移时如果不利用这些边界，而是仍然用“先放在全局管理器里，联网时再看怎么同步”的思路，后期重构成本会很高。

### 一套更合理的职责映射思路
迁移时不建议做“Unity 类到 UE 类的一一映射表”，更实用的做法是按职责分层：

#### 一、应用级状态
例如账号登录、全局配置缓存、跨局服务、平台 SDK 状态。  
这类内容通常更接近 `GameInstance` 或 `GameInstanceSubsystem`。

#### 二、World 级状态
例如场景级导航服务、环境查询、世界级玩法服务、局内资源调度。  
这类内容更接近 `WorldSubsystem` 或少量专职 Actor。

#### 三、对局规则级状态
例如胜负规则、刷怪节奏、局内模式逻辑、回合推进。  
服务端规则通常更适合 `GameMode`，需要同步给各端的公共状态则更适合 `GameState`。

#### 四、玩家级状态
例如输入控制、镜头控制、玩家公共属性、阵营、分数、背包摘要。  
控制与本地表现更接近 `PlayerController`，需同步的玩家状态更接近 `PlayerState`。

#### 五、实体级状态
例如角色移动、技能、受击、动画驱动、武器组件、交互组件。  
这类内容通常分布在 `Pawn/Character`、`ActorComponent` 和部分独立 `UObject` 能力对象中。

这个分层的好处是，团队讨论系统归属时不再停留于“哪个类名更像”，而是先判断职责在哪一层。

### Gameplay 框架迁移的重点不是功能齐全，而是边界稳定
一套好的框架不是把所有能力都做出来，而是明确哪些能力该放在哪里、如何扩展、如何调试、如何被内容团队接入。迁移时必须重点关注以下几组边界：

#### 一、规则边界
游戏规则是服务端权威、客户端预测，还是纯本地表现？哪些逻辑可以延迟一致，哪些必须强一致？这会直接影响系统落点。

#### 二、数据边界
配置数据、运行时状态、持久化数据、同步数据要分开。Unity 项目中这些常被混在 ScriptableObject、Prefab 默认值和运行时脚本字段里。迁移到 UE 后，应尽量把静态配置交给 DataAsset/表，把运行时状态留在合适的 Actor 或状态类里。

#### 三、扩展边界
哪些能力允许内容团队在 Blueprint 中扩展，哪些必须固定在 C++ 层。这个边界不清楚，后期要么程序被各种蓝图碎逻辑拖垮，要么内容团队完全失去自主性。

#### 四、编辑器边界
哪些内容应该通过编辑器工具批量生成、批量校验、批量整理，哪些不应让人手工维护。UE 的编辑器扩展能力比很多 Unity 项目更值得投入，如果不用起来，团队会错过很大的生产力收益。

### Subsystem 是迁移时最值得认真理解的能力之一
很多 Unity 团队在 UE 中最初会继续寻找“单例放哪”。如果只是为了找一个全局访问点，那几乎必然走偏。Subsystem 的价值不在于“方便全局取到”，而在于：

- 让服务有明确宿主和生命周期。
- 减少手写单例管理代码。
- 更自然地和引擎运行时阶段结合。
- 更容易建立工程级服务边界。

典型做法包括：

- 跨局服务放入 `GameInstanceSubsystem`。
- 局内世界服务放入 `WorldSubsystem`。
- 编辑器服务放入对应 Editor 模块或工具扩展，而不是混进运行时单例。

把 Subsystem 用好，往往能少掉大量“自制框架胶水层”。

### 蓝图与 C++ 的边界必须从架构阶段定义
很多团队不是不会用 Blueprint，而是不知道 Blueprint 应该负责到什么程度。迁移阶段建议直接定义三层边界：

#### 一、C++ 负责骨架与稳定规则
包括核心生命周期、网络边界、关键性能路径、底层服务、强约束系统和需要长期维护的框架代码。

#### 二、Blueprint 负责资源装配与受控扩展
包括表现拼装、内容参数配置、轻量流程组合、关卡触发和少量局部逻辑扩展。

#### 三、DataAsset/配置层负责静态内容
包括技能参数、角色成长、掉落表、资源映射、表现包和模块化配置。

这三层一旦混乱，就会出现非常典型的问题：底层规则被分散到蓝图节点里，运行时状态和静态配置混在一起，框架层无法做统一治理。

### 实现方案

### 推荐的框架分层
迁移时建议至少拆成以下几层：

#### 一、基础框架层
包含生命周期抽象、基础 Subsystem、事件分发、日志、配置访问、通用组件基类。这一层应尽量稳定、克制，不要急着做成“大而全引擎二封装”。

#### 二、玩法骨架层
围绕 `GameMode/GameState/PlayerController/PlayerState/Pawn` 建立项目级规范，明确每层放什么、不放什么。

#### 三、业务系统层
包括战斗、任务、AI、背包、交互、UI 驱动等系统。它们必须依赖玩法骨架，而不是反过来把骨架层污染成业务特例集合。

#### 四、内容配置层
通过 DataAsset、表、GameplayTag 等方式表达静态数据和事件语义，减少硬编码。

#### 五、表现扩展层
让动画、特效、音频、UI 和关卡逻辑通过 Blueprint 与配置层接入，而不是直接侵入底层框架。

这套分层的目标，不是追求学术意义上的优雅，而是让团队在未来两三年里还能持续改。

### 一个典型的落地判断方法
当团队面对某个新系统，例如“技能冷却、伤害结算与表现播放”时，可以按如下顺序判断归属：

1. 它是规则还是表现。
2. 它是否需要网络权威。
3. 它是否属于单个实体、单个玩家、单局世界还是跨局应用。
4. 它的数据是静态配置、运行时状态还是持久化数据。
5. 它是否需要开放给 Blueprint 或编辑器内容流。

只要坚持这样判断，很多“应该挂在哪”的争论都会自然收敛。

### 典型反模式与修正思路

#### 反模式一：用一个全局 GameplayManager 接管一切
这通常是 Unity 经验的惯性延续。短期上手快，长期会把生命周期、联机边界和职责分层全部打乱。修正方式不是把 manager 改名，而是按宿主重新拆职责。

#### 反模式二：所有东西都先放 Actor 上，后面再抽
Actor 很强，但不是所有逻辑都该挂 Actor。大量纯数据对象、配置对象、临时计算对象和跨实体服务硬塞 Actor，只会制造额外生命周期复杂度与性能负担。

#### 反模式三：把 Blueprint 当作“临时脚本区”
如果团队习惯于“先在蓝图里搭起来，以后再说”，最后通常会积累一堆没人敢动的脆弱逻辑。Blueprint 应该是受控扩展层，而不是架构兜底层。

#### 反模式四：忽视服务端与客户端视角差异
在 UE 中，很多对象在服务端和客户端的存在方式、职责和可见性都不同。如果团队仍然用单机心智理解问题，后期多人玩法开发会非常痛苦。

### 推荐的迁移节奏
Gameplay 框架迁移不宜一口气“搭完总框架”。更稳妥的节奏是：

1. 先挑一条最小主链路，打通玩家进入、控制、实体生成、规则驱动和基础表现。
2. 在这个闭环上验证框架宿主是否合理，例如玩家状态、世界服务、战斗实体分别落在哪。
3. 再逐步加入配置、联机、表现、工具和调试支持。
4. 最后把经验沉淀为项目规范，而不是继续靠最初那几个搭框架的人口口相传。

### 为什么“先封一层像 Unity 的接口”往往会适得其反
这几乎是所有迁移团队都会心动的一条路。表面上看，它能降低学习成本，让旧成员迅速上手；实际上它常常会带来三个长期问题：

#### 一、屏蔽了真正重要的框架差异
团队只学会了怎么调用一层自制接口，却没有理解 `GameMode`、`GameState`、`PlayerState`、`Subsystem`、反射和复制机制的原生职责。后续一旦遇到复杂问题，所有人都会重新掉回“这层封装下面到底发生了什么”的黑盒焦虑。

#### 二、把临时适配层做成了永久债务层
很多所谓“过渡封装”最后都没有被拆掉，反而不断吸纳新需求，变成项目里最难改的一层。它既不像 Unity 原生，也没有真正贴合 UE 原生，长期只能靠少数作者维护。

#### 三、新成员学到的是错误心智
如果团队一开始就被训练成“在 UE 里也按 Unity 老习惯想问题”，后面每次做框架决策都会天然回避 UE 提供的更优解。

更稳妥的做法，是在少量高频入口上提供迁移期辅助封装，但绝不隐藏原生职责边界。

### 一套可操作的项目级框架治理规则
为了防止架构在中后期重新失控，建议至少固化以下规则：

- 任何新系统在立项时必须明确宿主层级，是应用级、世界级、对局级、玩家级还是实体级。
- 任何需要网络同步的状态必须说明权威来源、复制路径和调试入口。
- 任何 Blueprint 扩展点都要定义“允许改什么、不允许改什么”。
- 任何跨模块公共服务优先考虑 Subsystem，而不是新增全局单例。
- 任何静态配置都优先进入 DataAsset 或表层，不允许长期寄生在角色默认实例上。

这些规则看起来像流程，但本质是在用制度保护框架边界。

### 框架验收的真正标准
引擎架构与 Gameplay 框架迁移是否到位，不应只看“功能能不能做”，而应看：

- 新系统接入时，团队是否能快速判断宿主与边界。
- 联机逻辑是否在框架层就具备清晰落点。
- 内容团队是否能在不破坏底层结构的前提下扩展功能。
- 调试时能否顺着生命周期和宿主关系定位问题。
- 一年后新成员加入，是否仍然能读懂并稳定扩展。

如果这些问题的答案还是“主要看谁写的”，那说明框架迁移仍未完成。

### 总结
引擎架构与 Gameplay 框架迁移，迁移的不是类名，而是团队组织玩法系统的方式。只有当对象模型、生命周期、联机边界、数据层次、扩展策略和编辑器协作全部形成清晰分工，UE 的框架优势才会真正转化为长期生产力。否则项目只是从 Unity 的自由生长，搬到了 UE 的强能力环境里继续野蛮生长，问题不会减少，只会更难察觉。
