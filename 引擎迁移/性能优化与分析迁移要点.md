# 性能优化与分析迁移要点

## 摘要
引擎迁移之后，性能优化的方法论也必须随之迁移。Unity 团队熟悉的 Profiler、Frame Debugger、Memory Profiler、Addressables 分析、Draw Call、GC、Script Update 和渲染批次经验，不能直接套用到 Unreal Engine 的线程模型、渲染管线、资源加载、内存管理、Cook 产物和工具链中。UE 中的 Game Thread、Render Thread、RHI Thread、GPU、Task Graph、Async Loading、Animation、Physics、Slate/UMG、Niagara、Streaming、Shader Compilation 和网络复制都有独立指标与诊断入口，性能问题需要通过可重复场景、明确预算、工具采样和回归机制定位，而不是凭体感或旧经验判断。

本文围绕 UE 迁移过程中的性能分析体系展开，重点说明如何建立基线场景、如何使用 Stat 命令与 Unreal Insights、如何区分 CPU/GPU/内存/加载/网络瓶颈、如何设计平台预算、如何避免过早优化和错误归因、如何把性能检查纳入 CI 与内容生产流程。迁移目标不是学习几条零散优化技巧，而是重建团队“看见问题、解释问题、验证收益、持续防退化”的能力。

## 正文

### 背景
Unity 团队通常已经熟悉自己的性能诊断路径：查看 Profiler 中脚本耗时、渲染耗时、GC Alloc、Draw Call、Batch、内存快照和资源加载尖峰；遇到卡顿时检查 Update、协程、Instantiate、材质实例、图集、粒子和 AssetBundle 加载。迁移到 UE 后，很多概念仍然存在对应方向，但表现路径、工具入口和优先级会发生变化。比如一次卡顿可能来自异步加载阻塞、Shader 编译、Texture Streaming、蓝图 Tick、UMG 失效、动画图求值、Chaos 物理、Niagara 粒子、RHI 提交或 GPU Pass，而不是 Unity 时代最熟悉的脚本 Update。

UE 性能分析更强调多线程视角和 Trace 数据。`stat unit` 可以快速观察 Game、Draw、GPU；`stat game`、`stat slate`、`stat anim`、`stat physics`、`stat net` 等命令提供专项统计；Unreal Insights 可以记录 CPU 任务、加载、内存、网络和资产事件；GPU Visualizer、RenderDoc 或平台图形工具帮助定位 GPU Pass；MemReport、LLM、Asset Audit、Size Map、包体报告用于分析内存和资源。迁移团队必须先掌握这些工具的语义，再谈优化策略。

性能迁移最常见的错误，是在没有基线和证据的情况下直接改代码或改资源。团队看到帧率低就压缩贴图，看到卡顿就删 Tick，看到包体大就清资源，看到 GPU 高就关后处理，但没有确认真正瓶颈。结果是做了很多修改，却无法说明收益来自哪里，也无法防止后续退化。性能优化应从指标和复现场景开始，任何优化动作都要有前后数据。

### 核心内容

#### 一、先建立性能基线，再判断迁移成本
性能基线是迁移决策的起点。没有基线，团队无法判断 UE 版本相比 Unity 版本是变好还是变差，也无法判断某次重构是否带来收益。基线应包含目标平台、目标分辨率、画质等级、典型场景、玩家数量、NPC 数量、特效数量、UI 状态、网络条件和测试时长。

基线场景不应只有空场景或编辑器预览。至少应包含启动进入、主界面、普通战斗、峰值战斗、大型关卡、资源加载、UI 高频操作、多人场景和长时间运行。每个场景记录平均帧率、P95/P99 帧时间、Game Thread、Render Thread、GPU、内存、显存、加载耗时、卡顿次数、网络带宽和包体体积。不同项目可增加自己的关键指标，例如 VR 帧时间、移动端温度、主机认证指标或开放世界流式加载指标。

迁移初期可以先建立粗基线，再逐步精细化。重要的是让团队形成习惯：性能讨论必须引用场景、平台、版本和数据，而不是“我感觉卡”。基线数据应归档，便于后续对比。

#### 二、CPU、GPU、内存、加载和网络瓶颈必须分开诊断
性能问题不能只用帧率描述。帧率低可能是 Game Thread 高、Render Thread 高、GPU 高、RHI 提交高、等待同步、加载阻塞、内存换页或网络延迟。第一步应使用 `stat unit`、Insights 和平台工具判断主瓶颈归属，再进入专项分析。

Game Thread 瓶颈通常与 Actor Tick、蓝图逻辑、AI、动画更新、物理、UMG、任务调度、对象生成销毁和游戏规则有关。Render Thread 或 RHI 瓶颈可能与渲染命令提交、材质状态切换、实例化、阴影、后处理和 Niagara 有关。GPU 瓶颈可能来自分辨率、光照、阴影、透明、后处理、Nanite、Lumen、Virtual Shadow Maps、粒子和复杂材质。内存问题可能来自纹理、网格、动画、音频、蓝图类、对象泄漏和 Streaming 配置。加载问题可能来自同步加载、软引用策略、包体组织、Shader 和 IO。

诊断顺序应固定：先确定瓶颈类型，再定位子系统，再找到具体对象或资源，再验证优化。跳过这个顺序，容易把 GPU 问题当成逻辑问题，把加载问题当成渲染问题，把网络延迟当成本地卡顿。

#### 三、Unreal Insights 应成为性能分析主工具
Unreal Insights 是 UE 性能分析的核心工具之一。它可以记录 CPU 线程、任务、定时器、加载事件、网络事件、内存和自定义 Trace。迁移团队应尽早建立采样流程：如何启动 Trace，采样哪些通道，如何命名文件，如何标记测试场景，如何分析 Game Thread、Render Thread、Async Loading 和 Task Graph。

Insights 的价值在于时间线。很多卡顿不是平均耗时高，而是偶发 Hitch。通过时间线可以看到某一帧中哪些任务串行阻塞、哪些异步加载回到主线程、哪些蓝图或动画任务耗时异常、哪些资源加载触发 IO。Unity 时代习惯看聚合 Profiler 的团队，需要适应 UE 中“按时间线复盘”的方法。

项目还应添加自定义统计或 Trace 标记。关键系统如资源加载、战斗结算、UI 打开、技能释放、关卡流式、AI 群体更新、网络消息处理，都可以增加作用域统计。这样在 Insights 中能以领域语言定位问题，而不是只看到泛化函数名。

#### 四、Stat 命令用于快速定位，不能替代完整分析
UE 的 Stat 命令适合快速观察运行时指标。`stat unit` 看线程瓶颈，`stat fps` 看帧率，`stat game` 看游戏线程分类，`stat slate` 看 UI，`stat anim` 看动画，`stat physics` 看物理，`stat rhi` 看渲染资源，`stat gpu` 或 GPU Visualizer 看 GPU Pass，`stat net` 看网络。迁移团队应整理一套常用命令清单，并规定在问题单中附带哪些截图或日志。

Stat 命令的局限是它们通常给出当前或聚合视图，不一定能解释偶发卡顿和跨线程依赖。因此快速定位后应进入 Insights、GPU 工具、内存报告或日志复盘。不要只凭一张 `stat unit` 截图就下结论。

Stat 结果还要结合运行环境。编辑器模式、PIE、Standalone、Development 包、Shipping 包、Dedicated Server、移动真机、主机 DevKit 的性能差异很大。正式性能结论应基于目标平台包，而不是编辑器窗口。

#### 五、蓝图和 Tick 优化要先看调用规模
Unity 团队迁移到 UE 后，容易把蓝图性能问题简单归因于“蓝图慢”。蓝图确实需要谨慎使用，但真正的问题通常是高频 Tick、过多 Actor、重复 Cast、动态查找、同步加载、UMG 绑定、复杂循环和无预算的事件广播。优化时应先看调用规模和热点，而不是一刀切改 C++。

蓝图适合组织表现、配置、编辑器协作和低频逻辑；高频核心逻辑、复杂算法、大量对象更新、网络权威逻辑和性能敏感路径应考虑 C++ 或更明确的数据驱动方案。迁移时应建立蓝图使用边界：哪些系统允许 Tick，Tick 频率是多少，哪些函数禁止在 Tick 中调用，哪些资源加载必须异步，哪些 Cast 需要缓存，哪些 UI 更新必须事件驱动。

不要为了“工业化”把所有蓝图改成 C++。没有数据证明的重写可能浪费时间。更专业的做法是用 Insights 和统计找出热点，再决定是减少调用次数、改事件驱动、缓存引用、拆分对象、降低频率，还是迁移到 C++。

#### 六、UI 性能要关注 Slate/UMG 的失效和重建
Unity UI 优化经验不能直接套到 UMG。UE UI 性能问题常与 Slate Tick、Widget 重建、绑定函数、布局失效、动画、材质、Retainer、Invalidation、列表虚拟化、纹理加载和过度层级有关。迁移时如果把 Unity 中的 Canvas 和 Prefab 思路直接搬到 UMG，很容易得到层级复杂、频繁刷新、打开页面卡顿的 UI。

UI 性能分析应使用 `stat slate`、Widget Reflector、Insights 和资源加载日志。重点检查页面打开时是否同步加载大量资源，列表是否创建过多条目，属性绑定是否每帧执行，动画是否导致整树失效，Retainer 使用是否合理，纹理是否进入正确资源组，关闭页面后引用是否释放。

UI 优化原则是事件驱动、数据快照、按需加载、虚拟化列表、减少全局失效、拆分重型页面和建立页面预算。每个核心页面应有打开耗时、峰值内存、Widget 数量、Draw Element 数量和资源加载预算。

#### 七、渲染性能要以目标平台和画质策略为前提
UE 渲染功能强大，但不能默认全开。Lumen、Nanite、Virtual Shadow Maps、后处理、透明材质、粒子、动态光源、反射、体积雾、抗锯齿和高分辨率纹理都会影响 GPU。迁移时不能只在高端 PC 上验证画面，再期望移动端或主机低档自动适配。

渲染性能分析应先确定目标平台、分辨率、帧率、画质档位和场景复杂度。GPU Visualizer、RenderDoc、平台 GPU 工具和 `stat gpu` 可用于定位 Pass。需要区分 Base Pass、Shadow、Lighting、Translucency、PostProcess、Niagara、UI、Reflection、Nanite、Lumen 等成本。美术效果上线前应有预算，而不是后期由程序逐个关闭。

材质复杂度和 Shader 变体是迁移重点。Unity 中的 Shader Graph 或手写 Shader 迁到 UE 材质系统后，可能产生不同指令成本和变体规模。材质函数、静态开关、透明、像素深度偏移、世界位置偏移、复杂遮罩和多层材质都要有使用规范。

#### 八、内存与资源加载要结合 Asset Manager 和 Streaming
UE 内存问题通常与资源引用、同步加载、对象生命周期、纹理 Streaming、关卡 Streaming、蓝图类引用、音频和特效池有关。迁移时如果硬引用泛滥或软引用未治理，内存会快速失控。性能分析必须与资源管线联动。

内存分析可以使用 MemReport、LLM、Insights Memory、Asset Audit、Size Map 和平台工具。重点关注常驻纹理、材质、骨骼网格、动画、音频、Widget、蓝图类、关卡和对象数量。应区分包体体积、磁盘占用、运行内存和显存，它们不是同一指标。

加载优化要避免同步加载进入关键路径。打开 UI、切换场景、生成角色、播放技能和进入战斗时，如果临时同步加载资源，会造成明显卡顿。应通过 Asset Manager、预加载、异步加载、资源池和加载进度管理提前处理。Streaming 策略要与地图、角色、活动和热更新组织一致。

#### 九、动画、物理、AI 和 Niagara 都需要专项预算
性能迁移不能只看渲染和脚本。动画系统可能因为复杂 Animation Blueprint、IK、Control Rig、Pose Cache 使用不当、骨骼数量、Montage 层数和 Update Rate 问题成为热点。物理系统可能因为大量 Trace、Overlap、刚体模拟、布娃娃和约束耗时。AI 系统可能因为感知、行为树 Service、EQS 和路径请求产生峰值。Niagara 可能因为粒子数量、碰撞、光照、透明和 GPU/CPU 模拟成本影响帧率。

每个系统应建立专项预算和降级策略。远处角色降低动画更新频率，普通 NPC 降低 AI Service 频率，群体粒子使用 LOD，物理道具及时休眠，EQS 限制查询频率，高成本特效限制同屏数量。专项预算要与内容生产工具结合，否则内容团队不知道自己的资产成本。

专项分析还要建立固定复现场景。例如动画压力场景、AI 群体场景、物理破碎场景、Niagara 峰值场景、UI 列表场景和多人网络场景。只有场景固定，优化收益才可比较。

#### 十、Hitch 分析要关注加载、GC、Shader 和对象生成
帧率平均值达标不代表体验稳定。UE 项目中 Hitch 可能来自同步加载、Shader 编译或预热不足、对象大量生成销毁、纹理 Streaming、关卡流式、异步加载回主线程、日志峰值、蓝图类首次加载、Niagara 或音频资源首次实例化。迁移时应特别关注 P95/P99 帧时间和卡顿次数。

Hitch 分析应使用 Insights 时间线和加载事件，结合资源日志和场景操作复现。常见策略包括启动或加载阶段预热关键资源，避免战斗中首次加载技能特效，使用对象池，分帧生成，异步加载并显示进度，控制关卡 Streaming 触发距离，提前编译或缓存 Shader，避免运行时创建大量动态材质。

卡顿优化必须验证真实操作路径。编辑器中手动打开资源预热后再测试，会掩盖首次加载问题。应使用干净客户端、清缓存或标准测试流程验证。

#### 十一、性能预算要与画质、玩法和平台共同制定
性能预算不是程序单方面制定。画面目标、玩法规模、平台范围、商业包体和用户体验共同决定预算。30 FPS、60 FPS、90 FPS VR、120 FPS 竞技游戏的预算完全不同。移动端、主机、PC、低端设备也有不同限制。迁移时应由程序、TA、美术、策划和产品共同确认每个平台的目标。

预算应拆到系统和内容：CPU 每帧预算、GPU 每 Pass 预算、内存预算、显存预算、包体预算、加载时间预算、同屏角色数、特效数量、UI 页面复杂度、AI 数量、物理对象数量、网络带宽。预算不是为了限制创作，而是让创作有边界。超预算内容可以存在，但必须有明确取舍和审批。

画质档位和 Scalability 设置也应纳入预算。不同档位应有明确差异：阴影距离、后处理质量、纹理质量、特效数量、视距、植被密度、反射、Lumen/Nanite 策略、UI 特效等。不要等上线后再临时做低配适配。

#### 十二、性能回归要进入 CI 和日常构建
性能优化如果没有回归机制，很快会被新内容抵消。迁移项目应建立每日或每周性能测试，固定场景、固定平台、固定参数，记录帧时间、线程耗时、内存、加载和包体。指标异常时自动生成报告，并关联版本和提交范围。

CI 不一定一开始就做到完全自动化，但至少要形成可重复流程。性能报告应显示趋势，而不是只显示单次结果。某次提交让 UI 打开时间增加 200ms，某个资源让包体增加 100MB，某个蓝图让 Game Thread 增加 1ms，都应能被发现。

回归门槛应分级。阻断级问题包括启动崩溃、严重帧率退化、内存超限和包体超限；警告级问题包括小幅退化和预算接近上限；提示级问题进入优化债务。这样团队既能保持速度，又不会让性能债务失控。

#### 十三、优化结论必须可验证、可回滚、可沉淀
每次优化都应记录问题、证据、修改、收益、风险和回滚方案。不要只提交“优化性能”。例如“在 30 NPC 战斗场景中，将 AI Service Tick 从 0.1s 调整为 0.25s，Game Thread P95 从 19ms 降到 16ms，行为响应延迟可接受”。这样的结论可复查、可传播、可复用。

优化也可能带来副作用。降低 Tick 频率可能影响手感，降低阴影质量可能影响画面，异步加载可能引入占位资源，资源池可能增加常驻内存。每个优化都要说明取舍。成熟团队不会把性能优化当成神秘经验，而是当成工程实验。

沉淀方式包括性能手册、常见问题库、工具命令清单、场景预算表、资产规范和代码审查清单。迁移完成后，这些沉淀会成为团队量产能力的一部分。

### 实现方案

#### 一、建立目标平台性能基线
选择目标平台和代表场景，建立基线测试流程。每次测试记录版本、平台、画质、分辨率、场景、操作脚本、测试时长和环境温度或设备状态。指标至少包括平均 FPS、P95/P99 帧时间、Game、Draw、GPU、内存、显存、加载耗时、Hitch 次数和包体。

基线场景应覆盖主界面、核心战斗、峰值战斗、大型地图、UI 重页面、资源加载、多人和长时间运行。初期可以手动执行，后续逐步自动化。

#### 二、制定性能分析流程
团队统一使用“判断瓶颈类型、专项采样、定位对象、修改验证、记录结论”的流程。快速阶段使用 Stat 命令，深入阶段使用 Insights、GPU 工具、内存报告和平台工具。问题单必须附带平台、版本、场景、指标和采样文件。

不要允许只凭体感创建优化结论。任何性能改动都要有前后对比。若无法复现或无法量化，应先补测试场景和工具。

#### 三、建立常用工具清单
整理项目常用命令和工具：`stat unit`、`stat game`、`stat slate`、`stat anim`、`stat physics`、`stat gpu`、`stat net`、Unreal Insights、MemReport、LLM、GPU Visualizer、RenderDoc、平台性能工具、Asset Audit、Size Map、包体报告。为每个工具写明用途、使用步骤和输出解读。

工具清单应面向全团队，不只面向程序。技术美术、美术、策划和测试至少应能理解基础指标和问题反馈格式。

#### 四、为重点系统建立专项场景
建立动画压力、AI 群体、物理碰撞、Niagara 峰值、UI 列表、资源加载、开放世界 Streaming、多人网络和大型战斗场景。每个场景有固定操作和预算。新增功能进入量产前，先在专项场景验证成本。

专项场景不要做成一次性 Demo，而要纳入回归。性能问题越早在专项场景暴露，越不会在真实关卡后期集中爆发。

#### 五、建立系统预算表
按平台制定 CPU、GPU、内存、显存、加载、包体、网络预算，再拆到系统：渲染、UI、动画、AI、物理、音频、特效、资源加载、网络。预算表包含目标值、警戒值、阻断值和 owner。

预算表应随项目阶段调整，但不能没有。项目早期可粗略，量产和上线前必须精细。预算超限需要说明原因和取舍。

#### 六、将性能检查接入内容生产
资源导入、UI 页面、特效、角色、地图和技能都应有性能检查。新增特效需说明粒子数量和 LOD，新增 UI 页面需说明打开耗时和 Widget 数量，新增角色需说明骨骼数量和动画成本，新增地图需说明可见对象和 Streaming 策略。

内容团队不是性能问题的旁观者。工具和报告应让内容团队能看到自己的成本，并知道如何修复。

#### 七、处理典型迁移热点
迁移项目优先检查以下热点：蓝图 Tick 过多、UMG 属性绑定、同步加载资源、硬引用导致常驻内存、复杂透明材质、粒子同屏过多、动画图过大、AI Service 过频、物理 Overlap 过多、对象生成销毁峰值、Shader 首次编译、纹理 Streaming 不合理。

这些热点不一定都是瓶颈，但它们是 Unity 转 UE 最常见误区。每个热点都应有检查方式和修复模板。

#### 八、建立性能回归报告
每次里程碑构建输出性能报告，展示核心场景趋势、指标变化、异常项和责任模块。报告应能和版本、提交、资源变更关联。对于严重退化，要求在进入下一阶段前修复或明确豁免。

报告不要只给程序看。项目负责人需要看到风险，内容负责人需要看到资产成本，测试需要看到复现路径。

#### 九、形成性能迁移验收标准
性能优化与分析迁移完成应满足以下标准：

- 目标平台、目标帧率、画质档位和核心场景基线已建立。
- 团队掌握 Stat、Insights、GPU、内存和资源分析工具的使用流程。
- CPU、GPU、内存、加载、网络瓶颈可以分开诊断。
- 核心系统有专项性能场景和预算表。
- UI、动画、AI、物理、Niagara、资源加载有独立检查项。
- 性能问题单包含平台、版本、场景、指标、采样和复现步骤。
- 优化结论有前后数据、风险说明和回滚方案。
- CI 或固定构建流程能生成性能趋势报告。
- 内容生产流程包含资源成本、包体和运行时预算检查。
- 迁移后的性能债务有清单、优先级和 owner。


#### 工业化补充：迁移的核心不是“找到对应功能”，而是重建一套新的工程工作方式
引擎迁移类文档最常见的误区，是把迁移理解成大规模 API 对照或功能搬运。事实上，Unity 到 UE 这样的迁移真正困难的地方，从来不是“某个功能能不能做出来”，而是原来建立在旧引擎工作方式上的组织习惯、调试习惯、资源流转、代码边界、性能判断和协作接口，都需要重新落位。也正因为如此，成熟的迁移文档绝不会只列功能映射，而会把重点放在“为什么原来的做法在新引擎里不再成立”“应该先建立什么基础心智”“哪些能力必须按新引擎哲学重构，而不是照抄旧习惯”。只有当团队接受迁移本质上是工程工作方式迁移，而不是语法迁移，这类文档才真正有指导价值。

从项目管理角度看，迁移也是一项高风险系统工程。语言、脚本协作、UI、输入、网络、资源、物理、渲染、构建发布、性能分析和工具链，看似分别属于不同专题，实际上都共享同一个迁移前提：旧项目里哪些抽象可以保留，哪些必须舍弃，哪些需要通过中间层过渡，哪些必须在新引擎原生能力之上重新组织。文档若不能回答这些问题，就很容易让团队陷入“局部上手很快、整体越来越乱”的假熟悉状态。

#### 双轨验证、分阶段退出与学习顺序：迁移不能只靠勇气，更要靠可回退的计划
大型迁移项目最怕的不是进度慢，而是没有可验证、可回退的阶段边界。很多团队一开始雄心很大，计划把旧系统整体搬过去，结果中途才发现角色框架、资源引用、蓝图协作、打包发布和网络复制并没有想象中那么容易对齐。真正成熟的做法，是在文档里提前定义双轨验证策略：哪些模块先做概念验证，哪些模块先做最小可用版本，哪些关键路径需要同时保留旧实现以便对照，哪些阶段满足什么条件后才能正式切换。只有当退出条件和切换条件被写清楚，迁移才不会在版本压力下变成一场只能向前硬冲的赌博。

学习顺序同样关键。Unity 开发者转向 UE 时，最危险的不是学不会，而是太快找到表面对应关系，从而过早获得虚假的熟悉感。文档应反复强调，正确顺序应优先建立对象模型、资源引用、编辑器工作流、调试与分析工具、Gameplay Framework 和代码/蓝图边界，然后再逐步扩展到 UI、网络、物理、输入、特效和构建链路。只要顺序错了，团队就会在早期形成大量看似能跑、实则违背新引擎哲学的做法，后续返工代价极高。

#### 协同面、资产面与工具链：迁移永远不是程序单线任务
引擎迁移失败的另一个高频原因，是把它误判成程序组自己的技术升级。实际上，只要项目进入实战迁移阶段，美术、策划、技术美术、测试、构建发布和工具链都会被牵动。UI 不只是控件换一套；资源管理不只是目录重新组织；蓝图与 C++ 协作不只是脚本语言换个写法；构建发布不只是新打包工具；性能优化也不只是换个 Profiler。每一项迁移都会改变上下游的交付接口和验证方式。因此，文档必须把协同面显式写出来：哪些改动只影响程序内部，哪些会改变内容团队的工作流，哪些需要培训支持，哪些需要先补工具再迁移，哪些要同步建立回归样例。

资产与工具链尤其需要被正式化管理。新引擎中的资源引用、Cook、软硬引用、DataAsset、Primary Asset、蓝图继承链和自动化构建链路，都决定了项目能否在规模扩大后仍然维持稳定。迁移文档不能只写“在 UE 里对应哪个概念”，还必须要求团队回答：资源如何组织、引用如何约束、编辑器配置如何沉淀、自动构建与校验如何接入、历史资产如何批量迁移、团队如何确认迁移后内容生产效率并未明显倒退。只有当这些问题被纳入主线，迁移才真正拥有工业化落地条件。

#### 评审、风险清单与验收矩阵：把迁移从“专家经验”变成“团队流程”
迁移项目之所以容易依赖少数专家，是因为很多判断天然跨越多个系统。某个模块该先迁还是后迁，某个旧框架该保留还是拆掉，某项蓝图逻辑是否应该转回 C++，某个资源系统是否已经适应新管线，如果缺少统一评审问题，团队就会持续依赖“最懂的人拍板”。更稳定的做法，是把迁移评审问题写进文档：这项能力在新引擎中的原生承载点是什么；旧项目里的假设在新引擎里是否还成立；迁移后上下游工作方式如何变化；最小验证样例是什么；失败时如何回退；成功后如何验收；哪些风险需要先补工具或先建约束再迁移。持续围绕这些问题工作，团队才会逐步从“依赖专家直觉”转向“依赖共享流程”。

验收矩阵也必须足够具体。一个专题完成迁移，不应只以“功能表现看起来差不多”为标准，而应同时检查：行为一致性是否通过；资源/配置是否能被新流程稳定承载；性能与内存是否满足目标机预算；调试入口是否明确；编辑器和自动化链路是否可用；新人是否能按照文档接手；旧实现是否已经具备退出条件；相关岗位是否完成最小培训。只有当这些维度一起通过，迁移才算真正完成，而不是停留在 Demo 成功阶段。


#### 迁移回归矩阵与培训补充
迁移类文档如果要真正支撑团队落地，还应补一层回归矩阵与培训计划。也就是说，每个专题迁移完成后，不仅要验证功能是否能跑，还要验证旧功能行为、新引擎工作流、调试入口、资源流程和协作接口是否已经被团队真正接住。同时，新旧思维切换往往不是一次讲解就能完成，团队需要为对象模型、蓝图/C++ 边界、资源引用、调试分析工具和构建流程安排持续培训与示例沉淀。没有回归矩阵和培训闭环，迁移就很容易停留在“少数人已经会了”，而不是“团队整体已经稳定迁过去了”。


#### 迁移完成判定补充
迁移专题在项目里真正“完成”的标志，不应只是功能可运行，而应包括团队是否掌握了新的默认做法、旧路径是否具备退出条件、文档和示例是否足以支撑接手、以及相应回归是否已纳入常规流程。把这层完成判定写进去，能避免迁移长期停留在半新半旧、谁都不敢彻底切换的状态。


#### 文档治理与版本维护补充
一份技术文档是否真正成熟，最终并不只取决于它有没有把知识点讲明白，还取决于它能否在项目生命周期里持续发挥作用。也就是说，这份文档在版本推进、人员流动、需求变化、线上问题和技术债积累的过程中，是否仍然能被团队拿来指导判断、约束实现、解释问题和组织回归。很多团队的文档一开始写得并不差，后面之所以失效，往往不是知识本身过时，而是缺少文档治理意识：没有固定更新入口，没有明确责任人，没有版本相关的补充规则，没有把新问题和新例外回写进文档，最终导致文档逐渐变成“历史上某一次讨论的快照”，而不是项目的正式工程资产。

因此，无论主题更偏架构、性能、算法、数学、物理、资源、协作、迁移还是职业成长，这类文档都应当补一层共同的维护要求。第一，关键结论要能被回到真实项目中验证，而不是只停留在抽象建议。第二，文档应有最小责任归属，知道谁来更新、何时更新、什么变更会触发更新。第三，涉及高风险主题时，应当能指向对应的回归样例、排障入口或验收清单，而不是只告诉读者“这里很重要”。第四，历史妥协、临时方案和版本性例外不应被默默埋掉，而应被显式记录并定期审视。第五，文档要能服务接手和培训，让后来者不必依赖作者口头补充才能理解核心边界。

当这些治理要求被写进正式文档之后，知识才会真正转化为组织能力。团队也会逐步从“出了问题再找最懂的人”转向“先回文档、回基线、回约定，再定位责任和修正方案”。这不仅能提高技术判断的一致性，也能显著降低协作中的解释成本和重复犯错概率。从这个意义上说，技术文档的成熟，并不只是内容变长，而是它已经具备了被长期维护、被多人复用、被版本持续校准的能力。

### 总结
性能优化与分析迁移的重点，不是学习几条 UE 优化技巧，而是重新建立“如何看见问题”的能力。Unity 时代的经验仍然有参考价值，但必须转换到 UE 的线程模型、工具链、资源体系和渲染管线中。没有基线、没有采样、没有预算、没有回归的优化，最终只能停留在经验猜测。

成熟的迁移团队应把性能工作做成闭环：建立目标，采集数据，定位瓶颈，验证收益，记录结论，防止退化。做到这一点后，性能不再是项目后期的救火任务，而是伴随内容生产持续运行的工程能力。
