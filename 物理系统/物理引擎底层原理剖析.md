# 物理引擎底层原理剖析

## 摘要
理解物理引擎底层原理的意义，并不在于要求每个游戏开发者都从零手写一套引擎，而在于帮助团队建立一套能够解释常见物理现象、定位异常来源并设计合理优化路径的统一语言。碰撞为什么不是一步完成的、约束求解为什么天然是近似的、为什么长链关节和大质量比会让系统不稳定、为什么同样数量的物体在不同结构下成本差异极大，这些问题如果只停留在 API 层面几乎无法回答。本文从实时物理引擎的一帧内部流水线出发，系统说明碰撞检测、接触生成、岛屿划分、约束构建、迭代求解、状态积分、睡眠与缓存等核心机制，并进一步讨论这些机制在 Unity / PhysX 一类通用物理系统中的工程意义。目标是让开发者把“黑盒物理”拆成可理解、可诊断、可治理的阶段性过程。

## 正文

### 背景
在日常开发中，很多人与物理引擎的接触都发生在组件与接口层：添加 Collider、挂 Rigidbody、设置质量、切换连续检测、绑定 Joint、调用 AddForce、做 Raycast。只看这些接口，物理引擎似乎更像一个“给输入、出结果”的服务组件，因此团队很容易形成两种极端认知。第一种是把它神化，认为它会自动帮你处理所有现实问题，只要参数调好一切都会稳定。第二种是把它妖魔化，一旦出现穿透、抖动、卡顿和不可解释行为，就认为物理引擎本身“不靠谱”。这两种认知都不利于项目。

更接近事实的理解是：通用实时物理引擎是一套在固定计算预算中持续做近似求解的工程系统。它并不试图完美复制真实世界，而是在可接受的时间内给出足够可信、可交互、可连续的结果。为了做到这一点，引擎会把问题拆成多个阶段：先快速筛掉不可能碰撞的物体，再为候选对生成接触信息，再把所有接触和关节转成约束，再通过有限次迭代调整速度和位置，最后推进状态并维护缓存。这条流水线中的每一步都带有明显的工程取舍，因此任何物理表现都是“算法能力、实时预算、数据结构与场景建模”共同作用的结果。

只要缺少这种流水线视角，开发者在面对问题时就很容易失焦。例如堆叠不稳时一味增加重力或迭代次数，穿透时只知道改 Collision Detection，性能高时只会粗暴减少物体数量。真正成熟的做法，是先理解问题更可能出在哪个阶段，然后再对症下药。也正因为如此，物理引擎底层原理并不是离项目很远的“底层知识炫技”，而是高质量使用现成引擎所必需的工程基础。

### 核心内容
#### 一、物理引擎的核心任务，不是“模拟真实”，而是“在实时预算内给出足够可信的交互结果”
从研究视角看，物理模拟可以追求更高精度、更复杂材料、更严格连续性和更完整的力学模型；但从游戏引擎视角看，物理系统首先必须满足实时性。也就是说，引擎不能为了精确而无限计算，它必须在每个固定时间步里完成求解，并把结果交给渲染和逻辑系统继续使用。因此，通用物理引擎的目标通常不是“尽可能真实”，而是“在可负担成本内足够稳定、足够可控、足够符合玩家感知”。

这个目标导向决定了物理引擎的几乎所有底层设计。很多问题之所以会出现近似、误差和不完全稳定，不是引擎“偷懒”，而是实时系统必须做的工程取舍。理解这一点，能够帮助开发者放弃“为什么它不是完全正确”的执念，转而关注“在当前项目需求下，怎样让它足够好且足够稳”。

#### 二、一帧物理流水线通常由广相检测、窄相检测、接触生成、约束求解和积分推进构成
尽管不同引擎实现细节不同，但大多数实时刚体物理引擎都遵循相似的阶段化流程。首先是 Broadphase，也就是广相检测，用来从大量物体中快速找出可能相交的候选对。接着是 Narrowphase，也就是窄相检测，对候选对做更精细几何判断，确认是否真正接触，并生成接触法线、穿透深度、接触点等信息。然后是约束构建与求解，把接触、不穿透、摩擦、关节限制等都转成需要满足的约束条件。最后再做积分和状态更新，把新的速度和位姿推进到下一步。

把这条流水线记住很重要，因为项目中几乎所有物理问题都可以被重新表述为“更像哪个阶段出错或超负荷”。穿透可能是离散采样和接触检测问题，抖动可能更像约束求解与稳定性问题，性能爆炸可能出在候选对过多或约束链太复杂。只要开发者开始用这条流水线来观察问题，诊断质量就会比单纯调 API 高出很多。

#### 三、Broadphase 的本质，是用更便宜的几何近似筛掉绝大多数不可能碰撞的组合
如果一个场景有 N 个可能参与碰撞的物体，理论上的两两检测数量会非常快地膨胀。显然，引擎不可能对所有物体对都直接做精确几何求交，因此 Broadphase 的意义就在于通过更粗糙但更便宜的表示，例如 AABB 包围盒和空间划分结构，先快速筛掉明显不可能发生接触的组合，只把少量“有机会相碰”的候选送进下一阶段。

工程上，这意味着很多性能问题在 Broadphase 阶段就已经埋下。碰撞层不合理、动态体数量过多、对象密度过高、细碎碎片过多、无意义的 trigger 与 query 进入同一空间，都可能让候选对数量异常膨胀。很多团队只看见最后整体物理耗时高，却忽视了大量成本实际上在“还没开始精确碰撞前”就已经花掉了。

#### 四、Narrowphase 的难点，在于从“可能相交”变成“准确描述接触条件”
候选对被筛出来后，引擎要做的就不再是简单判断“有没有碰到”，而是尽量准确地描述“如果碰到了，接触点在哪里、法线朝哪、穿透多少、相对速度怎样”。这些信息决定后续约束求解器如何生成冲量，因此其质量直接影响碰撞响应的稳定性和真实性。

碰撞体形状复杂度在这里非常关键。对 primitive collider 来说，接触几何更规则，成本也更稳定；对复杂网格碰撞来说，接触可能更细碎、法线更跳跃、候选面更多。于是很多“感觉像求解器不稳”的问题，其实源头来自接触信息本身就难以保持干净稳定。底层原理上，这说明高质量碰撞建模并不是“资源替代引擎”，而是物理稳定性的一部分。

#### 五、接触并不是简单事件，而是一组需要在当前时间步内尽量满足的约束条件
物体发生接触后，引擎并不会像很多人想象的那样“发现重叠了就把它们分开”。更普遍的方式是，把接触视为一组约束：物体不能沿法线继续相互压入；若存在摩擦，则切向相对运动要受到限制；若还有关节、铰链或其他机制，则还需同时满足额外限制。于是，问题被转化为“在当前时间步内，用有限迭代尽量让所有这些约束同时成立”。

这也是为什么约束越多、链路越长、质量差越大、接触关系越密，系统越容易不稳。因为求解器面对的不是单一方程，而是大量相互耦合、且通常只能近似满足的条件集合。堆叠、车辆悬挂、布娃娃和长关节链都难，根源就在这里。

#### 六、岛屿划分（Island Solving）决定了引擎是否能把大问题拆成若干小问题
在底层实现中，引擎通常不会把场景里所有物体与约束放进一个统一巨大系统中求解，因为那样既昂贵也没必要。更常见的做法是把相互关联的动态物体、接触和关节划分成若干“岛屿”，每个岛屿内部单独求解。两个完全无交互的区域自然也不应彼此影响。

岛屿划分的意义有两个。第一，它让求解范围局部化，减少无关计算。第二，它让睡眠和唤醒策略更容易生效，因为某个局部岛屿静止后可整体退出高成本状态。工程上，若项目中存在大量通过接触或关节形成的长链结构，就会导致岛屿规模变大、求解成本上升，也更容易出现“本来只是改动了一个局部，为什么整串都抖”的现象。

#### 七、约束求解器本质上是在有限迭代内逼近解，而不是一次性精确求解
大多数实时物理引擎不会对整套约束做昂贵的全局精确求解，而更倾向使用迭代方法逐步逼近可接受结果。简单理解，可以把它看成求解器不断扫描接触与关节条件，逐步施加冲量修正速度和位置，希望在有限轮数后达到一个“误差足够小”的状态。

这就解释了很多熟悉的现象。为什么增加 Solver Iteration 有时会改善稳定性？因为更多轮数给了系统更充分的修正机会。为什么它不是万能的？因为问题的结构若本身就过难，例如长链关节、大质量差、极高速度或过大时间步，有限迭代仍然可能收敛不足。理解“近似迭代”这一本质，是理解所有稳定性边界的基础。

#### 八、Warm Starting、接触缓存和持久化接触，是现代物理引擎保持实时稳定性的关键技巧
若每个时间步都从零开始求解，系统会更容易抖动，也更难在有限迭代中快速收敛。为此，现代物理引擎常会利用上一个时间步的接触与冲量信息作为初始猜测，也就是常说的 warm starting。若两个物体在连续帧里保持接触，上一步求出的冲量往往对下一步仍有参考价值。接触缓存和持久化接触点也有类似作用：减少不必要的重新发现和剧烈跳变。

从工程意义上说，这说明稳定性不仅来自单步求解，还来自时间上的连续性。若开发者通过脚本频繁瞬移刚体、强制修改姿态、不断销毁重建对象或让接触关系剧烈跳变，就会破坏这类缓存收益，使系统更难稳定。很多“为什么一切都像刚从零开始算一样不稳”的问题，本质上正是破坏了这种时间连续性。

#### 九、位置修正与速度修正的分工，决定了引擎如何在“避免穿透”和“保持能量合理”之间平衡
仅靠速度修正，有时无法快速消除已经发生的穿透；仅靠粗暴位置修正，又可能产生不自然弹跳或能量注入。因此，许多引擎会在速度层和位置层同时做一定的误差处理。常见做法包括基于偏差的速度修正、Baumgarte 一类稳定化思想，以及针对深度穿透的分离处理。不同实现细节有所差异，但目标相似：既别让物体长期互相压进，又尽量别因为修正本身制造过大伪能量。

这层原理对项目很有帮助，因为它解释了为什么某些深度穿透在一两帧内会被“弹开”，为什么某些堆叠即使没完全穿进去也会持续微抖。系统在实时条件下必须在误差清除速度和视觉稳定性之间做平衡，而不是绝对精确地恢复物理正确状态。

#### 十、时间步长、速度尺度和尺寸尺度共同决定了数值条件是否健康
底层物理求解从来不是脱离尺度独立工作的。一个很小的物体高速运动、一个很轻的物体去推动极重物体、一个极薄碰撞体承担关键阻挡、一个很大的时间步却要描述很精细的接触边界，这些都会让数值条件变差。通俗地说，问题并不是“引擎算错了”，而是你在让引擎用过粗的时间采样、过低的精度和过少的迭代去描述过难的问题。

因此，成熟团队通常会非常重视项目中的单位规范、尺寸范围、速度上限和质量比例。它们看起来像内容规则，实际上直接决定了底层数值是否健康。若这些尺度失控，任何后续参数优化都只是在补救。

#### 十一、睡眠与唤醒机制的意义，在于让不再变化的局部系统退出高成本求解
如果所有刚体无论静止还是运动都持续参与完整解算，物理系统成本会非常高。为此，引擎会判断某些物体或岛屿是否已足够静止，并将其进入睡眠状态。睡眠不是“把物理关掉”，而是在当前条件下认为它们短时间内无需继续高频求解，直到有新的碰撞、力或状态变化将其唤醒。

项目中很多性能问题都与睡眠策略有关。某些对象因为被轻微持续扰动而一直无法睡眠，某些链状结构因末端轻微运动而整串保持激活，某些脚本每帧微调 transform 让本应静止的物体不断被唤醒。这些问题如果只看业务层，很难意识到本质上是在持续扩大活动岛屿范围；若理解底层睡眠机制，就更容易发现真正症结。

#### 十二、查询系统虽然不直接参与刚体求解，但同样属于物理引擎的一部分预算
许多开发者在谈物理时只盯着刚体和碰撞，其实 Raycast、SphereCast、Overlap、CapsuleCast 等查询同样会占据物理预算，且常在 AI 感知、角色地面检测、武器命中、交互扫描和导航辅助中高频使用。底层看，这些查询虽然不一定进入完整求解，但也会复用碰撞世界中的空间结构和几何测试能力。

因此，物理引擎的负载并不只来自“发生了多少真实碰撞”，还来自“项目每帧在用多少查询去问这个世界”。理解这点，有助于团队避免把查询当成无成本工具无限滥用。

#### 十三、确定性并不是通用物理引擎的默认承诺
很多项目在做回放、多人同步或战斗复盘时，都会碰到一个问题：物理结果在不同运行环境、不同帧节奏甚至不同执行顺序下，并不总能完全一致。底层原因在于实时通用物理引擎通常并不把跨平台强确定性作为第一优先级，它更关注实时性、稳定性和通用功能。浮点误差、求解顺序、平台差异和时间步细节都可能让结果渐渐分叉。

这意味着，若产品对确定性有强要求，就不能简单假设“用了物理引擎就自然可回放、可同步”。很多时候需要额外策略，例如锁步友好设计、关键结果脚本化、服务端权威结算、回放只记录结果而非完全重演通用物理。理解底层不保证确定性，能帮助团队更早选择正确架构。

#### 十四、典型误区一：认为“碰撞发生了”就代表引擎已经完全知道如何合理响应
碰撞检测与合理响应并不是同一件事。系统即便知道两个物体接触了，也还需要在接触法线、相对速度、摩擦和其他约束共同作用下决定怎样修正。很多开发者看到 OnCollision 触发了，就误以为“后面肯定该很稳定”，实际上后续求解质量仍然取决于数值条件、迭代和接触结构。

#### 十五、典型误区二：把所有异常都归因于参数，而忽略建模与结构
质量、阻尼、迭代、CCD、插值、时间步等参数当然重要，但它们并不是独立于建模工作的万能旋钮。若一个链条太长、碰撞体太薄、质量比太极端、控制方式相互冲突、对象尺度离谱，那么再漂亮的参数也只能局部缓解。很多底层问题真正的解决办法，不是继续找“神奇参数”，而是重新设计结构条件。

#### 十六、典型误区三：把物理引擎当作默认的一切交互解
并不是所有“看起来像物理”的效果都值得用通用刚体求解去实现。角色平台跟随、某些脚本控制机关、部分抛物线视觉、演出性碎片、特定受击反馈、部分车辆手感都可能更适合“物理近似 + 脚本控制”的混合方案。通用物理引擎非常强大，但它不是项目里所有动态表现的唯一答案。

#### 十七、接触缓存与求解顺序，解释了为什么“同一场景偶尔会有细微差别”
许多开发者在观察物理现象时，会注意到一些微妙情况：同样的堆叠有时更稳一点，有时轻微更滑；同样的碰撞在帧率波动后呈现出略不同的结果。这类差异并不一定意味着引擎随机出错，很多时候与接触缓存、warm starting、求解顺序和浮点误差传播有关。只要求解器依赖上一帧信息做初始猜测，并且多个约束的处理顺序会影响中间结果，那么在边界条件附近出现细微差别就是可以理解的。

这也是为什么“可解释的黑盒”比“绝对确定的幻觉”更重要。团队若知道接触缓存和求解顺序会带来怎样的实际影响，就更容易在设计与测试阶段接受某些轻微近似差异，同时把真正值得治理的问题聚焦到结构风险和明显异常上，而不是对所有微小变化都做错误归因。

#### 十八、底层原理的真正项目价值，是让团队知道“该在哪一层停手”
理解底层后，开发者常会陷入另一种风险：看到所有问题都想继续往更深一层挖，仿佛必须完全掌握引擎每个内部细节才算真正理解。对量产项目而言，这并不是最优路径。真正成熟的使用方式，是理解到底层到什么程度足以支撑当前项目判断，然后在那个层级停手，把精力投向更高价值的建模、验证和治理上。

也就是说，底层原理的价值不是逼迫团队人人成为物理引擎开发者，而是让团队能更快判断：这个问题值得继续追到求解器层，还是已经能在碰撞建模、时间步、控制边界或玩法结构层解决。知道“该在哪一层停手”，其实正是底层理解成熟的表现。

### 实现方案
#### 一、在团队内部建立“物理流水线语言”
建议把广相、窄相、接触、约束、积分、睡眠、查询这些术语转化为团队共用语言。这样一来，遇到问题时大家讨论的就不再只是“这里又抖了”“那里又穿了”，而能更清楚地说“更像 Broadphase 过载”“更像接触不稳定”“更像求解链太长”“更像显示插值问题”。统一语言本身就是工程效率。

#### 二、把碰撞建模与单位规范前置到资源和玩法设计阶段
底层原理告诉我们，许多问题是由尺寸、速度、质量和碰撞体形状共同决定的。因此，建议尽早定义项目的尺度规范、速度上限、关键体质量比例和碰撞体建模原则，而不是把这些问题留到性能或稳定性事故后再倒推。越早规范，后续越少被动补救。

#### 三、为关键物理玩法建立专项样例
对项目中最核心的物理玩法，例如车辆、推箱、绳索、布娃娃、高速投射体、堆叠掉落等，建议建立独立样例场景并长期保留。样例场景的作用，是让底层问题在可控条件下被重复验证，而不是总要等到大型正式关卡里才看见症状。只要样例场景长期存在，团队就能更客观地比较参数、结构和引擎版本变化带来的影响。

#### 四、分离“模拟正确性问题”和“显示平滑性问题”
在排查时，先判断问题属于求解不稳还是显示抖动。若是显示问题，应优先考虑插值、相机、跟随和渲染同步；若是模拟问题，再回到碰撞建模、约束结构和时间步。把这两类问题混在一起，是最常见的误判来源之一。

#### 五、限制长链、高质量差和极端速度结构的默认使用
长关节链、大质量差堆叠、超高速小物体、极薄碰撞器、持续瞬移刚体，这些都属于“数值高风险结构”。建议在技术规范中明确这些场景必须专项评估或给出替代设计，而不是默认为可随意大量使用。底层原理的价值，正体现在能让团队更早识别这类高风险结构。

#### 六、把查询预算和刚体预算一起管理
若项目大量依赖 Raycast、Overlap 和 Cast 查询，也应将其纳入物理预算治理。高频查询应按系统分类、按层级过滤、按必要性节流。只有当团队认识到“查询也是物理负载的一部分”，物理优化才算完整。

#### 七、验收标准：什么样的底层理解算真正落地到项目中
如果一个团队已经能够把物理异常按流水线阶段定位、能在设计时主动规避高风险结构、能区分显示问题与求解问题、能为关键玩法建立专项样例、能把碰撞建模和单位尺度纳入正式规范，并且不再把物理引擎当成所有交互问题的默认解，那么就可以认为“物理引擎底层原理”已经不只是文档知识，而是进入了项目治理层。反之，如果大家仍然只是记住几个参数名字，却无法解释问题更像 Broadphase 还是 Solver，那说明理解仍停留在表面。

### 总结
物理引擎底层原理的真正价值，不在于炫耀知道多少术语，而在于让开发者能把“黑盒输出”拆成一条可解释的实时求解流水线：Broadphase 负责筛候选，Narrowphase 负责接触信息，约束系统负责不穿透与摩擦限制，求解器在有限迭代中逼近结果，积分系统推进状态，睡眠与缓存帮助控制预算。只要团队掌握了这套视角，很多曾经看似随机、玄学、只能盲调的问题，都会逐渐变成可分析、可验证、可治理的工程问题。这正是理解底层原理对项目最现实的意义。
