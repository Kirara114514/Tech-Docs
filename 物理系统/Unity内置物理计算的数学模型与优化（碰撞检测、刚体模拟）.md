# Unity内置物理计算的数学模型与优化（碰撞检测、刚体模拟）

## 摘要
Unity 内置物理系统经常被误用成“勾上 Rigidbody 和 Collider 就会自然工作”的黑箱。很多项目真正出问题时，团队才开始追问：为什么会穿透、为什么会抖动、为什么物体堆叠不稳、为什么 FixedUpdate 一改全局都乱、为什么加几个刚体 CPU 就飙高。其实这些现象背后并不神秘，它们几乎都可以回到几类基本数学模型：离散时间步长下的数值积分、粗筛与精筛结合的碰撞检测、接触约束与冲量求解、速度与位置修正，以及浮点误差和时间同步造成的稳定性问题。真正成熟的物理优化，不是把参数表一个个试过去，而是理解当前问题到底是出在检测阶段、求解阶段、时间推进阶段，还是业务层对物理系统的错误使用。本文围绕 Unity 内置物理的数学基础、碰撞检测链路、刚体模拟的数值本质、常见失稳原因、性能账本、调试与验收方式，以及在项目中最容易踩的反模式，系统说明如何把“碰到问题再猜参数”升级成“能定位模型、能解释问题、能做工程优化”的物理治理能力。

## 正文

### 背景
游戏里的物理问题常常具有两个非常讨厌的特点：

- 看起来是偶发的。
- 一旦出现往往很影响玩家体验。

例如：

- 高速物体偶发穿透。
- 角色贴墙抖动或爬坡异常。
- 一堆刚体叠在一起时轻微振荡。
- 远距离或低帧率场景下物理结果不稳定。
- 联机下某些物理交互无法稳定重现。

这类问题之所以难，不是因为公式有多高级，而是因为真实引擎里的物理系统是多个阶段共同作用的结果：

- 时间推进：物理是按离散步长推进，而不是连续解析世界。
- 碰撞检测：先缩小候选，再做精确接触计算。
- 约束求解：接触、摩擦、弹性和关节都依赖近似求解。
- 数值稳定性：积分误差、浮点误差和时间步长会持续放大问题。

如果开发者不理解这条链路，就很容易把任何问题都归因到“引擎不稳定”或“参数没调好”。事实上，大多数问题都有模型层原因。

### 核心内容

### 第一原则：Unity 物理不是连续世界，而是离散时间近似
这是理解一切问题的前提。真实世界的运动是连续的，但游戏引擎中的物理模拟通常不是。Unity 物理主要在固定时间步长中推进：

- 每个物理步里，根据当前速度、力和约束更新状态。
- 在离散时刻检查碰撞。
- 根据接触情况进行求解和修正。

这意味着：

- 物体并不是连续移动，而是在一个个离散时刻被更新。
- 如果步长过大，高速物体可能在两个时刻之间直接跨过障碍。
- 时间步长变化会影响模拟稳定性、接触一致性和求解质量。

这就是为什么物理问题里最常见的概念不是“绝对正确”，而是“在当前步长和预算下足够稳定”。

### 第二原则：碰撞检测本质上是“候选收缩 + 精确判定”
很多人把碰撞检测想得太简单，认为就是两个 Collider 是否相交。实际流程更像这样：

#### 一、Broad Phase（粗筛）
先用包围体、空间分区等方式快速缩小可能发生碰撞的对象对。

#### 二、Narrow Phase（精筛）
再对候选对象做更精确的接触判定，计算接触点、法线、穿透深度等。

#### 三、接触缓存与持续碰撞管理
很多情况下，系统还会利用接触对的连续性提高效率和稳定性。

这条链路带来的直接工程含义是：

- 物体数量变多时，Broad Phase 会先吃掉大量预算。
- Collider 形状和分布会影响候选对数量。
- 高速、小体积、复杂形状和高动态场景更容易暴露检测边界问题。

因此，碰撞优化不能只盯着某个 Collider，而要看整个候选收缩账本。

### 第三原则：刚体模拟的核心不是“自动运动”，而是数值积分与约束求解
刚体物理更新不是魔法，它本质上包括：

- 力和加速度影响速度。
- 速度影响位置。
- 接触和约束修正速度与位置。

在数值上，这通常依赖离散积分。积分本身就有误差，而约束求解又往往不是一次解析完成，而是通过迭代逼近。所以出现这些现象并不奇怪：

- 堆叠轻微振荡。
- 弹性和摩擦不是完全理想。
- 关节链末端抖动。
- 低迭代数下接触不稳。

这说明：

- 刚体系统本来就是近似数值模拟。
- 参数并不是“调对了就绝对真实”，而是在稳定性、性能和体验之间取平衡。

### 为什么会穿透：高速、步长、厚度和检测模式共同决定结果
“穿透”是最典型也最容易误解的物理问题。其根因通常不是只有一个，而是多个条件叠加：

#### 一、离散步长过大
在两个物理步之间，物体已经跨越了障碍厚度。

#### 二、速度过高
即便步长固定，高速物体也会放大离散检测的盲区。

#### 三、碰撞体过薄
越薄的障碍，越容易被离散跃迁直接越过。

#### 四、检测模式不适合
Continuous / Continuous Dynamic / Discrete 等模式选择不当，会明显影响高速对象稳定性。

#### 五、业务层手动改位置
如果开发者在错误时机直接修改 Transform，而不是走合适的刚体接口，也会破坏物理系统预期。

因此，修复穿透时不能只会说“把 Continuous 打开”，还要同时检查：

- Fixed Timestep 是否合理。
- 对象速度上限和厚度设计是否合理。
- 是否存在脚本层瞬移和不规范位置修正。

### 为什么会抖动：接触边界、状态切换和求解稳定性通常一起出问题
抖动经常出现在：

- 角色贴墙。
- 斜坡边缘。
- 多接触面切换。
- 堆叠物体。
- 相机或角色与刚体混合驱动场景。

常见根因有：

- 接触法线在边界来回跳。
- 位置修正和速度修正互相对抗。
- 脚本层与物理层同时在改状态。
- FixedUpdate 和 Update 间时序错配。
- 约束迭代数不够。

一个非常常见的坏模式是：

- 刚体系统在 FixedUpdate 推位置。
- 脚本又在 Update 或 LateUpdate 做贴地、纠偏、对齐。
- 两套系统都试图“修正正确结果”。

结果通常就是震荡和不可预测。

### 物理性能账本：别只看刚体数量
很多人评估物理性能时，第一反应是“场景里刚体太多”。刚体数量当然重要，但真正决定成本的因素更复杂：

- Broad Phase 候选对数量。
- Collider 复杂度和分布。
- 活跃刚体数量而非总数量。
- 接触点数量。
- 约束和关节数量。
- 求解迭代次数。
- 高速检测模式带来的额外成本。
- 脚本层对物理结果的高频查询与改写。

在实际项目里，几百个行为简单的刚体可能完全没问题，而几十个高复杂接触、持续高速碰撞、关节串联的对象就足以形成热点。

### 常见优化方向不能脱离问题模型

#### 一、减少无意义接触
通过 Layer、Collision Matrix、触发器设计和场景分层，让根本不需要相撞的对象不进入候选对。

#### 二、控制活跃对象数量
休眠、距离裁剪、区域激活和远距离简化，往往比微调求解参数更划算。

#### 三、合理使用碰撞检测模式
高速度关键对象使用连续检测，普通对象保持离散，避免全局都开最重模式。

#### 四、减少不必要的脚本层物理查询
频繁 `Raycast`、`Overlap`、`ClosestPoint` 若缺乏候选收缩和频率控制，也会成为显著负担。

#### 五、避免用物理系统承担不该承担的逻辑
某些纯规则判断完全没必要交给重型刚体模拟去做。

### FixedUpdate、Update 和 Transform 操作的边界必须非常清楚
这是 Unity 物理里最常见、也最容易被忽略的工程边界之一。

正确认知通常应是：

- 物理相关驱动放在 FixedUpdate。
- 展示层插值、相机跟随等显示逻辑谨慎区分。
- 刚体位置修改优先走 Rigidbody 接口，而不是直接乱改 Transform。

如果团队没有统一这个边界，物理系统迟早会因为时序错配而出现：

- 偶发抖动。
- 回调顺序难以解释。
- 碰撞结果不稳定。

### 物理问题的调试，必须从“画面现象”回到“接触与步长”
物理 Bug 只看日志通常不够。更高效的调试方式是：

- 可视化速度、接触法线、碰撞体和射线。
- 记录物理步中的关键状态快照。
- 比较不同 Fixed Timestep、不同检测模式和不同质量参数下的变化。
- 把问题归类到检测、求解、时序还是脚本层干预。

例如穿透问题，至少要问：

- 是 Broad Phase 没候选到，还是 Narrow Phase 没接住。
- 是离散步长问题，还是业务层瞬移问题。
- 是单个物体条件异常，还是全局时间参数不合理。

### 常见反模式

#### 反模式一：任何位移都直接改 Transform
这会绕过物理系统预期时序和状态管理。

#### 反模式二：为了“更稳”无脑提高全局迭代或连续检测
可能掩盖问题，但成本巨大，而且未必治本。

#### 反模式三：把物理和展示逻辑混在一起修
最终两套系统互相打架。

#### 反模式四：用物理系统做纯逻辑筛选
重型系统承担了轻量问题。

#### 反模式五：只在编辑器里试，不做真机与低帧率验证
很多离散步长问题只在特定帧率和负载下暴露。

### 验收标准：物理优化要同时看正确性、稳定性和成本
一项物理修复或优化至少应验证：

- 穿透 / 抖动 / 堆叠异常是否真实改善。
- 不同帧率和不同机型下是否仍稳定。
- CPU 成本是否在预算内。
- 是否引入新的副作用，例如更强抖动或更重求解负担。
- 脚本层与物理层的职责边界是否更清晰。

### 实现方案

### 推荐的处理顺序
遇到 Unity 内置物理问题时，建议按以下顺序处理：

#### 一、先判断问题属于哪一类
- 检测问题。
- 求解问题。
- 时间步长问题。
- 脚本干预问题。

#### 二、先缩小重现场景
固定步长、固定速度、固定碰撞体条件，去掉无关变量。

#### 三、再调物理参数
不要一上来盲调全局配置。

#### 四、最后回看业务设计
确认是否让物理系统承担了不适合它的职责。

### 总结
Unity内置物理计算的数学模型与优化，真正关键的不在于会不会调几个 Rigidbody 参数，而在于能否理解离散步长、碰撞检测、刚体积分和约束求解是如何共同构成结果的。穿透、抖动、堆叠不稳和性能飙高，几乎都不是“玄学问题”，而是模型边界和工程使用方式被放大后的表现。只有当团队能把问题拆回检测、求解、时序和脚本层职责这几本账，并用统一边界、合适检测模式、合理时间配置和可视化调试手段去治理，物理系统才会从“靠试参数”变成真正可控制、可优化、可维护的工程系统。
