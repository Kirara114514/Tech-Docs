# Unity内置物理计算的数学模型与优化（碰撞检测、刚体模拟）

## 摘要
Unity 内置物理系统在日常开发中经常以组件、开关和少量参数的形式出现，因此很多团队会不自觉地把它理解为“主要靠经验调试”的黑盒能力。但只要项目开始依赖高速碰撞、复杂堆叠、刚体交互、连续检测、角色与环境混合控制、物理驱动玩法和大量查询，物理系统就不再只是“勾几个选项就能用”的工具，而是一套受到离散时间步、碰撞检测、接触生成、约束求解和数值积分共同约束的实时近似求解过程。本文从工程视角系统梳理 Unity 内置物理计算的数学模型与优化方法，重点解释碰撞检测为什么会漏、刚体为什么会抖、CCD 为什么不能乱开、插值为什么不等于精度提升，以及项目里常见的物理性能热点究竟来自哪里。目标不是把开发者训练成物理引擎研究者，而是让团队在使用 Unity 物理时，能够从“盲调参数”进化到“按模型理解、按边界优化、按场景取舍”。

## 正文

### 背景
在大多数 Unity 项目里，物理系统问题往往以非常具体的症状出现，例如子弹偶尔穿墙、箱体堆叠时抖动、角色站在斜坡上微微打滑、绳索或关节链晃动异常、某些碰撞明明应该发生却似乎没有命中、某个房间只多了几十个动态物体帧率却突然掉得很厉害。很多团队面对这些问题时，第一反应是调参数：提高 Solver Iteration、改 Collision Detection、增大 Fixed Timestep 精度、切换 Interpolate、增加质量或阻尼。参数当然有用，但如果不知道这些参数作用在物理流水线的哪个环节，就很容易出现一种典型局面：局部问题似乎缓解了一点，总体成本却明显升高，而且换一个场景后问题又以另一种形式出现。

根本原因在于，Unity 的内置物理系统并不是“真实世界的连续精确求解”，而是一种在有限实时预算下对刚体动力学和碰撞响应的离散近似。它需要在固定时间步内完成碰撞候选筛选、接触计算、约束构建、冲量求解和状态积分，然后把结果交给渲染系统显示。每一个阶段都在做近似，每一个阶段都可能成为瓶颈，每一个阶段也都对应不同类型的稳定性与性能问题。只要开发者不了解这一套过程，就很容易误把“近似求解的边界”当成“随机异常”，从而用错误的方向消耗优化预算。

另一方面，Unity 物理系统并不只服务于传统刚体堆叠或简单机关。今天的项目往往会把它接入角色推进、子弹命中、爆炸冲击、车辆、布娃娃、场景机关、碎片掉落、道具交互、环境查询、AI 感知和特效联动等多种场景。不同场景对精度、稳定性、确定性和性能的要求完全不同。若团队把所有需求都简单丢给通用物理系统，既容易造成不必要的成本，也会在一些关键玩法上得到比预期更差的控制感。因此，对 Unity 内置物理的理解，必须同时包含两层：一层是“它数学上大致在算什么”，另一层是“哪些需求真的适合交给它，哪些更适合做脚本化近似或专门方案”。

### 核心内容
#### 一、Unity 物理模拟首先是一种离散时间系统，而不是连续时间的完美复制
理解 Unity 物理的第一前提，是接受它在固定时间步上工作。真实世界中的物体运动是连续发生的，但游戏物理并不是每时每刻都在解算，而是在每个固定步长上采样和推进状态。也就是说，系统看到的是“上一时刻状态”和“这一时刻要推进到哪里”的近似过程，而不是连续曲线本身。

这件事带来两个关键后果。第一，任何高速运动都可能在离散采样之间跨过碰撞体，从而产生经典的 tunneling 问题，也就是常说的穿透。第二，物理系统的稳定性会显著受时间步大小影响，因为时间步越大，单步内发生的位移、速度变化和接触误差都会更大，求解器需要在更粗糙的信息上逼近合理结果。因此，很多物理问题的根其实不在某个碰撞器设置，而在于当前场景是否已经超出了给定固定时间步下可稳定求解的范围。

#### 二、刚体模拟的数学骨架，是“力与约束共同作用下的状态推进”
Unity 的刚体更新，并不是简单地把速度乘时间再加到位置上那么朴素。更准确地说，系统会在固定步中综合考虑外力、重力、速度、质量、阻尼、接触和约束，然后更新刚体的位置与姿态。若只看最基础形式，线性运动大致可理解为：外力决定加速度，加速度改变速度，速度再推进位置；角运动同理，只不过作用对象变成角速度与角位姿。

但一旦物体发生碰撞或被关节限制，系统就不能只依靠单纯积分，因为它还必须满足“不穿透”“接触面相互分离速度合理”“摩擦方向受限”“关节长度或角度不越界”等条件。这时，问题从单纯动力学推进变成了“动力学推进 + 约束修正”的组合问题。正是这个组合，使得 Unity 物理既能在大多数常规场景下表现得像真实物理，又注定无法在所有复杂场景下都稳定且精确。

#### 三、积分方法决定了误差如何积累，插值则决定了画面如何呈现
很多开发者会把 Interpolate 误解为“提高物理精度”，这其实混淆了两件不同的事。物理积分决定的是模拟状态如何随时间步推进，插值决定的是在渲染帧率与固定物理帧率不完全对齐时，画面如何更平滑地显示这些状态。也就是说，插值主要改善视觉连续性，而不是提升求解质量。

在工程实践中，这个区别非常重要。若角色移动发抖，是因为物理步与渲染步不同步导致的显示抖动，那么 Interpolate 很有价值；若箱体堆叠发抖，是因为求解本身不稳、质量比过大或接触链太长，那么开插值只会让结果看起来略平滑一点，并不会从根本上解决稳定性问题。理解“积分解决模拟、插值解决显示”的分工，能帮助团队避免大量错误调参。

#### 四、碰撞检测不是一步完成的，而是“先筛可能接触，再精确确认”
Unity 物理中的碰撞检测通常可分为两大阶段：Broadphase 和 Narrowphase。Broadphase 的任务不是精确判断谁撞上了谁，而是在大量物体中快速筛掉明显不可能相交的组合，只保留候选对。Narrowphase 再对这些候选对做更精细的接触计算，生成碰撞点、法线和穿透深度等信息。

这个分层设计本身就是性能优化，因为任何项目只要动态物体和碰撞体数量一多，理论上的物体对数量就会增长得非常快。如果每个物体都与所有其他物体精确检测，成本会立刻失控。Broadphase 的存在，本质上是在用空间划分、包围盒重叠和层级过滤等手段，把问题先压缩到“有可能碰”的子集里。

理解这一点后，很多优化判断就会更清楚。比如碰撞层矩阵为什么这么重要、为什么让大量不该交互的物体进入同一碰撞层会显著增加成本、为什么复杂 Mesh Collider 对动态场景不友好、为什么把大量细碎动态刚体塞进一个高密度区域会让 Broadphase 也开始变贵。这些都不是参数玄学，而是流水线结构决定的。

#### 五、Narrowphase 和接触生成，决定了“撞上以后系统到底知道了什么”
当 Broadphase 判定两个物体有相交可能后，Narrowphase 会进一步计算它们是否真正发生接触，以及接触的几何信息。对后续求解器而言，最关键的数据并不是“撞了没撞”这个布尔值，而是接触点在哪里、法线朝向如何、穿透或接近程度是多少、接触面相对速度怎样。因为后续所有冲量修正和摩擦计算都建立在这些信息之上。

工程上，这意味着碰撞体形状的选择非常关键。原始 primitive collider 往往比复杂网格碰撞体更容易得到稳定、可预测且廉价的接触信息。若一个物体明明可以用 Box、Capsule、Sphere 组合近似，却被直接使用复杂 Mesh Collider，不仅检测成本更高，接触结果往往也更容易出现不必要的细碎法线和接触抖动。很多“看起来是求解器不稳”的问题，根因其实出在接触信息本身就不够干净。

#### 六、刚体碰撞后的响应，本质上是冲量求解，而不是“引擎自动想办法弹开”
物理系统在得到接触信息后，接下来要解决的是：如何修改两个物体的速度，使它们满足不穿透、相对运动合理、摩擦受限等条件。这个过程通常不是直接改位置，而是通过求解接触约束产生冲量，冲量再作用于刚体的线速度和角速度。简化理解可以是：系统在不断寻找“最小但足够的修正”，让物体不再以不合理方式互相穿透。

这也是为什么质量、惯量、速度规模、约束数量和迭代次数都会影响最终结果。因为冲量求解是近似迭代过程，不存在“一次就精确完美”的万能解。场景越复杂，系统越需要在有限迭代中取得“看起来足够好”的结果。理解这一点后，开发者就更容易接受一个事实：很多时候所谓“稳定”，并不是绝对正确，而是在给定预算内误差被控制到玩家可接受。

#### 七、摩擦与反弹不是附属小参数，而是会深度影响求解稳定性的接触条件
在项目里，开发者经常把物理材质上的摩擦和反弹看成体验参数，例如“箱子更滑一点”“弹球更弹一点”。这当然是它们的表层效果，但从求解角度看，它们也是约束的一部分。尤其是高摩擦、大量接触点叠加、细碎表面、倾斜面和堆叠场景下，摩擦模型会显著影响解算收敛性和稳定性。

这意味着，不合理的摩擦设置不仅会让物体“感觉不对”，还可能让求解器更难稳定。很多团队在追求“更真实”时把摩擦系数设得很激进，结果箱体堆叠更容易抖、角色贴墙更容易卡、车辆接地行为更难调。对于游戏项目而言，摩擦和反弹更适合被视为“可控的交互参数”，而不是物理真实性崇拜的对象。

#### 八、连续碰撞检测（CCD）的价值，在于弥补离散时间步的天然盲区，但它绝不是全局默认选项
CCD 最常见的用途，是高速小物体或关键高速物体在固定时间步中不至于直接跨过碰撞体。它本质上是在“两个离散状态之间”做额外的碰撞考虑，使得系统不只看到当前位置和下一步位置，还在路径上估计是否发生过接触。对子弹、快速飞行道具、高速坠落关键物体和特定机关而言，这非常重要。

但 CCD 并不便宜，也不是所有穿透都靠 CCD 解决。若一个场景中大量普通刚体都被无差别打开连续检测，成本会迅速上升，而且有些问题其实是碰撞体建模、时间步设置或高速逻辑不合理导致的。更成熟的做法是明确“哪些对象对穿透敏感到必须使用 CCD”，并尽量局部启用，而不是把它当作默认保险。

#### 九、睡眠、激活范围和参与解算规模，决定了刚体系统是否能在复杂场景中维持预算
从性能角度看，Unity 物理中最昂贵的并不是“有多少碰撞体存在”，而往往是“有多少动态刚体正处于需要持续解算的状态”。静态碰撞体通常相对便宜，而大量动态刚体、频繁唤醒、密集接触和持续运动链则会让解算范围迅速扩大。物理睡眠机制的意义，正是在对象长时间趋于静止时停止不必要的持续求解。

因此，项目若存在大量“理论上是物理体，实际上大多数时间并不需要活跃”的对象，就应认真设计激活与睡眠策略。例如远处碎片是否真的要长期保持动态；被收纳进背包的道具是否还应保留动态刚体；场景中装饰性小件是否需要参与长时间物理模拟。很多看似物理预算紧张的问题，实际上只是系统没有把不必要的活跃体及时移出高成本状态。

#### 十、Fixed Timestep 不是“越小越高级”，而是需要与玩法需求和平台预算匹配
很多团队遇到物理不稳时，第一反应是减小 Fixed Timestep，例如从 0.02 调到 0.01，甚至更低。这样做确实能让单步推进更细、碰撞与求解误差减小，但它也会直接提高单位真实时间内的物理解算次数。因此，这不是免费的精度提升，而是拿 CPU 预算换稳定性。

更重要的是，Fixed Timestep 的价值必须结合具体场景判断。若问题只集中在少数关键高速物体或长约束链，全局减小时间步往往代价过大。很多时候，更合理的做法是先检查是否存在错误建模、是否可局部启用 CCD、是否可调整质量比和碰撞体形状、是否应对关键玩法做专门方案。只有当全局时间步确实是主要瓶颈，才值得考虑统一收紧。

#### 十一、Unity 物理优化的核心不是“减少对象”，而是“减少无意义的物理参与”
在很多团队里，物理优化很容易被粗暴理解为“把物体数量降下来”。这当然常有帮助，但更高质量的表达应当是：减少不必要参与物理流水线的对象、接触和查询。一个对象是否存在，不等于它是否必须动态刚体；一个碰撞体是否存在，不等于它是否必须参与所有层级的碰撞；一个交互是否需要“看起来有物理感”，不等于它必须走完整刚体模拟。

只要团队学会把“是否存在”和“是否参与昂贵物理解算”区分开，优化思路就会成熟很多。比如某些机关可以通过动画和脚本轨迹驱动，再在关键时刻用查询校验命中；某些碎片完全可以只做短时间物理后快速休眠；某些角色推进逻辑应当以角色控制器为主、物理反馈为辅，而不是让通用刚体承担所有移动控制责任。

#### 十二、角色控制、脚本瞬移和刚体物理混用，是造成异常最常见的工程问题之一
很多 Unity 物理问题并不是求解器本身有多难，而是不同控制范式混在一起使用。最典型的情况是：刚体一方面被物理系统解算，另一方面又在 Update 中被脚本直接改位置、直接设 transform、直接旋转，或者角色控制器逻辑和刚体运动同时生效。这种混用会破坏物理系统对连续状态的预期，使得碰撞、接触和插值结果都变得不稳定。

因此，任何物理驱动对象都应尽量明确控制来源：是以物理驱动为主，脚本只施加力和目标；还是以脚本控制为主，物理只做辅助检测；还是使用 CharacterController / Kinematic 路径。越早划清这一点，后续问题越少。

#### 十三、典型误区一：把所有穿透都归结为 CCD 不够强
穿透问题当然常与 CCD 有关，但并不是所有穿透都该靠开更强的碰撞检测解决。高速跨步穿透是一类问题，复杂 Mesh 细缝、极端时间步、脚本瞬移、错误碰撞层、触发器使用混乱和不合理的尺寸/速度比例也是问题来源。若团队一看到穿透就无差别提升碰撞检测模式，最终得到的往往是成本升高而问题并未系统消失。

更成熟的诊断方式应先问：这是离散时间步采样造成的路径遗漏，还是接触建模本身不合理，还是对象运动方式违背物理系统预期。不同原因对应完全不同的治理方向。

#### 十四、典型误区二：把 Solver Iteration 当成万能稳定性旋钮
迭代次数增加确实可能改善接触和约束求解质量，但它并不能修复所有失稳，更不能弥补错误结构。长链关节、大质量差、极高速度、小尺寸薄碰撞体、脚本强行改位姿、复杂摩擦条件等问题，单纯加迭代次数往往收益有限却很昂贵。尤其在大场景里，盲目提高全局迭代通常会把每个活跃物体的成本都一起抬高。

因此，迭代次数更适合作为“最后校准项”，而不是“第一反应”。在动它之前，更值得先检查建模是否合理、时间步是否匹配、局部关键体是否应该专项处理。

#### 十五、典型误区三：把视觉上的平滑需求和物理解算精度混为一谈
项目里常见一种现象：角色移动不够顺滑，于是有人建议提高物理精度；刚体旋转看起来抖，于是有人建议减少时间步。事实上，这类问题很多只是渲染显示和固定物理步不同步导致的视觉不连续，而不是物理本身算错。此时，插值、相机跟随策略和显示层同步方式通常比提高全局求解精度更有效。

理解“显示问题”和“模拟问题”的区别，可以为团队节省大量无意义的 CPU 预算。否则，很多本可通过显示层解决的问题，最后都会被误导到更贵的物理设置上。

### 实现方案
#### 一、先建立按流水线排查的诊断顺序
面对物理问题时，建议团队统一按以下顺序判断：这是碰撞候选筛选问题、接触生成问题、约束求解问题、时间步采样问题，还是显示插值问题。只要排查顺序稳定，团队就不会一开始就被参数面板牵着走，而能更快收敛到真正的责任环节。

#### 二、把碰撞体建模视为首要优化项
对大多数项目而言，优化物理的第一收益点通常不是复杂调参，而是碰撞体建模。优先使用 primitive collider；避免让高复杂网格承担动态交互主体；控制碰撞层矩阵，减少无意义候选对；对只做环境阻挡的区域保持静态和简化。只要这一步做得好，Broadphase 和 Narrowphase 的基础成本就会明显降低，后续很多稳定性问题也会自然减轻。

#### 三、建立动态刚体分级制度
建议将物理对象至少区分为：关键动态体、普通动态体、脚本驱动辅助体、纯查询体、静态阻挡体。关键动态体可接受更高精度和更严格验证；普通动态体应强调成本控制；脚本驱动辅助体尽量减少完整求解依赖；纯查询体不应随意进入动态刚体池；静态阻挡体则应尽可能保持廉价稳定。对象分级越清晰，优化越不容易陷入一刀切。

#### 四、按场景启用 CCD、插值和高迭代
连续碰撞检测、插值和更高迭代次数都属于“有价值但不能滥用”的能力。建议只对高速关键体、玩家强感知物体或少量高风险链路局部启用。若某个场景需要全局开启某种高成本设置，通常应先反问：当前玩法设计或物理建模是否已经在逼迫系统做不合理工作。

#### 五、把脚本控制边界写入开发规范
对带 Rigidbody 的对象，明确哪些情况下允许直接改 transform，哪些情况下必须通过 MovePosition / MoveRotation、速度或施力接口驱动，哪些对象只允许 Kinematic 方式移动。只要这条边界被写进规范，团队会少掉很多“偶现穿透”“接触很怪”“插值失效”类问题。

#### 六、建立关键物理样例场景与回放验证
建议为项目里最关键的物理玩法建立专项样例场景，例如高速投射体、车辆悬挂、堆叠道具、可推动机关、布娃娃死亡反馈、破碎物掉落等。每次参数或系统升级后，都在这些样例场景中做回放和对比。这样做的价值在于，物理问题不再依赖主观体感，而是有明确可重复验证的基线。

#### 七、验收标准：什么样的物理优化算真正有效
一个有效的 Unity 物理优化方案，至少应满足以下条件：性能热点能被明确归类到碰撞、求解、时间步或显示层；目标场景中的穿透、抖动或堆叠异常得到可重复改善；改动没有通过全局高成本设置粗暴掩盖局部问题；关键对象和普通对象在物理预算上有明确分级；后续开发者能够理解为什么某些对象使用 CCD、为什么某些对象必须简化碰撞体、为什么某些控制逻辑不能直接改 transform。若做不到这些，即便某个场景暂时顺了一点，也很难称之为真正成熟的物理优化。

### 总结
Unity 内置物理计算的数学模型，并不神秘，它本质上是离散时间上的碰撞检测、接触生成、冲量求解与状态积分的组合过程。真正让项目中的物理问题显得复杂的，不是公式本身，而是团队经常在不了解这套近似边界的情况下，把各种高速度、高密度、高耦合和混合控制需求一起塞进通用物理系统。只要开发者开始按流水线理解问题、按场景启用高成本特性、按对象重要性分配精度预算，并把碰撞建模和控制边界视为正式工程规范，物理系统就会从“只能靠经验盲调”的黑盒，变成“可以被解释、被验证、被治理”的可控基础设施。
