# C#委托与事件高级应用

## 摘要
委托与事件是 C# 中实现回调机制和组件通信的核心技术，在 Unity 开发中广泛应用。本文深入探讨委托的原理、事件的封装模式和在 Unity 中的高级应用，帮助开发者构建灵活可扩展的通信系统。

## 正文

### 背景
上次我们聊了 **C# 反射**，它让程序拥有了在运行时"看清自己"的能力。但光能看清还不够，我们还需要让代码能够灵活地"沟通"和"响应"。这就不得不提到 C# 中另外两个非常重要的概念：**委托（Delegate）** 和 **事件（Event）**。

作为 Unity 开发者，你可能每天都在使用它们，比如 Unity UI 按钮的 `OnClick` 事件、`SendMessage` 等等，它们背后都离不开委托和事件的思想。今天我们来深入探讨它们的进阶用法，以及如何构建起高效、解耦的回调和消息系统。

### 1. 委托基础回顾

委托是 C# 中的类型安全的函数指针。它定义了方法的签名，任何匹配签名的方法都可以赋值给委托实例。

```csharp
// 声明委托类型
public delegate void DamageHandler(int damage);

// 使用委托
public class Player
{
    public DamageHandler OnTakeDamage;
    
    public void TakeDamage(int amount)
    {
        OnTakeDamage?.Invoke(amount);
    }
}
```

C# 提供了预定义的泛型委托：`Action`（无返回值）和 `Func`（有返回值），大多数场景无需自定义委托类型。

### 2. 多播委托与调用链

委托本质上是多播的——一个委托实例可以同时引用多个方法：

```csharp
DamageHandler handler = ShowDamageEffect;
handler += PlayDamageSound;
handler += UpdateHealthBar;

handler(10); // 三个方法依次执行
```

多播委托的调用顺序与注册顺序一致，且所有方法共享同一个参数实例。如果某个方法抛出异常，后续方法不会执行。

### 3. 事件：受保护的委托

```csharp
public class Player
{
    // event 关键字提供封装
    public event DamageHandler OnDamageTaken;
    
    public void TakeDamage(int amount)
    {
        OnDamageTaken?.Invoke(amount);
    }
}
```

`event` 关键字本质上是一个语法糖，它在委托的基础上增加了访问限制：外部类只能 `+=` 或 `-=`，不能 `=` 赋值或直接 `Invoke`。这种封装保护了事件的发布者，防止外部代码清空或伪造事件调用。

### 4. 弱事件模式与内存泄漏

在 Unity 中，对象 A 订阅了对象 B 的事件，如果 A 被销毁但未取消订阅，B 仍然持有对 A 的引用（通过委托），导致 A 无法被 GC 回收。这就是事件泄漏的常见形式。

```csharp
// 订阅者被销毁前必须取消订阅
void OnDestroy()
{
    someManager.OnEvent -= HandleEvent;
}
```

在大型项目中，手动管理大量订阅容易遗漏。可以考虑使用弱事件模式（通过 `WeakReference`）或借助 UniRx 的 `AddTo` 机制统一管理生命周期。

### 5. UnityEvent：编辑器友好的事件系统

`UnityEvent` 是 Unity 对 C# 事件的封装，支持在 Inspector 中绑定：

```csharp
[System.Serializable]
public class FloatEvent : UnityEvent<float> { }

public class HealthController : MonoBehaviour
{
    public FloatEvent onHealthChanged;
    
    void UpdateHealth(float newValue)
    {
        onHealthChanged.Invoke(newValue);
    }
}
```

`UnityEvent` 的优势是编辑器可视化和序列化，但性能比原生 C# 事件差（约 10-20 倍开销）。因此高频事件路径不建议使用 `UnityEvent`。

### 工程化深化：从语法能力到事件治理

#### 1. 委托的核心价值是把"调用谁"从"何时调用"中分离出来
委托让方法调用变得可配置——调用者不需要知道具体接收方，接收方也不需要知道触发时机。这种"发布-订阅"解耦是委托和事件架构的核心思想。但解耦程度需要控制：全局事件总线的滥用会带来调用链追踪困难。

#### 2. 事件封装的粒度决定了可维护边界
好的事件封装应按照"谁拥有状态，谁触发事件"的原则划分边界。一个常见的错误是让无关模块互相订阅对方的事件。事件是一种耦合形式，只不过是被约束的耦合。合理的设计应在模块接口文档中列出发布和订阅的事件类型，事件风暴（Event Storming）是领域驱动设计中常用的设计方法。

#### 3. 委托链的异常隔离
多播委托中一个订阅者抛异常会阻断整条链。生产级事件系统可以选择让每个委托在 try/catch 中执行，或者为事件增加中间件管道。至少应确保异常不扩散到事件发布者和其他订阅者。

#### 4. 事件生命周期管理是 Unity 项目的关键难点
界面的打开关闭、对象池的激活回收、场景的加载卸载，都伴随着事件的注册和注销。一个容易遗漏的场景是对象池对象被回收时是否取消了对全局事件的订阅。规范的 Unity 项目应在基类中提供生命周期绑定机制，或要求所有订阅必须持有一个可释放令牌。

#### 5. 事件参数设计——从 object 到强类型消息
大多数初级项目使用 `object` 作为事件参数，这导致订阅者需要类型转换和文档查阅。推荐为每个事件定义强类型消息类或结构，或者使用 `UniRx` 的 `IObservable<T>`。参数不可变也能避免事件传递中的意外修改。

#### 6. 事件与异步——超时、取消与并发处理
事件本质上是同步的——`Invoke` 会按顺序阻塞执行所有订阅者。对于需要异步通知的场景，建议在事件层处理超时、取消和并发。如果事件发布后希望订阅者有异步反馈，可在事件参数中包含回调或完成令牌。

#### 7. 跨线程事件应进入主线程调度
在 Unity 中，大部分代码在主线程执行，但网络回包、文件 IO 和部分第三方 SDK 回调用在后台线程触发。直接在这些线程中调用 Unity API 会导致错误。应通过主线程调度层统一处理跨线程事件。

#### 8. 事件接口应有可测试性设计
事件发布不应和 Unity 场景对象强绑定，应放在普通 C# 类中。测试应覆盖顺序、重复订阅、解绑、异常隔离和生命周期结束后的不再回调。

### 实现方案

1. **优先使用 `event` 关键字而非裸委托**：提供外部封装，防止事件被清空或伪造。
2. **高频路径避免 `UnityEvent`**：改用原生 C# 事件或 UniRx 流。
3. **生命周期统一管理**：基类提供 `AddTo` 机制或 `IDisposable`，确保订阅在对象销毁时自动释放。
4. **强类型事件参数**：为每个重要事件定义专用消息类，避免 `object` 类型传递。
5. **异常隔离机制**：生产环境为多播事件增加 try/catch 包装，防止单点异常阻断整条链。

### 总结
委托和事件是 C# 中实现解耦通信的核心机制。在 Unity 中，合理使用 event 关键字保护事件边界、规范生命周期管理避免泄漏、以及选择合适的事件类型（原生事件 vs UnityEvent vs ReactiveCommand）是构建健壮通信系统的关键。
