
## 摘要

GC（Garbage Collection，垃圾回收）是 Unity/C# 开发中绕不开的基础机制。它的职责不是“优化性能”，而是自动管理托管内存：当托管对象不再被程序引用时，运行时会在合适的时机识别这些对象，并将它们占用的托管堆空间重新标记为可用。GC 让 C# 开发者避免了大量手动释放内存的工作，也降低了悬空指针、二次释放等传统内存错误的概率。

但在游戏开发场景中，GC 也会带来非常现实的性能问题。游戏主循环通常以 30 FPS、60 FPS、90 FPS 甚至更高帧率稳定运行，每一帧的时间预算十分有限。以 60 FPS 为例，单帧预算约为 16.67 毫秒；任何突发的托管堆扫描、对象图标记、堆扩展或回收暂停，都可能造成帧时间尖峰，最终表现为卡顿、掉帧、输入延迟或动画不连续。

本文面向 Unity 工程实践，对 GC 的基础概念、存在原因、托管堆工作方式、对象可达性、GC Roots、常见回收流程、Unity 中增量 GC 的意义与边界、GC Alloc 的来源、内存泄漏与引用链、Profiler 分析方法以及工程化治理策略进行系统整理。本文不追求把所有运行时实现细节讲成虚拟机源码级别，而是强调开发者在项目中真正需要理解和使用的知识：哪些对象会进入托管堆、哪些写法会产生分配、为什么“业务上不用了”的对象仍然可能无法回收、为什么减少 GC Alloc 不等于解决所有内存问题，以及如何用工业化标准建立可验证、可回归、可长期维护的 GC 优化体系。

经过重写后，本文会移除原稿中的重复段落和容易误导的表述，并将内容从“入门说明”升级为可用于团队内部知识库、性能优化培训和代码评审规范的正式技术文档。

## 正文

### 背景

Unity 项目的性能问题往往不是由单一因素造成的。CPU 主线程逻辑、渲染提交、物理模拟、动画采样、UI 重建、资源加载、脚本热路径、内存分配、原生资源管理、显存压力都会影响最终帧率。GC 只是其中一个环节，却是最容易被开发者感知、也最容易被误解的环节之一。

很多开发者第一次关注 GC，通常是因为在 Unity Profiler 中看到了 `GC Alloc`，或者在 Timeline 中看到某一帧出现了 `GarbageCollect`、`GC.Collect`、`CollectIncremental` 等相关标记。也有一些项目是在低端移动设备上出现周期性卡顿后，才发现运行过程中每隔几秒就会触发一次托管堆回收。更常见的情况是，编辑器里看起来问题不大，一上真机就暴露出明显帧抖动；或者高端设备可以承受分配压力，低端设备却因为 CPU 余量不足而把 GC 成本放大成肉眼可见的卡顿。

要正确处理这类问题，首先要明确一个前提：GC 管理的是托管内存，不是 Unity 项目中的全部内存。Unity 项目中的内存大致可以分为以下几类：

1. **托管内存**：由 C# 运行时管理，典型对象包括普通类实例、字符串、数组、装箱后的值类型、委托对象、闭包对象、协程状态机、部分集合对象等。
2. **原生内存**：由 Unity 引擎、插件或底层系统分配，典型对象包括 Texture、Mesh、AudioClip、AnimationClip、AssetBundle、引擎内部对象、NativeArray 背后的 native buffer 等。
3. **显存**：由图形 API 和 GPU 资源占用，例如纹理、RenderTexture、顶点缓冲、索引缓冲、各种渲染中间资源。
4. **栈内存与线程相关内存**：方法调用栈、线程栈、部分运行时内部结构等。

GC 主要处理第一类，也就是托管内存。它不会因为一张纹理“业务上不用了”就自动释放显存，也不会替代 `Destroy`、Addressables Release、AssetBundle Unload、对象池清理、NativeContainer Dispose 等资源生命周期管理工作。反过来，一个 C# 托管包装对象被 GC 回收，也不等价于其背后的原生资源一定已经被正确释放。Unity 内存优化必须同时理解托管对象和原生资源的边界，否则很容易把所有内存上涨都错误归因于 GC。

原稿中反复强调“GC 是卡顿罪魁祸首”，这个方向有一定现实依据，但表述需要更严谨。GC 确实可能造成卡顿，尤其是高频分配导致托管堆频繁回收时；但 GC 不是所有卡顿的根源，也不是所有内存问题的答案。工业化项目中更准确的说法是：**GC 是 Unity 脚本层内存分配和对象生命周期治理的重要组成部分，它的风险主要表现为托管分配压力、回收暂停、堆增长、引用链泄漏和低端设备帧稳定性下降。**

### 核心内容

#### 1. 什么是 GC

GC，全称 Garbage Collection，中文通常称为垃圾回收。它是一种自动内存管理机制，运行时会在程序运行过程中识别不再被引用的托管对象，并将这些对象占用的内存重新变为可分配空间。

在没有 GC 的语言或场景中，开发者往往需要自己管理对象生命周期。例如在 C++ 中，开发者可能通过 `new` 分配对象，再通过 `delete` 释放对象。如果忘记释放，容易产生内存泄漏；如果提前释放，容易产生悬空指针；如果重复释放，则可能导致崩溃或未定义行为。手动内存管理虽然能提供更强控制力，但对工程纪律要求极高，尤其在大型项目、多人协作、异步流程复杂、对象关系庞大的游戏工程中，错误概率会显著上升。

C# 选择了另一种模式：绝大多数普通对象由运行时托管。开发者负责创建对象和维护引用关系，运行时负责在合适时机回收不再可达的对象。GC 的核心价值在于降低内存管理复杂度，使业务开发者不必在每个对象生命周期结束时显式释放托管对象。

但是，GC 不是免费的。它至少会带来以下成本：

- **分配成本**：对象进入托管堆需要运行时分配空间并维护元数据。
- **扫描成本**：回收时需要从根对象出发扫描对象引用关系。
- **标记成本**：运行时需要标记哪些对象仍然存活。
- **暂停或切片成本**：某些阶段需要暂停或协调用户代码执行。
- **堆增长成本**：如果现有托管堆空间不够，运行时可能扩展堆。
- **长期对象成本**：大量长期存活对象会增加之后扫描对象图的压力。

因此，GC 的定位应该是“安全性与开发效率机制”，而不是“性能优化机制”。在 Unity 游戏中，我们通常不是要关闭 GC，也不是要完全消灭所有分配，而是要让分配发生在可控时机、让高频路径尽量无分配、让对象生命周期清晰、让回收成本不会破坏帧稳定性。

#### 2. 托管堆是什么

托管堆是由脚本运行时管理的一块内存区域。C# 中绝大多数引用类型对象都会分配在托管堆上，例如：

- 普通 `class` 实例；
- `string` 字符串；
- 数组；
- `List<T>`、`Dictionary<TKey, TValue>` 等集合对象本身；
- 委托与事件相关对象；
- Lambda 捕获外部变量生成的闭包对象；
- 协程编译生成的状态机对象；
- `async/await` 编译生成的状态机及相关 continuation；
- 装箱后的值类型对象；
- 反射、LINQ、格式化字符串等运行时产生的临时对象。

值类型并不必然进入托管堆。局部的 `int`、`float`、`Vector3` 等值类型通常可以位于栈上或内嵌在对象内部。但当值类型发生装箱时，它会被包装成托管堆对象。例如把 `int` 当作 `object` 传递，或者通过非泛型接口处理值类型，都可能产生装箱分配。

一个非常重要的结论是：**GC 关注的是托管堆对象，而不是变量名本身。** 局部变量只是引用对象的路径之一。对象能否被回收，不取决于某个变量是否“看起来不用了”，而取决于运行时是否还能从 GC Roots 出发找到它。

可以用一个简化模型理解托管堆：

```text
托管堆：
[Object A] [Object B] [String X] [Array Y] [Closure Z] ...

引用关系：
Root -> Object A -> Array Y
Root -> Closure Z -> Object B
String X 无任何可达路径
```

在这个模型中，`Object A`、`Array Y`、`Closure Z`、`Object B` 仍然可达，因此应被视为存活；`String X` 无法从任何 Root 到达，因此可以被回收。

#### 3. GC Roots 与可达性分析

GC 判断对象是否应该保留，通常基于可达性分析。可达性分析的起点称为 GC Roots。GC Roots 可以理解为一组运行时认为“当前仍然有效、不能随意丢弃”的引用来源。

在 Unity/C# 场景中，常见 Root 或近似 Root 来源包括：

- 当前线程调用栈上的局部变量和参数；
- 静态字段；
- 当前正在执行的方法上下文；
- 委托和事件订阅链；
- 协程状态机持有的字段；
- async 状态机和 continuation；
- 运行时内部对象；
- Unity 引擎与脚本对象之间的桥接引用；
- 单例对象、全局管理器、缓存容器；
- 未清理的对象池、字典、列表、队列；
- Native 与托管互操作中被保持的托管引用。

当 GC 运行时，它会从这些根出发，沿着引用关系遍历对象图。能被遍历到的对象被认为是存活对象，不能被遍历到的对象被认为是垃圾对象。

这也是为什么“关闭 UI 面板”并不等于“UI 面板对象可以被 GC”。例如：

```csharp
public class GlobalEventHub
{
    public static event Action OnMoneyChanged;
}

public class BagPanel
{
    public void Open()
    {
        GlobalEventHub.OnMoneyChanged += Refresh;
    }

    public void Close()
    {
        // 如果这里忘记 -= Refresh
        // GlobalEventHub 仍然通过事件持有 BagPanel 实例
    }

    private void Refresh()
    {
    }
}
```

在业务语义上，`BagPanel` 已经关闭了；但在引用关系上，静态事件仍然持有它的方法引用，方法引用又指向实例对象。因此 GC 会认为这个面板仍然可达，不会回收它。这类问题本质上不是“GC 不工作”，而是“引用链仍然存在”。

工业化内存排查中，最关键的能力不是背 GC 名词，而是看懂引用链：某个对象为什么还活着？谁持有它？这个持有关系是否符合业务生命周期？它应该在哪个阶段解除？

#### 4. GC 的典型工作流程

不同运行时的 GC 实现细节并不完全相同，但从开发者角度，可以抽象成几个阶段：

1. **触发条件出现**：托管堆空间不足、分配压力达到阈值、显式调用回收、运行时认为需要回收等。
2. **枚举根对象**：运行时识别当前所有 GC Roots。
3. **标记存活对象**：从 Roots 出发遍历引用关系，标记可达对象。
4. **处理不可达对象**：不可达对象被认为可以回收。
5. **更新堆状态**：释放出的空间被加入可用区域，供后续分配使用。
6. **必要时扩展堆**：如果回收后仍不足以满足分配需求，运行时可能扩展托管堆。

可以用伪代码描述：

```text
CollectGarbage():
    roots = EnumerateGCRoots()

    for root in roots:
        MarkReachableObjects(root)

    for obj in managedHeap:
        if obj is not marked:
            Reclaim(obj)
        else:
            ClearMark(obj)
```

这个伪代码刻意省略了复杂实现细节，例如写屏障、对象头、卡表、并发/增量标记、终结器、固定对象、线程同步等。对 Unity 业务开发者而言，最重要的是理解两个事实：

- GC 必须知道哪些对象仍然被引用；
- 对象图越大、引用关系越复杂、存活对象越多，回收工作量通常越大。

#### 5. 常见 GC 算法的工程意义

经典教材中常见的 GC 算法包括标记-清除、复制、标记-整理、分代回收等。理解这些算法有助于建立基本概念，但不能简单把桌面 .NET 或 Java 虚拟机的所有行为直接套到 Unity 项目中。Unity 在不同版本、不同后端、不同平台上的运行时细节可能不同。工程文档中更推荐讲“算法思想”和“项目影响”，不要把不确定实现写成绝对结论。

**标记-清除**的思想是先标记存活对象，再回收未标记对象。它的优点是概念直接，缺点是可能产生碎片，并且标记过程需要扫描对象图。

**复制算法**将存活对象从一个区域复制到另一个区域，天然有整理效果，但需要额外空间，并且当存活对象很多时复制成本会增加。

**标记-整理**会在标记存活对象后移动对象，使存活对象更紧凑，从而降低碎片问题，但移动对象和更新引用本身成本较高。

**分代 GC**基于“多数对象生命周期很短”的经验，将对象按生命周期划分为不同代。桌面 .NET 的 GC 是典型分代 GC；但 Unity 文档更强调其托管堆、标记、增量回收与分配行为。在写 Unity GC 文档时，分代 GC 可以作为通用理论背景介绍，但不应直接断言 Unity 的 GC 行为完全等同于现代桌面 .NET 分代 GC。否则团队成员在分析 Profiler 时可能产生错误预期。

对 Unity 开发者来说，更实用的结论是：

- 高频创建短生命周期对象会造成持续 GC Alloc；
- 长期存活对象会扩大对象图扫描压力；
- 静态引用、事件、闭包、缓存会延长对象生命周期；
- 增量 GC 可以摊薄部分回收工作，但不能消除分配成本；
- GC 优化必须结合目标平台和真实 Profiler 数据。

#### 6. Unity 中的 GC 与普通 .NET 的差异意识

Unity 使用 C# 作为主要脚本语言，但 Unity 项目的运行环境并不等价于普通服务器 .NET 或桌面 .NET 应用。Unity 项目存在以下特殊性：

1. **帧循环敏感**：游戏以帧为单位运行，几十毫秒的暂停就可能明显影响体验。
2. **主线程压力集中**：大量 Unity API 必须在主线程使用，GC 与主线程逻辑竞争帧预算。
3. **托管与原生混合**：C# 对象经常只是 Unity 原生对象的包装层。
4. **平台差异明显**：PC、主机、移动端、WebGL、VR/AR 对内存和 CPU 的容忍度不同。
5. **编辑器与真机差异大**：Editor 下的分配、采样、脚本调试、域重载、Profiler 开销都可能影响判断。
6. **IL2CPP 与 Mono 后端差异**：构建后端、平台运行时和编译方式会影响性能表现。

因此，Unity GC 优化不能只靠“C# 语言经验”，必须以 Unity Profiler、Memory Profiler、真机测试和目标帧率预算为依据。

#### 7. 增量 GC 的意义与边界

Unity 现代版本提供增量 GC。增量 GC 的核心思想是把一次完整 GC 工作拆成多个较小的切片，分散到多帧执行，从而降低单帧长暂停的概率。它对于降低明显卡顿很有价值，尤其是在对象图较大、一次性完整扫描成本较高的项目中。

但增量 GC 不是“GC 免费化”。它只是改变了成本分布：从一次较大的暂停，变成多帧中持续的小段工作。对帧稳定性来说，这通常更友好；但如果项目每帧持续分配大量临时对象，增量 GC 仍然要处理持续增长的回收压力。

可以这样理解：

```text
非增量 GC：
某一帧集中做大量回收工作 -> 可能出现明显尖峰

增量 GC：
多帧分摊回收工作 -> 单帧尖峰降低，但总工作量仍然存在
```

增量 GC 的工程边界包括：

- 它不能阻止代码产生 GC Alloc；
- 它不能释放仍然可达的对象；
- 它不能管理原生资源和显存；
- 它仍然会消耗 CPU 时间；
- 它的收益需要在目标设备上验证；
- 它不能替代对象生命周期治理和资源释放规范。

对于 CPU 余量充足的平台，增量 GC 可能表现很好；但在低端移动设备、复杂战斗、UI 大量重建、同屏实体多、热更新框架开销重的场景中，增量切片本身也可能造成帧时间抖动。因此，项目不能因为开启了增量 GC 就放弃 GC Alloc 控制。

#### 8. GC Alloc 的真正含义

Unity Profiler 中的 `GC Alloc` 通常表示某段代码在该帧或采样范围内产生了托管堆分配。它不是“GC 已经发生”的意思，而是“产生了可能由 GC 管理的托管分配”。持续的 GC Alloc 会增加托管堆压力，并最终提高 GC 触发概率。

很多初学者看到 `GC Alloc` 会立刻紧张，但更成熟的判断应该结合以下问题：

- 分配发生在加载阶段、初始化阶段，还是运行时高频阶段？
- 分配是一次性的，还是每帧持续发生？
- 分配量是几十字节、几 KB，还是几 MB？
- 分配对象是否很快变成垃圾，还是会长期存活？
- 分配是否发生在战斗、主城、滚动列表、相机跟随、输入响应等关键路径？
- 分配是否会在低端机上触发明显 GC？
- 分配是否有简单、安全、低侵入的替代写法？

工程上不应追求全项目绝对 0 GC Alloc。合理目标应该是：**关键运行路径稳定低分配或零持续分配；加载、切场景、打开大型 UI 等低频阶段允许可解释、可预算的一次性分配；所有持续分配必须能被定位、解释和验收。**

#### 9. 为什么 GC 会导致卡顿

GC 导致卡顿的直接原因是回收工作需要时间，并且某些阶段会阻塞或协调用户代码。即使使用增量 GC，运行时也需要在某些时间片执行扫描、标记或相关维护工作。游戏帧预算非常敏感，因此这些成本会直接反映到帧时间上。

以 60 FPS 为例：

```text
单帧预算 ≈ 16.67ms
主线程逻辑  : 6ms
渲染提交    : 4ms
动画/物理/UI : 4ms
剩余余量    : 2.67ms
```

如果某一帧额外出现 8ms 的 GC 或托管堆相关开销，那么该帧就会超过预算。玩家看到的不是“GC 执行了 8ms”，而是画面卡了一下、输入反馈慢了一下、动画突然跳了一下。

GC 卡顿通常来自以下几类情况：

1. **高频临时分配**：每帧创建字符串、集合、闭包、数组、委托等对象。
2. **对象图过大**：大量长期存活对象、缓存、池对象、管理器引用，使扫描成本增加。
3. **堆频繁扩展**：持续分配导致托管堆不断增长。
4. **泄漏式引用保留**：对象业务上失效但仍被静态事件、缓存、闭包等引用。
5. **显式调用 `GC.Collect`**：在错误时机强制完整回收，造成尖峰。
6. **编辑器误判**：Editor 下额外分配或 Profiler 开销掩盖真机问题。

#### 10. 常见 GC Alloc 来源

Unity 项目中大量 GC Alloc 并不是来自显眼的 `new SomeClass()`，而是来自看起来很“正常”的语法糖或 API 使用方式。

**字符串操作**是最常见来源之一：

```csharp
scoreText.text = "Score: " + score;
```

如果这段逻辑每帧执行，就会持续产生新字符串。更合理的策略是仅在分数变化时更新，或者降低刷新频率。

```text
if score changed:
    scoreText = cachedPrefix + score
```

**装箱**也很隐蔽：

```csharp
object value = 10; // int 装箱
Debug.Log("hp=" + hp); // 可能产生字符串分配
```

**LINQ** 在热路径中也容易产生迭代器、闭包或临时集合：

```csharp
var alive = units.Where(u => u.Hp > 0).ToList();
```

在初始化工具、编辑器脚本、低频配置处理里使用 LINQ 没问题；但在 `Update`、战斗 tick、滚动列表刷新中使用，就需要慎重。

**闭包和 Lambda 捕获**常常同时带来分配和生命周期风险：

```csharp
button.onClick.AddListener(() => OpenItem(itemId));
```

如果这个按钮属于可复用 UI Item，且回收时没有清理监听，那么旧闭包可能持有旧数据，甚至持有整个 Item 对象图。

**协程**会生成状态机对象：

```csharp
StartCoroutine(WaitAndPlay());
```

协程很适合低频流程控制，但如果大量实体每帧或高频反复启动协程，就会产生额外分配和调度成本。

**async/await** 也会生成状态机及 continuation，尤其在捕获上下文、闭包、异常路径、Task 分配等情况下更明显。异步代码的核心风险不只是 GC，还包括生命周期取消、旧结果写回、对象销毁后 continuation 仍执行。

**Unity API 返回数组或集合**也可能分配：

```csharp
var components = GetComponents<ComponentType>();
```

在高频路径中应优先使用可复用 List 的重载或缓存结果，具体取决于 API 和 Unity 版本。

**日志**是非常容易被忽略的来源：

```csharp
Log.Debug("pos=" + transform.position + ", state=" + state);
```

即使日志系统内部最终没有输出，只要调用前已经完成字符串拼接，分配就已经发生。发布构建和高频路径中应避免无条件构建复杂日志字符串。

#### 11. 字符串与 UI 文本优化

UI 文本是 Unity 项目 GC 优化高发区。倒计时、血条、伤害数字、FPS、延迟、货币数量、任务进度、排行榜刷新等逻辑，都会频繁修改文本。一旦每帧拼接字符串，就容易产生持续分配。

优化原则包括：

1. **只在值变化时刷新**。
2. **降低刷新频率**，例如 FPS、Ping、倒计时可以每 0.2 秒或 0.5 秒刷新一次。
3. **避免重复格式化**，对常见数字或枚举文案做缓存。
4. **区分 GC 和 UI 重建成本**，TextMeshPro 文本变化还可能触发网格重建，不能只看 GC Alloc。
5. **避免在隐藏 UI 上继续刷新文本**。
6. **避免每帧构建富文本字符串**。

伪代码如下：

```text
UpdateHpText(currentHp):
    if currentHp == lastHp:
        return

    lastHp = currentHp
    hpText.SetText(currentHp)
```

对于大量飘字、伤害数字、聊天消息等对象，除了减少字符串分配，还要考虑对象池、布局重建、字体材质、网格生成和批处理影响。GC 只是 UI 性能的一部分，不应孤立优化。

#### 12. 集合分配与集合池

临时集合是另一个常见分配来源。例如在一次技能范围搜索中创建一个新 `List<Unit>`，在敌人筛选中创建一个新列表，在 UI 刷新中创建临时字典，都可能造成持续分配。

简单写法：

```csharp
List<Unit> targets = new List<Unit>();
FindTargets(targets);
```

如果这段代码在高频路径中执行，可以改为复用集合：

```text
targets.Clear()
FindTargets(targets)
Use(targets)
```

或者使用集合池：

```text
list = ListPool.Get()
try:
    Fill(list)
    Use(list)
finally:
    list.Clear()
    ListPool.Release(list)
```

但集合池不是无脑万能药。它有非常严肃的所有权风险：**归还到池中的集合不能再被任何外部逻辑使用。**

错误示例：

```text
list = ListPool.Get()
AsyncUseLater(list)
ListPool.Release(list)
```

如果 `AsyncUseLater` 在下一帧才读取列表，而列表已经被归还并被其他逻辑清空或复用，就会出现非常隐蔽的 bug。集合池适合短生命周期、同步使用、调用边界清晰的临时缓冲；不适合跨帧、跨异步、保存到字段、返回给外部长期持有的集合。

工程规范中应明确：

- 池化集合的 owner 必须清晰；
- Release 后不得继续访问；
- 不允许把池化集合直接暴露给外部保存；
- 跨帧数据使用普通集合或复制必要数据；
- Debug 构建中可以增加“已释放标记”辅助排查。

#### 13. 对象池的收益与代价

对象池可以减少频繁创建和销毁带来的分配，常用于子弹、特效、飘字、UI Item、伤害数字、临时战斗对象等。对象池的基本思路是对象不使用时不立即销毁，而是重置状态后放回池中，下次再取出复用。

伪代码：

```text
Spawn():
    if pool not empty:
        obj = pool.Pop()
    else:
        obj = CreateNew()

    obj.ResetForSpawn()
    obj.SetActive(true)
    return obj

Despawn(obj):
    obj.StopCoroutines()
    obj.UnsubscribeEvents()
    obj.ClearRuntimeRefs()
    obj.SetActive(false)
    pool.Push(obj)
```

对象池的收益是减少分配和实例化成本；代价是增加常驻内存和生命周期复杂度。池中对象仍然是存活对象，GC 需要知道它们存在。池容量过大，会让内存长期占用偏高；池中对象如果没有清理事件、协程、异步请求、临时引用，就可能把旧状态和旧对象图一直保留下来。

对象池验收不能只看 `GC Alloc` 降低，还要看：

- 常驻内存是否可接受；
- 池容量是否有上限；
- 场景退出是否清理；
- 对象复用后是否存在旧状态残留；
- 是否解绑事件；
- 是否停止协程和取消异步；
- 是否清理引用字段；
- 是否避免池对象互相引用导致泄漏。

在工业项目中，对象池属于性能工具，不是默认架构。只有高频创建、生命周期短、复用收益明确的对象才适合池化。低频对象无脑池化，往往只是把简单问题变成复杂问题。

#### 14. 闭包、事件与生命周期泄漏

闭包是 C# 很方便的语言特性，但在 Unity 中经常制造隐藏引用链。Lambda 捕获外部变量时，编译器会生成一个闭包对象，用来保存被捕获的变量。如果这个 Lambda 被长期保存，那么闭包对象和被捕获对象也会被长期保存。

例如：

```csharp
public void Bind(ItemData data)
{
    button.onClick.AddListener(() => ShowDetail(data));
}
```

如果 `button` 复用，但没有在解绑时清理旧 listener，就可能产生以下问题：

- 点击一次触发多个旧回调；
- 旧 `ItemData` 被闭包持有，无法回收；
- UI Item 或 Panel 被事件链持有；
- 对象池复用后出现旧数据串台。

更安全的模式是：

```text
OnBind(data):
    currentData = data
    button.RemoveListener(OnClick)
    button.AddListener(OnClick)

OnClick():
    ShowDetail(currentData)

OnRecycle():
    button.RemoveListener(OnClick)
    currentData = null
```

静态事件风险更大。静态对象生命周期通常接近整个进程，如果业务对象订阅静态事件却不解绑，就会被长期持有。常见泄漏路径如下：

```text
StaticEvent -> Delegate -> Target Instance -> Panel -> Child Views -> Textures/Models/Data
```

排查这类问题时，不要只看某个对象是否 `SetActive(false)` 或是否从父节点移除，而要看是否仍存在从静态字段、管理器、事件、缓存到该对象的引用链。

#### 15. 协程与 async 的 GC 风险

协程和 async/await 都会把执行过程拆成状态机。状态机需要保存局部变量、执行位置和后续逻辑，因此可能产生托管对象。低频使用没有问题，例如加载流程、剧情演出、网络请求、UI 打开动画等；高频创建则需要注意。

协程常见风险：

- 每次 `StartCoroutine` 产生状态机对象；
- `new WaitForSeconds(x)` 可能产生分配；
- 对象销毁后协程仍然持有引用；
- 对象池回收时未停止协程，复用后旧逻辑继续执行；
- 在大量实体上用协程模拟帧循环，调度和分配成本上升。

async 常见风险：

- 状态机和 continuation 分配；
- 捕获 `this` 延长对象生命周期；
- 没有 CancellationToken，导致对象关闭后异步结果仍写回；
- 异常路径分配和日志成本；
- fire-and-forget 任务失控。

更工程化的原则是：

```text
低频流程：允许协程/async，重点处理取消和生命周期
高频循环：优先集中 Update/Tick，避免大量状态机创建
对象池对象：回收时必须停止协程、取消异步、清理回调
UI 面板：关闭时取消未完成请求，避免旧结果写回
```

GC 优化不能只盯着分配数量，还要同时治理异步生命周期。否则即使分配降低，仍可能出现对象泄漏、旧数据写回、界面关闭后回调执行等问题。

#### 16. Unity 原生资源不是 GC 的职责

Unity 资源管理中最常见的误区之一，是把原生资源释放和 GC 回收混为一谈。以 `Texture2D` 为例，C# 层对象只是托管包装，真正的大块数据可能位于原生内存或显存。GC 只能处理托管包装对象是否可达，不能替代 Unity 对底层资源的生命周期管理。

例如：

```text
Texture2D wrapper: 托管对象，由 GC 关注
Texture native data: 原生内存，由 Unity 资源系统管理
GPU texture: 显存资源，由图形系统管理
```

如果业务逻辑不再需要一张运行时创建的纹理，通常需要通过合适的 Unity API 释放其底层资源，而不是指望 GC 自动解决全部问题。同理，Addressables、AssetBundle、Resources、RenderTexture、NativeArray、ComputeBuffer 等都有自己的释放规范。

工程上应建立两个不同的问题分类：

- **GC 问题**：托管分配、托管堆增长、GC Pause、可达对象泄漏；
- **资源问题**：贴图、网格、音频、材质、Bundle、显存、NativeBuffer 生命周期错误。

这两类问题可能同时出现，但分析工具和修复方式不同。GC 问题重点看 `GC Alloc`、Managed Heap、对象引用链；资源问题重点看 Native Object、Asset 引用、显存占用、加载卸载流程和资源系统引用计数。

#### 17. Profiler 如何分析 GC

Unity 提供了多个层级的内存分析工具。常用工具包括：

1. **CPU Profiler**：查看每帧调用栈、GC Alloc、GC 相关采样点。
2. **Memory Profiler Module**：查看内存类别、托管堆、对象数量等高层指标。
3. **Memory Profiler Package**：抓取内存快照，分析对象保留、引用链、Native/Managed 内存布局。
4. **Timeline 视图**：查看 GC 是否造成主线程帧尖峰。
5. **真机 Profiler**：验证目标设备上的真实表现。

分析 GC Alloc 时，建议流程如下：

```text
1. 在目标场景录制 Profiler
2. 找到持续 GC Alloc 的帧
3. 展开 CPU 调用栈
4. 定位具体方法和代码路径
5. 判断发生频率和分配量
6. 区分一次性分配与持续分配
7. 修改代码后再次录制对比
```

分析泄漏或对象无法回收时，建议流程如下：

```text
1. 进入场景并抓取 Memory Snapshot A
2. 执行打开/关闭 UI、进入/退出战斗等流程
3. 手动触发合理的卸载和清理流程
4. 抓取 Memory Snapshot B
5. 对比对象数量和 Managed Object 增长
6. 查看异常保留对象的引用链
7. 找到静态字段、事件、缓存、池、协程或异步持有者
8. 修复生命周期并复测
```

注意：Profiler 本身也有开销，Editor 与 Player 表现差异明显。最终结论应以目标平台、目标画质、目标帧率、真实玩法路径为准。

#### 18. 显式调用 GC.Collect 的风险

`GC.Collect()` 可以请求立即执行垃圾回收，但在 Unity 项目中不应随意使用。它可能造成明显暂停，并且如果调用时机不当，会比自动 GC 更破坏帧稳定性。

合理使用场景通常非常有限，例如：

- 大型场景切换完成后；
- 加载界面期间；
- 玩家不可交互的过渡阶段；
- 已经完成大量临时对象释放，并且能接受一次性成本；
- 经过真机验证，确认该调用比自动触发更稳定。

不合理场景包括：

- 每隔几秒固定调用；
- 战斗中调用；
- UI 打开关闭时无条件调用；
- 因为看到内存上涨就调用；
- 试图用它掩盖引用泄漏。

`GC.Collect()` 不能回收仍然可达的对象。如果某个 UI 面板被静态事件持有，强制 GC 也不会释放它。泄漏问题必须修引用链，不是靠强制回收解决。

#### 19. GC 优化预算

成熟项目不会只说“少 GC”或“禁止 GC”，而是建立明确的 GC 预算。预算应按场景、设备等级和玩法频率制定。

示例预算：

```text
战斗运行期：
- 稳态每帧 GC Alloc 目标：0 B 或接近 0 B
- 技能释放瞬间：允许少量可解释分配，但不得持续
- 单次 GC Pause：不得造成可感知帧尖峰

主城待机：
- 每帧持续 GC Alloc：原则上为 0
- 心跳、红点、活动入口刷新：不得产生周期性大分配

大型 UI 打开：
- 打开瞬间允许一次性分配
- 打开后静止状态不得持续分配
- 关闭后对象数量应回落或进入可解释对象池

列表滚动：
- 滚动过程中不得因每个 Item 绑定产生持续闭包/字符串分配
- 复用 Item 必须清理旧事件和旧数据引用

场景切换：
- 允许加载阶段集中分配
- 切换完成后应执行资源释放与托管引用清理
- 必须验证多次切换后内存平台稳定
```

预算的价值在于让优化从口号变成可验证指标。没有预算，团队很容易陷入两种极端：要么完全不管 GC，直到上线前卡顿爆炸；要么机械禁止所有 `new` 和 LINQ，使代码复杂度上升但收益不明确。

#### 20. 代码评审中的 GC 检查点

GC 相关代码评审应关注“分配位置、发生频率、对象生命周期、引用清理”四个维度，而不是简单看到 `new` 就打回。

高风险检查点包括：

- `Update`、`LateUpdate`、`FixedUpdate` 中创建字符串、集合、闭包、数组；
- 战斗 Tick 中使用 LINQ、反射、装箱、格式化字符串；
- UI Item 绑定时添加 Lambda 但回收不清理；
- 静态事件订阅没有解绑路径；
- 对象池回收不停止协程、不取消异步、不清字段；
- 集合池对象跨帧持有；
- 日志在高频路径中无条件拼接字符串；
- 字典、列表、缓存只增不减；
- async 任务没有取消令牌或 owner；
- 关闭 UI 后仍有计时器、订阅、协程、网络回调运行；
- 资源释放只清 C# 引用，不处理 Addressables/Bundle/Destroy/Dispose。

更合理的评审问题是：

```text
这个分配发生在什么频率？
它是否位于关键帧路径？
这个对象谁持有？什么时候释放？
如果 UI 关闭/对象回收/场景切换，它会不会继续被引用？
有没有 Profiler 数据证明这里是热点？
替代方案是否会显著降低可读性或引入生命周期风险？
```

#### 21. 常见误区澄清

**误区一：只要没有 `new` 就没有 GC Alloc。**

错误。字符串拼接、装箱、闭包、协程、LINQ、数组返回、委托组合、反射、日志格式化都可能产生托管分配。

**误区二：GC Alloc 越少，内存问题就越少。**

不完全正确。GC Alloc 主要反映分配压力，但内存泄漏常常来自对象仍然可达。一个系统可能几乎不分配，却因为静态缓存一直持有对象导致托管堆越来越大。

**误区三：对象池一定优化性能。**

不一定。对象池会降低创建销毁成本，但也会增加常驻内存和生命周期复杂度。池容量过大、状态清理不完整、事件不解绑，都会制造更隐蔽的问题。

**误区四：增量 GC 开启后就不用管分配。**

错误。增量 GC 降低单次暂停，不消除总工作量。持续分配仍会带来 CPU 开销和堆压力。

**误区五：调用 `GC.Collect()` 可以解决内存泄漏。**

错误。GC 只能回收不可达对象。仍被引用的对象不会因为强制 GC 而消失。

**误区六：Editor Profiler 结果等于真机结果。**

错误。Editor 有额外开销和额外对象，构建后 IL2CPP/Mono、目标平台、脚本调试开关都会影响结果。性能结论必须真机验证。

**误区七：所有 LINQ 都必须禁止。**

过度。LINQ 在编辑器工具、构建流程、低频初始化中可以提升可读性。真正需要限制的是高频运行路径中的不可控分配。

**误区八：所有字符串拼接都不能用。**

过度。低频日志、初始化文案、一次性 UI 构建可以接受。问题是每帧或高频循环中的持续字符串分配。

#### 22. 从“技巧优化”到“工程治理”

GC 优化的成熟度可以分为几个阶段：

```text
阶段 1：看到卡顿后临时查 Profiler
阶段 2：知道避免 Update 中字符串/集合分配
阶段 3：能定位引用链和泄漏来源
阶段 4：为关键场景建立 GC Alloc 预算
阶段 5：把预算接入性能回归和代码评审
阶段 6：形成生命周期、资源释放、对象池、异步取消的统一规范
```

真正稳定的项目不会依赖某个高手上线前救火，而是依赖流程。每个系统上线前都应知道自己的内存分配特征；每个核心玩法都应有性能基线；每次需求变更都应验证是否引入持续分配；每个 UI 生命周期都应有清理路径；每个异步任务都应有取消策略。

GC 优化不是为了把代码写得“极端硬核”，而是为了让玩家体验稳定、让团队问题可控、让长期迭代不被历史包袱拖垮。

### 实现方案

本节给出一套可落地的 Unity GC 优化与治理方案。它不是单个技巧，而是一套从编码、分析、验收到回归的流程。

#### 1. 建立 GC 基线

首先选择项目核心场景，建立初始性能基线。建议至少覆盖：

- 冷启动进入主界面；
- 主城待机 5 到 10 分钟；
- 打开并关闭常用 UI；
- 快速滚动大型列表；
- 进入战斗、战斗待机、释放技能、退出战斗；
- 场景来回切换多次；
- 长时间挂机；
- 低端目标设备测试。

每个场景记录：

```text
场景名称
设备型号
Unity 版本
构建后端
目标帧率
平均帧耗时
P95/P99 帧耗时
每帧 GC Alloc
托管堆大小
GC 触发频率
GC Pause 或增量 GC 切片表现
异常对象数量
```

没有基线，就无法判断优化是否有效，也无法判断某次提交是否让性能退化。

#### 2. 定位持续分配

对每个关键场景录制 Profiler，优先处理持续分配，而不是一次性分配。持续分配的危害在于它会随着运行时间不断累积压力。

处理顺序建议：

```text
1. 每帧分配
2. 高频交互分配
3. 周期性定时器分配
4. 列表滚动分配
5. 战斗技能/AI/寻敌分配
6. UI 文本刷新分配
7. 日志和调试信息分配
8. 加载阶段一次性大分配
```

一次性大分配不一定需要优先优化，尤其是发生在 Loading 阶段且玩家不可交互时。真正危险的是运行中每帧几十字节到几 KB 的“看起来很小”的持续分配。它短时间不明显，长时间就会触发频繁 GC。

#### 3. 分类处理分配来源

定位到分配后，应按类型选择修复方式。

**字符串分配**：

```text
问题：每帧拼接文本
方案：值变化时刷新、降低刷新频率、缓存格式、避免隐藏 UI 刷新
```

**集合分配**：

```text
问题：高频 new List/Dictionary
方案：成员集合复用、临时集合池、预设容量、避免 ToList/ToArray
```

**闭包分配**：

```text
问题：循环中创建 Lambda 或 UI 绑定捕获变量
方案：改为具名方法、保存委托、绑定时先解绑、回收时清理
```

**装箱分配**：

```text
问题：值类型传 object/interface/非泛型容器
方案：使用泛型接口、避免 object 参数、避免格式化路径装箱
```

**协程/async 分配**：

```text
问题：高频创建状态机
方案：集中 Tick、复用等待对象、减少 fire-and-forget、增加取消令牌
```

**Unity API 分配**：

```text
问题：调用返回数组的 API
方案：使用 NonAlloc 或 List 重载、缓存结果、降低调用频率
```

**日志分配**：

```text
问题：发布构建或高频路径仍拼接日志
方案：条件编译、日志等级前置判断、延迟格式化、采样输出
```

#### 4. 治理对象生命周期

减少分配只能降低 GC 压力，不能解决仍可达对象泄漏。生命周期治理应覆盖以下对象：

- UI Panel；
- UI Item；
- 战斗实体；
- 特效与飘字；
- 数据 Controller；
- 网络请求对象；
- 协程 owner；
- async owner；
- 全局管理器缓存；
- Addressables/AssetBundle 句柄；
- NativeContainer 和 ComputeBuffer。

每类对象应回答四个问题：

```text
谁创建它？
谁持有它？
什么时候业务失效？
失效时如何解除所有引用和资源？
```

UI Panel 示例规范：

```text
OnOpen:
    注册事件
    启动请求
    刷新数据

OnClose:
    解绑事件
    取消请求
    停止协程
    清理临时引用
    停止文本刷新计时器

OnDestroy:
    释放资源句柄
    清空集合
    从管理器注销
```

对象池对象示例规范：

```text
OnSpawn:
    重置运行时状态
    绑定必要 owner
    启动表现

OnDespawn:
    停止协程
    取消异步
    解绑事件
    清空引用字段
    重置动画和粒子
    返回池
```

#### 5. 建立代码规范

建议团队建立 GC 相关编码规范，但规范要有边界，不能机械禁止所有高级语法。

推荐规范：

```text
关键运行路径禁止无理由托管分配。
Update/Tick 中禁止字符串拼接、ToList、ToArray、LINQ 查询、反射。
UI Item 复用必须清理事件和闭包。
静态事件订阅必须有解绑路径。
对象池回收必须清状态、停协程、取消异步。
集合池不得跨帧持有。
发布构建中高频日志必须可裁剪。
资源释放必须区分托管引用和原生资源。
新增持续 GC Alloc 必须说明原因和预算。
```

允许例外：

```text
编辑器工具代码。
加载阶段一次性处理。
低频配置解析。
非性能关键路径。
为了可读性保留且经 Profiler 验证无明显影响的写法。
```

规范的目标不是让代码变得僵硬，而是让风险可见。

#### 6. 建立性能回归流程

GC 问题最怕“今天只加了一点点”。一个活动 UI 每帧分配 40B，一个红点系统每秒分配 1KB，一个日志系统偶尔拼接字符串，看起来都不严重；但项目长期迭代后，这些小问题会叠成稳定性问题。

建议为核心场景建立性能回归：

```text
每周固定设备跑主城待机和战斗待机。
每次大功能合入后跑对应玩法场景。
大型 UI 合入前录制打开、关闭、滚动、静止状态。
版本封板前跑长时间挂机和场景切换。
记录 GC Alloc、堆大小、GC 次数和帧尖峰。
发现新增持续分配必须定位 owner。
```

如果条件允许，可以把关键数据接入自动化测试或性能看板。即使无法完全自动化，也应保留人工基线记录，避免性能问题只靠主观体感发现。

#### 7. 验收标准

一次 GC 优化是否完成，应以数据和风险共同验收。

验收项包括：

```text
持续 GC Alloc 是否下降或归零？
帧时间尖峰是否减少？
托管堆是否趋于稳定？
对象数量是否在流程结束后回落？
引用链是否符合生命周期预期？
对象池是否增加过多常驻内存？
是否引入集合池所有权风险？
是否降低代码可读性？
是否在目标设备复测？
是否记录优化前后数据？
```

不要只凭 Editor 中某一帧的 Alloc 数字做结论，也不要只看优化后“没有红色 GC 标记”。真正可靠的验收应包含长时间运行、多次流程循环和低端设备验证。

#### 8. 推荐落地清单

以下清单适合作为团队内部 GC 优化 Checklist：

```text
[ ] 战斗稳态是否存在每帧 GC Alloc？
[ ] 主城待机是否存在周期性 GC Alloc？
[ ] UI 静止后是否仍在刷新文本？
[ ] 大型列表滚动是否产生闭包或字符串分配？
[ ] 对象池回收是否清理事件、协程、异步和引用字段？
[ ] 静态事件是否有解绑路径？
[ ] async 是否有 owner 和取消机制？
[ ] 集合池是否存在跨帧持有？
[ ] 发布构建高频日志是否被裁剪？
[ ] Addressables/Bundle/Destroy/Dispose 是否按资源类型释放？
[ ] Memory Snapshot 对比后对象数量是否回落？
[ ] 真机结果是否与 Editor 结论一致？
```

这份清单不是为了追求形式，而是为了防止 GC 风险在多人协作中被遗漏。

### 总结

GC 是 Unity/C# 开发中的基础运行时机制，它通过自动识别并回收不再可达的托管对象，降低了开发者手动管理内存的复杂度。它的核心判断标准不是“业务上是否还需要”，而是“对象是否仍然能从 GC Roots 被访问到”。因此，静态事件、闭包、协程、async 状态机、对象池、缓存字典、全局管理器等都可能让业务上已经失效的对象继续存活。

在 Unity 项目中，GC 的主要风险集中在托管分配压力、回收暂停、托管堆增长和引用链泄漏。`GC Alloc` 表示托管分配，不等于 GC 已经执行；减少 GC Alloc 能降低回收压力，但不能解决仍可达对象造成的泄漏。增量 GC 可以把回收工作分摊到多帧，降低单次长暂停，但不能消除分配成本，也不能替代生命周期治理。

工业化 GC 优化不应停留在“少用 new”“少用 LINQ”这类零散技巧上，而应形成完整体系：明确托管内存与原生资源边界，建立关键场景 GC 预算，使用 Profiler 和 Memory Profiler 定位分配与引用链，治理 UI、战斗、对象池、异步、事件和资源生命周期，并通过代码评审和性能回归防止问题反复出现。

对 Unity 开发者来说，最重要的不是把 GC 视为敌人，而是理解它的工作边界，减少不必要的高频分配，避免错误引用链，让 GC 在可控时机、可控成本下完成它该做的事情。真正优秀的 GC 优化，不是把代码写得过度复杂，而是在性能、稳定性、可读性和维护成本之间取得工程化平衡。
